## 🎯 Unity对比：场景数据管理差异

| 维度 | UE5.7 GPUScene | Unity 公开 SRP / 传统 CPU-driven 路径 | 差异说明 |
|------|---------------|----------------------|----------|
| **数据存储** | GPU全局StructuredBuffer | 公开层是 Renderer/Component；内部 native render data、SRP Batcher CBUFFER、BRG/Entities buffer 等路径并存 | UE把主路径的 primitive/instance 数据集中成统一 GPU 场景镜像 |
| **更新方式** | 增量更新（只传脏数据） | 依实现可能全量或批量SetData | UE静态或低频变化场景通常显著减少上传；Unity项目需看SRP/Entities路径 |
| **剔除执行** | GPU Compute Shader | 传统 SRP 路径偏 CPU；BRG/Entities 等路径可更显式地交给 Job/GPU 数据流 | UE把逐实例剔除转到GPU并行，收益随规模和平台变化 |
| **剔除粒度** | Instance级（每个实例独立） | Renderer级（整个物体） | UE可剔除ISM中的单个实例 |
| **Draw参数生成** | GPU填充DrawIndirectArgs | 传统 CPU-driven 路径由 CPU/SRP 组织 draw；BRG/Entities 路径可更批量化 | UE减少 CPU draw 提交瓶颈 |
| **百万级Instance示意** | 通常受GPU剔除、绘制和带宽共同限制 | 传统CPU路径常受主线程/提交限制 | 差距需按项目实测 |
| **内存模型** | Persistent GPU Buffer复用 | 依 SRP Batcher、BRG、Entities 或自定义 buffer 路径而定 | UE主路径把 slot 分配、dirty upload、清理集中在 Renderer 内部 |
| **适用规模** | 适合大规模实例化和GPU-Driven路径 | 传统CPU-Driven路径更容易遇到提交瓶颈 | 上限由数据布局、剔除、材质和平台共同决定 |

**关键差异**：Unity 公开层通常让你以 `Renderer` / `GameObject` / SRP culling result 的心智组织绘制，但底层会按 SRP Batcher、BatchRendererGroup、Entities Graphics 或自定义 buffer 路径同步成不同的 native/GPU 数据形态。UE 的 GPUScene 把主渲染路径需要的 Primitive/Instance 数据集中镜像到 GPU 端 StructuredBuffer，Compute Shader 可以按 Primitive/Instance id 读取并生成 DrawIndirect 参数，CPU 侧主要负责维护 dirty 状态和 slot 生命周期。

# 第一部分：理解GPUScene的存在意义

> **核心目标**：从传统渲染管线的实际瓶颈出发，理解GPUScene解决了什么问题、付出了什么代价

---

## 一、传统渲染管线的数据瓶颈

### 1.1 瓶颈一：CPU逐个传递场景数据

假设你正在开发一个开放世界场景，场景中有50,000个物体，每个物体有Transform矩阵、Bounds、材质参数等数据。

**Unity URP的做法**：

```csharp
// Unity URP：CPU每帧为每个可见物体设置Constant Buffer
void RenderVisibleObjects()
{
    foreach (var renderer in visibleRenderers)  // 遍历可见物体
    {
        // 每个DrawCall都要设置一次cbuffer
        material.SetMatrix("_ObjectToWorld", renderer.localToWorldMatrix);  // 64 bytes
        material.SetVector("_WorldSpaceBounds", renderer.bounds.center);    // 16 bytes
        material.SetFloat("_CustomData", renderer.customValue);             // 4 bytes
        
        Graphics.DrawMesh(renderer.mesh, renderer.transform, material, 0);
    }
    // 50,000个物体 = 50,000次cbuffer更新 + 50,000次DrawCall
    // CPU耗时会随平台、SRP路径、合批和材质状态明显变化
}
```

**问题本质**：每个DrawCall都需要CPU设置一次Constant Buffer，GPU通过固定槽位（`register(b0)`）读取。Constant Buffer有64KB大小限制，无法一次传递所有物体数据。

### 1.2 瓶颈二：CPU剔除成为串行瓶颈

```csharp
// Unity传统CPU-driven示意：由CPU/SRP公开层组织可见Renderer
void CullScene(Camera camera)
{
    visibleRenderers.Clear();
    foreach (var renderer in allRenderers)  // 遍历全部100万个实例
    {
        // Frustum Culling
        if (!GeometryUtility.TestPlanesAABB(camera.frustumPlanes, renderer.bounds))
            continue;
        
        // Occlusion Culling（如果开启）
        if (IsOccluded(renderer.bounds))
            continue;
        
        visibleRenderers.Add(renderer);  // 通过剔除
    }
    // 100万Instance → CPU串行遍历很容易成为瓶颈
    // 即使用Job System并行，收益也受核心数、内存访问和任务拆分影响
}
```

**问题对比**：

| 场景规模 | CPU剔除（Unity Job示意） | GPU剔除（UE GPUScene示意） | 趋势 |
|---------|---------------------|----------------------|------|
| 10万 Instance | 常可控 | 通常仍较轻 | GPU并行优势开始出现 |
| 50万 Instance | 更依赖任务拆分和缓存 | 更依赖GPU占用和带宽 | 差距可能扩大 |
| 100万 Instance | 容易压到主线程/任务系统 | 受剔除Pass和绘制成本共同影响 | 需按场景实测 |
| 500万 Instance | CPU路径常成为高风险瓶颈 | GPU路径也会受带宽/可见实例数限制 | 需要分层/流送/剔除协同 |

在数据布局和可见实例比例合适时，规模越大通常越能体现GPU并行优势；但最终收益仍取决于项目实测。

### 1.3 瓶颈三：数据带宽浪费

传统做法即使物体没有移动，每帧也要重新传递全部数据：

```
传统方式（每帧全量传递）：
  50,000个物体 × 80 bytes/物体 = 4MB/帧 × 60帧 = 240MB/秒
  
  但实际上：
  - 95%的物体是静态的（建筑、地形、植被）
  - 只有5%在移动（角色、车辆、粒子）
  - 每帧实际变化：2,500个 × 80 bytes = 200KB
  
  浪费了95%的带宽！
```

**GPUScene的增量更新**：

```
GPUScene方式（增量更新）：
  首次加载：50,000个物体 × 80 bytes = 4MB（一次性上传）
  之后每帧：只上传变化的 2,500个 × 80 bytes = 200KB/帧
  
  带宽趋势：dirty比例越低，每帧上传越接近变化子集
  静态场景：上传量通常显著下降，但仍有Buffer维护、剔除和绘制成本
```

## 二、GPUScene的解决思路：GPU侧场景镜像

### 2.1 核心思想

GPUScene的核心思想极其简单：**把整个场景的渲染数据复制一份到GPU端的大Buffer里，Shader直接用Index读取任意物体的数据**。

```
【传统方式】CPU每帧传递 → Constant Buffer → Shader读取固定槽位
  CPU: SetConstant(b0, ObjectA_Data)  →  Shader: cbuffer ObjectData : register(b0)
  CPU: SetConstant(b0, ObjectB_Data)  →  Shader: cbuffer ObjectData : register(b0)
  每个DrawCall前都要切换一次

【GPUScene方式】一次上传所有 → StructuredBuffer → Shader用ID随机访问
  GPU Buffer: [Object0][Object1][Object2]...[Object49999]
  Shader: StructuredBuffer<FPrimitiveSceneData> GPUScenePrimitiveSceneData;
          FPrimitiveSceneData data = GPUScenePrimitiveSceneData[PrimitiveID];
  任意Shader、任意时刻、直接用ID访问
```

**类比**：
- 传统方式 = **图书管理员逐本递书**（每次要一本新的，管理员跑一趟）
- GPUScene方式 = **把整个图书馆搬到你桌上**（随时翻阅任何一本，管理员只更新变动的书）

### 2.2 三大核心Buffer

GPUScene在GPU端维护的三大Buffer：

```
GPU Memory
┌─────────────────────────────────────────────────┐
│ PrimitiveBuffer (StructuredBuffer<float4>)       │
│ ├─ Primitive 0: LocalToWorld, Bounds, Flags...   │  每个Primitive占
│ ├─ Primitive 1: ...                              │  FPrimitiveSceneShaderData
│ └─ Primitive N: ...                              │  (~13个float4)
├─────────────────────────────────────────────────┤
│ InstanceSceneDataBuffer (StructuredBuffer<float4>)│
│ ├─ Instance 0: Transform, PrimitiveID, Flags... │  每个Instance占
│ ├─ Instance 1: ...                               │  FInstanceSceneShaderData
│ └─ Instance M: ...                               │  (~8个float4)
├─────────────────────────────────────────────────┤
│ InstancePayloadDataBuffer (StructuredBuffer<float4>)│
│ ├─ Payload 0: CustomData, LightmapUV...         │  可变长度
│ └─ Payload K: ...                                │  每Instance stride不同
├─────────────────────────────────────────────────┤
│ LightmapDataBuffer                               │  光照贴图数据
│ LightDataBuffer (ByteAddressBuffer)              │  动态光源数据
└─────────────────────────────────────────────────┘
```

**关系**：Primitive是"物体类型"（一个StaticMesh），Instance是"物体实例"（ISM中的每棵树）。一个Primitive可以有1到数万个Instance。

## 三、代价与限制

GPUScene并非万能方案，它引入了以下代价：

| 代价 | 具体表现 | 影响 |
|------|---------|------|
| **GPU内存占用** | Instance/Primitive/Payload/Lightmap等Buffer会随实例规模增长 | 中端GPU需关注峰值和保留资源策略 |
| **帧延迟** | GT修改到RT/GPU消费存在跨线程排队 | 快速移动物体需结合渲染帧率和并行管线观察 |
| **调试困难** | 数据在GPU端，无法直接Inspector | 需RenderDoc/Insights |
| **全量重传开销** | 切换Tiled布局、调试强制上传或首次加载可能触发大量上传 | 加载/切换时可能出现短暂峰值 |
| **Feature Level要求** | 需SM5+（StructuredBuffer支持） | 不支持SM4及以下 |

**何时GPUScene最有效**：
- ✅ 大规模静态或低频变化场景（dirty比例低时，增量更新优势更明显）
- ✅ ISM/HISM密集植被（百万Instance GPU剔除）
- ✅ Nanite场景（Nanite直接读取GPUScene做Cluster Culling）
- ⚠️ 全动态场景（每帧所有物体都动，增量优势消失，但GPU剔除仍有效）

## 2.1 FGPUScene：全局场景镜像管理器

`FGPUScene`是GPUScene系统的核心类，每个`FScene`持有一个实例。

**源码锚点**：`Engine/Source/Runtime/Renderer/Private/GPUScene.h`，搜索`class FGPUScene`、`FGPUSceneResourceParameters`、`PrimitiveUploadBuffer`、`InstanceSceneUploadBuffer`。事实类型：类结构、Shader参数绑定、Buffer成员。

### 核心成员

```cpp
class FGPUScene
{
public:
    // === 核心接口 ===
    void AddPrimitiveToUpdate(FPersistentPrimitiveIndex, EPrimitiveDirtyState);  // 标记脏数据
    void Update(FRDGBuilder&, FSceneUniformBuffer&, ...);                        // 主更新入口
    int32 AllocateInstanceSceneDataSlots(FPersistentPrimitiveIndex, int32);       // 分配Instance槽位
    void FreeInstanceSceneDataSlots(int32 Offset, int32 Count);                  // 释放Instance槽位
    void BeginRender(FRDGBuilder&, FGPUSceneDynamicContext&);                    // 帧开始
    void EndRender();                                                            // 帧结束

    // === GPU Buffer ===
    TRefCountPtr<FRDGPooledBuffer> PrimitiveBuffer;          // Primitive数据
    FRDGAsyncScatterUploadBuffer   PrimitiveUploadBuffer;    // Primitive上传器

    TRefCountPtr<FRDGPooledBuffer> InstanceSceneDataBuffer;  // Instance数据
    FRDGAsyncScatterUploadBuffer   InstanceSceneUploadBuffer;// Instance上传器

    FSpanAllocator InstancePayloadDataAllocator;             // Payload槽位分配
    TRefCountPtr<FRDGPooledBuffer> InstancePayloadDataBuffer;// Instance附加数据

    FSpanAllocator LightmapDataAllocator;                    // Lightmap槽位分配
    TRefCountPtr<FRDGPooledBuffer> LightmapDataBuffer;       // Lightmap数据
    TPersistentByteAddressBuffer<FLightSceneData> LightDataBuffer; // 光源数据

    // === 更新追踪 ===
    TArray<FPersistentPrimitiveIndex> PrimitivesToUpdate;    // 待更新Primitive列表
    TArray<EPrimitiveDirtyState> PrimitiveDirtyState;        // 每个Primitive的脏状态

private:
    FSpanAllocator InstanceSceneDataAllocator;  // Instance槽位分配器
    bool bUseTiledInstanceDataLayout = true;    // 是否使用Tiled布局
    int32 InstanceDataTileSizeLog2 = -1;        // Tile大小（log2）
    int32 DynamicPrimitivesOffset;              // 动态Primitive起始偏移
    FScene& Scene;                              // 所属Scene引用
};
```

### 脏状态枚举

```cpp
enum class EPrimitiveDirtyState : uint8
{
    None            = 0,
    ChangedId       = 1 << 0,  // PrimitiveID变化
    ChangedTransform= 1 << 1,  // Transform变化
    ChangedOther    = 1 << 2,  // 其他属性变化
    ChangedAll      = ChangedId | ChangedTransform | ChangedOther,
    Added           = 1 << 3,  // 新增Primitive
    Removed         = 1 << 4,  // 删除Primitive
};
```

```cpp
// 帧内：先移动，再换材质
AddPrimitiveToUpdate(ID=5, ChangedTransform);  // State = 0b00010
AddPrimitiveToUpdate(ID=5, ChangedOther);      // State = 0b00010 | 0b00100 = 0b00110

```

## 2.2 FSpanAllocator：连续槽位分配

GPUScene使用`FSpanAllocator`（而非TSparseArray）管理Instance槽位。FSpanAllocator支持**连续范围分配**，保证同一ISM的Instance在Buffer中相邻存储，利于GPU Cache预取。

### 工作原理

```cpp
// FSpanAllocator：基于Span（起始偏移+长度）的连续分配器
class FSpanAllocator
{
    // 空闲列表：每个Span记录一段连续的空闲区域
    TArray<FLinearAllocation> FreeSpans;  // {Offset, Num}
    int32 MaxSize;      // 当前Buffer总大小
    int32 SearchIndex;  // 上次搜索位置（加速查找）
    
    // 分配：找到足够大的连续空闲Span
    int32 Allocate(int32 Num)
    {
        // First-fit策略：从SearchIndex开始，找到第一个Num大小的Span
        for (int32 i = SearchIndex; i < FreeSpans.Num(); ++i)
        {
            if (FreeSpans[i].Num >= Num)
            {
                int32 Offset = FreeSpans[i].Offset;
                FreeSpans[i].Offset += Num;   // 缩小Span
                FreeSpans[i].Num -= Num;
                if (FreeSpans[i].Num == 0)
                    FreeSpans.RemoveAt(i);     // Span用完了
                return Offset;
            }
        }
        // 没有足够空间 → 扩容MaxSize
        int32 Offset = MaxSize;
        MaxSize += Num;
        return Offset;
    }
    
    // 释放：将区域加回FreeSpans，尝试与相邻Span合并
    void Free(int32 Offset, int32 Num)
    {
        // 插入FreeSpans并合并相邻Span
        MergeAdjacentSpans(Offset, Num);
    }
};
```

### 为什么选FSpanAllocator而不是TSparseArray

| 维度 | FSpanAllocator | TSparseArray |
|------|---------------|-------------|
| 分配方式 | **连续范围**（1000个Instance一起分配） | **逐个**（每个独立分配） |
| 内存布局 | ISM的1000个Instance连续存储 | 可能分散在Buffer各处 |
| GPU Cache | 顺序访问通常更利于预取 | 分散访问更容易造成Cache Miss |
| 碎片合并 | ✅ 自动合并相邻Span | ❌ 需手动Compact |
| 适用场景 | 批量分配（ISM、Foliage） | 逐个分配（独立Actor） |

**ISM场景实例**：

```
分配1000棵树的Instance（FSpanAllocator）：
  Allocate(1000) → Offset=5000, 连续区域[5000, 5999]
  
GPU剔除时读取：
  Thread 0: InstanceBuffer[5000]  ← 更可能命中相邻缓存线
  Thread 1: InstanceBuffer[5001]  ← 同一局部区域
  Thread 2: InstanceBuffer[5002]  ← 同一局部区域
  ...顺序读取通常更利于GPU缓存和内存合并

如果用TSparseArray逐个分配：
  Instance分散在 [47], [1523], [8901], [234]...
  每次读取跨越不同缓存区域 → Cache Miss风险上升
```

## 2.3 FRDGAsyncScatterUploadBuffer：异步Scatter上传

GPUScene的数据上传核心是`FRDGAsyncScatterUploadBuffer`——一个CPU异步填充、GPU端Compute Shader执行Scatter Write的上传机制。

### 上传流程

```
CPU端（RenderThread + TaskGraph并行）        GPU端
┌──────────────────────────┐               ┌───────────────────┐
│1. Lock: 映射Staging Buffer│              │                   │
│2. 并行填充N个Primitive数据 │              │                   │
│   ├─ Thread 0: Prim[0-99] │              │                   │
│   ├─ Thread 1: Prim[100-199]             │                   │
│   └─ Thread N: ...        │              │                   │
│3. Unlock: 提交给GPU       │──────────>   │4. Scatter Copy CS │
└──────────────────────────┘               │   ├─ Thread 0:    │
                                           │   │  Staging[0]   │
                                           │   │  → Dst[ID_0]  │
                                           │   ├─ Thread 1:    │
                                           │   │  Staging[1]   │
                                           │   │  → Dst[ID_5]  │
                                           │   └─ ...          │
                                           └───────────────────┘
```

**Scatter Write vs 逐个CopyBuffer**：

```
【逐个CopyBufferRegion】（传统方式）
  for (Batch : 1000个更新)
      RHICmdList.CopyBufferRegion(Dst, DstOffset, Src, SrcOffset, Size);
  结果：大量Driver Call，CPU提交开销随批次数增加

【Scatter Write CS】（GPUScene方式）
  少量Dispatch组织Scatter写入
  GPU端并行写入目标Buffer
  结果：通常更适合大量非连续小更新
  
  趋势：更新批次越多，Scatter方式越容易摊薄CPU提交成本
```

### FRDGAsyncScatterUploadBuffer的关键特性

```cpp
// Staging Buffer是CPU可写的Upload Heap
// 通过RDG GraphBuilder调度，自动处理Barrier

// 使用流程：
// 1. Begin(): 分配Staging Buffer空间
FRDGScatterUploader* Uploader = UploadBuffer.Begin(GraphBuilder, DstBuffer, 
    NumUploads, ElementSize, TEXT("PrimitiveUpload"));

// 2. Lock(RHICmdList): 映射CPU地址（可在TaskGraph中并行执行）
Uploader->Lock(RHICmdList);

// 3. Add(Index, Data): 填充数据（CPU端，可并行）
Uploader->Add(DstIndex, &PrimitiveShaderData);

// 4. Unlock(RHICmdList): 解除映射
Uploader->Unlock(RHICmdList);

// 5. End(): RDG自动添加Compute Shader Pass执行Scatter Write
UploadBuffer.End(GraphBuilder, Uploader);
```

## 2.4 Tiled Instance Data Layout：分块数据布局

UE5.7引入了**Tiled Instance Data Layout**（CVar `r.GPUScene.InstanceDataTileSizeLog2`），将Instance数据按Tile分块存储，优化GPU的数据访问模式。

### 传统SoA vs Tiled布局

```
【传统线性布局】（Field 0连续，Field 1连续）
Buffer:
  [Inst0.F0][Inst1.F0][Inst2.F0]...[InstN.F0]  ← 所有Instance的Field 0
  [Inst0.F1][Inst1.F1][Inst2.F1]...[InstN.F1]  ← 所有Instance的Field 1
  ...
  访问单个Instance的所有Field → 跨越巨大偏移 → Cache Miss

【Tiled布局】（TileSize个Instance为一组）
Buffer:
  Tile 0:
    [Inst0.F0][Inst1.F0]...[Inst(T-1).F0]  ← Tile 0的Field 0
    [Inst0.F1][Inst1.F1]...[Inst(T-1).F1]  ← Tile 0的Field 1
  Tile 1:
    [InstT.F0][Inst(T+1).F0]...             ← Tile 1的Field 0
    [InstT.F1][Inst(T+1).F1]...             ← Tile 1的Field 1
  
  T = TileSize (2^TileSizeLog2)
```

### 索引计算

```cpp
// 源码锚点：Engine/Source/Runtime/Renderer/Private/GPUScene.cpp，搜索 CalcInstanceDataIndex
uint32 CalcInstanceDataIndex(uint32 InstanceId, uint32 ArrayIndex)
{
    uint32 TileId = InstanceId >> InstanceDataTileSizeLog2;          // 哪个Tile
    uint32 IdInTile = InstanceId & InstanceDataTileSizeMask;         // Tile内偏移
    return InstanceDataTileStride * TileId                           // Tile起始
         + (ArrayIndex << InstanceDataTileSizeLog2)                  // Field偏移
         + IdInTile;                                                 // Instance偏移
}
```

**Shader端对应**（SceneData.ush）：

```hlsl
// Shader端读取Instance数据
FInstanceSceneData GetInstanceSceneData(uint InstanceId, uint FieldIndex)
{
    uint TileId = InstanceId >> GPUSceneInstanceDataTileSizeLog2;
    uint IdInTile = InstanceId & GPUSceneInstanceDataTileSizeMask;
    uint Index = GPUSceneInstanceDataTileStride * TileId 
               + (FieldIndex << GPUSceneInstanceDataTileSizeLog2) 
               + IdInTile;
    return GPUSceneInstanceSceneData[Index];
}
```

**UE5.7路径**：当GPU支持Reserved Resources时（D3D12 Tiled Resources），GPUScene可通过`r.GPUScene.UseReservedResources`保留较大的虚拟地址空间，用来降低Buffer扩容时的完整重分配风险；实际显存压力和映射行为需按平台验证。

## 3.1 AddPrimitiveToUpdate：脏标记与状态合并

### 函数概览

**源码锚点**：`Engine/Source/Runtime/Renderer/Private/GPUScene.cpp`，搜索`FGPUScene::AddPrimitiveToUpdate`、`ResizeDirtyState`、`PrimitiveDirtyState`。事实类型：Dirty标记、状态合并。  
**函数**：`FGPUScene::AddPrimitiveToUpdate()`  
**职责**：将Primitive标记为"脏数据"，等待下一次Update()批量处理

### 核心流程

```cpp
void FGPUScene::AddPrimitiveToUpdate(FPersistentPrimitiveIndex PersistentPrimitiveIndex, 
                                      EPrimitiveDirtyState DirtyState)
{
    if (bIsEnabled && PersistentPrimitiveIndex.IsValid())
    {
        // 【步骤1】确保脏状态数组足够大
        ResizeDirtyState(PersistentPrimitiveIndex.Index + 1);

        // 【步骤2】首次标记：加入待更新列表
        if (PrimitiveDirtyState[PersistentPrimitiveIndex.Index] == EPrimitiveDirtyState::None)
        {
            PrimitivesToUpdate.Add(PersistentPrimitiveIndex);
        }

        // 【步骤3】合并脏状态（位运算OR）
        EPrimitiveDirtyState NewState = PrimitiveDirtyState[PersistentPrimitiveIndex.Index] | DirtyState;

        // 【步骤4】特殊处理：Removed时清除Added标记
        if (EnumHasAnyFlags(DirtyState, EPrimitiveDirtyState::Removed))
        {
            EnumRemoveFlags(NewState, EPrimitiveDirtyState::Added);
        }
        PrimitiveDirtyState[PersistentPrimitiveIndex.Index] = NewState;
    }
}
```

### 关键细节

**Remove + Add的状态机**：

```
场景：Primitive被删除，同一帧内其ID被新Primitive复用

时序：
  1. RemovePrimitive(ID=100)  → State = Removed
  2. AddPrimitive(ID=100)     → State = Removed | Added（两者同时存在）

UpdateInternal中的处理：
  if (HasRemoved && !HasAdded)  → 跳过上传（已删除）
  if (HasRemoved && HasAdded)   → 正常上传（ID被复用，需要上传新数据）
```

**性能特性**：

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 首次标记 | O(1) | TArray::Add |
| 重复标记 | O(1) | 位运算合并，不重复加入列表 |
| 内存占用 | 1 byte/Primitive | EPrimitiveDirtyState::uint8 |

### 💡 核心要点

一句话：O(1)标记脏数据，位运算合并多次修改，Remove+Add状态机处理ID复用。

## 3.2 UpdateInternal：主更新流程

### 函数概览

**源码锚点**：`Engine/Source/Runtime/Renderer/Private/GPUScene.cpp`，搜索`FGPUScene::UpdateInternal`、`RDG_EVENT_SCOPE(GraphBuilder, "GPUScene.Update")`、`CVarGPUSceneUploadEveryFrame`、`AddClearInstancesPass`。事实类型：帧更新入口、全量上传触发、清除Pass调度。  
**函数**：`FGPUScene::UpdateInternal()`  
**职责**：每帧由`Update()`调用一次，协调整个GPUScene的数据更新流程

### 核心流程

```cpp
void FGPUScene::UpdateInternal(FRDGBuilder& GraphBuilder, FSceneUniformBuffer& SceneUB, 
    FRDGExternalAccessQueue& ExternalAccessQueue, const UE::Tasks::FTask& UpdateTaskPrerequisites,
    const FUpdateFromComputeCommands& UpdatesFromCompute)
{
    RDG_EVENT_SCOPE(GraphBuilder, "GPUScene.Update");

    // 【步骤1】检测Tiled布局变更（CVar改变时触发全量重传）
    bool bPrevUseTiledLayout = bUseTiledInstanceDataLayout;
    bUseTiledInstanceDataLayout = CVarGPUSceneInstanceDataTileSizeLog2.GetValueOnRenderThread() >= 0;
    if (bPrevUseTiledLayout != bUseTiledInstanceDataLayout)
    {
        InstanceSceneDataBuffer.SafeRelease();  // 释放旧Buffer
        bUpdateAllPrimitives = true;            // 触发全量重传
    }

    // 【步骤2】Reserved Resource预分配（UE5.7新增）
    if (bUseReservedResources && !InstanceSceneDataBuffer.IsValid())
    {
        const uint64 MaxSizeInBytes = uint64(2048) << 20;  // 2GB虚拟地址
        FRDGBufferDesc Desc = FRDGBufferDesc::CreateStructuredDesc(
            sizeof(FVector4f), uint32(MaxSizeInBytes / sizeof(FVector4f)));
        Desc.Usage |= EBufferUsageFlags::ReservedResource;
        InstanceSceneDataBuffer = AllocatePooledBuffer(Desc, TEXT("GPUScene.InstanceSceneData"));
    }

    // 【步骤3】全量重传模式（如果需要）
    if (CVarGPUSceneUploadEveryFrame.GetValueOnRenderThread() || bUpdateAllPrimitives)
    {
        PrimitivesToUpdate.Reset();
        for (FPrimitiveSceneInfo* Info : Scene.Primitives)
        {
            PrimitiveDirtyState[Info->GetPersistentIndex().Index] |= EPrimitiveDirtyState::ChangedAll;
            PrimitivesToUpdate.Add(Info->GetPersistentIndex());
        }
        bUpdateAllPrimitives = false;
    }

    // 【步骤4】过滤已删除的Primitive（Removed且未重新Added）
    for (int32 Index = 0; Index < PrimitivesToUpdate.Num();)
    {
        const FPersistentPrimitiveIndex Id = PrimitivesToUpdate[Index];
        bool bRemoved = EnumHasAnyFlags(PrimitiveDirtyState[Id.Index], EPrimitiveDirtyState::Removed);
        bool bAdded = EnumHasAnyFlags(PrimitiveDirtyState[Id.Index], EPrimitiveDirtyState::Added);
        if (bRemoved && !bAdded)
            PrimitivesToUpdate.RemoveAtSwap(Index, EAllowShrinking::No);  // 跳过已删除
        else
            ++Index;
    }

    // 【步骤5】构造上传数据适配器
    FUploadDataSourceAdapterScenePrimitives& Adapter = 
        *GraphBuilder.AllocObject<FUploadDataSourceAdapterScenePrimitives>(
            Scene, SceneFrameNumber, MoveTemp(PrimitivesToUpdate), MoveTemp(PrimitiveDirtyState));

    // 【步骤6】更新Buffer分配（可能扩容）
    FRegisteredBuffers BufferState = UpdateBufferAllocations(GraphBuilder, SceneUB);

    // 【步骤7】清除需要重置的Instance（设为Hidden）
    AddClearInstancesPass(GraphBuilder, ...);

    // 【步骤8】重置脏标记（为下一帧准备）
    PrimitivesToUpdate.Reset();
    PrimitiveDirtyState.Init(EPrimitiveDirtyState::None, PrimitiveDirtyState.Num());

    // 【步骤9】执行批量上传
    UploadGeneral<FUploadDataSourceAdapterScenePrimitives>(
        GraphBuilder, BufferState, &ExternalAccessQueue, Adapter, UpdateTaskPrerequisites);

    // 【步骤10】处理GPU端Compute更新（Niagara等）
    if (!UpdatesFromCompute.IsEmpty())
    {
        FGPUSceneWriteDelegateParams Params;
        Params.GPUSceneWriterUB = GetWriteParameters(GraphBuilder, ...);
        for (const auto& Command : UpdatesFromCompute)
        {
            Command.Payload.GPUSceneWriter.Execute(GraphBuilder, Params);
        }
    }
}
```

### 关键细节

**AddClearInstancesPass**：在上传新数据前，先将需要重置的Instance标记为`INSTANCE_SCENE_DATA_FLAG_HIDDEN`。这通过`GPUSceneSetInstancePrimitiveIdCS` Compute Shader完成：


一句话：UpdateInternal管理GPUScene的完整帧更新——检测布局变更、过滤已删除Primitive、清除旧Instance、批量Scatter上传新数据、执行GPU端Compute更新。

## 3.3 UploadGeneral：批量Scatter上传

### 函数概览

**源码锚点**：`Engine/Source/Runtime/Renderer/Private/GPUScene.cpp`，搜索`FGPUScene::UploadGeneral`、`FGPUSceneUploadTaskContext`、`ParallelForTemplate`、`InstanceSceneUploadBuffer.BeginPreSized`。事实类型：批量Scatter上传、并行填充、Tiled索引。  
**函数**：`FGPUScene::UploadGeneral<FUploadDataSourceAdapter>()`  
**职责**：将Primitive、Instance、Lightmap等数据通过Scatter Upload批量传到GPU

### 核心流程

```cpp
template<typename FUploadDataSourceAdapter>
void FGPUScene::UploadGeneral(FRDGBuilder& GraphBuilder, const FRegisteredBuffers& BufferState, 
    FRDGExternalAccessQueue* ExternalAccessQueue, const FUploadDataSourceAdapter& Adapter,
    const UE::Tasks::FTask& PrerequisiteTask)
{
    const int32 NumPrimitiveDataUploads = Adapter.NumPrimitivesToUpload();
    if (!NumPrimitiveDataUploads) return;

    // 【步骤1】统计上传量
    FGPUSceneUploadTaskContext& TaskContext = *GraphBuilder.AllocObject<FGPUSceneUploadTaskContext>();
    for (int32 i = 0; i < NumPrimitiveDataUploads; ++i)
    {
        FPrimitiveUploadInfoHeader UploadInfo;
        Adapter.GetPrimitiveInfoHeader(i, UploadInfo);
        
        // MAX_INSTANCE_ID限制检查
        if (UploadInfo.InstanceSceneDataOffset + UploadInfo.NumInstanceUploads >= MAX_INSTANCE_ID)
            UploadInfo.NumInstanceUploads = FMath::Max(0, int32(MAX_INSTANCE_ID) - UploadInfo.InstanceSceneDataOffset);
        
        TaskContext.NumInstanceSceneDataUploads += UploadInfo.NumInstanceUploads;
        TaskContext.NumLightmapDataUploads += UploadInfo.LightmapUploadCount;
        TaskContext.NumInstancePayloadDataUploads += UploadInfo.NumPayloadUploadInstances * UploadInfo.InstancePayloadDataStride;
        TaskContext.PrimitiveUploadInfos[i] = UploadInfo;
    }

    // 【步骤2】为每种数据类型分配Scatter Uploader
    TaskContext.PrimitiveUploader = PrimitiveUploadBuffer.Begin(
        GraphBuilder, BufferState.PrimitiveBuffer, 
        Adapter.GetItemPrimitiveIds().Num(), sizeof(FPrimitiveSceneShaderData::Data), 
        TEXT("PrimitiveUploadBuffer"));

    if (TaskContext.NumInstanceSceneDataUploads > 0)
    {
        TaskContext.InstanceSceneUploader = InstanceSceneUploadBuffer.BeginPreSized(
            GraphBuilder, BufferState.InstanceSceneDataBuffer,
            TaskContext.NumInstanceSceneDataUploads * FInstanceSceneShaderData::GetDataStrideInFloat4s(),
            sizeof(FVector4f), TEXT("InstanceSceneUploadBuffer"));
    }

    // 【步骤3】决定是否并行执行
    const bool bParallel = ParallelExecuteThreshold > 0 
        && (TaskContext.NumPrimitiveDataUploads > ParallelExecuteThreshold);

    // 【步骤4】通过RDG AddCommandListSetupTask异步填充
    GraphBuilder.AddCommandListSetupTask([&TaskContext, &Adapter, bParallel](FRHICommandListBase& RHICmdList)
    {
        // Lock所有Uploader
        LockIfValid(RHICmdList, TaskContext.PrimitiveUploader);
        LockIfValid(RHICmdList, TaskContext.InstanceSceneUploader);
        LockIfValid(RHICmdList, TaskContext.InstancePayloadUploader);
        LockIfValid(RHICmdList, TaskContext.LightmapUploader);

        // 并行/串行填充Staging Buffer
        auto FillTask = [&](int32 ItemIndex)
        {
            FPrimitiveUploadInfoHeader& Header = TaskContext.PrimitiveUploadInfos[ItemIndex];
            
            // 填充Primitive数据
            Adapter.GetPrimitiveData(ItemIndex, TaskContext.PrimitiveUploader);
            
            // 填充Instance数据（考虑Tiled Layout）
            for (int32 InstanceIdx = 0; InstanceIdx < Header.NumInstanceUploads; ++InstanceIdx)
            {
                uint32 InstanceId = Header.InstanceSceneDataOffset + InstanceIdx;
                // Tiled布局：计算实际Buffer索引
                for (uint32 ArrayIdx = 0; ArrayIdx < DataStride; ++ArrayIdx)
                {
                    uint32 DstIndex = TaskContext.CalcInstanceDataIndex(InstanceId, ArrayIdx);
                    TaskContext.InstanceSceneUploader->Add(DstIndex, &InstanceData[ArrayIdx]);
                }
            }
        };

        if (bParallel)
            ParallelFor(NumPrimitiveDataUploads, FillTask);
        else
            for (int32 i = 0; i < NumPrimitiveDataUploads; ++i) FillTask(i);

        // Unlock所有Uploader
        UnlockIfValid(RHICmdList, TaskContext.PrimitiveUploader);
        UnlockIfValid(RHICmdList, TaskContext.InstanceSceneUploader);
        // ...
    });
}
```

### 关键细节

**CalcInstanceDataIndex与Tiled布局**：

```cpp
// 源码锚点：Engine/Source/Runtime/Renderer/Private/GPUScene.cpp，搜索 CalcInstanceDataIndex
uint32 CalcInstanceDataIndex(uint32 InstanceId, uint32 ArrayIndex)
{
    uint32 TileId = InstanceId >> InstanceDataTileSizeLog2;       // 例：InstanceId=300, TileSize=256 → TileId=1
    uint32 IdInTile = InstanceId & InstanceDataTileSizeMask;      // 300 & 0xFF = 44
    return InstanceDataTileStride * TileId                        // Tile 1的起始位置
         + (ArrayIndex << InstanceDataTileSizeLog2)               // 第几个Field的偏移
         + IdInTile;                                              // Tile内的Instance偏移
}
```

**并行填充阈值**：由CVar `r.GPUScene.ParallelUpdate`控制（默认1024）。超过阈值时使用`ParallelFor`多线程填充Staging Buffer。

### 💡 核心要点

一句话：UploadGeneral为每种数据类型创建Scatter Uploader，CPU端（可并行）填充Staging Buffer，RDG自动调度GPU端Compute Shader执行Scatter Write到目标Buffer。

## 3.4 AllocateInstanceSceneDataSlots：槽位分配

### 函数概览

**源码锚点**：`Engine/Source/Runtime/Renderer/Private/GPUScene.cpp`，搜索`FGPUScene::AllocateInstanceSceneDataSlots`、`FGPUScene::FreeInstanceSceneDataSlots`、`AddOrMergeInstanceRange`。事实类型：Instance槽位分配、释放、清除范围合并。  
**函数**：`FGPUScene::AllocateInstanceSceneDataSlots()`  
**职责**：为新增Instance分配GPU Buffer中的连续槽位

### 核心流程

```cpp
int32 FGPUScene::AllocateInstanceSceneDataSlots(
    FPersistentPrimitiveIndex PersistentPrimitiveIndex, 
    int32 NumInstanceSceneDataEntries)
{
    if (bIsEnabled && NumInstanceSceneDataEntries > 0)
    {
        // 【步骤1】从FSpanAllocator分配连续范围
        const int32 InstanceSceneDataOffset = InstanceSceneDataAllocator.Allocate(NumInstanceSceneDataEntries);
        
        // 【步骤2】记录需要清除的Instance范围
        AddOrMergeInstanceRange(InstanceRangesToClear, 
            FInstanceRange{ PersistentPrimitiveIndex, 
                           uint32(InstanceSceneDataOffset), 
                           uint32(NumInstanceSceneDataEntries) });

        return InstanceSceneDataOffset;
    }
    return INDEX_NONE;
}
```

**释放流程**：

```cpp
void FGPUScene::FreeInstanceSceneDataSlots(int32 Offset, int32 Count)
{
    if (bIsEnabled)
    {
        InstanceSceneDataAllocator.Free(Offset, Count);      // 归还给FSpanAllocator
        AddOrMergeInstanceRange(InstanceRangesToClear,        // 标记为需要清除
            FInstanceRange{ {}, uint32(Offset), uint32(Count) });
    }
}
```

### AddOrMergeInstanceRange优化

```cpp
// 如果新Range与上一个Range相邻且属于同一Primitive → 合并
inline void AddOrMergeInstanceRange(TArray<FGPUSceneInstanceRange>& Output, FGPUSceneInstanceRange Range)
{
    if (!Output.IsEmpty())
    {
        FGPUSceneInstanceRange& Last = Output.Last();
        if (Range.InstanceSceneDataOffset == (Last.InstanceSceneDataOffset + Last.NumInstanceSceneDataEntries) 
            && Range.Primitive == Last.Primitive)
        {
            Last.NumInstanceSceneDataEntries += Range.NumInstanceSceneDataEntries;
            return;  // 合并成功，无需新增
        }
    }
    Output.Add(Range);
}
```

**实际场景**：一个HISM组件添加10个Instance Range（分10批），如果它们是连续分配的，最终合并为1个大Range → 只需1次GPU清除操作。

### 💡 核心要点

一句话：FSpanAllocator保证连续分配利于GPU Cache，AddOrMergeInstanceRange合并相邻范围减少GPU清除操作次数。

## 3.5 Instance Culling Pipeline：GPU剔除管线

Instance Culling是GPUScene的核心消费者——读取GPUScene数据，在GPU端完成可见性判断，生成DrawIndirect参数。

### 架构概览

```
FInstanceCullingManager（帧级管理器）
  ├─ RegisterView()：注册View（可多个View：主视角、阴影、反射）
  ├─ BeginDeferredCulling()：启动延迟剔除
  └─ 管理多个 FInstanceCullingContext

FInstanceCullingContext（每个MeshPass一个）
  ├─ AddInstancesToDrawCommand()：CPU端收集DrawCommand + Instance映射
  ├─ BuildRenderingCommands()：提交GPU剔除
  └─ FInstanceCullingDrawParams：输出的Draw参数

GPU端：
  InstanceCullBuildInstanceIdBufferCS（BuildInstanceDrawCommands.usf）
    ├─ 读取GPUScene InstanceBuffer → Frustum Culling
    ├─ 读取HZB → Occlusion Culling
    ├─ 填充InstanceIdsBufferOut（通过剔除的Instance ID列表）
    └─ 填充DrawIndirectArgsBufferOut（DrawIndexedIndirect参数）
```

### GPU Shader核心逻辑

**源码锚点**：`Engine/Shaders/Private/InstanceCulling/BuildInstanceDrawCommands.usf`，搜索`InstanceCullBuildInstanceIdBufferCS`、`LoadInstanceCullingSetup`、`DrawIndirectArgsBufferOut`、`InstanceIdsBufferOut`。事实类型：GPU端Instance剔除、间接参数输出。

```hlsl
[numthreads(NUM_THREADS_PER_GROUP, 1, 1)]
void InstanceCullBuildInstanceIdBufferCS(uint3 GroupId : SV_GroupID, int GroupThreadIndex : SV_GroupIndex)
{
    // 【步骤1】负载均衡：通过LoadBalancer分配工作
    FInstanceCullingSetup Setup = InstanceCullingSetup(GroupId, GroupThreadIndex);
    if (!Setup.bValid) return;

    uint InstanceId = Setup.InstanceId;
    
    // 【步骤2】读取GPUScene Instance数据
    FInstanceSceneData InstanceData = GetInstanceSceneData(InstanceId, ...);

    // 【步骤3】Frustum Culling
    bool bVisible = true;
    if (bCullInstances)
    {
        FBoxSphereBounds Bounds = GetInstanceBounds(InstanceData);
        bVisible = ViewFrustumIntersectSphere(Bounds.Origin, Bounds.SphereRadius);
        if (bVisible)
            bVisible = ViewFrustumIntersectBox(Bounds.Origin, Bounds.BoxExtent);
    }

    // 【步骤4】Occlusion Culling（使用HZB）
    if (bVisible && bOcclusionCull)
    {
        bVisible = IsVisibleHZB(InstanceData.Bounds, HZBTexture);
    }

    // 【步骤5】通过剔除 → 写入输出
    if (bVisible)
    {
        uint OutputIndex;
        InterlockedAdd(DrawIndirectArgsBufferOut[IndirectArgIndex * 5 + 1], 1, OutputIndex);
        InstanceIdsBufferOut[InstanceIdOffset + OutputIndex] = InstanceId;
    }
}
```

### InstanceCullingLoadBalancer：负载均衡

传统方式每个线程处理一个Instance，但不同DrawCommand的Instance数量差异巨大（有的100个，有的10000个），导致线程利用率不均。

UE5.7使用**Load Balancer**：将所有DrawCommand的Instance打包成统一的Batch，通过Prefix Sum均匀分配给每个线程组。

```hlsl
// Engine/Shaders/Private/InstanceCulling/InstanceCullingLoadBalancer.ush
FInstanceWorkSetup InstanceCullingLoadBalancer_Setup(uint3 GroupId, int GroupThreadIndex, uint BatchInfoIndex)
{
    // 通过预计算的PrefixSum找到当前线程应处理的DrawCommand和Instance
    uint WorkIndex = GroupId.x * NUM_THREADS_PER_GROUP + GroupThreadIndex;
    
    // 二分查找：找到WorkIndex落在哪个Batch
    FInstanceBatchItem Item = BinarySearchBatch(WorkIndex, BatchBuffer, ItemBuffer);
    
    FInstanceWorkSetup Result;
    Result.bValid = (WorkIndex < TotalWork);
    Result.Item = Item;
    Result.LocalItemIndex = WorkIndex - Item.BatchPrefixOffset;
    return Result;
}
```

### Compaction Pass（可选）

某些场景需要保持Instance绘制顺序（如半透明物体）。UE5.7通过`CompactVisibleInstances.usf`实现GPU端Compaction：

```
原始Instance ID列表（有空洞）：
  [3, _, _, 7, _, 12, _, _, 20, _]

Block Count统计 → Prefix Sum → 紧凑输出：
  [3, 7, 12, 20]  ← 连续无空洞，保持原始顺序
```

### 💡 核心要点

一句话：Instance Culling读取GPUScene Buffer在GPU端执行Frustum/HZB剔除，LoadBalancer改善线程利用率，输出DrawIndirectArgs和InstanceID列表；CPU仍负责Pass设置、命令组织和RDG调度。

# 第五部分：与其他系统的关系

---

## 5.1 与Nanite的协同

Nanite的Instance Culling直接读取GPUScene的PrimitiveBuffer和InstanceSceneDataBuffer：

```
GPUScene.Update()
  → PrimitiveBuffer / InstanceSceneDataBuffer 更新完毕
    → Nanite::InstanceCull（NaniteInstanceCulling.usf）
      → 读取FPrimitiveSceneData判断Nanite开关
      → 读取FInstanceSceneData获取Transform和Bounds
      → 执行Frustum/Distance Culling
      → 输出可见Nanite Instance列表
```

**关键**：Nanite不使用通用的`InstanceCullBuildInstanceIdBufferCS`，而是有自己专用的`InstanceCull`入口（支持层次化Culling `InstanceHierarchyCellChunkCull_CS`），但它们都从同一个GPUScene Buffer读取数据。

## 5.2 与Lumen的协同

Lumen需要知道场景中所有Primitive的信息来维护Surface Cache：

```
GPUScene.Update()
  → PrimitiveBuffer更新
    → Lumen::UpdateLumenScene()
      → 遍历变化的Primitive
      → 更新对应的Mesh Cards
      → 重新Capture变化区域的Surface Cache
```

当Primitive的Transform变化时，GPUScene的脏标记同时触发Lumen的Surface Cache局部更新。

## 5.3 与MeshDrawCommand的协同

MeshDrawCommand（MDC）通过`FInstanceCullingContext`与GPUScene交互：

```
FMeshPassProcessor生成FMeshDrawCommand
  → FInstanceCullingContext.AddInstancesToDrawCommand()
    → 记录DrawCommand ↔ Instance映射
  → FInstanceCullingContext.BuildRenderingCommands()
    → GPU端读取GPUScene
    → Culling + 生成DrawIndirectArgs
  → DrawIndexedIndirect()
    → GPU用DrawIndirectArgs自驱动绘制
    → Shader通过InstanceID读取GPUScene数据
```

**传统 vs GPUScene驱动的MDC执行**：

| 维度 | 传统MDC | GPU-Driven MDC |
|------|--------|----------------|
| DrawCall参数 | CPU填充 | GPU Compute生成 |
| Instance Count | CPU统计可见数 | GPU原子计数器累加 |
| 状态切换 | 每个DrawCall一次 | 合并为少量Indirect Draw |
| CPU开销 | 随可见Mesh和提交路径增长 | 通常转为少量Dispatch/RDG调度和命令组织成本 |
