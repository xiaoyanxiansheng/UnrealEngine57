## 🎯 Unity对比：多光源方案差异

| 维度 | UE5.7 MegaLights | Unity URP Forward+ | 差异说明 |
|------|------------------|---------------------|----------|
| **光源上限** | 无实际上限（测试过数十万） | ~256/Tile | UE用随机采样消除上限，Unity受Light List容量限制 |
| **阴影策略** | 全部光源可投射阴影 | 仅少量光源投射Shadow Map | UE通过VSM/RT高效处理大量带阴影光源 |
| **算法基础** | 随机采样 + 光线追踪 | 确定性Tile Light List | UE有噪点需降噪，Unity无噪点但受光源数限制 |
| **GPU开销曲线** | 近乎恒定（光源数对开销影响小） | 线性增长（每增加光源增加开销） | UE适合大规模光源，Unity适合少量光源 |
| **硬件要求** | 需要RT硬件或VSM支持 | 通用GPU | UE门槛更高 |
| **噪点** | 有（需时空降噪） | 无（确定性渲染） | 各有取舍 |
| **适用场景** | 密集光源（城市夜景、粒子发光） | 常规游戏光照（<100光源） | 各有侧重 |

**关键差异**：Unity的Forward+/Clustered Deferred对每个Tile遍历所有影响光源，光源越多开销越高；MegaLights对每个像素只随机采样少量光源（2-4个），通过时空降噪恢复完整光照，光源总数对每帧开销影响极小。

# 第一部分：理解MegaLights的存在意义

> **核心目标**：从传统多光源方案的实际瓶颈出发，理解MegaLights解决了什么问题、付出了什么代价

---

## 一、传统多光源方案的三大瓶颈

### 1.1 瓶颈一：Clustered/Tiled Deferred的光源数上限

假设你正在开发一个赛博朋克城市夜景，场景中有10,000个霓虹灯、路灯、车灯。

**Unity URP Forward+的做法**：

```
Forward+ Light Culling：
┌──────────────────────────────────────┐
│ 屏幕划分为 16×16 像素的Tile           │
│                                      │
│  每个Tile维护一个Light List：         │
│  ┌─────────────────────────────┐     │
│  │ Light 0, Light 1, ... Light N│    │
│  │ 最大容量：256个光源          │     │  ← 硬性上限
│  └─────────────────────────────┘     │
│                                      │
│  Fragment Shader：                   │
│  for (i = 0; i < LightCount; i++)   │
│      color += CalculateLight(i);    │  ← 每个光源都要计算
│                                      │
│  问题：                              │
│  · 256个光源/Tile的硬性上限          │
│  · 超出后Light List溢出，光照异常     │
│  · 10,000个光源 → 许多Tile溢出       │
└──────────────────────────────────────┘
```

**UE5 Clustered Deferred类似**：每个Cluster有Light List容量限制，光源密集区域容易溢出。

**问题本质**：传统方案要求每个像素**确定性地**遍历所有影响它的光源。光源数×像素数 = 着色次数，线性增长无法避免。

### 1.2 瓶颈二：大量光源的阴影开销

传统方案中，每个投射阴影的光源需要独立的Shadow Map：

```
100个阴影光源的传统方案：

光源1 → Shadow Map 1 (1024×1024) → 渲染所有Caster → ~0.5ms
光源2 → Shadow Map 2 (1024×1024) → 渲染所有Caster → ~0.5ms
...
光源100 → Shadow Map 100 → ~0.5ms

总阴影开销：100 × 0.5ms = 50ms  ← 已超出16ms帧预算

实际做法：只让5-10个光源投射阴影
其余光源：无阴影 → 漏光严重
```

**Unity URP的妥协**：主光源（Directional）投射阴影，其余Additional Lights大多不投射阴影。

**问题本质**：Shadow Map是逐光源的，光源数与阴影开销线性关系。想要所有光源都有阴影，传统方案不可行。

### 1.3 瓶颈三：GPU Overdraw

传统Deferred Lighting对每个光源执行一个全屏/区域DrawCall：

```
传统Deferred Lighting流程：
for each light in visibleLights:
    DrawLightVolume(light)  ← 每个光源一个DrawCall
    // 光源体积内的所有像素都要着色
    // 即使大部分像素被其他光源覆盖

10,000个光源 → 10,000个DrawCall
每个DrawCall覆盖的像素存在大量重叠（Overdraw）
```

## 二、MegaLights的解决思路：随机采样 + 降噪

MegaLights的核心设计哲学：**不遍历所有光源，而是随机采样少量光源，用降噪恢复完整光照**。

| 传统方案 | MegaLights方案 | 设计意图 |
|---------|---------------|---------|
| 遍历所有光源 | 每像素随机选2-4个光源 | 开销与光源总数无关 |
| 逐光源Shadow Map | 光线追踪/VSM阴影 | 所有光源都有阴影 |
| 确定性渲染（无噪点） | 随机采样（有噪点）→ 时空降噪 | 用降噪换取恒定开销 |
| Light List容量限制 | 无上限 | 支持任意数量光源 |

**Unity开发者类比**：想象Unity不再为每个像素遍历Tile内所有光源，而是"掷骰子"随机选2个光源采样，然后用类似TAA的时间滤波把多帧结果混合成完整光照——这就是MegaLights的核心思路。

**官方定义**（源码注释）：
> "MegaLights uses stochastic sampling to render many shadow casting lights efficiently, with a consistent low GPU cost."
> — `Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `MegaLights uses stochastic sampling`

## 三、三阶段管线概览

```
MegaLights三阶段管线（每帧执行）：

╔══════════════════════════════════════════════════════════╗
║  阶段1：Stochastic Sampling（随机采样）                   ║
║  · Tile Classification：按材质复杂度分类屏幕区域          ║
║  · Sample Generation：为每个降采样像素选择光源             ║
║  · 历史引导：根据上一帧结果调整采样概率                    ║
║  · 输出：LightSampleRays（每像素2-4个光源采样）           ║
╚══════════════════════════════════════════════════════════╝
                          ↓
╔══════════════════════════════════════════════════════════╗
║  阶段2：Ray Tracing / Shadow Testing（阴影测试）          ║
║  · Hardware RT路径：DXR内联光线追踪（硬件加速）            ║
║  · VSM路径：Virtual Shadow Maps查询                       ║
║  · 光照计算：BRDF着色（Diffuse + Specular分离）           ║
║  · 输出：LightSamples（每采样的光照贡献）                  ║
╚══════════════════════════════════════════════════════════╝
                          ↓
╔══════════════════════════════════════════════════════════╗
║  阶段3：Spatiotemporal Denoising（时空降噪）              ║
║  · Resolve：Upscale + 累积到全分辨率Diffuse/Specular      ║
║  · Spatial Filter：空间域双边滤波（保边去噪）              ║
║  · Temporal Accumulation：时间域累积（混合历史帧）         ║
║  · Visible Light Hash：光源可见性追踪（防止ghost）         ║
║  · 输出：降噪后的光照 → 写入SceneColor                    ║
╚══════════════════════════════════════════════════════════╝
```

**与传统方案的核心差异**：

| 步骤 | 传统Clustered Deferred | MegaLights |
|------|----------------------|------------|
| 光源剔除 | CPU/GPU构建Light List | 随机采样（隐式剔除） |
| 着色次数/像素 | = 影响该像素的光源数 | = 固定2-4次 |
| 阴影 | 少量光源Shadow Map | 全部光源RT/VSM |
| 后处理 | 无需降噪 | 必须时空降噪 |
| 光源数扩展性 | O(N × Pixels) | O(Samples × Pixels) |

## 四、代价与限制

MegaLights不是银弹，它有明确的代价和适用范围：

| 代价 | 说明 |
|------|------|
| **噪点** | 随机采样天然产生噪点，降噪无法完全消除（静止场景收敛后可接受） |
| **时间延迟** | Temporal Accumulation需要多帧收敛，快速移动时噪点明显 |
| **硬件要求** | RT路径需要DXR硬件（RTX 2060+/RX 6000+），VSM路径需要Virtual Shadow Maps |
| **基础开销** | 即使少量光源，采样+降噪的固定开销也比传统方案高 |
| **不支持方向光**（默认） | `r.MegaLights.DirectionalLights`默认关闭，需要`r.Forward.LightBuffer.Mode > 0` |
| **实验性功能** | 官方标记为Experimental，API可能变化 |

**决策原则**：场景光源数量大（>100个Point/Spot/Rect Light）且需要阴影时，MegaLights有显著优势；少量光源（<20个）的常规场景，传统Deferred Lighting开销更低且无噪点。

## 一、EMegaLightsMode：RT与VSM双路径

MegaLights支持两种阴影策略，通过每个光源的`MegaLightsShadowMethod`属性选择：

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.h`，搜索 `enum class EMegaLightsMode`

```cpp
// 真实代码：MegaLights.h，搜索 enum class EMegaLightsMode
enum class EMegaLightsMode
{
    Disabled,    // MegaLights未启用
    EnabledRT,   // 使用Hardware Ray Tracing做阴影测试
    EnabledVSM   // 使用Virtual Shadow Maps做阴影测试
};
```

**双路径选择逻辑**（`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `GetMegaLightsMode`）：

```
光源启用MegaLights？
    ├─ 否 → EMegaLightsMode::Disabled（走传统光照）
    └─ 是 → 检查ShadowMethod
         ├─ Default → 根据r.MegaLights.DefaultShadowMethod决定
         ├─ RayTracing → 检查View是否允许RT
         │    ├─ 是 → EnabledRT
         │    └─ 否 → Disabled
         └─ VirtualShadowMap → EnabledVSM
```

**两种路径对比**：

| 维度 | EnabledRT（Hardware RT） | EnabledVSM（Virtual Shadow Maps） |
|------|------------------------|----------------------------------|
| 硬件要求 | RTX 2060+（DXR支持） | 通用GPU（需要VSM系统） |
| 阴影精度 | 像素级精确 | 取决于VSM分辨率 |
| 性能特征 | RT核心利用率高 | 传统Compute Shader |
| 适用场景 | 高端硬件 | 中端硬件/AMD GPU |

### 2.2 设计要点

**为什么是per-View？** 分屏或VR双眼各自有独立的采样、降噪历史和光源可见性状态。

**为什么有GBuffer和HairStrands两套？** 毛发渲染使用特殊的Visibility Buffer，无法复用GBuffer的材质分类和降采样逻辑，需要独立流水线。

## 五、完整数据流时序图

```
DeferredShadingRenderer::Render()
    │
    ├─ GBuffer Pass（BasePass）
    │   └─ 输出：SceneDepth, GBuffer（Normal, BaseColor, Metallic, Roughness, ...）
    │
    ├─ RenderLights()（传统Deferred光照：Directional等）
    │
    ├─ GenerateMegaLightsSamples()  ← MegaLights采样阶段（在VSM标记前完成）
    │   └─ per View:
    │       ├─ Setup()               → 初始化参数、分配RDG资源
    │       ├─ GenerateSamples()     → Tile分类 + 随机光源采样
    │       └─ MarkVSMPages()        → 标记VSM需要渲染的页面
    │
    ├─ （VSM渲染Shadow Maps）
    │
    └─ RenderMegaLights()  ← MegaLights渲染阶段
        └─ per View:
            └─ RenderMegaLightsViewContext()
                └─ per ShadingPass (通常1次，Reference模式可多次):
                    ├─ TileClassificationMark() → 更新Tile分类（第2+次Pass）
                    ├─ GenerateSamples()        → 新采样（第2+次Pass）
                    ├─ RayTrace()               → 光线追踪 / 光照着色
                    └─ Resolve()                → 累积到全分辨率
                └─ DenoiseLighting()            → 时空降噪 → 写入SceneColor
```

> "Do MegaLights sampling before VSM pages are marked and rendered so they can be specialized based on the selected samples."
> — `Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp`，搜索 `Do MegaLights sampling before VSM pages are marked`

## 一、GenerateMegaLightsSamples：采样生成入口

### 1.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `FDeferredShadingSceneRenderer::GenerateMegaLightsSamples`  
**函数**：`FDeferredShadingSceneRenderer::GenerateMegaLightsSamples()`  
**职责**：为所有View创建MegaLights渲染上下文，执行采样生成和VSM页面标记

### 1.2 核心流程

```cpp
// 真实代码：MegaLights.cpp，搜索 FDeferredShadingSceneRenderer::GenerateMegaLightsSamples
TSharedPtr<FMegaLightsFrameTemporaries> FDeferredShadingSceneRenderer::GenerateMegaLightsSamples(
    FRDGBuilder& GraphBuilder,
    const FSceneTextures& SceneTextures,
    const FLumenSceneFrameTemporaries& LumenFrameTemporaries,
    FRDGTextureRef LightingChannelsTexture)
{
    // 【步骤1】：前置检查
    if (!MegaLights::IsEnabled(ViewFamily) || !ViewFamily.EngineShowFlags.DirectLighting)
    {
        return nullptr;  // MegaLights未启用或直接光照被关闭
    }

    // 【步骤2】：检测是否有光源使用VSM路径
    const bool bUseVSM = ShadowSceneRenderer.AreAnyLightsUsingMegaLightsVSM();

    // 【步骤3】：准备Blue Noise（用于高质量随机数生成）
    FBlueNoise BlueNoise = GetBlueNoiseGlobalParameters();
    TUniformBufferRef<FBlueNoise> BlueNoiseUniformBuffer = CreateUniformBufferImmediate(BlueNoise, ...);

    // 【步骤4】：为每个View创建渲染上下文
    TSharedPtr<FMegaLightsFrameTemporaries> MegaLightsFrameTemporaries(new FMegaLightsFrameTemporaries);
    for (int32 ViewIndex = 0; ViewIndex < Views.Num(); ++ViewIndex)
    {
        FViewInfo& View = Views[ViewIndex];

        // 创建ViewContext（GBuffer路径）
        FMegaLightsViewContext& ViewContext = MegaLightsFrameTemporaries->ViewContexts.Add_GetRef(...);
        ViewContext.Setup(LightingChannelsTexture, LumenFrameTemporaries, ...);
        ViewContext.GenerateSamples(LightingChannelsTexture, 0 /* ShadingPassIndex */);
        ViewContext.MarkVSMPages(VirtualShadowMapArray);

        // 创建ViewContext（HairStrands路径，如果有毛发）
        if (bHairStrands)
        {
            FMegaLightsViewContext& HairContext = MegaLightsFrameTemporaries->ViewContextsHairStrands.Add_GetRef(...);
            HairContext.Setup(..., EMegaLightsInput::HairStrands);
            HairContext.GenerateSamples(LightingChannelsTexture, 0);
            HairContext.MarkVSMPages(VirtualShadowMapArray);
        }
    }

    return MegaLightsFrameTemporaries;
}
```

### 1.3 关键细节

**Blue Noise**：MegaLights使用Blue Noise（蓝噪声）而非白噪声作为随机数源。蓝噪声的频谱分布使得采样点在空间上更均匀，相邻像素的采样模式差异最大化，降噪效果更好。

**FMegaLightsFrameTemporaries结构**（`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `struct FMegaLightsFrameTemporaries`）：

```cpp
// 真实代码：MegaLights.cpp，搜索 struct FMegaLightsFrameTemporaries
struct FMegaLightsFrameTemporaries
{
    TArray<FMegaLightsViewContext, SceneRenderingAllocator> ViewContexts;
    TArray<FMegaLightsViewContext, SceneRenderingAllocator> ViewContextsHairStrands;
};
```

使用`SceneRenderingAllocator`（帧线性分配器），所有临时数据在帧结束时批量释放，零碎片。

### 💡 核心要点

`GenerateMegaLightsSamples`的执行时机是精心设计的——在VSM渲染之前，这样VSM只需渲染被采样选中的光源页面。

## 二、RenderMegaLights与渲染主循环

### 2.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `FDeferredShadingSceneRenderer::RenderMegaLights`  
**函数**：`FDeferredShadingSceneRenderer::RenderMegaLights()`  
**职责**：驱动每个View的光线追踪、着色、降噪

### 2.2 核心流程

```cpp
// 真实代码：MegaLights.cpp，搜索 FDeferredShadingSceneRenderer::RenderMegaLights
void FDeferredShadingSceneRenderer::RenderMegaLights(
    FRDGBuilder& GraphBuilder,
    TSharedPtr<FMegaLightsFrameTemporaries> MegaLightsFrameTemporaries,
    const FSceneTextures& SceneTextures,
    const TArrayView<FRDGTextureRef> NaniteShadingMasks,
    FRDGTextureRef LightingChannelsTexture)
{
    if (!MegaLightsFrameTemporaries.IsValid())
        return;

    RDG_EVENT_SCOPE_STAT(GraphBuilder, MegaLights, "MegaLights");
    RDG_GPU_STAT_SCOPE(GraphBuilder, MegaLights);

    for (int32 ViewIndex = 0; ViewIndex < MegaLightsFrameTemporaries->ViewContexts.Num(); ++ViewIndex)
    {
        FViewInfo& View = Views[ViewIndex];
        FMegaLightsViewContext& ViewContext = MegaLightsFrameTemporaries->ViewContexts[ViewIndex];
        FMegaLightsViewContext& ViewContextHairStrands = MegaLightsFrameTemporaries->ViewContextsHairStrands[ViewIndex];

        // GBuffer路径：渲染不透明物体的MegaLights光照
        RenderMegaLightsViewContext(
            GraphBuilder, ViewContext, VirtualShadowMapArray,
            NaniteShadingMasks, LightingChannelsTexture,
            &View.GetOwnMegaLightsVolume(),      // 输出体积光照
            SceneTextures.Color.Target);          // 输出到SceneColor

        // HairStrands路径：渲染毛发的MegaLights光照
        if (ViewContextHairStrands.AreSamplesGenerated())
        {
            RenderMegaLightsViewContext(
                GraphBuilder, ViewContextHairStrands, VirtualShadowMapArray,
                NaniteShadingMasks, LightingChannelsTexture,
                nullptr,                          // 毛发不使用体积光照
                View.HairStrandsViewData.VisibilityData.SampleLightingTexture);
        }
    }
}
```

### 2.3 RenderMegaLightsViewContext：核心渲染循环

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLights.cpp`，搜索 `RenderMegaLightsViewContext`

```cpp
// 真实代码：MegaLights.cpp，搜索 RenderMegaLightsViewContext
static void RenderMegaLightsViewContext(
    FRDGBuilder& GraphBuilder,
    FMegaLightsViewContext& ViewContext,
    const FVirtualShadowMapArray& VirtualShadowMapArray,
    const TArrayView<FRDGTextureRef> NaniteShadingMasks,
    FRDGTextureRef LightingChannelsTexture,
    FMegaLightsVolume* MegaLightsVolume,
    FRDGTextureRef OutputColorTarget)
{
    check(ViewContext.AreSamplesGenerated());

    // Reference模式可循环多次（用于质量测试），正常模式ShadingPassCount=1
    for (uint32 ShadingPassIndex = 0; ShadingPassIndex < ViewContext.GetReferenceShadingPassCount(); ++ShadingPassIndex)
    {
        // 第2+次Pass需要重新分类和采样
        if (ShadingPassIndex > 0)
        {
            ViewContext.TileClassificationMark(ShadingPassIndex);
            ViewContext.GenerateSamples(LightingChannelsTexture, ShadingPassIndex);
        }

        ViewContext.RayTrace(VirtualShadowMapArray, NaniteShadingMasks, ShadingPassIndex);
        ViewContext.Resolve(OutputColorTarget, MegaLightsVolume, ShadingPassIndex);
    }

    ViewContext.DenoiseLighting(OutputColorTarget);
}
```

**Reference模式**：通过`r.MegaLights.ReferenceShadingPassCount`可设置多次循环（最高10240次），每次使用不同的采样模式。用于质量对比测试，生产环境始终为1次。

### 💡 核心要点

渲染循环的结构是：`[TileClassify → GenerateSamples → RayTrace → Resolve]× N → DenoiseLighting`。降噪在所有Shading Pass完成后统一执行。

## 三、Tile Classification：材质分类优化

### 3.1 设计目标

**问题**：不同材质区域的着色复杂度差异巨大。粗糙地面只需要Lambert Diffuse，光滑金属需要完整GGX Specular，面光源（Rect Light）需要额外的面积积分。如果所有像素都运行最复杂的Shader，浪费严重。

**解决**：将屏幕划分为Tile（8×8像素），根据GBuffer中的材质属性将每个Tile分类到不同的着色路径，使用Indirect Dispatch只对需要复杂着色的Tile执行完整Shader。

### 3.2 分类策略

```
Tile分类维度（三个正交维度的组合）：

维度1：材质复杂度
├─ Simple/Single：单层BRDF（地面、墙壁等常规材质）
└─ Complex：多层BRDF或特殊材质（透涂层、次表面散射等）

维度2：光源类型
├─ 无Rect Light：只有Point/Spot Light
├─ Rect Light：有Rect Light（需要面积积分）
└─ Textured Rect Light：有纹理Rect Light（需要纹理采样）

维度3：是否为空
└─ Empty：Tile内无几何体（天空），直接跳过

组合示例：
· SimpleShading                  → 简单材质 + Point/Spot
· ComplexShading_Rect_Textured   → 复杂材质 + 纹理面光源
· Empty                          → 跳过
```

### 3.3 Indirect Dispatch

Tile分类后，使用GPU Indirect Dispatch按类型批量执行着色：

```
GPU执行：
TileClassifyCS → 输出各类型Tile数量到IndirectArgs Buffer
                  ↓
DispatchIndirect(SimpleShadingCS,  IndirectArgs[SimpleShading])  ← 只处理Simple Tile
DispatchIndirect(ComplexShadingCS, IndirectArgs[ComplexShading]) ← 只处理Complex Tile
...

优势：空Tile零开销，Simple Tile用轻量Shader，Complex Tile用完整Shader
```

## 四、采样生成：随机光源选择与历史引导

### 4.1 采样策略概述

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsSampling.cpp`，搜索 `FGenerateLightSamplesCS`  
**Shader**：`Engine/Shaders/Private/MegaLights/MegaLightsSampling.usf`，搜索 `GenerateLightSamplesCS`

MegaLights的采样核心是**从Forward Light Grid中选择光源**。UE5维护一个Forward Light Data Buffer，包含所有光源的信息。MegaLights不构建独立的BVH——它复用Forward Lighting已有的光源数据结构。

### 4.2 降采样

采样不在全分辨率执行，而是在降采样分辨率上：

```
r.MegaLights.DownsampleMode的三种模式：

Mode 0：1×1（全分辨率）
  1920×1080 → 1920×1080
  性能最低，质量最高

Mode 1：2×1（棋盘格）
  1920×1080 → 960×1080
  性能中等

Mode 2：2×2（半分辨率，默认）
  1920×1080 → 960×540
  性能最高，像素数减少75%
```

降采样后，每个"超级像素"生成`NumSamplesPerPixel`个光源采样（默认4个），然后Resolve阶段Upscale回全分辨率。

### 4.3 历史引导采样（GuideByHistory）

MegaLights利用上一帧的光源可见性信息来优化当前帧的采样策略：

```
r.MegaLights.GuideByHistory的三种模式：

Mode 0：禁用
  每帧独立随机采样，不参考历史

Mode 1：引导到可见光源
  上一帧被遮挡的光源 → 降低采样概率
  上一帧可见的光源   → 提升采样概率
  效果：减少阴影区域的噪点

Mode 2：引导到光源的可见部分（默认）
  不仅考虑光源是否可见，还考虑光源的哪个方向可见
  效果：对面光源（Rect Light）特别有效
```

**Visible Light Hash**：MegaLights维护一个per-Tile的哈希表（`VisibleLightHash`），记录每个Tile区域哪些光源在上一帧可见。这个信息用于引导采样并检测光源可见性变化（用于Temporal Filter的ghost检测）。

## 五、光线追踪：Hardware RT与VSM双路径

### 5.1 模块概览

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsRayTracing.cpp`，搜索 `RayTraceLightSamples`、`FHardwareRayTraceLightSamples`

这是MegaLights中最大的源文件，负责：
1. 为采样选中的光源执行阴影测试
2. 计算BRDF着色（Diffuse + Specular分离）
3. 支持Screen Traces（屏幕空间追踪，HZB加速）、Hardware RT（DXR）、VSM三种阴影来源

### 5.2 阴影测试策略

```
每个采样的阴影测试路径选择：

                  ┌─ Screen Traces（r.MegaLights.ScreenTraces=1）
                  │   用HZB做屏幕空间遮挡测试（最快，但只能检测屏幕内遮挡）
采样的光源 ──→    │
                  ├─ Hardware RT（r.MegaLights.HardwareRayTracing=1）
                  │   DXR Inline Ray Tracing（硬件加速，最精确）
                  │   支持Inline模式（在Compute Shader内追踪）
                  │
                  └─ VSM Query（EMegaLightsMode::EnabledVSM）
                      查询Virtual Shadow Map（无需RT硬件）
```

**Screen Traces优先**：如果启用，先尝试用HZB做屏幕空间遮挡测试。Screen Trace成功则直接使用结果，失败（无法判定）才fallback到RT或VSM。这减少了昂贵的光追调用。

**Shader**：`Engine/Shaders/Private/MegaLights/MegaLightsHardwareRayTracing.usf`，搜索 `HardwareRayTraceLightSamples`

使用DXR 1.1的Inline Ray Tracing（`RayQuery`），在Compute Shader中内联执行：

Ray.Origin = WorldPosition;
Ray.Direction = normalize(LightPosition - WorldPosition);
Ray.TMin = 0.001;
Ray.TMax = distance(LightPosition, WorldPosition);

**Inline RT vs Ray Generation Shader**：MegaLights选择Inline模式（在Compute Shader中调用`TraceRayInline`），而非独立的Ray Generation Shader。原因：Inline模式可以和光照计算在同一Compute Shader中执行，减少Shader调度开销。

### 5.4 Trace Compaction

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsRayTracing.cpp`，搜索 `FCompactLightSampleTracesCS`

为了提高GPU利用率，MegaLights在追踪前执行**Trace Compaction**——将需要追踪的采样紧凑排列，跳过不需要追踪的（被Screen Trace已解决的、权重过低的、空Tile的）。这避免了GPU线程空转。

### 5.5 材质评估模式

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsRayTracing.cpp`，搜索 `enum class EMaterialMode`

{
    enum class EMaterialMode

着色时根据光源特性选择不同的材质评估路径。纹理Rect Light需要额外的纹理采样，Light Function需要评估光照函数，组合模式同时处理两者。

### 💡 核心要点

MegaLights的光追路径不是"为了光追而光追"——它是解决"大量光源阴影"问题的手段。Screen Traces优先、Trace Compaction减少浪费、Inline RT减少调度开销，三层优化使得光追开销可控。

## 六、Resolve与Denoising：时空降噪管线

### 6.1 Resolve阶段

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsResolve.cpp`，搜索 `FMegaLightsViewContext::Resolve`

Resolve负责将降采样的光照结果累积到全分辨率：

```
Resolve流程：
1. 读取降采样的LightSamples（每像素2-4个光源的着色结果）
2. 重要性采样权重修正（除以PDF）
3. 累积到全分辨率的Diffuse/Specular分离Buffer
4. 同时处理Volume数据（体积雾、半透明）
```

**Diffuse/Specular分离**：MegaLights将Diffuse光照和Specular光照分别存储和降噪。这是因为Diffuse光照空间频率低（适合激进降噪），Specular光照空间频率高（需要保守降噪）。分离存储允许对两者使用不同的降噪参数。

### 6.2 Spatial Filter（空间域滤波）

**来源**：`Engine/Source/Runtime/Renderer/Private/MegaLights/MegaLightsDenoising.cpp`，搜索 `FMegaLightsViewContext::DenoiseLighting`  
**Shader**：`Engine/Shaders/Private/MegaLights/MegaLightsDenoiserSpatial.usf`，搜索 `DenoiserSpatialCS`

```
空间域滤波原理：

对每个像素，读取周围邻域像素的光照值，
根据几何相似性（深度差、法线差）加权平均：

FilteredColor = Σ (Color[i] × Weight[i]) / Σ Weight[i]

Weight = DepthWeight × NormalWeight
  DepthWeight  = exp(-|depth_diff| × scale)    ← 深度差越大权重越低
  NormalWeight = pow(max(0, dot(N1, N2)), exp)  ← 法线差越大权重越低

效果：平滑噪点，但保留几何边缘（深度/法线不连续处不滤波）
```

**CVar控制**：`r.MegaLights.Spatial`（默认true）

### 6.3 Temporal Accumulation（时间域累积）

**Shader**：`Engine/Shaders/Private/MegaLights/MegaLightsDenoiserTemporal.usf`，搜索 `DenoiserTemporalCS`

```
时间域累积原理：

1. 重投影：用运动矢量找到当前像素在上一帧的位置
   PrevUV = CurrentUV - MotionVector

2. 读取历史光照
   HistoryColor = HistoryTexture.Sample(PrevUV)

3. Ghost检测：
   · 用VisibleLightHash检测光源可见性是否变化
   · 光源出现/消失 → 降低历史权重（防止ghost）
   · 深度/法线变化过大 → 降低历史权重（防止错误重投影）

4. 混合：
   Alpha = f(NumFramesAccumulated, MotionVector, GhostDetection)
   FinalColor = lerp(HistoryColor, CurrentColor, Alpha)

   静止场景：Alpha ≈ 0.05（95%历史权重，快速收敛）
   运动场景：Alpha ≈ 0.3-1.0（更多当前帧权重，减少ghost）
```

**NumFramesAccumulated**：MegaLights追踪每个像素已累积的有效帧数。新暴露区域（遮挡解除、相机运动暴露的新区域）帧数为0，使用更高的Alpha快速收敛；已累积多帧的稳定区域使用低Alpha保持平滑。

**Visible Light Hash**：每个Tile维护一个光源可见性哈希值。当哈希值与历史帧不同（说明该Tile区域的可见光源集合发生了变化），Temporal Filter降低历史权重以快速适应变化。这是防止"ghost"（残影）的关键机制。

### 💡 核心要点

MegaLights的降噪质量取决于三个因素：(1) 采样数越多，单帧噪点越少；(2) 空间滤波平滑邻域噪点但可能模糊细节；(3) 时间累积是最有效的降噪手段，但受运动和场景变化限制。三者协同才能达到可接受的视觉质量。

# 第五部分：与其他系统的关系

> **核心目标**：理解MegaLights在UE5渲染管线中的集成位置

---

## 一、与Virtual Shadow Maps的协同

MegaLights和VSM深度集成：

```
VSM集成流程：
1. MegaLights采样阶段选择需要追踪的光源
2. 对使用VSM路径的光源，标记需要渲染的VSM页面（MarkVSMPages）
3. VSM系统渲染被标记的页面
4. MegaLights RayTrace阶段查询VSM获取阴影结果
```

**优势**：MegaLights只标记被采样选中的光源的VSM页面，大幅减少VSM需要渲染的Shadow Map量。对于10,000个光源的场景，每帧可能只有几百个光源被采样选中需要VSM页面。

**相关CVar**：`r.MegaLights.VSM.MarkPages`（默认1）

---

## 二、与Lumen的集成

MegaLights处理**直接光照**（Direct Lighting），Lumen处理**间接光照**（Indirect Lighting/GI）。两者互补：

```
光照管线分工：
                  ┌─ 方向光（Directional）→ 传统Deferred Lighting
直接光照 Direct ──┤
                  └─ 局部光（Point/Spot/Rect）→ MegaLights

间接光照 Indirect → Lumen（Surface Cache + Screen Probe）

最终光照 = Direct + Indirect
```

**共享基础设施**：
- 硬件光追TLAS（Top-Level Acceleration Structure）被MegaLights和Lumen Hardware RT共享
- GPUScene的光源数据被两者共同访问
- RenderGraph的资源管理和Pass调度统一处理

---

## 三、与Nanite的协同

```
Nanite → MegaLights数据流：

Nanite输出：
├─ Visibility Buffer → Depth → MegaLights降采样深度
├─ GBuffer（Normal, BaseColor, etc.）→ MegaLights Tile分类 + BRDF着色
└─ Nanite Shading Masks → MegaLights RayTrace遮挡检测

MegaLights使用NaniteShadingMasks优化追踪：
对Nanite渲染的几何，可利用Nanite的高精度遮挡信息
```

## 四、适用场景决策指南

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| **< 20个光源** | 传统Deferred Lighting | 固定开销低，无噪点 |
| **20-100个光源** | 传统/MegaLights均可 | MegaLights对阴影有优势 |
| **100-1000个光源** | MegaLights | 传统方案Light List接近上限 |
| **1000+光源** | MegaLights（必须） | 只有MegaLights可行 |
| **粒子发光** | MegaLights | 大量小光源正是其优势 |
| **城市夜景** | MegaLights | 霓虹灯/路灯/车灯密集 |
| **无RT硬件** | MegaLights(VSM) 或传统 | VSM路径不需要RT硬件 |
| **VR/低延迟** | 谨慎使用 | Temporal降噪有延迟 |
