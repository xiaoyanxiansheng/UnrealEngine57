## 🎯 Unity对比：几何渲染管线差异

| 维度 | UE5.7 Nanite | Unity 传统 MeshRenderer / LODGroup 路径 | 差异说明 |
|------|-------------|----------------------|----------|
| **LOD管理** | 自动生成，像素级连续LOD | 手动制作5-7个离散LOD | UE无Pop-in，Unity有LOD切换跳变 |
| **剔除执行位置** | GPU（Compute Shader） | 传统 LOD/Renderer 路径多由 CPU/SRP 组织；BRG/Entities 等路径另算 | UE把 Nanite cluster 剔除固定放到 GPU 主路径 |
| **剔除粒度** | Cluster级（128三角形） | Mesh级（整个物体） | UE剔除更精细 |
| **光栅化** | 软件+硬件混合 | 纯硬件固定管线 | UE对小三角形更高效 |
| **DrawCall数量** | 通常大幅批量化 | 传统 CPU-driven 场景可能产生大量 draw；SRP Batcher/instancing/BRG 会改善 | 具体数量必须按项目和路径实测 |
| **材质着色** | 与 Nanite material bins / 后续 mesh pass 协作 | 传统硬件光栅路径 | Nanite 优化几何提交和可见性，不等于所有像素成本消失 |
| **运行时内存** | 支持流式加载 | 传统 LOD 资源常需要多级资产 | 节省比例依内容和平台而定 |
| **适用场景** | 静态/半静态高多边形 | 通用场景 | 各有侧重 |

**关键差异**：和传统 MeshRenderer + LODGroup 的 CPU-driven 心智相比，Nanite把 cluster 剔除、LOD 选择和部分小三角形处理固定纳入 GPU-driven 路径，大幅降低 CPU 逐物体提交压力。不要把它理解成“Unity 永远 CPU 剔除、UE 永远零成本”；Unity 的 BRG/Entities/instancing 路径和 UE 的材质、阴影、带宽成本都要单独评估。

# 第一部分：理解Nanite的存在意义

> **核心目标**：从传统渲染管线的实际瓶颈出发，理解Nanite解决了什么问题、付出了什么代价

---

## 一、传统渲染管线的三大瓶颈

### 1.1 瓶颈一：CPU DrawCall瓶颈

假设你正在开发一个开放世界场景，场景中有10,000个物体（岩石、植被、建筑）。

**Unity URP的做法**：

```csharp
// Unity URP传统CPU-driven示意：按 Renderer 组织可见对象
void RenderScene()
{
    foreach (var renderer in visibleRenderers)  // 遍历10,000个
    {
        renderer.material.SetPass(0);  // 绑定材质状态
        Graphics.DrawMesh(              // 提交一个DrawCall
            renderer.mesh,
            renderer.transform.localToWorldMatrix,
            renderer.material, 0
        );
    }
    // 结果：10,000+个DrawCall
    // CPU耗时：~20ms（仅DrawCall提交）
    // 瓶颈：CPU主线程被DrawCall淹没
}
```

**UE5传统BasePass类似**：每个MeshBatch生成一个`FMeshDrawCommand`，CPU端提交数千个DrawCall。

**问题本质**：CPU串行提交DrawCall，每个DrawCall都有固定开销（状态验证、资源绑定、驱动层处理），物体越多CPU越慢。

### 1.2 瓶颈二：LOD管理瓶颈

传统LOD需要美术手动制作5-7个细节级别：

```
LOD0：50,000三角形（近距离）
LOD1：20,000三角形
LOD2：5,000三角形
LOD3：1,000三角形
LOD4：200三角形（远距离）

问题1：美术工作量巨大（每个物体×5个LOD）
问题2：LOD切换时有明显Pop-in跳变
问题3：CPU计算LOD距离有延迟
问题4：LOD粒度粗（整个Mesh统一切换）
```

### 1.3 瓶颈三：小三角形光栅化效率

GPU硬件光栅化的固有限制——**Quad Overdraw问题**：

```
GPU硬件以2×2像素的Quad为最小处理单元：

三角形覆盖64像素：
┌──────────────┐
│ Quad占满     │  效率：97%（几乎无浪费）
│ Setup 2cyc   │
│ Raster 64cyc │
└──────────────┘

三角形只覆盖1像素：
┌──────────────┐
│ Quad中3/4浪费│  效率：25%（浪费75%算力）
│ Setup 2cyc   │
│ Raster 1cyc  │  一个Quad只有1个像素有效
└──────────────┘

Nanite场景中60%的三角形<4像素 → 大量GPU算力浪费在Quad Overdraw上
```

**三大瓶颈总结**：

| 瓶颈 | 根因 | 影响 |
|------|------|------|
| **CPU DrawCall** | 串行提交，每个DrawCall固定开销 | CPU成为帧率瓶颈 |
| **LOD管理** | 手动制作，离散切换 | 美术量大，Pop-in明显 |
| **小三角形** | 硬件Quad粒度浪费 | GPU效率下降到25% |

---

## 二、Nanite的解决思路：GPU-Driven一切

Nanite的核心设计哲学：**把传统在CPU上做的事全部搬到GPU**。

| 传统方案（CPU-Driven） | Nanite方案（GPU-Driven） |
|----------------------|------------------------|
| 传统CPU-driven路径逐个组织Draw | GPU Indirect Draw / 批量提交 |
| CPU计算LOD距离 | GPU Compute Shader实时计算屏幕空间误差 |
| CPU Frustum Culling | GPU Compute Shader并行剔除 |
| 硬件光栅化所有三角形 | 小三角形走软件光栅化（Compute） |
| Pixel Shader即时着色 | Material Classify后Compute Shader批量着色 |

**Unity开发者类比**：想象Unity把`OnBecameVisible()`、LOD Group计算、`Graphics.DrawMesh()`全部替换成一组Compute Shader在GPU上并行执行，CPU只负责调度Dispatch——这就是Nanite的思路。

## 四、代价与限制

Nanite不是银弹，它有明确的代价：

| 代价 | 说明 |
|------|------|
| **仅静态/半静态几何** | 骨骼动画支持有限（5.7有实验性Skinning支持，通过`bSkinning`标志） |
| **不支持半透明** | Visibility Buffer方案天然要求不透明 |
| **GPU负载增加** | Compute Shader占用GPU算力（但总帧时间更短） |
| **额外磁盘占用** | 全LOD层次树数据需要存储 |
| **DX12/Vulkan必需** | 依赖Compute Shader + 64-bit Atomics + Indirect Draw |
| **材质数量敏感** | Material Shading阶段耗时与材质种类数成正比 |

**决策原则**：场景中高多边形静态物体（岩石、建筑、植被）>50%时，Nanite有显著收益；低多边形或大量动态物体场景，传统BasePass可能更合适。

# 第二部分：Nanite架构与核心数据结构

> **核心目标**：掌握Cluster、层次树、Page、Visibility Buffer四大核心概念，C++位运算知识在讲解数据压缩时自然融入

---

## 一、Cluster：渲染管线的最小调度单元

### 1.1 为什么需要Cluster？

传统渲染以整个Mesh为调度单位——一个10万三角形的岩石，要么全部渲染，要么全部剔除。这太粗糙了：如果岩石只有一角露出屏幕，也得渲染全部10万个三角形。

Nanite将Mesh切割成**Cluster**（约128个三角形/Cluster），以Cluster为最小调度单元：

```
传统方式：
┌─────────────────────────────┐
│ 整个Mesh（10万三角形）      │
│ 剔除粒度：全部或不渲染      │
│ 问题：角落露出也渲染10万面  │
└─────────────────────────────┘

Nanite方式：
┌─────┬─────┬─────┬─────┬─────┐
│ C0  │ C1  │ C2  │ C3  │ ... │  780个Cluster
│ 128 │ 128 │ 128 │ 128 │     │  每个128三角形
└─────┴─────┴─────┴─────┴─────┘
  ✅    ❌    ❌    ✅    ...   只渲染可见Cluster
```

### 1.2 Cluster划分：Metis图分割

离线阶段使用Metis图分割算法将Mesh划分为Cluster：

1. **构建图**：三角形为节点，共享边为连接
2. **K-way分割**：目标K = 三角形总数 / 128
3. **优化边界**：最小化跨Cluster边数（空间局部性好 → Cache友好）

**为什么是128三角形？** 平衡粒度与开销——太大则剔除不够精细，太小则Cluster元数据占比过高。

## 二、FPackedCluster与FCluster：CPU压缩 vs GPU解压

Nanite对Cluster数据做了两级表示：
- **`FPackedCluster`**（CPU端，`NaniteResources.h`）：位域紧凑打包，存储在磁盘和内存
- **`FCluster`**（GPU端，`NaniteDataDecode.ush`）：解压后的完整结构，Shader中使用

### 2.1 FPackedCluster：位域打包实战

**来源**：`Engine/Source/Runtime/Engine/Public/Rendering/NaniteResources.h`，搜索 `struct FPackedCluster`

`FPackedCluster`将多个字段打包进单个`uint32`，减少内存占用和带宽：

```cpp
// 真实代码节选（NaniteResources.h，搜索 struct FPackedCluster）
struct FPackedCluster
{
    // ═══ 光栅化所需 ═══
    uint32  NumVerts_PositionOffset;     // NumVerts:14, PositionOffset:18
    uint32  NumTris_IndexOffset;         // NumTris:8, IndexOffset:24
    uint32  ColorMin;
    uint32  ColorBits_GroupIndex;        // R:4, G:4, B:4, A:4, GroupIndex:16

    FIntVector PosStart;
    uint32  BitsPerIndex_PosPrecision_PosBits_NormalPrecision_TangentPrecision;
    //      BitsPerIndex:3+1, PosPrecision:6, PosBits:5.5.5, Normal:4, Tangent:4

    // ═══ 剔除所需 ═══
    FSphere3f LODBounds;                 // xyz=中心, w=半径
    FVector3f BoxBoundsCenter;
    uint32    LODErrorAndEdgeLength;     // 两个float16打包
    FVector3f BoxBoundsExtent;
    uint32    Flags_NumClusterBoneInfluences;  // ClusterFlags:4, BoneInfluences:5

    // ═══ 材质所需 ═══
    uint32  AttributeOffset_BitsPerAttribute;  // Offset:22, Bits:10
    uint32  DecodeInfoOffset_HasTangents_Skinning_NumUVs_ColorMode;
    uint32  UVBitOffsets;
    uint32  PackedMaterialInfo;

    // ═══ 扩展数据 ═══
    uint32  ExtendedDataOffset_Num;
    uint32  BrickDataOffset_Num;
    uint32  Dummy0, Dummy1;
    uint32  VertReuseBatchInfo[4];

    // ═══ 访问器（位提取） ═══
    uint32  GetNumVerts() const   { return GetBits(NumVerts_PositionOffset, 14, 0); }
    uint32  GetNumTris() const    { return GetBits(NumTris_IndexOffset, 8, 0); }
    uint32  GetFlags() const      { return GetBits(Flags_NumClusterBoneInfluences, 4, 0); }
    // ...
};
```

**位域打包原理**（C++知识融入）：

```cpp
// 位打包：多个小字段存入一个uint32
// NumVerts_PositionOffset = NumVerts:14位 | PositionOffset:18位

// 提取NumVerts（低14位）：
uint32 GetNumVerts() const { return NumVerts_PositionOffset & 0x3FFF; }
//                                                           ^^^^^^ 14个1

// 提取PositionOffset（高18位）：
uint32 GetPositionOffset() const { return NumVerts_PositionOffset >> 14; }

// 写入：
void SetNumVerts(uint32 NumVerts) {
    NumVerts_PositionOffset &= ~0x3FFF;         // 清除低14位
    NumVerts_PositionOffset |= NumVerts & 0x3FFF; // 写入低14位
}
```

**与Unity对比**：Unity C#不支持位域语法，只能手动位运算或用`[FieldOffset]`模拟。UE C++通过访问器封装位操作，Shader端通过`ReadBits`/`WriteBits`工具函数解压。

### 3.2 LOD层次树构建与遍历

```
原始Mesh（100万三角形）：

LOD 0（最细节）：7812个Cluster → 叶子节点
    ↑ QEM简化50%
LOD 1：3906个Cluster
    ↑ QEM简化50%
LOD 2：1953个Cluster
    ↑ ...
LOD N（最粗糙）：1个Cluster → 根节点

树结构：
            [Root: LOD N]          ← bLeaf=false
           /              \
      [LOD N-1]        [LOD N-1]
      /      \          /      \
  [LOD 0] [LOD 0]  [LOD 0] [LOD 0]  ← bLeaf=true
```

**LOD选择核心公式**：

```
ScreenSpaceError = LODError × LODScale / Distance

其中：
- LODError: Cluster的世界空间简化误差（米）
- LODScale: 预计算的 ScreenHeight / (2 × tan(FOV/2))
- Distance: Cluster到相机的距离

如果 ScreenSpaceError < 阈值（默认1像素）→ 使用当前LOD（够精细）
如果 ScreenSpaceError >= 阈值 → 展开子节点（需要更精细）
```

**与传统LOD对比**：

| 维度 | 传统LOD | Nanite LOD |
|------|---------|-----------|
| 计算位置 | CPU | GPU |
| 级别 | 固定5-7个 | 连续像素级 |
| 切换方式 | 距离阈值，离散跳变 | 屏幕空间误差，连续平滑 |
| 粒度 | 整个Mesh | 每个Cluster独立 |
| Pop-in | 明显 | 无（误差<1像素时切换） |

## 四、Page-based流式加载

Nanite将Cluster数据组织成固定大小的**Page**（类似虚拟内存的Page机制）：

```
Page结构（~128KB）：
┌────────────────────────────────────────┐
│ Page Header                            │
│  ├─ NumClusters                        │
│  ├─ MaxClusterBoneInfluences           │  ← FPageHeader
│  └─ MaxVoxelBoneInfluences             │
├────────────────────────────────────────┤
│ FPackedCluster[0..N]                   │  ← 压缩的Cluster元数据
├────────────────────────────────────────┤
│ Compressed Vertex Data                 │
│  ├─ Position Data（量化uint16×3）      │
│  ├─ Index Data（自适应位宽）           │
│  ├─ Normal Data（Octahedral uint8×2）  │
│  └─ UV/Color Data（量化）              │
└────────────────────────────────────────┘

寻址方式：
ClusterID → (PageIndex, ClusterIndex)
通过 PackPoolClusterRef / UnpackPoolClusterRef 打包/解包
```

**来源**：`Engine/Shaders/Private/Nanite/NaniteDataDecode.ush`，搜索 `PackPoolClusterRef` / `UnpackPoolClusterRef`

**流式加载策略**：
1. 预测下一帧可见Cluster（基于当前帧结果）
2. 异步加载对应Page到GPU显存
3. LRU淘汰长时间不可见的Page
4. 运行时仅加载~20%的全量数据（节省80%显存）

## 五、Visibility Buffer编码

Nanite光栅化输出的不是传统GBuffer，而是64-bit的**Visibility Buffer**：

```
Visibility Buffer（每像素64位）：
┌─────────────────────────────────────────────┐
│ UlongType (uint2 in shader, uint64 on GPU)  │
│                                              │
│ 编码内容（通过位打包）：                     │
│  - VisibleCluster标识                        │
│  - 深度值                                    │
│                                              │
│ 解码后可获得：                               │
│  - InstanceId → 哪个物体                     │
│  - PageIndex + ClusterIndex → 哪个Cluster    │
│  - TriangleIndex → 哪个三角形                │
│  - Depth → 深度                              │
└─────────────────────────────────────────────┘
```

**为什么不直接写GBuffer？**
- Visibility Buffer只有64-bit/像素（GBuffer通常128-256-bit/像素）
- 光栅化阶段更快（写入数据量小）
- Material Shading阶段只对**最终可见**像素着色 → **零Overdraw**
- 材质可以按MaterialID批量处理 → 减少状态切换

```hlsl
void InstanceCullingCS(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    uint InstanceIndex = DispatchThreadID.x;
    if (InstanceIndex >= TotalInstanceCount) return;

    // 【步骤1】加载Instance数据（从GPUScene）
    FInstanceSceneData InstanceData = GetInstanceSceneData(InstanceIndex);

        InstanceData.LocalToTranslatedWorld,

        InstanceCenter - NaniteView.CullingViewOriginTranslatedWorld,
        InstanceCenter - NaniteView.CullingViewOriginTranslatedWorld

    // 【步骤5】通过所有剔除，原子计数写入可见列表
    uint OutputIndex;
    InterlockedAdd(VisibleInstanceCount[0], 1, OutputIndex);
    VisibleInstanceIndices[OutputIndex] = InstanceIndex;
```

### 2.1 层次遍历：Persistent Thread模型

**来源**：`Engine/Shaders/Private/Nanite/NaniteHierarchyTraversal.ush`

Nanite使用**Persistent Thread**模型遍历层次树——一组线程持续运行，不断从队列中取节点处理，而非为每层启动新的Dispatch：

```hlsl
// 概念流程（基于 NaniteHierarchyTraversal.ush 中的 ProcessNodeBatch 简化）

template<typename FNaniteTraversalCallback>
void ProcessNodeBatch(uint BatchSize, uint GroupIndex, uint QueueStateIndex)
{
    const uint ChildIndex = GroupIndex & NANITE_MAX_BVH_NODE_FANOUT_MASK;

    // 【步骤1】从GroupShared Memory获取当前节点
    FHierarchyNodeSlice NodeSlice = GetHierarchyNodeSlice(
        TraversalCallback.GetHierarchyNodeOffset(), ChildIndex
    );

    bool bVisible = NodeSlice.bEnabled;
    bool bLoaded = NodeSlice.bLoaded;

    // 【步骤2】回调判断是否应该访问此子节点
    // 内部执行：Frustum Culling + LOD Selection
    bVisible = TraversalCallback.ShouldVisitChild(NodeSlice, bVisible);

    // 【步骤3】如果可见且已加载，且不是叶子节点，加入下一轮队列
    if (bVisible && bLoaded && !NodeSlice.bLeaf)
    {
        WaveInterlockedAddScalar_(GroupNumCandidateNodes, 1, CandidateNodesOffset);
    }

    // 【步骤4】如果是叶子节点，输出为可见Cluster
    if (bVisible && bLoaded && NodeSlice.bLeaf)
    {
        // 输出ClusterRef到VisibleCluster列表
        OutputVisibleCluster(NodeSlice);
    }
}
```

```hlsl
// 真实代码节选（NaniteCullingCommon.ush，搜索 TransformNodeCullingBounds）
FNodeCullingBounds TransformNodeCullingBounds(
    FNodeCullingBounds Bounds,
    float4x4 CullingTransform,
    float4x4 LODTransform
)
{
    const float3x3 CullingScaleRot = (float3x3)CullingTransform;
    const float3x3 LODScaleRot = (float3x3)LODTransform;

    // 计算最大缩放（影响LOD误差计算）
    const float3 LODScaleSq = {
        length2(LODScaleRot[0]),
        length2(LODScaleRot[1]),
        length2(LODScaleRot[2])
    };
    const float LODMaxScale = sqrt(max3(LODScaleSq.x, LODScaleSq.y, LODScaleSq.z));

    // 变换AABB
    Bounds.BoxCenter = mul(float4(Bounds.BoxCenter, 1.0f), CullingTransform).xyz;
    Bounds.BoxExtent = mul(Bounds.BoxExtent, abs(CullingScaleRot));

    // 变换LOD包围球（用LODTransform，可能与CullingTransform不同）
    Bounds.Sphere.xyz = mul(float4(Bounds.Sphere.xyz, 1.0f), LODTransform).xyz;
    Bounds.Sphere.w *= LODMaxScale;

    return Bounds;
}
```

### 2.3 HZB遮挡剔除

**来源**：`Engine/Shaders/Private/Nanite/NaniteHZBCull.ush`

HZB（Hierarchical Z-Buffer）使用上一帧的深度Mipmap金字塔判断Cluster是否被遮挡：

```hlsl
// 概念流程（基于 NaniteHZBCull.ush 中的 MipLevelForRect / GetScreenRect 简化）

// 核心函数：选择合适的HZB Mip级别
int MipLevelForRect(int4 RectPixels, int DesiredFootprintPixels)
{
    // 计算覆盖矩形的最小Mip级别
    // 使用firstbithigh代替log2（GCN上FULL rate vs QUARTER rate）
    int2 MipLevelXY = firstbithigh(RectPixels.zw - RectPixels.xy);
    int MipLevel = max(max(MipLevelXY.x, MipLevelXY.y) - MipOffset, 0);

    // 处理对齐量化：可能需要高一级
    MipLevel += any((RectPixels.zw >> MipLevel) - (RectPixels.xy >> MipLevel) > MaxPixelOffset)
                ? 1 : 0;
    return MipLevel;
}

// GetScreenRect：计算Cluster在屏幕上的像素矩形
FScreenRect GetScreenRect(int4 ViewRect, float3 CullRectMin, float3 CullRectMax, int DesiredFootprintPixels)
{
    FScreenRect Rect;
    Rect.Depth = CullRectMax.z;

    // NDC→UV：X[-1,1]→[0,1], Y[-1,1]→[1,0]
    float4 RectUV = saturate(
        float4(CullRectMin.xy, CullRectMax.xy) * float2(0.5, -0.5).xyxy + 0.5
    ).xwzy;

    // UV→像素坐标（使用像素中心而非保守边界，减少~5%无效Cluster）
    float2 ViewSize = ViewRect.zw - ViewRect.xy;
    Rect.Pixels = int4(RectUV * ViewSize.xyxy + ViewRect.xyxy + float4(0.5f, 0.5f, -0.5f, -0.5f));

    // HZB从半分辨率开始
    Rect.HZBTexels = Rect.HZBTexels >> 1;
    Rect.HZBLevel = MipLevelForRect(Rect.HZBTexels, DesiredFootprintPixels);

    return Rect;
}
```

**HZB剔除效果**：
- 密集场景剔除率：30-50%
- 误判率：<5%（使用上一帧数据，有一帧延迟）

```hlsl
    FRasterTri Tri;
    Tri.bIsValid = true;
    Tri.InvW = float3( Verts[0].w, Verts[1].w, Verts[2].w );

    // 16.8固定点数子像素精度
    float2 Vert0 = Verts[0].xy;
    float2 Vert1 = Verts[1].xy;
    float2 Vert2 = Verts[2].xy;

    // 计算边向量
    Tri.Edge01 = Vert0 - Vert1;
    Tri.Edge12 = Vert1 - Vert2;
    Tri.Edge20 = Vert2 - Vert0;

    // 行列式判断正反面
    float DetXY = Tri.Edge01.y * Tri.Edge20.x - Tri.Edge01.x * Tri.Edge20.y;
    Tri.bBackFace = (DetXY >= 0.0f);

    if (bBackFaceCull)
        Tri.bIsValid = !Tri.bBackFace;

    // 如果不剔除背面，则翻转绕序
    if (!bBackFaceCull && Tri.bBackFace)
    {
        Tri.Edge01 *= -1.0f;
        Tri.Edge12 *= -1.0f;
        Tri.Edge20 *= -1.0f;
    }

    // 计算包围矩形
    const float2 MinSubpixel = min3(Vert0, Vert1, Vert2);
    const float2 MaxSubpixel = max3(Vert0, Vert1, Vert2);
    Tri.MinPixel = (int2)floor((MinSubpixel + (SubpixelSamples/2) - 1) * (1.0/SubpixelSamples));
    Tri.MaxPixel = (int2)floor((MaxSubpixel - (SubpixelSamples/2) - 1) * (1.0/SubpixelSamples));

    // Scissor裁剪
    Tri.MinPixel = max(Tri.MinPixel, ScissorRect.xy);
    Tri.MaxPixel = min(Tri.MaxPixel, ScissorRect.zw - 1);

    // 限制光栅化范围为64×64像素（避免极端大三角形占用过长时间）
    Tri.MaxPixel = min(Tri.MaxPixel, Tri.MinPixel + 63);

    // 无像素覆盖则无效
    if (any(Tri.MinPixel > Tri.MaxPixel))
        Tri.bIsValid = false;

    return Tri;
}
```

### 3.3 为什么需要软件光栅化？

```
三角形大小      硬件效率    软件效率    原因
───────────    ────────    ────────   ─────────────────────
> 16像素        ~95%        ~60%      硬件Quad利用率高
4-16像素        ~70%        ~85%      软件避免Quad浪费
1-4像素         ~30%        ~90%      软件1线程=1像素，无浪费
< 1像素         ~10%        ~90%      软件可精确处理亚像素

Nanite场景中~60%三角形<4像素 → 硬件光栅化大量浪费
混合策略：大三角形→硬件，小三角形→软件Compute Shader
```

## 四、Material Classify与Shading

### 4.1 为什么需要Material Classify？

传统Forward Rendering对每个三角形立即着色，导致：
1. **Overdraw**：被遮挡的像素也计算了完整材质
2. **状态切换频繁**：每个DrawCall可能切换不同材质

Nanite的方案：先光栅化产生Visibility Buffer，然后按MaterialID将像素分组，最后批量着色。

```
传统Forward：
  Draw岩石A（材质1）→ Draw树木B（材质2）→ Draw岩石C（材质1）→ ...
  材质切换：10,000次
  Overdraw：200万多余像素

Nanite延迟着色：
  阶段4：Visibility Buffer → 按MaterialID分组像素
  阶段5：材质1的所有像素 → 一次Dispatch着色
         材质2的所有像素 → 一次Dispatch着色
         ...
  材质切换：50次（材质种类数）
  Overdraw：0（只着色最终可见像素）
```

### 4.2 Material Classify流程

```hlsl
// 概念流程（基于Nanite实际架构简化）

[numthreads(8, 8, 1)]
void MaterialClassifyCS(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    uint2 PixelPos = DispatchThreadID.xy;
    if (any(PixelPos >= View.ViewSizeAndInvSize.xy)) return;

    // 【步骤1】读取Visibility Buffer
    uint2 VisData = VisibilityBuffer[PixelPos];
    if (VisData.x == 0 && VisData.y == 0) return;  // 背景像素

    // 【步骤2】解码获取Cluster和三角形索引
    FVisibleCluster VisCluster = UnpackVisibleCluster(VisData);
    FCluster Cluster = DecodeCluster(VisCluster.PageIndex, VisCluster.ClusterIndex);

    // 【步骤3】获取材质ID（快路径 vs 慢路径）
    uint MaterialID;
    if (Cluster.MaterialTableLength == 0)
    {
        // 快路径：1-3种材质直接查
        if (TriIdx < Cluster.Material0Length)
            MaterialID = Cluster.Material0Index;
        else if (TriIdx < Cluster.Material0Length + Cluster.Material1Length)
            MaterialID = Cluster.Material1Index;
        else
            MaterialID = Cluster.Material2Index;
    }
    else
    {
        // 慢路径：查材质表
        MaterialID = ClusterPageData.Load(Cluster.MaterialTableOffset + TriIdx);
    }

    // 【步骤4】原子写入按材质分组的像素列表
    uint OutputIndex;
    InterlockedAdd(MaterialPixelCounts[MaterialID], 1, OutputIndex);
    MaterialPixelLists[MaterialID][OutputIndex] = EncodePixelPos(PixelPos);
}
```

### 4.3 Material Shading流程

```hlsl
// 概念流程

[numthreads(64, 1, 1)]
void MaterialShadingCS(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    uint PixelIndex = DispatchThreadID.x;
    if (PixelIndex >= MaterialPixelCounts[MaterialID]) return;

    // 【步骤1】解码像素位置
    uint2 PixelPos = DecodePixelPos(MaterialPixelLists[MaterialID][PixelIndex]);

    // 【步骤2】从Visibility Buffer重建顶点属性
    uint2 VisData = VisibilityBuffer[PixelPos];
    FVisibleCluster VisCluster = UnpackVisibleCluster(VisData);
    FCluster Cluster = DecodeCluster(VisCluster.PageIndex, VisCluster.ClusterIndex);

    // 解码三角形三个顶点
    uint3 Indices = DecodeTriangleIndices(Cluster, TriangleIndex);
    float3 V0 = UnpackPosition(Cluster, Indices.x);
    float3 V1 = UnpackPosition(Cluster, Indices.y);
    float3 V2 = UnpackPosition(Cluster, Indices.z);

    // 【步骤3】计算重心坐标，插值属性
    float3 Bary = ComputeBarycentric(PixelPos, V0, V1, V2);
    float3 WorldNormal = normalize(Bary.x*N0 + Bary.y*N1 + Bary.z*N2);
    float2 UV = Bary.x*UV0 + Bary.y*UV1 + Bary.z*UV2;

    // 【步骤4】采样材质纹理，计算GBuffer
    float4 BaseColor = BaseColorTexture.SampleLevel(Sampler, UV, 0);
    float Metallic = MetallicTexture.SampleLevel(Sampler, UV, 0).r;
    float Roughness = RoughnessTexture.SampleLevel(Sampler, UV, 0).r;

    // 【步骤5】写入GBuffer
    GBufferA[PixelPos] = EncodeGBufferA(WorldNormal);
    GBufferB[PixelPos] = EncodeGBufferB(BaseColor, Metallic);
    // ...
}
```

## 五、顶点解码全流程

### 5.1 固定点数量化原理

Nanite将`float3`位置压缩为`uint16×3`，节省50%带宽：

```
编码（离线，编辑器端）：
  Normalized = (Position - PosStart) / PosScale
  Quantized = uint16(Normalized * 65535)

解码（运行时，GPU端）：
  Normalized = float(Quantized) / 65535.0
  Position = Normalized * PosScale + float3(PosStart)
```

**精度分析**（Cluster包围盒1米×1米×1米时）：
- uint16精度：1.0 / 65535 ≈ 0.015mm
- 人眼在正常距离（>0.5m）无法察觉
- Cluster越大精度越低，但LOD保证远处Cluster简化后不需要高精度

**FCluster中的量化参数**：
- `PosStart`：包围盒最小点（解码偏移基准）
- `PosScale`：缩放因子
- `PosRcpScale`：缩放因子倒数（编码时用，避免除法）
- `PosBits`：XYZ各轴量化位数（支持非均匀量化）

### 5.2 Octahedral法线编码

Nanite使用Octahedral编码将`float3`法线压缩为`uint8×2`（12字节→2字节，83%压缩）：

```hlsl
// 解码流程（概念）
float3 DecodeOctahedralNormal(uint8 NormalX, uint8 NormalY)
{
    // [0,255] → [-1,1]
    float2 Oct;
    Oct.x = (float(NormalX) / 255.0) * 2.0 - 1.0;
    Oct.y = (float(NormalY) / 255.0) * 2.0 - 1.0;

    // 重建Z分量
    float Z = 1.0 - abs(Oct.x) - abs(Oct.y);

    // 下半球翻转
    if (Z < 0)
    {
        float2 Sign = select(Oct >= 0, 1.0, -1.0);
        Oct = (1.0 - abs(Oct.yx)) * Sign;
    }

    return normalize(float3(Oct.x, Oct.y, Z));
}
```

### 5.3 自适应索引压缩

三角形索引根据Cluster内顶点数量选择最小位宽：

| 顶点数 | 索引位宽 | 每三角形大小 | vs uint32基准 |
|--------|---------|------------|--------------|
| ≤128 | 7-8 bit | ~3字节 | 节省75% |
| ≤256 | 8 bit | 3字节 | 节省75% |
| ≤65536 | 16 bit | 6字节 | 节省50% |

`FPackedCluster`中的`BitsPerIndex`字段：存储为3位（值+1），支持1-8 bit索引。
