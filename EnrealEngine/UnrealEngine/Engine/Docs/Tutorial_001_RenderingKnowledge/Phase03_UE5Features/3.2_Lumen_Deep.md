## 🎯 Unity对比：全局光照方案差异

| 维度 | UE5.7 Lumen | Unity 常见 Lightmap / Probe 路径 | 差异说明 |
|------|-------------|----------------------|----------|
| **GI类型** | 完全动态（实时计算） | 静态烘焙（Lightmap） | UE无需烘焙，Unity需要分钟-小时烘焙时间 |
| **动态物体GI** | 完全支持 | Light Probe（有限） | UE动态物体也有完整GI，Unity仅球谐近似 |
| **追踪方式** | Software RT + Hardware RT | SSGI（屏幕空间，实验性） | UE用Distance Field追踪，不限屏幕内 |
| **反射** | 动态（Surface Cache + HiZ） | Reflection Probe（静态） | UE反射实时更新，Unity需手动烘焙 |
| **内存开销** | ~280MB（Surface Cache Atlas） | ~500MB+（Lightmap纹理） | 各有优劣，Lumen更可控 |
| **性能开销** | ~10-15ms/帧 | ~0ms运行时（烘焙后） | Lumen有运行时开销，烘焙方案几乎免费 |
| **适用场景** | 动态场景、开放世界 | 静态场景、室内 | 各有侧重 |

**关键差异**：Unity依赖CPU端Lightmap烘焙+Light Probe插值，Lumen将整个GI计算搬到GPU实时执行，用Mesh Cards替代Lightmap UV、用Screen Probe替代Light Probe、用Software Ray Tracing替代预计算光线传输，从根本上消除了烘焙流程。

# 第一部分：理解Lumen的存在意义

> **核心目标**：从传统GI方案的实际局限出发，理解Lumen解决了什么问题、付出了什么代价

---

## 一、传统GI方案的四大局限

### 1.1 局限一：静态烘焙无法支持动态场景

假设你正在开发一个开放世界游戏，场景中有动态昼夜循环和可破坏建筑。

**Unity URP的做法**：

```csharp
// Unity URP：烘焙Lightmap
void SetupGI()
{
    // 1. 在编辑器中烘焙Lightmap（耗时5-30分钟）
    Lightmapping.BakeAsync();

    // 2. 运行时只读取预烘焙数据
    lightmapData = LightmapSettings.lightmaps;
    // 问题：太阳移动了 → Lightmap过时
    // 问题：建筑被破坏了 → Lightmap仍显示原来的阴影
    // 问题：新放置了物体 → 没有GI
}
```

**问题本质**：Lightmap是离线预计算的，运行时任何场景变化都无法反映在GI中。

| 场景变化 | Lightmap（Unity） | Lumen（UE5） |
|---------|-------------------|-------------|
| 太阳角度变化 | ❌ GI不更新 | ✅ 实时响应 |
| 物体移动 | ❌ 残留旧阴影 | ✅ 自动更新 |
| 物体破坏 | ❌ 阴影不消失 | ✅ 即时反映 |
| 新增物体 | ❌ 无GI | ✅ 自动加入 |

### 1.2 局限二：Light Probe精度不足

Unity用Light Probe为动态物体提供间接光照，但精度严重不足：

```
Light Probe方案：
  ● ── ● ── ● ── ●    ← 美术手动放置Probe（间距1-5米）
  │    │    │    │
  ● ── ● ── ● ── ●    ← 球谐函数（SH L2）只有9个系数
  │    │    │    │
  ● ── ● ── ● ── ●    ← 无法表达高频光照变化（阴影边界模糊）

问题1：精度差 — SH L2只能表达低频光照，锐利阴影变为模糊过渡
问题2：漏光严重 — Probe间距大于墙壁厚度时，光线"穿墙"
问题3：工作量大 — 需要美术手动放置和调整数百个Probe
```

### 1.3 局限三：SSGI受限于屏幕空间

Screen-Space Global Illumination（SSGI）只能利用屏幕可见的信息：

```
SSGI的局限：
┌──────────────────────────────┐
│ 屏幕可见区域                  │
│                              │
│  玩家视角 →                  │
│                              │
│         ┌──────┐             │
│         │ 墙壁 │             │  ← 屏幕内：SSGI能计算GI
│         │      │             │
│         └──────┘             │
│                              │
└──────────────────────────────┘
        ↑ 屏幕外区域
        墙壁的另一面、天花板、屋顶 → SSGI无法获取信息
        转动视角 → GI突然出现/消失（闪烁）
```

### 1.4 局限四：传统方案的美术工作流瓶颈

| 传统GI工作流 | 时间开销 | 迭代问题 |
|-------------|---------|---------|
| UV展开（Lightmap UV2） | 每个模型1-5分钟 | 修改模型需重新展开 |
| Lightmap烘焙 | 5分钟-数小时 | 每次修改场景需重新烘焙 |
| Light Probe放置 | 数小时手动调整 | 场景修改需重新调整 |
| Reflection Probe放置 | 数小时手动调整 | 场景修改需重新烘焙 |

**四大局限总结**：

| 局限 | 根因 | 影响 |
|------|------|------|
| **静态烘焙** | 离线预计算，运行时无法更新 | 动态场景GI失真 |
| **Probe精度不足** | 球谐函数+稀疏采样 | 漏光、模糊、手动放置 |
| **SSGI屏幕限制** | 仅利用屏幕可见信息 | 转头闪烁、缺失屏幕外GI |
| **美术流程重** | UV展开+烘焙+Probe放置 | 迭代慢，修改代价高 |

## 二、Lumen的解决思路：三层动态GI架构

Lumen的核心设计哲学：**用运行时追踪替代离线烘焙，用自动化替代手动放置**。

| 传统方案 | Lumen方案 | 设计意图 |
|---------|-----------|---------|
| Lightmap UV → 烘焙纹理 | Mesh Cards → Surface Cache Atlas | 自动6朝向卡片，无需UV展开 |
| Light Probe → SH插值 | Screen Probe → 射线追踪+时间滤波 | 每帧自动追踪，无需手动放置 |
| Reflection Probe → 静态Cubemap | Lumen Reflections → HiZ+Surface Cache追踪 | 实时动态反射 |
| 预计算光线传输 | Software RT（Distance Field步进） | GPU实时追踪 |

**Unity开发者类比**：想象Unity把Lightmap替换为实时渲染的6面体"截图"（Surface Cache），把Light Probe替换为屏幕空间自动放置的追踪探针（Screen Probe），把Reflection Probe替换为每帧实时追踪的反射——这就是Lumen的思路。

## 四、代价与限制

Lumen不是银弹，它有明确的运行时代价：

| 代价 | 说明 |
|------|------|
| **GPU性能开销** | ~10-15ms/帧（1080p），占帧预算的50-80% |
| **内存占用** | ~280MB（Surface Cache Atlas + Radiance Cache + 辅助Buffer） |
| **延迟** | Surface Cache更新有1-2帧延迟（异步更新） |
| **薄物体漏光** | Distance Field精度限制导致薄墙可能漏光 |
| **不支持半透明GI** | Surface Cache方案天然要求不透明表面 |
| **DX12/Vulkan必需** | 依赖Compute Shader + 64-bit Atomics |

**决策原则**：场景需要动态光照变化（昼夜循环、可破坏环境、开放世界）时，Lumen有显著优势；纯静态室内场景且GPU预算紧张时，传统Lightmap可能更合适。

## 一、Mesh Cards：场景表面的卡片表示

### 1.1 为什么需要Mesh Cards？

传统Lightmap需要为每个模型手动展开Lightmap UV——这对美术来说工作量巨大，而且UV接缝处会产生光照不连续。

Lumen的替代方案：**将每个物体的表面自动投影为6个轴对齐方向的2D卡片（Cards）**，类似Cubemap的6个面。

```
传统Lightmap：
  需要手动UV展开，美术工作量大
  ┌──────────────┐
  │  UV空间       │  ← 每个模型都需要第二套UV
  │  (手动展开)   │  ← 接缝处光照不连续
  └──────────────┘

Lumen Mesh Cards（自动生成）：
         +Z (Top)
          │
    ──────┼──────  +X (Right)
         ╱│
        ╱ │
      +Y  │
   (Front)

  6个Card：
  Card 0：+X（右侧）  Card 1：-X（左侧）
  Card 2：+Y（前侧）  Card 3：-Y（后侧）
  Card 4：+Z（顶部）  Card 5：-Z（底部）

  每个Card是一个轴对齐的2D投影
  自动生成，无需美术干预
```

### 1.2 FLumenCard结构体

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneData.h`，token：`class FLumenCard`

```cpp
// 真实代码锚点：Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneData.h + class FLumenCard
class FLumenCard
{
public:
    FLumenCard();
    ~FLumenCard();

    // Card的三种空间表示（OBB = Oriented Bounding Box）
    FLumenCardOBBf LocalOBB;      // 局部空间OBB
    FLumenCardOBBd WorldOBB;      // 世界空间OBB（双精度，支持LWC大世界坐标）
    FLumenCardOBBf MeshCardsOBB;  // MeshCards空间OBB

    bool bVisible = false;
    bool bHeightfield = false;    // 是否为地形Card
    bool bAxisXFlipped = false;   // X轴翻转标志（Card共享时使用）

    // Mip级别范围：Surface Cache按需分配不同分辨率
    uint8 MinAllocatedResLevel = UINT8_MAX;  // 最低已分配Mip
    uint8 MaxAllocatedResLevel = 0;          // 最高已分配Mip
    uint8 DesiredLockedResLevel = 0;         // 基于距离计算的期望Mip

    // 每个Mip级别一个Surface Cache分配，索引为 [ResLevel - Lumen::MinResLevel]
    FLumenSurfaceMipMap SurfaceMipMaps[Lumen::NumResLevels];

    int32 MeshCardsIndex = -1;               // 所属MeshCards索引
    int32 IndexInMeshCards = -1;             // 在MeshCards中的索引
    uint8 AxisAlignedDirectionIndex = UINT8_MAX;  // 朝向（0-5对应±X/±Y/±Z）
    float ResolutionScale = 1.0f;            // 分辨率缩放

    FLumenCardId CardSharingId;              // Card共享ID（复用相同几何的Card）
};
```

**关键设计点**：

1. **三种OBB**：LocalOBB（模型空间）、WorldOBB（世界空间，双精度LWC）、MeshCardsOBB（所属MeshCards空间）——三重表示确保精度和变换正确性
2. **MipMap层级**：每个Card有多个分辨率级别（2^3=8 到 2^11=2048），基于摄像机距离动态分配
3. **Card共享**：相同几何体的不同Instance可以共享Card数据（`CardSharingId`），节省Atlas空间

### 1.3 FLumenCardId：位域打包的唯一标识

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneData.h`，token：`union FLumenCardId`

```cpp
// 真实代码锚点：Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneData.h + union FLumenCardId
union FLumenCardId
{
    static constexpr uint64 InvalidPackedValue = -1;

    uint64 PackedValue;    // 整体作为uint64使用
    struct
    {
        uint32 ResLevelBiasX : 4;              // X方向分辨率偏移（4位）
        uint32 ResLevelBiasY : 4;              // Y方向分辨率偏移（4位）
        uint32 AxisAlignedDirectionIndex : 3;  // 朝向索引0-5（3位）
        uint32 Unused : 21;                    // 保留位
        uint32 CustomId;                       // 自定义ID（32位）
    };
};
```

**位域打包原理**（C++知识融入）：

UE大量使用C++ union + 位域来紧凑存储数据。`FLumenCardId`将6个字段打包进一个`uint64`：
- 低32位：ResLevelBiasX(4) + ResLevelBiasY(4) + AxisAlignedDirectionIndex(3) + Unused(21)
- 高32位：CustomId(32)

这种打包方式让Card ID可以作为单个`uint64`进行高效比较和哈希，同时保留了各字段的语义。

**关于位域**：详见[2.3 GBuffer](../Phase02_Pipeline/2.3_GBuffer_Deep.md)中的首次详细讲解。

## 二、Surface Cache：虚拟纹理Atlas

### 2.1 Surface Cache的设计目标

Surface Cache的核心思想：**为场景中每个Card维护一套低分辨率的表面属性纹理（Albedo、Normal、Depth、Emissive、Opacity），存储在一个共享Atlas中**。

这些纹理在运行时"捕获"——类似于从每个Card方向对物体拍摄一张照片，记录表面材质信息。当射线命中Surface Cache时，可以直接查询命中点的表面属性，而无需执行完整的材质着色。

```
Surface Cache Atlas（4096×4096，多张纹理）：
┌─────────────────────────────────────────┐
│  Albedo Atlas   │  Normal Atlas         │
│  (RGBA8)        │  (RG16F)              │
│                 │                       │
│ ┌───┬───┬───┐  │ ┌───┬───┬───┐        │
│ │C1 │C2 │C3 │  │ │C1 │C2 │C3 │        │
│ ├───┼───┼───┤  │ ├───┼───┼───┤        │
│ │C4 │C5 │C6 │  │ │C4 │C5 │C6 │        │
│ └───┴───┴───┘  │ └───┴───┴───┘        │
├─────────────────┼───────────────────────┤
│  Depth Atlas    │  Emissive Atlas       │
│  (R16F)         │  (RGBA16F)            │
│                 │                       │
│  Opacity Atlas  │  DirectLighting Atlas │
│  (R8)           │  (RGBA16F)            │
└─────────────────┴───────────────────────┘

每个Card (C1, C2...) 占据Atlas中一块区域
分辨率根据距离动态调整（8×8 到 2048×2048）
```

## 三、Global Distance Field：追踪加速结构

### 3.1 为什么需要Distance Field？

Software Ray Tracing需要沿射线方向步进，寻找与场景表面的交点。朴素做法是固定步长步进——但步长太大会跳过薄物体，步长太小则性能无法接受。

**Signed Distance Field（SDF）的加速原理**：SDF在空间中每个点存储到最近表面的距离。追踪时，以当前SDF值作为步长——离表面远时大步前进，离表面近时小步逼近：

```
Sphere Tracing（Distance Field加速步进）：

起点 O ─────●─────────────●───────●──●─● 命中
            ↑             ↑       ↑  ↑ ↑
         SDF=8m        SDF=5m  SDF=2m  SDF<阈值
         步长8m        步长5m  步长2m  → 命中表面

vs 固定步长：

起点 O ─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─● 命中
         步长1m × 16次 = 远慢于SDF步进

性能对比（估算）：
  固定步长（1m）：平均32次步进/射线
  SDF步进：平均5-8次步进/射线 → 提升4-6倍
```

### 3.2 Global Distance Field Clipmap

UE5使用多层级Clipmap结构覆盖不同距离范围：

```
以相机为中心的Clipmap层级：

Clipmap 0（近场）：覆盖 ~50m，分辨率最高
  └─ 用于近距离精确追踪
Clipmap 1（中场）：覆盖 ~100m
  └─ 用于中等距离追踪
Clipmap 2（远场）：覆盖 ~200m
  └─ 用于远距离追踪
Clipmap 3（极远场）：覆盖 ~400m
  └─ 用于极远距离追踪（Radiance Cache）

每层为 252³ 的3D纹理（BC4压缩格式）
相机移动时，Clipmap滚动更新（增量更新边缘区域）
```

**与Unity对比**：Unity没有全局SDF系统。Unity HDRP的SSGI仅在屏幕空间工作，无法追踪屏幕外物体。Lumen的Global Distance Field覆盖整个场景，支持任意方向的射线追踪。

## 四、Screen Probe与Radiance Cache：多尺度GI采样

### 4.1 Screen Probe：屏幕空间高频GI

**问题**：对每个像素追踪间接光照，性能开销巨大（1080p = 200万像素，每像素32条射线 = 6400万射线/帧，~150ms）。

**解决方案**：以较低分辨率在屏幕空间放置Probe（每16×16或8×8像素一个），每个Probe追踪少量射线（16-32条），然后插值到像素。

```
Screen Probe放置（1080p，16×16间距）：

┌────────────────────────────────────────┐
│ ●     ●     ●     ●     ●     ●       │  ← 每16×16像素一个Probe
│                                        │
│ ●     ●     ●     ●     ●     ●       │  约 120×68 = 8160个Probe
│                                        │
│ ●     ●     ●     ●     ●     ●       │  每个Probe追踪32条射线
│                                        │
│ ●     ●     ●     ●     ●     ●       │  总射线：~26万条（vs 6400万条）
└────────────────────────────────────────┘

性能对比（估算，1080p）：

| 策略 | Probe数 | 射线/Probe | 总射线 | 估算耗时 |
|------|---------|-----------|--------|---------|
| 每像素追踪 | 2,073,600 | 32 | 66.3M | ~150ms |
| Screen Probe 16×16 | ~8,100 | 32 | ~260K | ~4ms |
| Screen Probe 8×8 | ~32,400 | 32 | ~1M | ~2ms |
```

### 4.2 Radiance Cache：3D空间远场GI

Screen Probe只覆盖当前帧的屏幕空间。对于远场GI（射线追踪命中远处表面后需要二次反弹），Lumen使用3D空间的Radiance Cache：

```
Radiance Cache Probe Grid（以相机为中心）：

3D Grid：通常 32×32×32 或更高分辨率
每个Probe存储周围的辐射度（SH或Octahedral Map）

┌─────────────────────────────┐
│  ●    ●    ●    ●    ●      │  ← Probe层（Z=16）
│                             │
│  ●    ●    ●    ●    ●      │  总共 ~32K个Probe
│                             │
│  ●    ●    ●    ●    ●      │  每帧增量更新64-128个
│                             │
│  ●    ●    ●    ●    ●      │  ~256帧完全刷新一次
└─────────────────────────────┘

查询方式：三线性插值
给定世界坐标 → 找到周围8个Probe → 插值
```

### 4.3 两者协同

```
Screen Probe Trace 射线追踪结果：

射线从屏幕像素出发
    ├─ 命中 Surface Cache（近场） → 直接采样 Card Atlas → 获取辐射度
    │
    └─ 未命中 Surface Cache → 查询 Radiance Cache（远场） → 插值获取辐射度
                                  ↑
                            3D空间预计算，覆盖远场
```

## 五、渲染流程时序图

**Lumen在每一帧的执行顺序**（RenderGraph依赖图）：

```
帧开始（Frame N）
  ↓
【Pass 1】Lumen Scene Update（RenderThread）
  ├─ 检测Dirty Primitives（移动、材质变化、新增/删除）
  ├─ 更新Mesh Cards（AddMeshCards/RemoveMeshCards）
  ├─ 分配/释放Atlas空间（SurfaceCacheAllocator）
  └─ 上传GPU数据（ScatterUpload）
  估算耗时：~0.5ms
  ↓
【Pass 2】Surface Cache Capture（Async Compute）
  ├─ 渲染Card Capture（从6个方向渲染场景到Card纹理）
  ├─ 更新Direct Lighting（光源直接照射Card表面）
  └─ Radiosity（Card表面间的光线传递）
  估算耗时：~2.5ms（异步，与其他Pass并行）
  ↓
【Pass 3】Global Distance Field Update（Compute）
  ├─ 合并Object Distance Field到全局
  └─ 更新Clipmap层级（增量更新）
  估算耗时：~1.0ms
  ↓
【Pass 4】Radiance Cache Update（Async Compute）
  ├─ Probe放置（根据相机位置调整Grid）
  ├─ Probe追踪（追踪64-128个Probe的射线）
  └─ 插值权重计算
  估算耗时：~1.5ms（异步）
  ↓
【Pass 5】Screen Probe Placement（Compute）
  ├─ 根据GBuffer深度/法线放置Probe Grid
  └─ 计算自适应Probe密度
  估算耗时：~0.3ms
  ↓
【Pass 6】Screen Probe Trace（Compute）
  ├─ 每个Probe追踪16-32条射线
  ├─ 采样Surface Cache（Software RT / Hardware RT）
  ├─ 采样Radiance Cache（远场补充）
  └─ Importance Sampling（根据上一帧结果自适应）
  估算耗时：~3.2ms
  ↓
【Pass 7】Screen Probe Filter（Compute）
  ├─ Temporal Filter（混合历史帧降噪）
  └─ Spatial Filter（双边滤波平滑）
  估算耗时：~0.8ms
  ↓
【Pass 8】Final Gather（Compute）
  ├─ 插值Screen Probe结果到每个像素
  └─ 合成Diffuse Indirect
  估算耗时：~1.5ms
  ↓
【Pass 9】Lumen Reflections（Compute）
  ├─ HiZ Screen Tracing（屏幕空间追踪，优先）
  ├─ Surface Cache Tracing（屏幕外追踪，兜底）
  └─ Hit Lighting + Denoising
  估算耗时：~3.0ms
  ↓
输出：Diffuse Indirect + Specular Indirect → 合成到场景光照
```

### 1.3 UpdateLumenScene核心流程

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneRendering.cpp`，token：`FDeferredShadingSceneRenderer::UpdateLumenScene`

```cpp
// 概念流程（基于 Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneRendering.cpp + FDeferredShadingSceneRenderer::UpdateLumenScene 实际逻辑简化）
void FDeferredShadingSceneRenderer::UpdateLumenScene(
    FRDGBuilder& GraphBuilder,
    FLumenSceneFrameTemporaries& FrameTemporaries)
{
    FLumenSceneData& LumenScene = *Scene->GetLumenSceneData();

    // 【步骤1】处理待删除的Primitives
    for (FLumenPrimitiveGroupRemoveInfo& RemoveInfo : LumenScene.PendingRemoveOperations)
    {
        LumenScene.RemoveMeshCards(RemoveInfo.PrimitiveGroupIndex);
    }

    // 【步骤2】处理待添加的Primitives
    for (FPrimitiveSceneInfo* Primitive : LumenScene.PendingAddOperations)
    {
        int32 GroupIndex = FindOrCreatePrimitiveGroup(Primitive);
        LumenScene.AddMeshCards(GroupIndex);
    }

    // 【步骤3】处理待更新的Primitives（变换变化、材质变化）
    for (FPrimitiveSceneInfo* Primitive : LumenScene.PendingUpdateOperations)
    {
        LumenScene.UpdateMeshCards(Primitive->GetLocalToWorld(), ...);
        LumenScene.InvalidateSurfaceCache(...);
    }

    // 【步骤4】基于距离计算每个Card的期望分辨率
    UpdateSurfaceCacheResolution(LumenScene, Views);

    // 【步骤5】分配/重分配Surface Cache Atlas空间
    LumenScene.AllocateCardAtlases(GraphBuilder, FrameTemporaries, ViewFamily);

    // 【步骤6】通过ScatterUpload上传变化的数据到GPU
    LumenScene.UploadPageTable(GraphBuilder, ...);
    // CardBuffer、MeshCardsBuffer等的增量上传

    // 【步骤7】调度Surface Cache Capture Pass（异步）
    RenderLumenSceneLighting(GraphBuilder, FrameTemporaries, ...);
}
```

### 💡 核心要点

UpdateLumenScene是Lumen每帧的入口，采用**增量更新**策略——只处理变化的Primitives，只上传变化的数据。这是支撑大场景（10万+物体）的关键。

## 二、Surface Cache分配与Card Capture

### 2.1 Surface Cache分配流程

当一个新物体加入场景时，Lumen需要为其6个Card分配Atlas空间：

```
新物体加入场景
    ↓
AddMeshCards()
    ├─ 创建FLumenMeshCards（6个Card的容器）
    ├─ 为每个Card创建FLumenCard
    └─ 初始化Card的OBB、朝向、分辨率
    ↓
基于距离计算分辨率
    ├─ 近处（<10m）：高分辨率（512-2048）
    ├─ 中距（10-50m）：中分辨率（64-512）
    └─ 远处（>50m）：低分辨率（8-64）
    ↓
ReallocVirtualSurface()
    ├─ 计算需要的PageTable条目数量
    ├─ 在PageTable中分配虚拟页
    └─ 调用SurfaceCacheAllocator.Allocate()分配物理页
    ↓
标记Card为Dirty → 需要Card Capture
```

### 2.2 Card Capture：渲染Card表面

Card Capture是一个特殊的渲染Pass，从每个Card的方向渲染场景到Atlas：

```
Card Capture流程（类似从6个方向"拍照片"）：

Card方向 +X：从右侧看物体
    ├─ 设置正交投影（覆盖Card的OBB范围）
    ├─ 渲染场景到临时RenderTarget
    └─ 写入Atlas的对应区域
        ├─ AlbedoAtlas[CardRect] = 表面颜色
        ├─ NormalAtlas[CardRect] = 表面法线
        ├─ DepthAtlas[CardRect] = 深度
        └─ EmissiveAtlas[CardRect] = 自发光

关键文件：
  LumenSceneCardCapture.cpp  — Card Capture调度
  LumenCardComputeShader.usf — Card渲染Compute Shader
  LumenCardPixelShader.usf   — Card渲染Pixel Shader
```

### 2.3 Surface Cache Direct Lighting

Card表面的直接光照由`RenderLumenSceneLighting()`计算：

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneLighting.cpp`，token：`FDeferredShadingSceneRenderer::RenderLumenSceneLighting`

```cpp
// 真实代码签名锚点：Engine/Source/Runtime/Renderer/Private/Lumen/LumenSceneLighting.cpp + FDeferredShadingSceneRenderer::RenderLumenSceneLighting
void FDeferredShadingSceneRenderer::RenderLumenSceneLighting(
    FRDGBuilder& GraphBuilder,
    const FLumenSceneFrameTemporaries& FrameTemporaries,
    const FLumenDirectLightingTaskData* DirectLightingTaskData)
```

此函数为Surface Cache中每个Card计算直接光照（太阳光、点光源等直接照射），结果写入`DirectLightingAtlas`。后续追踪命中Surface Cache时，可以直接采样`DirectLightingAtlas`获取直接光照贡献。

### 2.4 Radiosity：Card间光线传递

在Surface Cache内部，Lumen还会计算Card之间的光线传递（Radiosity），实现一次反弹GI：

```
Radiosity流程：
  Card A的DirectLighting → 作为"光源"照射到Card B
  Card B接收到间接光照 → 写入IndirectLightingAtlas
  最终：FinalLightingAtlas = DirectLightingAtlas + IndirectLightingAtlas

关键文件：
  LumenRadiosity.cpp          — Radiosity调度
  LumenRadiosity.usf          — Radiosity追踪Shader
  LumenRadiosityCulling.usf   — Radiosity射线剔除
```

### 💡 核心要点

Surface Cache是Lumen的"场景材质数据库"——将完整的材质着色结果缓存在Atlas纹理中，让后续追踪可以通过简单的纹理采样获取表面属性，而非每次命中都执行完整的材质Shader。

### 3.2 Screen Probe Trace

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenScreenProbeTracing.cpp`  
**Shader**：`Engine/Shaders/Private/Lumen/LumenScreenProbeTracing.usf`

每个Probe沿法线半球追踪16-32条射线，使用Halton低差异序列保证均匀采样：

```hlsl
// 概念流程（基于LumenScreenProbeTracing.usf实际逻辑简化）

[numthreads(PROBE_THREADGROUP_SIZE_X, PROBE_THREADGROUP_SIZE_Y, 1)]
void ScreenProbeTraceCS(uint3 DispatchThreadId : SV_DispatchThreadID)
{
    uint2 ProbeCoord = DispatchThreadId.xy;

    // 【步骤1】获取Probe的世界位置和法线
    float3 ProbeWorldPos = GetProbeWorldPosition(ProbeCoord);
    float3 ProbeNormal = GetProbeNormal(ProbeCoord);

    // 【步骤2】为每条射线生成方向（Halton序列）
    float3 AccumulatedRadiance = 0;
    for (int RayIndex = 0; RayIndex < NumRaysPerProbe; ++RayIndex)
    {
        // 使用Halton序列生成均匀分布的半球方向
        float2 HaltonSample = GetHalton2D(ProbeIndex * NumRaysPerProbe + RayIndex);
        float3 RayDirection = HemisphereSampleFromHalton(HaltonSample, ProbeNormal);

        // 【步骤3】追踪射线
        float HitDistance;
        float3 HitRadiance;
        bool bHit = TraceLumenRay(ProbeWorldPos, RayDirection, HitDistance, HitRadiance);

        if (bHit)
        {
            AccumulatedRadiance += HitRadiance;
        }
        else
        {
            // 未命中Surface Cache → 查询Radiance Cache（远场）
            AccumulatedRadiance += SampleRadianceCache(ProbeWorldPos + RayDirection * FarFieldDistance);
        }
    }

    // 【步骤4】存储Probe结果
    StoreProbeRadiance(ProbeCoord, AccumulatedRadiance / NumRaysPerProbe);
}
```

**关于Halton低差异序列**：

```
为什么不用随机采样？

随机采样（伪随机数）：
  · · ·· ··· ·  ·  · ··· ·    ← 有聚簇，覆盖不均匀
  · ·· ·  · ·  · ·  ·· · ·
  ··· ·  · · · ·  · ·· · ·

Halton序列（低差异序列）：
  · · · · · · · · · · · · ·    ← 均匀分布，无聚簇
  · · · · · · · · · · · · ·
  · · · · · · · · · · · · ·

结果：Halton序列用更少的采样点就能覆盖整个半球，噪声降低~70%，
      收敛速度提升~3倍。
```

Halton序列的原理是将整数索引按特定基数（2, 3, 5...）进制展开并反转，生成[0,1)范围内的均匀分布数列。具体算法在`LumenScreenProbeImportanceSampling.usf`中使用。

### 3.3 Temporal Filter（时间滤波）

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenScreenProbeFiltering.cpp`  
**Shader**：`Engine/Shaders/Private/Lumen/LumenScreenProbeGatherTemporal.usf`

单帧32条射线噪声很大，Temporal Filter混合历史帧结果大幅降噪：

```hlsl
// 概念流程（基于LumenScreenProbeGatherTemporal.usf简化）

float3 FilteredRadiance = TemporalFilter(
    CurrentFrameRadiance,    // 当前帧：32条射线的结果（噪声大）
    HistoryFrameRadiance,    // 历史帧：已累积多帧的低噪结果
    MotionVector,            // 运动向量（处理相机移动）
    DepthSimilarity          // 深度相似度（避免错误混合）
);

// 混合公式：指数移动平均
// Filtered = lerp(Current, History, TemporalWeight)
// TemporalWeight通常0.9（90%历史，10%当前）
// 等效于：~10帧的积累 → 实际效果≈320条射线

// 效果对比（估算）：
// 无滤波：32条射线，MSE = 1.0（噪声明显）
// Temporal Filter：等效320条射线，MSE ≈ 0.1（噪声基本不可见）
```

### 3.4 Final Gather：插值到像素

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenScreenProbeGather.cpp`  
**Shader**：`Engine/Shaders/Private/Lumen/LumenScreenProbeGather.usf`

将Screen Probe的结果插值到每个像素，使用深度加权的双线性插值避免跨深度不连续边界的错误混合：

```hlsl
// 概念流程
float3 FinalGather(float2 ScreenUV, float PixelDepth)
{
    // 【步骤1】找到周围4个Probe
    float2 ProbeGridUV = ScreenUV * ProbeGridSize;
    int2 Probe00 = floor(ProbeGridUV);

    // 【步骤2】双线性插值 + 深度权重
    float TotalWeight = 0;
    float3 TotalRadiance = 0;

    for (int dy = 0; dy <= 1; dy++)
    for (int dx = 0; dx <= 1; dx++)
    {
        int2 ProbeCoord = Probe00 + int2(dx, dy);
        float3 ProbeRadiance = ReadProbeRadiance(ProbeCoord);
        float ProbeDepth = ReadProbeDepth(ProbeCoord);

        // 双线性权重
        float2 Frac = frac(ProbeGridUV);
        float BilinearWeight = (dx ? Frac.x : 1-Frac.x) * (dy ? Frac.y : 1-Frac.y);

        // 深度权重（深度差异大的Probe权重降低，避免漏光）
        float DepthWeight = exp(-abs(PixelDepth - ProbeDepth) * DepthScale);

        float Weight = BilinearWeight * DepthWeight;
        TotalRadiance += ProbeRadiance * Weight;
        TotalWeight += Weight;
    }

    return TotalRadiance / max(TotalWeight, 0.001);
}
```

### 💡 核心要点

Screen Probe Gather是Lumen中**耗时最大的阶段**（约占Lumen总耗时的30%）。核心优化策略：低分辨率Probe + Halton均匀采样 + Temporal Filter时间累积 + 深度加权插值。

## 四、Software Ray Tracing：DDA + Distance Field

### 4.1 追踪核心算法

**来源**：`Engine/Shaders/Private/Lumen/LumenSoftwareRayTracing.ush`

Lumen的Software Ray Tracing使用**Sphere Tracing**算法，借助Global Distance Field自适应步长：

```hlsl
// 概念流程（基于LumenSoftwareRayTracing.ush实际逻辑简化）

bool TraceSurfaceCache(
    float3 Origin,
    float3 Direction,
    float MaxTraceDistance,
    out float HitDistance,
    out float3 HitRadiance)
{
    float CurrentDistance = 0;
    const float MinStepSize = 0.5;
    const float HitThreshold = 0.1;

    // 【步骤1】Sphere Tracing循环
    while (CurrentDistance < MaxTraceDistance)
    {
        float3 SamplePos = Origin + Direction * CurrentDistance;

        // 【步骤2】采样Global Distance Field
        // 获取当前点到最近表面的距离
        float DistanceToSurface = SampleGlobalDistanceField(SamplePos);

        // 【步骤3】命中判断
        if (DistanceToSurface < HitThreshold)
        {
            // 命中表面，查找对应的Card
            uint CardIndex = FindCardAtWorldPosition(SamplePos);
            if (CardIndex != INVALID_INDEX)
            {
                // 【步骤4】世界坐标 → Card UV
                float2 CardUV = WorldPosToCardUV(SamplePos, CardIndex);

                // 【步骤5】采样Surface Cache Atlas
                float3 Albedo = SampleAtlas(AlbedoAtlas, CardUV);
                float3 DirectLighting = SampleAtlas(DirectLightingAtlas, CardUV);
                float3 Emissive = SampleAtlas(EmissiveAtlas, CardUV);

                // 【步骤6】计算辐射度
                HitRadiance = Albedo * DirectLighting + Emissive;
                HitDistance = CurrentDistance;
                return true;
            }
        }

        // 【步骤7】自适应步进（Distance Field加速）
        // 安全步长 = SDF值（保证不会跳过表面）
        CurrentDistance += max(abs(DistanceToSurface), MinStepSize);
    }

    return false;  // 未命中
}
```

### 4.2 Hardware RT加速（可选）

当GPU支持DXR（RTX系列）时，Lumen可以使用Hardware Ray Tracing加速：

```
混合追踪策略：

Software RT（默认，所有GPU）：
  ✅ 不需要RTX硬件
  ✅ 使用Distance Field加速
  ❌ 精度受Distance Field分辨率限制
  ❌ 薄物体可能穿透

Hardware RT（可选，需要RTX）：
  ✅ 精确追踪（使用BVH加速结构）
  ✅ 薄物体无穿透
  ❌ 需要RTX 2060+
  ❌ BVH构建有开销

关键CVar：
  r.Lumen.HardwareRayTracing 1   // 启用Hardware RT
  r.Lumen.HardwareRayTracing 0   // 仅使用Software RT

关键文件：
  LumenHardwareRayTracingCommon.h/.cpp  — 硬件追踪通用代码
  LumenScreenProbeHardwareRayTracing.cpp — Screen Probe硬件追踪
  LumenReflectionHardwareRayTracing.cpp  — 反射硬件追踪
```

### 💡 核心要点

Software RT是Lumen的默认追踪方案，不依赖RTX硬件。其核心加速来自Global Distance Field的Sphere Tracing——平均每条射线只需5-8次步进（vs 固定步长的30+次），性能提升4-6倍。

## 五、Lumen Reflections

### 5.1 两级追踪策略

**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenReflections.cpp`  
**来源**：`Engine/Source/Runtime/Renderer/Private/Lumen/LumenReflectionTracing.cpp`

Lumen Reflections采用"先廉价、后昂贵"的分级追踪：

```
反射追踪策略：

【第一级】HiZ Screen Tracing（屏幕空间追踪）
  输入：反射方向、HiZ深度金字塔
  方法：在屏幕空间沿反射方向步进，通过HiZ Mipmap快速跳过
  优点：极快（~1ms），利用已有的屏幕颜色
  缺点：只能追踪屏幕内可见的物体
    ↓ 未命中（反射目标在屏幕外）
【第二级】Surface Cache Tracing
  输入：反射方向
  方法：使用Software RT追踪Surface Cache
  优点：支持屏幕外物体
  缺点：较慢（~2ms），精度受Surface Cache分辨率限制
    ↓ 未命中（超出追踪距离）
【兜底】Skylight采样
  输入：反射方向
  方法：直接采样天空光Cubemap
```

```hlsl
// 概念流程（基于LumenReflectionTracing.usf简化）

float3 TraceLumenReflection(float3 WorldPos, float3 ReflectionDir)
{
    // 【第一级】HiZ屏幕追踪
    float2 HitScreenUV;
    if (HierarchicalScreenTrace(WorldPos, ReflectionDir, HitScreenUV))
    {
        return SampleSceneColor(HitScreenUV);  // 直接采样屏幕颜色
    }

    // 【第二级】Surface Cache追踪
    float HitDistance;
    float3 HitRadiance;
    if (TraceSurfaceCache(WorldPos, ReflectionDir, MaxDistance, HitDistance, HitRadiance))
    {
        return HitRadiance;
    }

    // 【兜底】天空光
    return SampleSkylight(ReflectionDir);
}
```

```
Nanite在渲染管线中的位置：

【阶段1】Nanite渲染（BasePass）
  └─ 输出：GBuffer（Depth/Normal/Albedo/Roughness/Metallic）
                        ↓
【阶段2】Lumen Screen Probe Placement
  ├─ 从GBuffer读取Depth → 重建世界坐标 → 放置Probe
  └─ 从GBuffer读取Normal → 确定Probe追踪方向
                        ↓
【阶段3】Lumen Trace
  ├─ 追踪射线 → 命中Surface Cache → 采样Card Atlas
  └─ Card Atlas中存储的也是Nanite渲染的结果
                        ↓
输出：Diffuse Indirect + Specular Indirect → 合成到场景
```

| 系统 | 与Lumen的关系 |
|------|---------------|
| **Nanite** | 上游：提供GBuffer（深度、法线）；Lumen Card Capture也渲染Nanite几何 |
| **GPUScene** | 上游：提供Instance Transform和Primitive数据 |
| **RenderGraph** | 基础设施：Lumen所有Pass通过RDG调度和资源管理 |
| **Virtual Shadow Maps** | 并行：VSM也使用Distance Field，与Lumen共享 |
| **Deferred Lighting** | 下游：Lumen输出的间接光照在Lighting阶段与直接光照合成 |

---

## 二、与RenderGraph的集成

Lumen完全基于RenderGraph实现，每个Pass都是RDG Pass。RenderGraph为Lumen提供：

1. **自动资源管理**：Surface Cache Atlas、Screen Probe纹理的生命周期自动管理
2. **自动Barrier插入**：根据纹理的读写模式自动插入Pipeline Barrier
3. **Pass依赖推导**：根据资源使用自动推导Lumen各Pass的执行顺序
4. **异步计算调度**：Surface Cache Capture、Radiance Cache Update等标记为Async Compute

详见[2.5 RenderGraph系统](../Phase02_Pipeline/2.5_RenderGraph_Deep.md)。

## 三、适用场景决策指南

```
✅ 推荐Lumen：
  - 动态昼夜循环（光照实时变化）
  - 可破坏环境（几何体运行时改变）
  - 开放世界（大量动态物体、NPC、载具）
  - 需要快速迭代的项目（无需等待烘焙）
  - 动态反射需求（水面、金属、玻璃）

❌ 不推荐Lumen：
  - GPU预算极紧张（移动端、低端PC，<15ms总帧预算）
  - 纯静态室内场景（Lightmap质量更好且零运行时开销）
  - 需要半透明物体GI（Lumen不支持半透明Surface Cache）
  - DX11/OpenGL项目（Lumen需要DX12/Vulkan）

⚖️ 混合方案：
  - 室外开放世界：Lumen
  - 室内精细场景：可考虑Lightmap与Lumen混合
  - 角色渲染：Lumen为角色提供动态GI（替代Light Probe）
```
