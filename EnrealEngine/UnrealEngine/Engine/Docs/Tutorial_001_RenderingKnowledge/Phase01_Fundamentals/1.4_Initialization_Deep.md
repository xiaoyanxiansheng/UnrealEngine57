# 第一部分：理解初始化流程的意义

## 1.1 从一个实际问题开始

假设你的UE项目启动很慢（5秒+），你需要定位问题。打开日志，看到这些：

```
LogInit: Display: SCOPED_BOOT_TIMING: RHIInit (312.4 ms)
LogInit: Display: SCOPED_BOOT_TIMING: InitializeShaderTypes (1245.7 ms)  ← 这里很慢！
LogInit: Display: SCOPED_BOOT_TIMING: LoadMap (678.3 ms)
```

**问题**：
- `RHIInit` 是什么？为什么要300ms？
- `InitializeShaderTypes` 为什么要1.2秒？能优化吗？
- 这些阶段的顺序是什么？为什么是这个顺序？

**另一个场景**：你想了解UE的UCLASS宏是如何工作的

```cpp
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()  // ← 这个宏做了什么？
    
    UPROPERTY(EditAnywhere)
    float Health;
};
```

**问题**：
- `GENERATED_BODY()` 展开后是什么？
- UE如何在运行时获取类的元数据？
- 为什么C++不支持原生反射，UE如何解决？

**这就是学习初始化流程的意义**：理解引擎如何启动、如何构建反射系统、如何加载模块，才能真正理解UE的架构。

**耗时分析**：

| 阶段 | 耗时 | 占比 | 优化方向 |
|------|------|------|---------|
| 平台层初始化 | 100ms | 3% | 无法优化（系统调用） |
| TaskGraph初始化 | 50ms | 2% | 无法优化 |
| 模块加载 | 50ms | 2% | 延迟加载非核心模块 |
| **RHI初始化** | **300ms** | **10%** | ✅ 预热GPU驱动 |
| **Shader加载** | **1000ms** | **33%** | ✅ DDC缓存、异步编译 |
| 默认材质初始化 | 200ms | 7% | ✅ 延迟创建 |
| 引擎初始化 | 200ms | 7% | - |
| 加载关卡 | 600ms | 20% | ✅ 流式加载 |

💡 **本节核心要点**
- 5秒速览：启动流程 = PreInit(平台+RHI+Shader) → Init(引擎+场景) → Tick(首帧)
- 关键词：`FEngineLoop::PreInit` `RHIInit` `GlobalShaderMap` `FScene`
- 最大瓶颈：Shader加载（33%），DDC缓存是关键优化点

# 第二部分：反射系统与UHT

## 2.1 问题：C++为什么需要反射系统？

在Unity的C#中，反射是语言原生支持的：

```csharp
// Unity C# - 原生反射
GameObject go = new GameObject();
Transform transform = go.GetComponent<Transform>();  // ← 反射查找组件

// 获取类型信息
Type type = typeof(Transform);
MethodInfo method = type.GetMethod("Translate");  // ← 反射查找方法
method.Invoke(transform, new object[] { Vector3.up });
```

但C++标准**不支持**反射：

```cpp
// ❌ 标准C++无法实现
MyClass* obj = new MyClass();
void* method = obj->GetMethod("MyFunction");  // ← 不存在
```

**为什么C++不支持反射？**

| 原因 | 说明 |
|------|------|
| 性能 | C++追求零开销抽象，反射需要运行时元数据（增加内存） |
| 哲学 | "You don't pay for what you don't use" |
| ABI | 不同编译器的类布局不同，标准化困难 |

**UE的解决方案**：编译时生成反射代码

```
源码编写 → UHT扫描 → 生成.generated.h → C++编译 → 运行时可用
```

---

## 2.2 UHT工作流程

**UHT（Unreal Header Tool）** 是UE的反射代码生成器：

```
【步骤1】源码编写
───────────────────────────────────────────────
// MyActor.h
UCLASS()                           // ← UHT标记
class AMyActor : public AActor
{
    GENERATED_BODY()               // ← UHT标记
    
    UPROPERTY(EditAnywhere)        // ← UHT标记
    float Health;
    
    UFUNCTION(BlueprintCallable)   // ← UHT标记
    void TakeDamage(float Amount);
};

【步骤2】UBT调用UHT扫描
───────────────────────────────────────────────
UnrealBuildTool.exe
  └─> 扫描所有.h文件
  └─> 查找UCLASS/UPROPERTY/UFUNCTION等宏
  └─> 调用 UnrealHeaderTool.exe MyActor.h

【步骤3】UHT生成反射代码
───────────────────────────────────────────────
UnrealHeaderTool.exe
  └─> 解析MyActor.h的AST（抽象语法树）
  └─> 生成 MyActor.generated.h
  └─> 生成 MyActor.gen.cpp（注册反射信息）

【步骤4】C++编译器编译
───────────────────────────────────────────────
cl.exe (MSVC) 或 clang++
  └─> 编译 MyActor.cpp
  └─> #include "MyActor.generated.h" 被展开
  └─> 反射代码被编译进最终二进制

【步骤5】运行时注册
───────────────────────────────────────────────
程序启动时
  └─> 调用静态初始化函数
  └─> 注册 AMyActor 的 UClass 元数据
  └─> 运行时可通过 UClass* 访问反射信息
```

**时序图**：

```
编译时                           运行时
──────────────────────────────────────────────
                                 │
.h (源码)                        │
   ↓                             │
UHT扫描                          │
   ↓                             │
.generated.h (生成)              │
   ↓                             │
C++编译器                        │
   ↓                             │
二进制程序  ──────────────────> 启动
                                 ↓
                              静态初始化
                                 ↓
                              注册UClass
                                 ↓
                         GetClass() 可用 ✅
```

## 2.3 GENERATED_BODY宏展开

**来源**：`Engine/Source/Runtime/CoreUObject/Public/UObject/ObjectMacros.h`  
**源码锚点**：搜索 `GENERATED_BODY`、`BODY_MACRO_COMBINE`、`CURRENT_FILE_ID`。  
**版本**：UE5.7

```cpp
// 展开前：
UCLASS()
class MYGAME_API AMyActor : public AActor
{
    GENERATED_BODY()
    
    UPROPERTY(EditAnywhere)
    float Health;
};

// 展开后（简化示意）：
class MYGAME_API AMyActor : public AActor
{
private:
    // 【步骤1】友元声明 - 让反射系统访问私有成员
    friend struct Z_Construct_UClass_AMyActor_Statics;
    
public:
    // 【步骤2】StaticClass() - 获取UClass*的静态方法
    static UClass* StaticClass()
    {
        static UClass* Singleton = nullptr;
        if (!Singleton)
        {
            Singleton = Z_Construct_UClass_AMyActor_Statics::ConstructUClass();
        }
        return Singleton;
    }
    
    // 【步骤3】构造函数
    AMyActor(const FObjectInitializer& ObjectInitializer = FObjectInitializer::Get());
    
private:
    // 【步骤4】静态初始化器 - 程序启动时自动注册
    static struct FStaticConstructor
    {
        FStaticConstructor()
        {
            UClassCompiledInDefer(
                &AMyActor::StaticClass,
                sizeof(AMyActor),
                TEXT("/Script/MyGame"),
                TEXT("AMyActor")
            );
        }
    } StaticConstructorInstance;
    
    // 原有成员
    float Health;
};
```

**关键技巧：获取成员偏移量**

```cpp
// 问题：如何在编译时获取成员变量Health的偏移量？

// 技巧：使用nullptr作为假对象
(size_t)(&((AMyActor*)0)->Health)

// 解释：
// 1. (AMyActor*)0        - 将地址0转换为AMyActor*指针
// 2. ->Health            - 访问Health成员
// 3. &                   - 取地址（偏移量 = 成员地址 - 对象地址 = X - 0 = X）
// 4. (size_t)            - 转换为整数

// 示例：假设Health偏移量是120字节
// ((AMyActor*)0)->Health 的地址就是 0 + 120 = 120
```

## 2.4 反射数据结构

UE的反射系统有三个核心类：

**UClass - 类元数据**

```cpp
class UClass : public UStruct
{
public:
    // 【基础信息】
    FName           ClassName;            // 类名（"AMyActor"）
    UClass*         SuperClass;           // 父类UClass*
    UObject*        ClassDefaultObject;   // CDO（Class Default Object）
    
    // 【成员信息】
    TArray<FProperty*>  Properties;       // 属性数组
    TArray<UFunction*>  Functions;        // 函数数组
    
    // 【大小和标志】
    int32           ClassSize;            // 类大小
    EClassFlags     ClassFlags;           // 标志（Abstract、Transient等）
    
    // 【核心方法】
    bool IsChildOf(const UClass* SomeBase) const;  // 继承判断
    UObject* GetDefaultObject() const;             // 获取CDO
};

// 使用示例：
UClass* ActorClass = AMyActor::StaticClass();
UObject* CDO = ActorClass->GetDefaultObject();
bool bIsActor = ActorClass->IsChildOf(AActor::StaticClass());
```

**FProperty - 属性元数据**

```cpp
class FProperty
{
public:
    FName           PropertyName;         // 属性名（"Health"）
    int32           Offset;               // 成员偏移量
    EPropertyFlags  PropertyFlags;        // 标志（EditAnywhere等）
    
    // 核心方法
    void* ContainerPtrToValuePtr(void* Container) const;  // 获取属性值指针
};

// 使用示例：
AMyActor* Actor = ...;
FProperty* HealthProp = AMyActor::StaticClass()->FindPropertyByName(TEXT("Health"));
float* HealthPtr = HealthProp->ContainerPtrToValuePtr<float>(Actor);
*HealthPtr = 100.0f;  // 反射修改Health值
```

**UFunction - 函数元数据**

```cpp
class UFunction : public UStruct
{
public:
    FName           FunctionName;         // 函数名（"TakeDamage"）
    FNativeFuncPtr  NativeFunc;           // 原生函数指针
    EFunctionFlags  FunctionFlags;        // 标志（BlueprintCallable等）
};

// 使用示例（反射调用函数）：
AMyActor* Actor = ...;
UFunction* Func = Actor->FindFunction(TEXT("TakeDamage"));
struct { float Amount; } Params = { 50.0f };
Actor->ProcessEvent(Func, &Params);
```

## 2.5 Unity vs UE反射对比

| 维度 | Unity C# | UE C++ |
|------|---------|--------|
| **实现方式** | 编译器生成IL元数据 | UHT生成C++反射代码 |
| **运行时开销** | 较高（IL解释+JIT） | 极低（编译时生成） |
| **类型安全** | 运行时检查 | 编译时检查 |
| **反射API** | `System.Reflection` | `UClass/FProperty/UFunction` |
| **性能** | 反射调用慢10-100倍 | 反射调用慢3-10倍 |
| **灵活性** | 高（所有类型都可反射） | 中（只有UCLASS标记的类） |

**性能对比**：

```cpp
// ===== UE C++ =====
// 直接调用
Actor->TakeDamage(50.0f);  // ~1ns

// 反射调用
Actor->ProcessEvent(Func, &Params);  // ~3-10ns（慢3-10倍）

// ===== Unity C# =====
// 直接调用
myComponent.TakeDamage(50.0f);  // ~5ns

// 反射调用
method.Invoke(myComponent, args);  // ~500-1000ns（慢100-200倍）
```

💡 **本节核心要点**
- 5秒速览：UE用UHT在编译时生成反射代码，实现零运行时开销的类型信息访问
- 关键词：`UHT` `GENERATED_BODY` `UClass` `FProperty` `UFunction`
- Unity对比：UE反射比Unity快10-20倍（编译时生成 vs IL解释）

# 第三部分：编译系统与UBT

## 3.1 问题：为什么需要UBT？

标准C++项目用CMake或MSBuild管理编译，但UE选择自己实现UBT，原因：

| 需求 | CMake/MSBuild | UBT |
|------|---------------|-----|
| 模块依赖管理 | 手动配置 | 自动解析Build.cs |
| 跨平台编译 | 需要多套配置 | 统一的Build.cs |
| UHT集成 | 需要手动调用 | 自动集成 |
| Unity Build | 需要手动配置 | 内置支持 |
| Hot Reload | 不支持 | 内置支持 |

---

## 3.2 UBT工作流程

```
【步骤1】扫描.Build.cs文件
─────────────────────────────────────
UBT启动
  └─> 扫描 Source/ 目录
  └─> 查找所有 *.Build.cs 文件
  └─> 编译Build.cs成C#程序集

【步骤2】解析模块依赖
─────────────────────────────────────
执行Build.cs
  └─> 读取PublicDependencyModuleNames
  └─> 构建依赖图（DAG - 有向无环图）
  └─> 检测循环依赖

【步骤3】生成编译任务
─────────────────────────────────────
遍历所有模块
  └─> 生成编译参数（Include路径、宏定义）
  └─> 生成链接参数（.lib文件）
  └─> 创建编译任务队列

【步骤4】调用编译器
─────────────────────────────────────
并行编译
  └─> 调用 cl.exe（MSVC）
  └─> 编译每个.cpp文件
  └─> 输出.obj文件

【步骤5】链接
─────────────────────────────────────
链接器（link.exe）
  └─> 合并所有.obj和.lib
  └─> 生成最终.exe或.dll
```

## 3.3 模块依赖图

**示例：Renderer模块的依赖树**

```
Renderer (渲染器模块)
├─> Core (核心模块 - 最基础)
├─> CoreUObject (对象系统)
├─> Engine (引擎基础)
├─> RHI (渲染硬件接口)
├─> RenderCore (渲染核心)
└─> Projects (项目管理)
    └─> Json (JSON解析)
        └─> Core
```

**Renderer.Build.cs示例**：

```csharp
// Engine/Source/Runtime/Renderer/Renderer.Build.cs
public class Renderer : ModuleRules
{
    public Renderer(ReadOnlyTargetRules Target) : base(Target)
    {
        // 【1】公开依赖（传递给依赖者）
        PublicDependencyModuleNames.AddRange(new string[] {
            "Core",
            "Engine",
            "RHI"
        });
        
        // 【2】私有依赖（不传递）
        PrivateDependencyModuleNames.AddRange(new string[] {
            "CoreUObject",
            "RenderCore",
```

## 3.4 循环依赖检测

**问题：循环依赖导致编译失败**

```
A.Build.cs:  PublicDependencyModuleNames.Add("B");
B.Build.cs:  PublicDependencyModuleNames.Add("C");
C.Build.cs:  PublicDependencyModuleNames.Add("A");  // ← 循环！

结果：UBT报错 "Circular dependency detected: A -> B -> C -> A"
```

**UBT检测算法（深度优先搜索）**：

```csharp
// 简化示意
bool DetectCycle(string moduleName, HashSet<string> visited, HashSet<string> stack)
{
    visited.Add(moduleName);
    stack.Add(moduleName);  // 加入当前递归栈
    
    foreach (var dep in Dependencies[moduleName])
    {
        if (!visited.Contains(dep))
        {
            if (DetectCycle(dep, visited, stack))
                return true;  // 发现循环
        }
        else if (stack.Contains(dep))
        {
            return true;  // 依赖在递归栈中 → 循环！
        }
    }
    
    stack.Remove(moduleName);  // 退出递归栈
    return false;
}
```

**解决方案**：
1. 接口分离：将公共接口提取到独立模块
2. 前向声明：使用 `class FMyClass;` 代替 `#include`
3. 重新设计：调整模块职责

```csharp
{
```

```cpp
    bool bForceD3D12 = FParse::Param(FCommandLine::Get(), TEXT("d3d12"));
    bool bForceVulkan = FParse::Param(FCommandLine::Get(), TEXT("vulkan"));

        ChosenRHI = ERHIInterfaceType::Vulkan;
    
    // 【4】加载RHI模块
    switch (ChosenRHI)
    {
    case ERHIInterfaceType::D3D12:
        return &FModuleManager::LoadModuleChecked<IDynamicRHIModule>(TEXT("D3D12RHI"));
    case ERHIInterfaceType::Vulkan:
        return &FModuleManager::LoadModuleChecked<IDynamicRHIModule>(TEXT("VulkanRHI"));
    }
}
```

# 第七部分：代码迭代机制

## 7.1 问题：如何在编辑器中快速测试代码修改？

传统C++开发需要：修改代码 → 关闭编辑器 → 编译 → 重新打开编辑器

这个流程在大型项目中极其痛苦——UE编辑器启动一次可能需要30秒到数分钟。UE为此提供了两代解决方案：

| 方案 | 引入版本 | 状态（UE5.7） | 原理 |
|------|---------|---------------|------|
| **Hot Reload（旧）** | UE4早期 | ⚠️ 已不推荐 | 卸载旧DLL → 加载新DLL → Patch VTable |
| **Live Coding（新）** | UE 4.22 | ✅ **官方推荐** | 进程内直接patch机器码，不替换DLL |

> ⚠️ **重要**：UE5.x 中 Live Coding 是**默认且推荐**的热重载方式。旧 Hot Reload 存在严重的稳定性问题（Blueprint引用损坏、编辑器崩溃等），Epic 已明确不推荐使用。本节先讲 Live Coding，再讲旧 Hot Reload（保留用于理解底层原理）。

---

## 7.2 ★ Live Coding（UE5官方推荐方案）

### 7.2.1 什么是Live Coding？

Live Coding 是 UE 4.22 引入、UE5 默认启用的代码热更新机制。与旧 Hot Reload 的本质区别：

```
旧 Hot Reload：
  编译新DLL → 卸载旧DLL → 加载新DLL → Patch VTable
  ↑ 整个DLL被替换，UClass需要重新注册，对象可能损坏

Live Coding：
  编译修改的.obj → 在运行中的进程内直接patch机器码
  ↑ 不替换DLL，不重新注册UClass，对象完全不受影响
```

### 7.2.2 Live Coding工作流程

```
【步骤1】用户修改代码
─────────────────────────────────────
MyActor.cpp:
  void AMyActor::Tick(float DeltaTime)
  {
      UE_LOG(LogTemp, Log, TEXT("New code!"));  // 修改
  }

【步骤2】触发Live Coding
─────────────────────────────────────
用户按下 Ctrl+Alt+F11（或编辑器菜单 Tools → Live Coding）
  └─> 编辑器调用编译器，只编译修改的.cpp文件
  └─> 生成新的.obj文件（不是DLL！）

【步骤3】进程内Patch
─────────────────────────────────────
Live Coding Runtime（集成在编辑器进程中）
  └─> 读取新.obj中的机器码
  └─> 定位旧函数在进程内存中的地址
  └─> 用JMP指令将旧函数入口跳转到新函数代码
  └─> 完成！无需暂停GameThread/RenderThread

【步骤4】立即生效
─────────────────────────────────────
  └─> 下一帧调用Tick()时，自动跳转到新代码
  └─> PIE（Play In Editor）运行中也能生效
  └─> 编辑器显示 "Live Coding Succeeded" ✅
```

### 7.2.3 Live Coding Patch原理

```
旧函数在内存中的样子（patch前）：

0x12345678:  push rbp           ← 函数入口
0x12345679:  mov rbp, rsp
0x1234567C:  sub rsp, 0x40
0x12345680:  ... (函数体)
0x123456FF:  ret

Patch后：

0x12345678:  jmp 0x87654321     ← 被替换为跳转指令！
0x1234567D:  (原指令被覆盖)
...

0x87654321:  push rbp           ← 新函数代码（新.obj加载到新内存区域）
0x87654322:  mov rbp, rsp
0x87654325:  ... (新函数体)
0x87654399:  ret
```

**关键区别**：
- 旧 Hot Reload：替换VTable中的函数指针 → 需要遍历所有对象的VTable
- Live Coding：直接在函数入口写JMP → 所有调用自动跳转，**无需遍历任何对象**

### 7.2.4 Live Coding 支持与限制

| 能做的 ✅ | 不能做的 ❌ |
|-----------|------------|
| 修改函数体内的逻辑 | 新增/删除 `UPROPERTY` |
| 修改局部变量 | 新增/删除 `UFUNCTION` |
| 调整数值、算法 | 修改类的内存布局（加/删成员变量） |
| 新增非UFUNCTION函数 | 修改类继承关系 |
| 修改函数内的控制流 | 新增 `UCLASS`、`USTRUCT`、`UENUM` |
| PIE运行中生效 | 修改构造函数签名 |

```cpp
// ✅ Live Coding 支持
void AMyActor::Tick(float DeltaTime)
{
    // 修改逻辑、调整数值、改变控制流 - 全部OK
    float Speed = 500.0f;  // 改成 500（原来是 200）✅
    if (bShouldMove)       // 新增条件判断 ✅
    {
        AddMovementInput(GetActorForwardVector(), Speed * DeltaTime);
    }
}

// ❌ Live Coding 不支持 - 需要重启编辑器
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()
    
    UPROPERTY(EditAnywhere)
    float Health;
    
    UPROPERTY(EditAnywhere)
    float Mana;  // ← 新增UPROPERTY，Live Coding无法处理
};
```

### 7.2.6 Live Coding 最佳实践

1. **日常迭代用 Live Coding**：修改函数逻辑后按 `Ctrl+Alt+F11`，1-3秒生效
2. **PIE中测试**：Live Coding在PIE运行中也能生效，不需要停止Play
3. **在IDE中编译，在编辑器中触发**：先在VS/Rider中保存，再回编辑器按快捷键
4. **出现异常时重启**：如果Live Coding报错或行为异常，重启编辑器解决
5. **关闭旧Hot Reload**：确保Editor Preferences中旧Hot Reload已关闭，避免两者冲突

## 7.3 旧 Hot Reload（遗留机制，了解原理用）

> ⚠️ **注意**：旧 Hot Reload 在 UE5.x 中**已不推荐使用**。本节保留是因为其DLL替换 + VTable Patch的原理对理解C++底层机制有教学价值。

### 7.3.1 旧 Hot Reload 流程

```
【步骤1】编译新DLL
─────────────────────────────────────
UBT启动
  └─> 只编译修改的模块（MyGame模块）
  └─> 生成新的 MyGame-Win64-Debug-1234.dll

【步骤2】卸载旧DLL
─────────────────────────────────────
引擎挂起GameThread和RenderThread
  └─> 调用所有UObject的PreEditChange()
  └─> FreeLibrary(MyGame.dll)  ← 卸载旧DLL

【步骤3】加载新DLL
─────────────────────────────────────
LoadLibrary(MyGame-Win64-Debug-1234.dll)
  └─> 重新注册UClass元数据
  └─> Patch函数指针（VTable重定向）

【步骤4】恢复运行
─────────────────────────────────────
调用所有UObject的PostEditChange()
  └─> 恢复GameThread和RenderThread
```

### 7.3.2 旧 Hot Reload 的严重问题

| 问题 | 说明 |
|------|------|
| **Blueprint引用损坏** | DLL卸载后UClass指针失效，Blueprint可能引用到旧的UClass |
| **CDO不一致** | Class Default Object在DLL替换后可能与新代码的默认值不匹配 |
| **编辑器崩溃** | VTable Patch失败时直接崩溃，尤其在修改涉及多个类时 |
| **需要暂停线程** | GameThread和RenderThread都要挂起，造成明显卡顿 |

这些问题是Epic引入Live Coding的直接原因。

## 7.4 VTable Patch原理（C++底层知识）

> 本节讲解旧Hot Reload使用的VTable Patch技术。虽然Live Coding使用了不同的patch方式（JMP指令），但理解VTable对C++底层学习仍有价值。

**C++虚函数表（VTable）**：

```cpp
class AActor
{
public:
    virtual void Tick(float DeltaTime) { }  // 虚函数
};

class AMyActor : public AActor
{
public:
    virtual void Tick(float DeltaTime) override { ... }
};

// 内存布局：
//
// AMyActor对象：
// ┌──────────────────┐
// │ VTable指针       │ ──────> VTable：
// ├──────────────────┤         ┌──────────────────────────┐
// │ 成员变量...      │         │ &AMyActor::Tick          │ ← Tick函数地址
// └──────────────────┘         ├──────────────────────────┤
//                              │ &AActor::BeginPlay       │
//                              └──────────────────────────┘
```

**旧Hot Reload的VTable Patch**：
- 旧Tick地址：`0x12345678`（旧DLL）
- 新Tick地址：`0x87654321`（新DLL）
- Patch：将VTable中的`0x12345678`替换为`0x87654321`

```cpp
// 简化示意
void PatchVTable(UObject* Obj, UClass* NewClass)
{
    // 获取旧VTable
    void** OldVTable = *(void***)Obj;
    
    // 获取新VTable
    void** NewVTable = *(void***)NewClass->GetDefaultObject();
    
    // 逐个函数替换
    for (int32 i = 0; i < VTableSize; i++)
    {
        if (OldVTable[i] != NewVTable[i])
        {
            OldVTable[i] = NewVTable[i];  // Patch！
        }
    }
}
```

**对比两种Patch方式**：

```
旧Hot Reload（VTable Patch）：            Live Coding（JMP Patch）：

VTable:                                    函数入口：
┌─────────────────────┐                    0x12345678:
│ Tick → 0x87654321   │ ← 改指针            jmp 0x87654321  ← 改指令
│ BeginPlay → ...     │
└─────────────────────┘

缺点：需要遍历所有对象的VTable                优点：只改一处，所有调用自动跳转
      DLL卸载可能导致悬垂指针                        不涉及DLL加载/卸载
      需要暂停线程                                  无需暂停线程
```

## 7.5 其他代码迭代方案

当 Live Coding 也无法满足需求时（比如需要新增 UPROPERTY），还有以下方案：

### 7.5.1 Blueprint 热迭代

```
C++ 基类（稳定框架）  →  Blueprint 子类（频繁迭代）

修改Blueprint = 即时生效，无需任何编译
```

**最佳实践**：把经常调整的玩法逻辑放在Blueprint中，C++只写性能敏感和框架代码。

### 7.5.2 Data-Driven 设计

```
DataTable / DataAsset / CurveTable → 运行时读取

修改数据 = 即时生效，不需要碰C++代码
```

### 7.5.3 脚本语言方案（第三方）

对于需要更自由热重载的团队，社区有成熟方案：

| 方案 | 语言 | 热重载能力 | 特点 |
|------|------|-----------|------|
| **AngelScript** | AngelScript | 几乎无限制（可加字段、改继承） | Hazelight工作室（It Takes Two）使用 |
| **UnrealSharp** | C# | 完整热重载 | 社区活跃，UE5.3+ |
| **UnLua** | Lua | 完整热重载 | 腾讯开源，国内广泛使用 |
| **puerts** | TypeScript | 完整热重载 | 国内流行 |

### 7.5.4 方案选择指南

| 你的场景 | 推荐方案 |
|---------|---------|
| 日常函数逻辑调整 | **Live Coding**（`Ctrl+Alt+F11`） |
| 需要加UPROPERTY/改类结构 | **重启编辑器**（无法避免） |
| 快速迭代玩法原型 | **Blueprint子类** + C++基类 |
| 调数值/参数 | **DataTable / DataAsset** |
| 团队需要极致热重载 | **AngelScript / UnrealSharp / UnLua** |

## 7.6 Unity vs UE 代码迭代对比

| 维度 | Unity（Domain Reload） | UE Live Coding | UE 旧Hot Reload（已不推荐） |
|------|----------------------|----------------|---------------------------|
| **原理** | 卸载/重载AppDomain | 进程内JMP patch | DLL替换 + VTable patch |
| **支持修改类大小** | ✅ 是 | ❌ 否 | ❌ 否 |
| **支持修改继承** | ✅ 是 | ❌ 否 | ❌ 否 |
| **重载时间** | 慢（5-30秒） | 快（1-3秒） | 中（1-5秒） |
| **对象保持** | ❌ 否（序列化/反序列化） | ✅ 是（对象完全不受影响） | ⚠️ 部分（可能损坏） |
| **稳定性** | 高 | 高 | ⚠️ 低（可能崩溃） |
| **PIE中可用** | ✅ 是 | ✅ 是 | ❌ 否（需停止Play） |
| **状态** | 当前方案 | ✅ **UE5官方推荐** | ⚠️ 已不推荐 |

💡 **本节核心要点**
- 5秒速览：UE5用**Live Coding**（进程内JMP patch）取代了旧Hot Reload（DLL替换），日常迭代按`Ctrl+Alt+F11`即可
- 关键词：`Live Coding` `JMP Patch` `Ctrl+Alt+F11`
- 限制：Live Coding不支持修改类内存布局（加/删UPROPERTY），此时需重启编辑器
- 替代方案：Blueprint热迭代、DataTable数据驱动、AngelScript/UnrealSharp等脚本方案
- 旧Hot Reload的VTable Patch原理仍有C++底层学习价值

# 第八部分：FScene创建详解

## 8.1 问题：FScene是什么？

`FScene` 是渲染层的场景表示，与游戏层的 `UWorld` 一一对应。

```
游戏层                     渲染层
┌─────────────┐           ┌─────────────────────────────┐
│   UWorld    │ ──1:1──→  │         FScene              │
│             │           │  ├─ Primitives (可渲染物体) │
│  AActor     │           │  ├─ Lights (光源)           │
│  UComponent │           │  ├─ GPUScene (GPU数据)      │
└─────────────┘           │  ├─ LumenSceneData          │
                          │  └─ VirtualShadowMapArray   │
                          └─────────────────────────────┘
```

## 8.3 FScene构造函数详解

```cpp
FScene::FScene(
    UWorld* InWorld,
    bool bInRequiresHitProxies,       // 是否需要HitProxy（编辑器选择）
    bool bInIsEditorScene,            // 是否编辑器场景
    bool bCreateFXSystem,             // 是否创建粒子系统
    ERHIFeatureLevel::Type FeatureLevel
)
{
    // 【步骤1】初始化基础容器
    Primitives.Reserve(1024);    // 预分配Primitive槽位
    Lights.Reserve(256);         // 预分配Light槽位
    
    // 【步骤2】创建空间加速结构
    PrimitiveOctree = MakeUnique<FScenePrimitiveOctree>(
        FVector::ZeroVector,     // 中心点
        HALF_WORLD_MAX           // 半径（大世界坐标）
    );
    
    // 【步骤3】创建粒子系统（可选）
    if (bCreateFXSystem)
    {
        FXSystem = FFXSystemInterface::Create(FeatureLevel, World->Scene);
    }
    
    // 【步骤4】★ 初始化GPUScene（UE5核心）
    GPUScene.bIsEnabled = UseGPUScene(GMaxRHIShaderPlatform, FeatureLevel);
    
    if (GPUScene.bIsEnabled)
    {
        // 创建PrimitiveBuffer（存储FPrimitiveSceneData）
        GPUScene.PrimitiveBuffer = CreateStructuredBuffer(
            sizeof(FPrimitiveSceneData),  // 每个Primitive约1KB
            MaxPrimitives                  // 最大数量（默认1000000）
        );
        
        // 创建InstanceDataBuffer
        GPUScene.InstanceDataBuffer = CreateStructuredBuffer(
            sizeof(FInstanceSceneData),   // 每个Instance约256字节
            MaxInstances
        );
        
        // 创建LightDataBuffer
        GPUScene.LightDataBuffer = CreateStructuredBuffer(
            sizeof(FLightSceneData),      // 每个Light约512字节
            MaxLights
        );
    }
    
    // 【步骤5】初始化Distance Field
    DistanceFieldSceneData.Init(FeatureLevel);
    
    // 【步骤6】★ 初始化Lumen（UE5全局光照）
    if (ShouldRenderLumen(FeatureLevel))
    {
        LumenSceneData = MakeUnique<FLumenSceneData>(FeatureLevel);
        // ~500MB内存（Surface Cache + Card Data）
    }
    
    // 【步骤7】★ 初始化Virtual Shadow Map（UE5虚拟阴影）
    if (UseVirtualShadowMaps(GMaxRHIShaderPlatform, FeatureLevel))
    {
        VirtualShadowMapArray = MakeUnique<FVirtualShadowMapArray>();
        // ~200MB内存（Page Pool + Page Table）
    }
}
```

# 第九部分：与Unity初始化对比

> 下表数字是教学示意，用于说明架构差异的量级，不是官方 Benchmark。启动时间、内存和反射性能需在目标项目、目标平台和目标构建配置下复测。

| 维度 | UE5.7 | Unity URP |
|------|-------|-----------|
| **启动入口** | `WinMain → GuardedMain → PreInit` | `PlayerLoop.Initialize` |
| **编译系统** | UBT + UHT（源码级反射） | MSBuild/Roslyn（IL反射） |
| **RHI初始化** | 直接操作D3D12 API | 封装GraphicsDevice |
| **Shader加载** | 阻塞式DDC，首次需编译 | 异步ShaderCache |

| **场景内存** | ~700MB（含Lumen/VSM） | ~200MB |
| **启动时间** | 示例为数秒级 | 示例为数秒级 |

**关键差异总结**：
1. **启动速度**：Unity更快（IL编译快），UE慢但运行时性能高
2. **内存管理**：UE手动（C++），Unity自动（GC）
3. **反射系统**：UE编译时生成，Unity运行时解释
4. **扩展性**：UE源码级修改，Unity插件系统
