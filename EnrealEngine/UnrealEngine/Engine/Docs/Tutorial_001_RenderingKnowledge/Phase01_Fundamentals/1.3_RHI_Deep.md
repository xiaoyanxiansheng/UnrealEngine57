# 第一部分：理解RHI的存在意义

## 1.1 从一个实际问题开始

假设你是一个引擎开发者，需要让你的渲染代码同时支持Windows（DirectX 12）、Linux（Vulkan）、macOS（Metal）三个平台。

现在有一个简单的需求：**创建一张1024x1024的RGBA纹理**。

### 1.1.1 不同平台的实现差异

**Windows/DirectX 12**：
```cpp
// D3D12需要：Device、HeapProperties、ResourceDesc、InitialState...
ID3D12Device* device = ...;

D3D12_HEAP_PROPERTIES heapProps = {};
heapProps.Type = D3D12_HEAP_TYPE_DEFAULT;

D3D12_RESOURCE_DESC desc = {};
desc.Dimension = D3D12_RESOURCE_DIMENSION_TEXTURE2D;
desc.Width = 1024;
desc.Height = 1024;
desc.DepthOrArraySize = 1;
desc.MipLevels = 1;
desc.Format = DXGI_FORMAT_R8G8B8A8_UNORM;
desc.SampleDesc.Count = 1;
desc.Layout = D3D12_TEXTURE_LAYOUT_UNKNOWN;
desc.Flags = D3D12_RESOURCE_FLAG_NONE;

ID3D12Resource* texture;
HRESULT hr = device->CreateCommittedResource(
    &heapProps,
    D3D12_HEAP_FLAG_NONE,
    &desc,
    D3D12_RESOURCE_STATE_COMMON,
    nullptr,
    IID_PPV_ARGS(&texture)
);
```

**Linux/Vulkan**：
```cpp
// Vulkan需要：Device、ImageCreateInfo、MemoryAllocateInfo...
VkDevice device = ...;

VkImageCreateInfo createInfo = {};
createInfo.sType = VK_STRUCTURE_TYPE_IMAGE_CREATE_INFO;
createInfo.imageType = VK_IMAGE_TYPE_2D;
createInfo.extent.width = 1024;
createInfo.extent.height = 1024;
createInfo.extent.depth = 1;
createInfo.mipLevels = 1;
createInfo.arrayLayers = 1;
createInfo.format = VK_FORMAT_R8G8B8A8_UNORM;
createInfo.tiling = VK_IMAGE_TILING_OPTIMAL;
createInfo.initialLayout = VK_IMAGE_LAYOUT_UNDEFINED;
createInfo.usage = VK_IMAGE_USAGE_SAMPLED_BIT;
createInfo.sharingMode = VK_SHARING_MODE_EXCLUSIVE;
createInfo.samples = VK_SAMPLE_COUNT_1_BIT;

VkImage image;
vkCreateImage(device, &createInfo, nullptr, &image);

// Vulkan还需要单独分配内存并绑定！
VkMemoryRequirements memRequirements;
vkGetImageMemoryRequirements(device, image, &memRequirements);

VkMemoryAllocateInfo allocInfo = {};
allocInfo.sType = VK_STRUCTURE_TYPE_MEMORY_ALLOCATE_INFO;
allocInfo.allocationSize = memRequirements.size;
allocInfo.memoryTypeIndex = FindMemoryType(memRequirements.memoryTypeBits, ...);

VkDeviceMemory memory;
vkAllocateMemory(device, &allocInfo, nullptr, &memory);
vkBindImageMemory(device, image, memory, 0);
```

**macOS/Metal**：
```objc
// Metal相对简洁，但仍然是不同的API
id<MTLDevice> device = ...;

MTLTextureDescriptor* desc = [MTLTextureDescriptor texture2DDescriptorWithPixelFormat:MTLPixelFormatRGBA8Unorm
                                                                                width:1024
                                                                               height:1024
                                                                            mipmapped:NO];
desc.usage = MTLTextureUsageShaderRead;
desc.storageMode = MTLStorageModePrivate;

id<MTLTexture> texture = [device newTextureWithDescriptor:desc];
```

### 1.1.2 问题分析

观察上面的代码，你会发现：

| 差异点 | D3D12 | Vulkan | Metal |
|--------|-------|--------|-------|
| 设备对象 | `ID3D12Device*` | `VkDevice` | `id<MTLDevice>` |
| 纹理对象 | `ID3D12Resource*` | `VkImage` + `VkDeviceMemory` | `id<MTLTexture>` |
| 格式枚举 | `DXGI_FORMAT_R8G8B8A8_UNORM` | `VK_FORMAT_R8G8B8A8_UNORM` | `MTLPixelFormatRGBA8Unorm` |
| 内存管理 | 自动（CommittedResource） | 手动分配+绑定 | 自动 |
| 调用风格 | COM接口 | C风格函数 | Objective-C |

**如果你的渲染代码直接使用这些API**：
- 你需要维护三套完全不同的代码
- 每增加一个平台，工作量翻倍
- 代码中到处是`#if PLATFORM_WINDOWS ... #elif PLATFORM_LINUX ...`
- 几乎不可能维护

### 1.1.3 RHI的解决方案

**RHI（Render Hardware Interface）** 的核心思想很简单：**提供一套统一的接口，隐藏平台差异**。

使用RHI后，创建纹理变成这样：

```cpp
// 使用RHI接口 - 任何平台都是相同的代码！
FRHITextureCreateDesc Desc = FRHITextureCreateDesc::Create2D(TEXT("MyTexture"))
    .SetExtent(1024, 1024)
    .SetFormat(PF_R8G8B8A8)
    .SetFlags(TexCreate_ShaderResource);

FRHITexture* Texture = RHICreateTexture(Desc);
```

**这就是RHI存在的根本原因**：让上层渲染代码只写一次，就能在所有平台运行。

## 1.2 RHI的核心职责

理解了RHI存在的原因，我们来明确它的职责：

### 1.2.1 五大核心职责

```
┌─────────────────────────────────────────────────────────────────┐
│                        RHI的核心职责                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 【平台抽象】                                                 │
│     定义统一的接口，让上层代码不需要知道底层是D3D12还是Vulkan     │
│                                                                 │
│  2. 【资源管理】                                                 │
│     统一管理GPU资源（纹理、缓冲区、着色器、PSO）的创建和销毁       │
│                                                                 │
│  3. 【Command录制】                                              │
│     提供DrawCall、状态设置等渲染命令的录制接口                    │
│                                                                 │
│  4. 【线程协调】                                                 │
│     协调RenderThread和RHIThread之间的工作分配                    │
│                                                                 │
│  5. 【类型转换】                                                 │
│     在UE类型和平台原生类型之间进行转换                           │
│     (例如：PF_R8G8B8A8 → DXGI_FORMAT_R8G8B8A8_UNORM)            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2.2 RHI不做什么

同样重要的是理解RHI**不**负责什么：

- **不负责**：决定渲染什么（这是Renderer的职责）
- **不负责**：场景管理、可见性剔除（这是Scene的职责）
- **不负责**：材质系统、Shader编译（这是材质系统的职责）
- **不负责**：渲染顺序决策（这是RenderGraph的职责）

**RHI是一个"翻译层"**：它接收上层的渲染指令，翻译成平台原生API调用。

## 1.3 架构概览：三层模型

在深入细节之前，建立一个宏观的架构认知：

```
┌─────────────────────────────────────────────────────────────────┐
│                     应用层（你的渲染代码）                        │
│                                                                 │
│   • SceneRenderer - 场景渲染逻辑                                 │
│   • MeshPassProcessor - 网格Pass处理                            │
│   • PostProcessing - 后处理                                     │
│   • 你写的自定义渲染代码                                         │
│                                                                 │
│   这一层【只使用RHI接口】，完全不知道底层是D3D12还是Vulkan        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 调用RHI统一接口
                              │ (FRHICommandList.DrawIndexedPrimitive等)
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                        RHI抽象层                                 │
│                                                                 │
│   • FDynamicRHI - 平台抽象接口（定义所有RHI操作的虚函数）         │
│   • FRHICommandList - Command录制器                             │
│   • FRHITexture/FRHIBuffer - 资源抽象基类                       │
│   • 枚举转换表 - UE格式 ↔ 平台格式                               │
│                                                                 │
│   这一层【定义接口和基类】，但不包含具体平台实现                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 虚函数调用，路由到具体平台
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                       平台实现层                                 │
│                                                                 │
│   ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐  │
│   │ FD3D12DynamicRHI│ │FVulkanDynamicRHI│ │ FMetalDynamicRHI│  │
│   │  (Windows/Xbox) │ │ (Linux/Android) │ │  (macOS/iOS)    │  │
│   └────────┬────────┘ └────────┬────────┘ └────────┬────────┘  │
│            │                   │                   │           │
│            ↓                   ↓                   ↓           │
│   ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐  │
│   │    D3D12 API    │ │   Vulkan API    │ │    Metal API    │  │
│   │ ID3D12Device    │ │    VkDevice     │ │   MTLDevice     │  │
│   └─────────────────┘ └─────────────────┘ └─────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 驱动程序
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                          GPU硬件                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键理解**：
1. 你写的渲染代码在最上层，只依赖RHI接口
2. RHI通过**虚函数多态**实现平台切换
3. 每个平台有一个独立的实现模块（D3D12RHI、VulkanRHI、MetalRHI）

# 第二部分：现代图形API背景知识

在深入RHI实现之前，我们需要理解现代图形API的设计理念。这对技术美术非常重要，因为它解释了很多RHI设计决策的原因。

## 2.1 从OpenGL到D3D12/Vulkan：范式转变

### 2.1.1 旧时代：OpenGL/D3D11的设计

**特点：高级抽象，驱动做很多事**

```cpp
// OpenGL风格（伪代码）
glBindTexture(GL_TEXTURE_2D, textureID);    // 绑定纹理
glBindBuffer(GL_ARRAY_BUFFER, vertexBuffer); // 绑定顶点缓冲
glUseProgram(shaderProgram);                 // 绑定着色器
glDrawArrays(GL_TRIANGLES, 0, vertexCount);  // 绘制！

// 驱动在背后做了很多事：
// - 自动管理资源状态
// - 自动处理同步
// - 自动优化命令提交
// - 隐式的状态机管理
```

**优点**：
- API简单，易于学习
- 驱动帮你处理很多细节

**缺点**：
- 驱动开销大（每次DrawCall都要验证状态）
- CPU性能瓶颈明显
- 多线程支持差
- 难以预测性能

### 2.1.2 新时代：D3D12/Vulkan的设计

**特点：显式控制，应用负责更多**

```cpp
// D3D12/Vulkan风格（概念伪代码）

// 1. 预先创建Pipeline State Object（PSO）
// 包含：着色器、顶点格式、混合状态、深度状态等ALL配置
PipelineStateObject* pso = CreatePSO(shaderConfig, blendState, depthState, ...);

// 2. 预先创建Command List
CommandList* cmdList = CreateCommandList();

// 3. 显式管理资源状态（Resource Barrier）
cmdList->ResourceBarrier(texture, STATE_SHADER_RESOURCE, STATE_RENDER_TARGET);

// 4. 绑定和绘制
cmdList->SetPipelineState(pso);
cmdList->SetDescriptorTable(0, textureDescriptor);
cmdList->DrawInstanced(vertexCount, instanceCount, 0, 0);

// 5. 显式提交
cmdList->Close();
commandQueue->ExecuteCommandLists(1, &cmdList);

// 6. 显式同步
fence->Signal(commandQueue);
fence->WaitForCompletion();
```

**显式控制的含义**：

| 方面 | D3D11/OpenGL | D3D12/Vulkan |
|------|--------------|--------------|
| 资源状态 | 驱动自动跟踪和转换 | 应用显式声明Barrier |
| 内存管理 | 驱动自动管理 | 应用控制分配和绑定 |
| 命令缓冲 | 驱动内部管理 | 应用创建和管理CommandList |
| 同步 | 驱动隐式处理 | 应用使用Fence显式同步 |
| 状态组合 | 每次DrawCall验证 | 预编译成PSO |
| 多线程 | 有限支持 | 原生支持多线程录制 |

### 2.1.3 为什么要这样设计？

**核心原因：减少CPU开销，提高多线程效率**

旧API的问题：
```
每次DrawCall，驱动需要：
├─ 验证当前状态是否合法
├─ 检查资源绑定是否正确
├─ 检测资源状态是否需要转换
├─ 优化命令序列
└─ 这些都在CPU上进行，开销巨大！
```

新API的解决方案：
```
应用预先做好准备：
├─ PSO预编译 → DrawCall时不需要验证状态
├─ 显式Barrier → 不需要运行时检测状态
├─ 多CommandList → 可以多线程录制
└─ 驱动变"瘦"，CPU开销大幅降低
```

**实际性能对比**：

| 场景 | D3D11 | D3D12 | 提升 |
|------|-------|-------|------|
| 10000 DrawCalls | ~8ms CPU | ~2ms CPU | 4x |
| 多线程录制 | 不支持 | 支持 | 线性扩展 |
| 状态切换 | 每次验证 | PSO直接切换 | 10x+ |

### 2.1.4 对RHI设计的影响

现代图形API的这些特点直接影响了UE RHI的设计：

1. **PSO抽象**：RHI需要管理`FRHIGraphicsPipelineState`
2. **Resource Barrier**：RHI需要提供状态转换接口
3. **Command录制**：RHI采用Command Pattern支持多线程
4. **显式同步**：RHI需要管理Fence

## 2.2 关键概念：Pipeline State Object (PSO)

### 2.2.1 什么是PSO？

**PSO是渲染管线所有状态的预编译组合**。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Pipeline State Object (PSO)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────┐  ┌───────────────────┐                  │
│  │   Vertex Shader   │  │   Pixel Shader    │                  │
│  └───────────────────┘  └───────────────────┘                  │
│                                                                 │
│  ┌───────────────────┐  ┌───────────────────┐                  │
│  │  Input Layout     │  │   Blend State     │                  │
│  │  (顶点格式)        │  │  (混合状态)        │                  │
│  └───────────────────┘  └───────────────────┘                  │
│                                                                 │
│  ┌───────────────────┐  ┌───────────────────┐                  │
│  │ Rasterizer State  │  │ DepthStencil State│                  │
│  │ (光栅化状态)       │  │  (深度模板状态)    │                  │
│  └───────────────────┘  └───────────────────┘                  │
│                                                                 │
│  ┌───────────────────┐  ┌───────────────────┐                  │
│  │ RenderTarget格式   │  │  MSAA设置         │                  │
│  └───────────────────┘  └───────────────────┘                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2.2 为什么需要PSO？

**没有PSO（旧API）**：
```cpp
// 每次DrawCall前设置各种状态
glUseProgram(shader);
glEnable(GL_BLEND);
glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA);
glEnable(GL_DEPTH_TEST);
glDepthFunc(GL_LESS);
// ... 更多状态设置

glDrawArrays(...);

// 驱动在内部：
// 1. 检查所有状态是否兼容
// 2. 编译/查找对应的硬件配置
// 3. 这个过程可能很慢！
```

**有PSO（新API）**：
```cpp
// 启动时预创建PSO（耗时，但只做一次）
PSO* pso = CreatePSO(shader, blendState, depthState, ...);

// DrawCall时直接绑定（非常快）
cmdList->SetPipelineState(pso);
cmdList->DrawIndexedInstanced(...);

// 没有运行时验证，没有查找，直接使用！
```

### 2.2.3 UE中的PSO

在UE中，PSO相关的类：

```cpp
// RHI层的PSO抽象
class FRHIGraphicsPipelineState : public FRHIResource {
    // 平台无关的基类
};

// 初始化器（描述PSO需要什么）
struct FGraphicsPipelineStateInitializer {
    FRHIVertexShader* VertexShader;
    FRHIPixelShader* PixelShader;
    FRHIVertexDeclaration* VertexDeclaration;
    FBlendStateInitializer BlendState;
    FRasterizerStateInitializer RasterizerState;
    FDepthStencilStateInitializer DepthStencilState;
    // ...
};

// 创建PSO
FRHIGraphicsPipelineState* PSO = RHICreateGraphicsPipelineState(Initializer);
```

**对TA的意义**：
- 材质的每个变体（Permutation）都对应一个PSO
- PSO编译需要时间，这就是为什么有"Shader编译"过程
- 运行时切换PSO很快，但创建PSO很慢

## 2.3 关键概念：Resource Barrier（资源屏障）

### 2.3.1 什么是Resource Barrier？

GPU资源（纹理、缓冲区）在不同操作中有不同的"状态"，需要显式转换。

**常见的资源状态**：

| 状态 | 含义 | 典型用途 |
|------|------|----------|
| ShaderResource (SRV) | 着色器可读 | 采样纹理 |
| RenderTarget (RTV) | 可写入的渲染目标 | 渲染到纹理 |
| DepthWrite | 深度缓冲写入 | Z-Buffer写入 |
| DepthRead | 深度缓冲只读 | 深度测试但不写 |
| UnorderedAccess (UAV) | 着色器读写 | Compute着色器 |
| CopySource | 复制源 | 纹理拷贝 |
| CopyDest | 复制目标 | 纹理拷贝 |
| Present | 呈现 | 显示到屏幕 |

### 2.3.2 为什么需要状态转换？

**GPU内部有缓存**，不同操作使用不同的缓存路径：

```
┌─────────────────────────────────────────────────────────────────┐
│                         GPU缓存架构（简化）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐                     ┌─────────────┐          │
│   │ Texture     │                     │ RenderTarget│          │
│   │ Cache (L1)  │                     │ Cache       │          │
│   │ 优化采样读取 │                     │ 优化写入     │          │
│   └──────┬──────┘                     └──────┬──────┘          │
│          │                                   │                 │
│          └─────────────┬─────────────────────┘                 │
│                        ↓                                       │
│               ┌─────────────────┐                              │
│               │    L2 Cache     │                              │
│               └────────┬────────┘                              │
│                        ↓                                       │
│               ┌─────────────────┐                              │
│               │   Video Memory  │                              │
│               └─────────────────┘                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**问题场景**：
```
Pass 1: 将纹理A作为RenderTarget写入数据
Pass 2: 将纹理A作为ShaderResource采样

如果没有Barrier：
- Pass 2可能读到的是L1 Cache中的旧数据
- 因为Pass 1写入的数据可能还在RenderTarget Cache中
```

**Barrier的作用**：
```cpp
// 强制GPU：
// 1. 等待Pass 1完成所有写入
// 2. 刷新RenderTarget Cache到显存
// 3. 使Texture Cache中的旧数据无效
// 4. 然后才能开始Pass 2

cmdList->ResourceBarrier(textureA, 
    D3D12_RESOURCE_STATE_RENDER_TARGET,    // 从RTV
    D3D12_RESOURCE_STATE_PIXEL_SHADER_RESOURCE  // 到SRV
);
```

### 2.3.3 UE中的Resource Barrier

```cpp
// RHI提供的Barrier接口
RHICmdList.Transition(
    FRHITransitionInfo(
        Texture,                           // 资源
        ERHIAccess::RTV,                   // 从哪个状态
        ERHIAccess::SRVMask                // 到哪个状态
    )
);

// RenderGraph会自动管理大部分Barrier
// 但理解这个概念很重要！
```

**对TA的意义**：
- 后处理链中的纹理需要状态转换
- RenderGraph帮你自动处理大部分情况
- 但写自定义Pass时需要注意状态管理

## 2.4 关键概念：Command List与多线程录制

### 2.4.1 什么是Command List？

**Command List是一个"命令录制器"**：不立即执行命令，而是记录下来，稍后批量提交给GPU。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Command List 概念                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  录制阶段（CPU）：                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CmdList.SetPipelineState(pso)      // 记录命令1         │   │
│  │ CmdList.SetVertexBuffer(vb)        // 记录命令2         │   │
│  │ CmdList.DrawIndexedInstanced(...)  // 记录命令3         │   │
│  │ CmdList.SetPipelineState(pso2)     // 记录命令4         │   │
│  │ CmdList.DrawIndexedInstanced(...)  // 记录命令5         │   │
│  │ ...                                                     │   │
│  │ CmdList.Close()                    // 录制完成          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  提交阶段：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CommandQueue.ExecuteCommandLists(CmdList)               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                     │
│                           ↓                                     │
│  执行阶段（GPU）：                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ GPU按顺序执行录制的命令                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4.2 多线程录制

**Command List的核心优势：可以多线程录制！**

```
┌─────────────────────────────────────────────────────────────────┐
│                    多线程录制                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Thread 1                Thread 2                Thread 3      │
│   ┌──────────┐           ┌──────────┐           ┌──────────┐   │
│   │CmdList 1 │           │CmdList 2 │           │CmdList 3 │   │
│   │录制Pass A │           │录制Pass B │           │录制Pass C │   │
│   └────┬─────┘           └────┬─────┘           └────┬─────┘   │
│        │                      │                      │          │
│        └──────────────────────┼──────────────────────┘          │
│                               ↓                                 │
│                    ┌─────────────────────┐                      │
│                    │   Submit All        │                      │
│                    │   CommandQueue.     │                      │
│                    │   Execute(1,2,3)    │                      │
│                    └─────────────────────┘                      │
│                               │                                 │
│                               ↓                                 │
│                    ┌─────────────────────┐                      │
│                    │   GPU执行           │                      │
│                    │   (按提交顺序)       │                      │
│                    └─────────────────────┘                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**UE的线程模型**：
- **RenderThread**：录制RHI Command
- **RHIThread**：将RHI Command转换为平台Command并提交

```
GameThread ──(同步)──> RenderThread ──(异步)──> RHIThread ──> GPU
                            │                      │
                            │                      │
                       录制FRHICommand        执行FRHICommand
                       (不执行，只记录)         (调用D3D12/Vulkan)
```

## 3.1 资源抽象：FRHIResource体系

### 3.1.1 设计目标

**问题回顾**：上层代码需要持有GPU资源（纹理、缓冲区）的引用，但不想知道具体是什么平台类型。

**解决方案**：定义一个平台无关的基类，所有平台实现都继承自它。

```
                    ┌─────────────────┐
                    │  FRHIResource   │  ← 所有资源的根基类
                    │  (平台无关)      │
                    └────────┬────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                 │                 │
           ↓                 ↓                 ↓
    ┌────────────┐    ┌────────────┐    ┌────────────┐
    │FRHITexture │    │FRHIBuffer  │    │FRHIShader  │
    │ (纹理基类)  │    │ (缓冲基类)  │    │ (着色器基类)│
    └─────┬──────┘    └─────┬──────┘    └─────┬──────┘
          │                 │                 │
    ┌─────┴─────┐     ┌─────┴─────┐     ┌─────┴─────┐
    │           │     │           │     │           │
    ↓           ↓     ↓           ↓     ↓           ↓
┌────────┐ ┌────────┐ ...       ...   ...       ...
│FD3D12  │ │FVulkan │
│Texture │ │Texture │
│(D3D12) │ │(Vulkan)│
└────────┘ └────────┘
```

### 3.1.2 FRHIResource基类

```cpp
// Engine/Source/Runtime/RHI/Public/RHIResources.h

class FRHIResource
{
public:
    FRHIResource()
        : NumRefs(0)
        , MarkedForDelete(false)
    {}
    
    virtual ~FRHIResource() {}
    
    // =====================================================
    // 引用计数管理
    // TRefCountPtr<FRHITexture>会自动调用这些方法
    // =====================================================
    
    uint32 AddRef() const
    {
        int32 NewValue = ++NumRefs;
        check(NewValue > 0);  // 溢出检查
        return NewValue;
    }
    
    uint32 Release() const
    {
        int32 NewValue = --NumRefs;
        if (NewValue == 0)
        {
            // 引用归零，可以删除
            // 注意：不是直接delete，而是加入延迟删除队列
            // 因为GPU可能还在使用这个资源
            if (!MarkedForDelete)
            {
                MarkedForDelete = true;
                PendingDeletes.Enqueue(const_cast<FRHIResource*>(this));
            }
        }
        return NewValue;
    }
    
    uint32 GetRefCount() const { return NumRefs; }
    
    // =====================================================
    // 获取平台原生资源（慎用）
    // =====================================================
    
    // 如果你确实需要访问底层平台资源，可以用这个
    // 但大多数情况下不需要
    virtual void* GetNativeResource() const { return nullptr; }
    
    // 获取原生资源的备用虚拟地址（如果有）
    virtual void* GetNativeShaderResourceView() const { return nullptr; }
    virtual void* GetNativeUnorderedAccessView() const { return nullptr; }
    
    // =====================================================
    // 调试支持
    // =====================================================
    
    void SetName(const FName& InName) { Name = InName; }
    FName GetName() const { return Name; }
    
private:
    mutable int32 NumRefs;        // 引用计数（mutable允许const对象修改）
    mutable bool MarkedForDelete; // 是否已标记删除
    FName Name;                   // 资源名称（调试用）
    
    // 延迟删除队列（静态成员）
    static TQueue<FRHIResource*> PendingDeletes;
};
```

**关键设计点**：

1. **引用计数**：使用`TRefCountPtr<FRHITexture>`时自动管理生命周期
2. **延迟删除**：资源不会立即删除，因为GPU可能还在使用
3. **虚析构函数**：确保通过基类指针删除时调用正确的析构函数

### 3.1.5 使用TRefCountPtr管理资源

**重要**：不要直接使用裸指针，使用`TRefCountPtr`自动管理生命周期。

```cpp
// ❌ 不好：裸指针，容易忘记释放或重复释放
FRHITexture* Texture = RHICreateTexture(Desc);
// ... 使用纹理
// 忘记释放了？或者在某个分支释放两次？

// ✅ 好：使用TRefCountPtr自动管理
TRefCountPtr<FRHITexture> Texture = RHICreateTexture(Desc);
// ... 使用纹理
// 超出作用域自动释放，或者被其他TRefCountPtr持有时不会释放

// 赋值给另一个TRefCountPtr
TRefCountPtr<FRHITexture> AnotherRef = Texture;  // 引用计数+1

// Texture和AnotherRef都可以使用
// 当两者都超出作用域时，资源才会被释放
```

```cpp
// Engine/Source/Runtime/D3D12RHI/Private/D3D12RHI.h

class FD3D12DynamicRHI : public FDynamicRHI
{
public:
    FD3D12DynamicRHI();
    virtual ~FD3D12DynamicRHI();
    
    // =====================================================
    // 实现所有虚函数
    // =====================================================
    
    virtual FRHITexture* RHICreateTexture(
        const FRHITextureCreateDesc& CreateDesc
    ) override
    {
        // 1. 转换描述符
        D3D12_RESOURCE_DESC D3D12Desc = {};
        D3D12Desc.Dimension = D3D12_RESOURCE_DIMENSION_TEXTURE2D;
        D3D12Desc.Width = CreateDesc.Extent.X;
        D3D12Desc.Height = CreateDesc.Extent.Y;
        D3D12Desc.DepthOrArraySize = CreateDesc.ArraySize;
        D3D12Desc.MipLevels = CreateDesc.NumMips;
        D3D12Desc.Format = ConvertPixelFormat(CreateDesc.Format);  // UE格式→D3D12格式
        D3D12Desc.SampleDesc.Count = CreateDesc.NumSamples;
        D3D12Desc.SampleDesc.Quality = 0;
        D3D12Desc.Layout = D3D12_TEXTURE_LAYOUT_UNKNOWN;
        D3D12Desc.Flags = ConvertTextureFlags(CreateDesc.Flags);   // UE标志→D3D12标志
        
        // 2. 确定堆属性
        D3D12_HEAP_PROPERTIES HeapProps = {};
        HeapProps.Type = D3D12_HEAP_TYPE_DEFAULT;  // GPU本地内存
        
        // 3. 确定初始状态
        D3D12_RESOURCE_STATES InitialState = ConvertResourceState(CreateDesc.InitialState);
        
        // 4. 创建D3D12资源
        TRefCountPtr<ID3D12Resource> D3D12Resource;
        HRESULT hr = Device->CreateCommittedResource(
            &HeapProps,
            D3D12_HEAP_FLAG_NONE,
            &D3D12Desc,
            InitialState,
            nullptr,  // Clear value (可选)
            IID_PPV_ARGS(D3D12Resource.GetInitReference())
        );
        
        if (FAILED(hr))
        {
            UE_LOG(LogD3D12RHI, Error, TEXT("Failed to create texture: %s"), 
                   *CreateDesc.DebugName);
            return nullptr;
        }
        
        // 5. 设置调试名称（方便GPU调试器查看）
        if (!CreateDesc.DebugName.IsEmpty())
        {
            D3D12Resource->SetName(*CreateDesc.DebugName);
        }
        
        // 6. 创建UE资源对象
        FD3D12Texture* Texture = new FD3D12Texture();
        Texture->Resource = D3D12Resource;
        Texture->SizeX = CreateDesc.Extent.X;
        Texture->SizeY = CreateDesc.Extent.Y;
        Texture->Format = CreateDesc.Format;
        Texture->NumMips = CreateDesc.NumMips;
        Texture->Flags = CreateDesc.Flags;
        
        // 7. 创建视图（SRV/RTV/DSV）
        if (EnumHasAnyFlags(CreateDesc.Flags, TexCreate_ShaderResource))
        {
            CreateShaderResourceView(Texture);
        }
        if (EnumHasAnyFlags(CreateDesc.Flags, TexCreate_RenderTargetable))
        {
            CreateRenderTargetView(Texture);
        }
        if (EnumHasAnyFlags(CreateDesc.Flags, TexCreate_DepthStencilTargetable))
        {
            CreateDepthStencilView(Texture);
        }
        
        return Texture;
    }
    
    virtual void RHIDrawIndexedPrimitive(
        FRHIBuffer* IndexBufferRHI,
        int32 BaseVertexIndex,
        uint32 FirstInstance,
        uint32 NumVertices,
        uint32 StartIndex,
        uint32 NumPrimitives,
        uint32 NumInstances
    ) override
    {
        // 1. 转换资源类型
        FD3D12Buffer* IndexBuffer = ResourceCast(IndexBufferRHI);
        
        // 2. 设置索引缓冲（如果变化了）
        FD3D12CommandContext& Context = GetCommandContext();
        if (Context.CurrentIndexBuffer != IndexBuffer)
        {
            D3D12_INDEX_BUFFER_VIEW IBView;
            IBView.BufferLocation = IndexBuffer->GetGPUVirtualAddress();
            IBView.SizeInBytes = IndexBuffer->GetSize();
            IBView.Format = (IndexBuffer->GetStride() == 2) ? 
                            DXGI_FORMAT_R16_UINT : DXGI_FORMAT_R32_UINT;
            
            Context.CommandList->IASetIndexBuffer(&IBView);
            Context.CurrentIndexBuffer = IndexBuffer;
        }
        
        // 3. 计算索引数量
        uint32 IndexCount = GetVertexCountForPrimitiveCount(NumPrimitives, PrimitiveType);
        
        // 4. 调用D3D12绘制
        Context.CommandList->DrawIndexedInstanced(
            IndexCount,      // IndexCountPerInstance
            NumInstances,    // InstanceCount
            StartIndex,      // StartIndexLocation
            BaseVertexIndex, // BaseVertexLocation
            FirstInstance    // StartInstanceLocation
        );
        
        // 5. 统计
        Context.NumDrawCalls++;
        Context.NumTriangles += NumPrimitives * NumInstances;
    }
    
private:
    // D3D12特有成员
    TRefCountPtr<IDXGIFactory4> DXGIFactory;
    TRefCountPtr<ID3D12Device> Device;
    TRefCountPtr<ID3D12CommandQueue> CommandQueue;
    
    // 转换函数
    static DXGI_FORMAT ConvertPixelFormat(EPixelFormat Format);
    static D3D12_RESOURCE_FLAGS ConvertTextureFlags(ETextureCreateFlags Flags);
    static D3D12_RESOURCE_STATES ConvertResourceState(ERHIAccess Access);
};
```

## 3.3 Command系统详解

### 3.3.1 为什么需要Command？

回顾线程模型：

```
GameThread ──────> RenderThread ──────> RHIThread ──────> GPU
                        │                   │
                        │                   │
                   录制渲染命令         执行GPU命令
```

**问题**：RenderThread和RHIThread是不同的线程。如何传递渲染命令？

**方案1：直接调用（不好）**
```cpp
// RenderThread中
void RenderScene()
{
    // 直接调用RHI接口
    GDynamicRHI->RHIDrawIndexedPrimitive(...);
    
    // 问题：
    // 1. 需要线程同步（RHI内部状态需要保护）
    // 2. RenderThread被阻塞，无法并行
}
```

**方案2：Command Pattern（UE的选择）**
```cpp
// RenderThread中
void RenderScene()
{
    FRHICommandListImmediate& RHICmdList = GetImmediateCommandList();
    
    // "录制"命令，不立即执行
    RHICmdList.DrawIndexedPrimitive(...);  // 只是创建一个Command对象
    RHICmdList.DrawIndexedPrimitive(...);  // 又一个Command对象
    
    // 批量提交给RHIThread
    RHICmdList.SubmitCommandsHint();
    
    // RenderThread继续工作，不等待
}

// RHIThread中
void ExecuteCommands()
{
    // 拉取RenderThread提交的Command
    FRHICommandListBase* CmdList = PullCommandList();
    
    // 逐个执行
    while (FRHICommandBase* Cmd = CmdList->GetNextCommand())
    {
        Cmd->Execute();  // 这里才真正调用RHI接口
    }
}
```

### 3.3.2 Command基类

```cpp
// 所有Command的基类
struct FRHICommandBase
{
    FRHICommandBase* Next;  // 链表，指向下一个Command
    
    // 虚析构，但没有虚Execute
    // 因为Execute使用CRTP模式避免虚函数开销
    virtual ~FRHICommandBase() {}
};

// CRTP模板基类
template<typename TCmd>
struct FRHICommand : public FRHICommandBase
{
    // 执行并析构
    void ExecuteAndDestruct(FRHICommandListBase& CmdList)
    {
        // static_cast而不是虚函数，避免VTable开销
        TCmd* ThisCmd = static_cast<TCmd*>(this);
        ThisCmd->Execute(CmdList);  // 调用派生类的Execute
        ThisCmd->~TCmd();           // 析构（内存由内存池管理）
    }
};
```

### 3.3.4 FRHICommandList：Command录制器

```cpp
class FRHICommandListBase
{
public:
    // =====================================================
    // 录制接口
    // =====================================================
    
    // 录制DrawIndexedPrimitive命令
    void DrawIndexedPrimitive(
        FRHIBuffer* IndexBuffer,
        int32 BaseVertexIndex,
        uint32 FirstInstance,
        uint32 NumVertices,
        uint32 StartIndex,
        uint32 NumPrimitives,
        uint32 NumInstances)
    {
        // 如果当前在Bypass模式（立即执行模式），直接调用RHI
        if (Bypass())
        {
            GDynamicRHI->RHIDrawIndexedPrimitive(
                IndexBuffer, BaseVertexIndex, FirstInstance,
                NumVertices, StartIndex, NumPrimitives, NumInstances
            );
            return;
        }
        
        // 正常模式：录制Command
        
        // 1. 从内存池分配Command对象的内存
        void* Memory = AllocCommand(
            sizeof(FRHICommandDrawIndexedPrimitive),
            alignof(FRHICommandDrawIndexedPrimitive)
        );
        
        // 2. Placement New：在分配的内存上构造对象
        FRHICommandDrawIndexedPrimitive* Cmd = 
            new (Memory) FRHICommandDrawIndexedPrimitive(
                IndexBuffer, BaseVertexIndex, FirstInstance,
                NumVertices, StartIndex, NumPrimitives, NumInstances
            );
        
        // 3. 加入Command链表
        if (CommandLink)
        {
            CommandLink->Next = Cmd;
        }
        else
        {
            Root = Cmd;
        }
        CommandLink = Cmd;
    }
    
    // 类似的，还有很多其他录制方法：
    void SetGraphicsPipelineState(FRHIGraphicsPipelineState* PSO, ...);
    void SetVertexBuffer(uint32 Index, FRHIBuffer* Buffer, uint32 Offset);
    void SetViewport(float MinX, float MinY, float MinZ, float MaxX, float MaxY, float MaxZ);
    void SetScissorRect(bool bEnable, uint32 MinX, uint32 MinY, uint32 MaxX, uint32 MaxY);
    void BeginRenderPass(const FRHIRenderPassInfo& Info, const TCHAR* Name);
    void EndRenderPass();
    // ... 还有很多
    
    // =====================================================
    // 提交接口
    // =====================================================
    
    void SubmitCommandsHint()
    {
        if (HasCommands())
        {
            // 将当前CommandList加入执行队列
            FRHICommandListExecutor::GetImmediateCommandList().QueueCommandList(this);
            
            // 唤醒RHIThread
            FRHIThread::Get().TriggerExecution();
            
            // 重置，准备录制下一批Command
            Reset();
        }
    }
    
    // =====================================================
    // 内存管理
    // =====================================================

    FMemStackBase MemManager;       // 内存池
    FRHICommandBase* Root;          // 链表头
    FRHICommandBase* CommandLink;   // 当前链表尾
    bool bExecuting;                // 是否正在执行

    void* AllocCommand(uint32 Size, uint32 Alignment)
    {
        return MemManager.Alloc(Size, Alignment);
    }
    
    // 重置（帧结束时调用）
    void Reset()
    {
        Root = nullptr;
        CommandLink = nullptr;
        MemManager.Flush();  // 重置内存池指针
    }
    
    bool HasCommands() const
    {
        return Root != nullptr;
    }
```

### 3.3.5 内存池：为什么不用new

**问题**：每帧可能录制数万个Command，如果每个都用`new`：
- `new`需要调用`malloc`，涉及锁和内存管理
- 每次分配约500ns
- 1万个Command = 5ms，太慢了！

**解决方案**：栈式内存池

```cpp
// 简化的内存池实现
class FMemStackBase
{
public:
    FMemStackBase()
    {
        // 预分配一大块内存（例如1MB）
        BlockSize = 1024 * 1024;
        Memory = (uint8*)FMemory::Malloc(BlockSize);
        Current = Memory;
        End = Memory + BlockSize;
    }
    
    // 分配内存：只是移动指针！
    void* Alloc(uint32 Size, uint32 Alignment)
    {
        // 对齐
        Current = Align(Current, Alignment);
        
        // 检查空间
        if (Current + Size > End)
        {
            // 当前块不够，分配新块
            AllocNewBlock(Size);
        }
        
        // "分配"：只是移动指针
        void* Result = Current;
        Current += Size;
        return Result;
    }
    
    // 重置：整块内存可以复用
    void Flush()
    {
        Current = Memory;
        // 不调用任何析构函数，Command的析构由ExecuteAndDestruct处理
    }
    
private:
    uint8* Memory;    // 内存块起始地址
    uint8* Current;   // 当前分配位置
    uint8* End;       // 内存块结束地址
    uint32 BlockSize;
};
```

**性能对比**：

| 分配方式 | 单次耗时 | 1万次耗时 |
|---------|---------|----------|
| `new`/`malloc` | ~500ns | ~5ms |
| 内存池 | ~20ns | ~0.2ms |
| **提升** | **25倍** | **25倍** |

### 3.3.6 Command执行流程

```cpp
// RHIThread的主循环（简化）
void FRHIThread::Run()
{
    while (!bExitRequested)
    {
        // 1. 等待RenderThread提交Command
        ExecutionEvent.Wait();
        
        // 2. 拉取CommandList
        FRHICommandListBase* CmdList;
        while (CommandListQueue.Dequeue(CmdList))
        {
            // 3. 遍历执行每个Command
            FRHICommandBase* Cmd = CmdList->Root;
            while (Cmd)
            {
                FRHICommandBase* Next = Cmd->Next;
                
                // 4. 执行并析构（调用派生类的Execute）
                Cmd->ExecuteAndDestruct(*CmdList);
                
                Cmd = Next;
            }
        }
    }
}
```

## 3.4 性能开销分析

### 3.4.1 各环节耗时

| 操作 | 耗时 | 说明 |
|------|------|------|
| AllocCommand（内存池） | ~20ns | 栈式分配，极快 |
| 构造Command对象 | ~10ns | 拷贝参数 |
| 链表插入 | ~2ns | 设置Next指针 |
| **录制总计** | **~32ns** | RenderThread的开销 |
| Command执行调度 | ~5ns | CRTP静态分发 |
| GDynamicRHI虚函数 | ~3ns | VTable查找 |
| D3D12 API调用 | ~10-50ns | 取决于具体API |
| **执行总计** | **~20-60ns** | RHIThread的开销 |

### 3.4.2 每帧总开销估算

假设每帧1万个DrawCall：
- 录制开销：10000 × 32ns = 0.32ms
- 执行开销：10000 × 40ns = 0.4ms
- **CPU总开销：约0.7ms**

**对比GPU耗时**：一帧渲染通常需要10-16ms的GPU时间，所以0.7ms的CPU开销是可接受的。

### 3.4.3 虚函数开销讨论

你可能注意到RHI使用了虚函数。虚函数有额外开销（VTable查找），为什么还用？

**开销分析**：
- 虚函数调用：~3ns
- 每帧调用次数：~1万次
- 总开销：~30μs（0.03ms）

**可以忽略的原因**：
1. 0.03ms相比16ms的帧时间微不足道
2. 虚函数带来的代码清晰度和可维护性收益巨大
3. 真正的性能瓶颈是GPU，不是CPU端的虚函数调用

```cpp
    return RHICreateBuffer(Size, Usage | BUF_StructuredBuffer, Stride, ERHIAccess::SRVMask, CreateInfo);
}
```

### 4.1.4 更新纹理数据

```cpp
// 锁定纹理进行CPU写入
void UpdateTextureData(FRHITexture* Texture, const TArray<FColor>& NewData)
{
    FRHICommandListImmediate& RHICmdList = FRHICommandListExecutor::GetImmediateCommandList();
    
    // 1. 锁定纹理
    uint32 DestStride;
    void* TextureData = RHICmdList.LockTexture2D(
        Texture,
        0,                    // Mip级别
        RLM_WriteOnly,        // 锁定模式：只写
        DestStride,           // 输出：每行的字节数
        false                 // 不锁定MipTail
    );
    
    // 2. 拷贝数据（注意Stride可能和宽度不一致！）
    uint32 Width = Texture->GetSizeX();
    uint32 Height = Texture->GetSizeY();
    
    for (uint32 Y = 0; Y < Height; ++Y)
    {
        FMemory::Memcpy(
            (uint8*)TextureData + Y * DestStride,
            NewData.GetData() + Y * Width,
            Width * sizeof(FColor)
        );
    }
    
    // 3. 解锁
    RHICmdList.UnlockTexture2D(Texture, 0, false);
}
```

## 4.3 如何录制渲染命令

### 4.3.1 基本渲染流程

```cpp
void RenderMyEffect(FRHICommandListImmediate& RHICmdList, FRHITexture* RenderTarget)
{
    // 1. 获取渲染目标视图
    FRHIRenderPassInfo RPInfo(RenderTarget, ERenderTargetActions::Clear_Store);
    
    // 2. 开始RenderPass
    RHICmdList.BeginRenderPass(RPInfo, TEXT("MyEffect"));
    
    // 3. 设置视口
    RHICmdList.SetViewport(
        0, 0, 0.0f,                                    // MinX, MinY, MinZ
        RenderTarget->GetSizeX(),                       // MaxX
        RenderTarget->GetSizeY(),                       // MaxY
        1.0f                                           // MaxZ
    );
    
    // 4. 设置管线状态
    RHICmdList.SetGraphicsPipelineState(GraphicsPSO, 0);
    
    // 5. 绑定资源
    RHICmdList.SetShaderTexture(PixelShader, 0, SourceTexture);
    RHICmdList.SetShaderSampler(PixelShader, 0, BilinearSampler);
    
    // 6. 设置顶点/索引缓冲
    RHICmdList.SetStreamSource(0, VertexBuffer, 0);
    
    // 7. 绘制
    RHICmdList.DrawIndexedPrimitive(
        IndexBuffer,
        0,          // BaseVertexIndex
        0,          // FirstInstance
        4,          // NumVertices
        0,          // StartIndex
        2,          // NumPrimitives (2个三角形)
        1           // NumInstances
    );
    
    // 8. 结束RenderPass
    RHICmdList.EndRenderPass();
}
```

## 4.5 性能优化建议

### 4.5.1 减少状态切换

```cpp
// ❌ 不好：频繁切换PSO
for (const FMeshBatch& Batch : Batches)
{
    RHICmdList.SetGraphicsPipelineState(Batch.PSO);  // 每次都切换
    RHICmdList.DrawIndexedPrimitive(...);
}

// ✅ 好：按PSO排序后绘制
Batches.Sort([](const FMeshBatch& A, const FMeshBatch& B) {
    return A.PSO < B.PSO;  // 按PSO指针排序
});

FRHIGraphicsPipelineState* CurrentPSO = nullptr;
for (const FMeshBatch& Batch : Batches)
{
    if (Batch.PSO != CurrentPSO)
    {
        RHICmdList.SetGraphicsPipelineState(Batch.PSO);
        CurrentPSO = Batch.PSO;
    }
    RHICmdList.DrawIndexedPrimitive(...);
}
```

### 4.5.2 批量提交

```cpp
// ❌ 不好：频繁提交
for (int i = 0; i < 1000; ++i)
{
    RHICmdList.DrawIndexedPrimitive(...);
    RHICmdList.SubmitCommandsHint();  // 每次都提交
}

// ✅ 好：批量提交
for (int i = 0; i < 1000; ++i)
{
    RHICmdList.DrawIndexedPrimitive(...);
}
RHICmdList.SubmitCommandsHint();  // 一次性提交
```

### 4.5.3 使用Instancing

```cpp
// ❌ 不好：多次DrawCall
for (const FTransform& Transform : InstanceTransforms)
{
    // 设置每个实例的变换...
    RHICmdList.DrawIndexedPrimitive(..., 1);  // 绘制1个实例
}

// ✅ 好：一次DrawCall绘制所有实例
// 使用InstanceBuffer提供变换数据
RHICmdList.DrawIndexedPrimitive(
    IndexBuffer,
    0, 0, VertexCount,
    0, PrimitiveCount,
    InstanceTransforms.Num()  // 一次绘制所有实例
);
```

# 第五部分：与其他系统的关系

## 5.1 RHI与RenderGraph

**RenderGraph**是UE5的高级渲染调度系统，建立在RHI之上：

```
┌─────────────────────────────────────────────────────────────────┐
│                      RenderGraph                                 │
│   • 声明式定义Pass和资源依赖                                      │
│   • 自动资源状态管理                                             │
│   • 自动并行化                                                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 内部使用
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         RHI                                      │
│   • 底层Command录制                                              │
│   • 资源创建                                                     │
│   • 平台抽象                                                     │
└─────────────────────────────────────────────────────────────────┘
```

**关系**：
- RenderGraph使用RHI来执行实际的渲染操作
- RenderGraph自动处理资源Barrier，你不需要手动调用`RHICmdList.Transition`
- 大多数情况下，你应该使用RenderGraph而不是直接使用RHI

## 5.2 RHI与材质系统

材质系统编译Shader，最终通过RHI创建和使用：

```
Material Editor
      ↓
Material Expression Graph
      ↓
HLSL代码生成
      ↓
Shader编译（FXC/DXC/SPIRV）
      ↓
RHI创建Shader对象（RHICreateVertexShader等）
      ↓
RHI创建PSO（RHICreateGraphicsPipelineState）
      ↓
渲染时绑定PSO（SetGraphicsPipelineState）
```

## 5.3 RHI与MeshPassProcessor

MeshPassProcessor负责决定如何绘制网格，最终调用RHI：

```cpp
// MeshPassProcessor内部（简化）
void FMyMeshPassProcessor::AddMeshBatch(const FMeshBatch& MeshBatch, ...)
{
    // 1. 获取材质的Shader
    FMeshMaterialShader* VertexShader = Material->GetShader<FMyVS>();
    FMeshMaterialShader* PixelShader = Material->GetShader<FMyPS>();
    
    // 2. 创建PSO初始化器
    FGraphicsPipelineStateInitializer PSOInit;
    PSOInit.BoundShaderState.VertexShaderRHI = VertexShader->GetVertexShader();
    PSOInit.BoundShaderState.PixelShaderRHI = PixelShader->GetPixelShader();
    // ... 设置其他状态
    
    // 3. 获取或创建PSO
    FRHIGraphicsPipelineState* PSO = GetGraphicsPipelineState(PSOInit);
    
    // 4. 录制绘制命令（通过RHI）
    DrawRenderState.SetPipelineState(PSO);
    DrawRenderState.DrawMeshBatch(MeshBatch);  // 内部调用RHI
}
```
