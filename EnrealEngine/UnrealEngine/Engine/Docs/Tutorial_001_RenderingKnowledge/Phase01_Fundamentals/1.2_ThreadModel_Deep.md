# 第一部分：理解线程模型的存在意义

## 1.1 设计目标

> 💡 单线程渲染的问题已在 [1.1 RenderLoop](./1.1_RenderLoop_Deep.md) 第一部分详细分析，这里直接进入解决方案。

**UE线程模型要解决的核心问题**：

| 问题 | 解决方案 |
|------|---------|
| CPU等待GPU | 流水线并行（GT/RT/RHIT同时工作） |
| 数据竞争 | 命令队列模式（ENQUEUE_RENDER_COMMAND） |
| 多核利用率低 | Task Graph并行调度 |
| 高频通信开销 | 无锁队列（Lock-Free） |

## 1.2 UE的三线程架构

### 1.2.1 架构总览

```
┌─────────────────────────────────────────────────────────────────┐
│                     UE5 三线程架构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐                                           │
│  │  GameThread     │  游戏逻辑线程                              │
│  │  (GT)           │  • Actor Tick                             │
│  │                 │  • 物理模拟（Chaos）                       │
│  │                 │  • AI计算                                  │
│  │                 │  • 动画更新                                │
│  │                 │  • 输入处理                                │
│  └────────┬────────┘                                           │
│           │ ENQUEUE_RENDER_COMMAND                             │
│           │ （发送渲染命令）                                     │
│           ↓                                                    │
│  ┌─────────────────┐                                           │
│  │  RenderThread   │  渲染线程                                  │
│  │  (RT)           │  • 可见性剔除                              │
│  │                 │  • 排序DrawCall                           │
│  │                 │  • 录制RHI命令                             │
│  │                 │  • 场景遍历                                │
│  └────────┬────────┘                                           │
│           │ Submit Commands                                    │
│           │ （提交命令）                                        │
│           ↓                                                    │
│  ┌─────────────────┐                                           │
│  │  RHIThread      │  RHI线程                                  │
│  │  (RHIT)         │  • 执行RHI命令                             │
│  │                 │  • 调用D3D12/Vulkan API                   │
│  │                 │  • 管理GPU命令队列                         │
│  └────────┬────────┘                                           │
│           │ GPU Command Buffer                                 │
│           ↓                                                    │
│  ┌─────────────────┐                                           │
│  │      GPU        │                                           │
│  └─────────────────┘                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2.2 为什么是三个线程？

| 线程 | 职责 | 为什么需要独立 |
|------|------|---------------|
| GameThread | 游戏逻辑 | 需要稳定的帧时间，不能被渲染阻塞 |
| RenderThread | 场景遍历、命令录制 | CPU密集型，可以和GT并行 |
| RHIThread | GPU命令提交 | D3D12/Vulkan API调用可能阻塞 |

**为什么不是两个线程？**

```
【两线程方案的问题】：
GameThread → RenderThread
                  │
                  └─ 既要生成命令，又要提交GPU
                     当D3D12/Vulkan API调用阻塞时，命令生成也停止

【三线程方案的优势】：
GameThread → RenderThread → RHIThread
                  │              │
                  │              └─ 专门处理GPU提交
                  │                  可以异步批量提交
                  │
                  └─ 专心生成命令
                     不被GPU提交阻塞
```

### 1.2.3 线程分离的收益

| 场景 | 单线程 | 三线程 | 提升 |
|------|--------|--------|------|
| 10000物体渲染 | 33ms | 16ms | 2x |
| 复杂物理场景 | 40ms | 20ms | 2x |
| CPU瓶颈场景 | 无法改善 | 利用多核 | 3-4x |

## 1.3 帧间流水线与同步

### 1.3.1 帧间流水线

```
时间 →
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
帧N-1                帧N                  帧N+1
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GameThread:
  [游戏逻辑 N-1]   [游戏逻辑 N]       [游戏逻辑 N+1]
       │               │                   │
       │ ENQUEUE       │ ENQUEUE           │
       ↓               ↓                   ↓

RenderThread:
       [场景遍历 N-1] [场景遍历 N]     [场景遍历 N+1]
            │              │                │
            │ Submit       │ Submit         │
            ↓              ↓                ↓

RHIThread:
            [RHI命令 N-1] [RHI命令 N]   [RHI命令 N+1]
                 │             │             │
                 ↓             ↓             ↓

GPU:
                 [执行 N-1]   [执行 N]    [执行 N+1]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**关键点**：
1. GT处理帧N时，RT处理帧N-1，GPU执行帧N-2
2. 最大并行度 = 3-4个阶段同时工作
3. **延迟（Latency）**：输入到显示有2-3帧延迟

### 1.3.2 帧同步点

虽然线程并行运行，但需要同步点防止跑得太远：

```
┌─────────────────────────────────────────────────────────────────┐
│                     同步点示意图                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  GameThread ─────────────┬──────────────────┬─────────────     │
│               帧N开始    │ FlushRenderingCommands             │
│                          │ （等待RT完成）    │                  │
│                          ↓                  │                  │
│  RenderThread ──────────────────────────────┼─────────────     │
│                                             │                  │
│                                             │                  │
│  同步原因：                                  │                  │
│  1. 读取渲染结果（截图、像素读取）            │                  │
│  2. 删除渲染资源（安全释放）                  │                  │
│  3. 帧结束同步（防止GT跑太远）                │                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**同步的代价**：
- GT等待RT = CPU空闲
- 应该尽量减少同步次数
- 典型：每帧末尾同步一次

## 1.4 对技术美术的意义

理解线程模型对TA非常重要：

| 场景 | 为什么需要理解 |
|------|--------------|
| 材质参数更新 | 需要通过ENQUEUE_RENDER_COMMAND发送到RT |
| 动态修改网格 | 必须正确同步，否则崩溃 |
| 读取渲染结果 | 需要等待RT完成 |
| 性能分析 | 理解瓶颈是GT、RT还是GPU |
| 自定义渲染Pass | 代码运行在RT，不能直接访问GT数据 |

💡 **核心要点**
- UE使用三线程架构：GT（游戏逻辑）→ RT（渲染准备）→ RHIT（GPU提交）
- 线程并行运行，通过流水线提升吞吐量
- 线程间通过命令队列通信，减少锁竞争
- 理解线程模型是写正确多线程渲染代码的基础

# 第二部分：并发与并行基础

## 2.1 并发 vs 并行

这两个概念经常混淆，但含义不同：

### 2.1.1 并发（Concurrency）

**定义**：多个任务在**同一时间段**内交替执行

```
单核CPU上的"并发"（时间片轮转）：

时间 →
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CPU │ A │ B │ A │ B │ A │ B │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
      ↑     ↑     ↑
      任务切换（上下文切换）

• 任何时刻只有一个任务运行
• 通过快速切换产生"同时"的错觉
• 开销：上下文切换约1-10μs
```

### 2.1.2 并行（Parallelism）

**定义**：多个任务在**同一时刻**真正同时执行

```
多核CPU上的"并行"：

时间 →
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Core0 │    任务A    │    任务C    │
Core1 │    任务B    │    任务D    │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        ↑
        同一时刻两个任务真正同时运行
```

### 2.1.3 UE中的应用

| 场景 | 类型 | 说明 |
|------|------|------|
| GT/RT/RHIT | 并行 | 三个线程在不同核心同时运行 |
| Task Graph | 并行 | 任务分发到Worker线程并行执行 |
| Async Loading | 并发 | IO操作和CPU计算交替 |
| 协程（Future） | 并发 | 等待结果时让出CPU |

## 2.4 C++线程基础

### 2.4.1 std::thread vs UE线程

UE不直接使用`std::thread`，而是封装了自己的线程系统：

```cpp
// ❌ 不推荐：直接使用std::thread
#include <thread>

std::thread MyThread([](){
    // 工作
});
MyThread.join();

// ✅ 推荐：使用UE的FRunnable
class FMyRunnable : public FRunnable
{
public:
    virtual uint32 Run() override
    {
        // 工作
        return 0;
    }
};

FRunnableThread* Thread = FRunnableThread::Create(
    new FMyRunnable(),
    TEXT("MyThread"),
    0,                    // StackSize (0=默认)
    TPri_Normal           // 优先级
);
```

**为什么UE要封装？**

| 特性 | std::thread | UE FRunnable |
|------|-------------|--------------|
| 平台抽象 | ❌ 有差异 | ✅ 统一接口 |
| 优先级设置 | ❌ 不支持 | ✅ 支持 |
| 核心绑定 | ❌ 不支持 | ✅ 支持 |
| 调试名称 | ❌ 无 | ✅ 有 |
| 性能统计 | ❌ 无 | ✅ 集成 |

### 2.4.2 互斥锁（Mutex）

```cpp
// 标准C++的mutex

std::mutex MyMutex;
MyMutex.lock();
// 临界区
MyMutex.unlock();

// UE封装的FCriticalSection

// 方法1：手动加锁
CriticalSection.Lock();
// 临界区
CriticalSection.Unlock();

// 方法2：RAII（推荐）

    // 临界区
}  // 自动解锁
```

### 2.4.3 原子操作

```cpp
// UE的原子类型
TAtomic<int32> Counter(0);

// 原子操作
Counter.Increment();        // ++Counter
Counter.Decrement();        // --Counter
Counter.Add(5);             // Counter += 5

int32 Value = Counter.Load();                    // 读取
Counter.Store(100);                              // 写入
bool Success = Counter.CompareExchange(100, 0); // CAS
```

**何时使用原子操作 vs 锁？**

| 场景 | 推荐 | 原因 |
|------|------|------|
| 简单计数器 | 原子操作 | 无锁，更快 |
| 标志位 | 原子操作 | 无锁，更快 |
| 复杂数据结构 | 锁 | 原子操作无法保护 |
| 多个变量协同 | 锁 | 需要原子性保证 |

💡 **核心要点**
- UE封装了跨平台的线程原语（FRunnable、FCriticalSection、TAtomic）
- 优先使用RAII模式（FScopeLock）管理锁
- 简单场景用原子操作，复杂场景用锁
- 理解帧预算和瓶颈分析对性能调优至关重要

## 3.2 ENQUEUE_RENDER_COMMAND详解

### 3.2.1 为什么需要这个宏？

**问题**：GameThread有一些数据需要发送给RenderThread，如何安全传递？

```cpp
// ❌ 错误：直接访问渲染资源
void AMyActor::UpdateMaterial()
{
    // 这在GameThread运行
    MaterialInstance->SetScalarParameter(TEXT("Time"), Time);
    // 问题：MaterialInstance可能正在被RenderThread使用！
    // 数据竞争 → 崩溃
}

// ✅ 正确：通过命令队列发送
void AMyActor::UpdateMaterial()
{
    float CapturedTime = Time;  // 捕获数据
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [CapturedTime](FRHICommandListImmediate& RHICmdList)
        {
            // 这在RenderThread运行
            // 安全地更新材质参数
        }
    );
}
```

### 3.2.2 宏展开分析

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderingThread.h`  
**源码锚点**：搜索 `ENQUEUE_RENDER_COMMAND`、`TEnqueueUniqueRenderCommandType`、`EnqueueUniqueRenderCommand`。

```cpp
// 宏定义
#define ENQUEUE_RENDER_COMMAND(TypeName) \
    struct TypeName##Name \
    { \
        static const char* CStr() { return #TypeName; } \
    }; \
    EnqueueUniqueRenderCommand<TypeName##Name>

// 展开示例：
ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
    [Time](FRHICommandListImmediate& RHICmdList)
    {
        // Lambda体
    }
);

// 展开后：
struct UpdateMaterialName
{
    static const char* CStr() { return "UpdateMaterial"; }  // 调试名称
};
EnqueueUniqueRenderCommand<UpdateMaterialName>(
    [Time](FRHICommandListImmediate& RHICmdList)
    {
        // Lambda体
    }
);
```

**为什么要生成唯一类型？**
1. **性能分析**：每个命令有名字，Profiler能显示
2. **类型安全**：通过模板参数区分不同命令

### 3.2.3 EnqueueUniqueRenderCommand实现

```cpp
template<typename TSTR, typename LAMBDA>
void EnqueueUniqueRenderCommand(LAMBDA&& Lambda)
{
    // 【步骤1】类型萃取 - 获取Lambda的类型
    typedef TDecay<LAMBDA>::Type DecayedLambda;
    
    // 【步骤2】检查是否需要排队
    if (IsInRenderingThread())
    {
        // 已经在RenderThread，直接执行
        FRHICommandListImmediate& RHICmdList = GetImmediateCommandList_ForRenderCommand();
        Lambda(RHICmdList);
        return;
    }
    
    // 【步骤3】包装成Task
    FRenderCommand<TSTR, DecayedLambda>* RenderCommand = 
        new FRenderCommand<TSTR, DecayedLambda>(Forward<LAMBDA>(Lambda));
    
    // 【步骤4】发送到RenderThread的任务队列
    FTaskGraphInterface::Get().QueueTask(
        ENamedThreads::RenderThread,   // 目标线程
        RenderCommand                   // 任务对象
    );
    
    // 【步骤5】唤醒RenderThread（如果在等待）
    GRenderCommandWaitEvent->Trigger();
}
```

### 3.2.4 Lambda捕获详解（重点！）

**这是ENQUEUE_RENDER_COMMAND最容易出错的地方！**

```cpp
// ====== 值捕获 vs 引用捕获 ======

// ❌ 危险：引用捕获
void AMyActor::UpdateMaterial()
{
    float Time = GetWorld()->GetTimeSeconds();
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [&Time](FRHICommandListImmediate& RHICmdList)  // 引用捕获！
        {
            // 当这个Lambda在RenderThread执行时，
            // Time变量可能已经超出作用域被销毁！
            // 访问悬空引用 → 崩溃或未定义行为
            UseTime(Time);  // 💥 危险！
        }
    );
}  // Time在这里销毁

// ✅ 安全：值捕获
void AMyActor::UpdateMaterial()
{
    float Time = GetWorld()->GetTimeSeconds();
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [Time](FRHICommandListImmediate& RHICmdList)  // 值捕获
        {
            // Time被拷贝到Lambda内部
            // 即使原变量销毁，Lambda内的副本仍有效
            UseTime(Time);  // ✅ 安全！
        }
    );
}
```

### 3.2.5 移动语义优化

对于大对象，使用移动语义避免拷贝开销：

```cpp
// 问题：TArray拷贝可能很慢
TArray<FVector> LargeArray;
LargeArray.SetNum(100000);  // 10万个元素

// ❌ 拷贝捕获（慢）
ENQUEUE_RENDER_COMMAND(SlowCopy)(
    [LargeArray](FRHICommandListImmediate& RHICmdList)  // 拷贝10万个元素！
    {
        // ...
    }
);

// ✅ 移动捕获（快）
ENQUEUE_RENDER_COMMAND(FastMove)(
    [LargeArray = MoveTemp(LargeArray)](FRHICommandListImmediate& RHICmdList)
    {
        // LargeArray 被"移动"进来，不是拷贝
        // 原来的 LargeArray 现在是空的
    }
);
// 注意：MoveTemp后原数组不能再使用！
```

**移动语义原理**（简化）：

```cpp
// 移动 vs 拷贝
TArray<int32> Source;
Source.SetNum(100000);

// 【拷贝】：分配新内存，逐元素复制
TArray<int32> Copy = Source;
// Source: [数据指针 → 10万个int]
// Copy:   [新指针   → 10万个int（复制）]

// 【移动】：转移指针所有权
TArray<int32> Moved = MoveTemp(Source);
// Source: [数据指针 → nullptr]  ← 指针被"偷走"
// Moved:  [原指针   → 10万个int] ← 获得原指针
// 没有数据复制！只是指针赋值！
```

### 3.2.6 捕获指针的注意事项

```cpp
// ❌ 危险：捕获裸指针
void AMyActor::UpdateMaterial()
{
    UMaterialInstanceDynamic* MID = DynamicMaterial;
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [MID](FRHICommandListImmediate& RHICmdList)
        {
            // MID指向的对象可能已被GC回收！
            MID->SetScalarParameter(...);  // 💥 可能崩溃！
        }
    );
}

// ✅ 安全：使用弱引用
void AMyActor::UpdateMaterial()
{
    TWeakObjectPtr<UMaterialInstanceDynamic> WeakMID = DynamicMaterial;
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [WeakMID](FRHICommandListImmediate& RHICmdList)
        {
            if (UMaterialInstanceDynamic* MID = WeakMID.Get())
            {
                MID->SetScalarParameter(...);  // ✅ 安全！
            }
        }
    );
}

// ✅ 安全：捕获渲染资源指针（引用计数）
void AMyActor::UpdateMaterial()
{
    FMaterialRenderProxy* RenderProxy = DynamicMaterial->GetRenderProxy();
    RenderProxy->AddRef();  // 增加引用计数
    
    ENQUEUE_RENDER_COMMAND(UpdateMaterial)(
        [RenderProxy](FRHICommandListImmediate& RHICmdList)
        {
            // 使用RenderProxy...
            RenderProxy->Release();  // 释放引用
        }
    );
}
```

### 3.2.7 Lambda捕获规则速查

| 语法 | 含义 | 安全性 |
|------|------|--------|
| `[=]` | 值捕获所有变量 | ✅ 安全（但可能拷贝过多） |
| `[&]` | 引用捕获所有变量 | ❌ 跨线程危险 |
| `[x]` | 值捕获x | ✅ 安全 |
| `[&x]` | 引用捕获x | ❌ 跨线程危险 |
| `[x, &y]` | 混合捕获 | ⚠️ 取决于y的生命周期 |
| `[this]` | 捕获this指针 | ⚠️ 对象可能被销毁 |

**最佳实践**：

```cpp
// 【规则1】跨线程Lambda永远使用值捕获
ENQUEUE_RENDER_COMMAND(MyCmd)([Value = SomeValue](...) { ... });

// 【规则2】大对象使用移动语义
TArray<FVector> LargeArray = ...;
ENQUEUE_RENDER_COMMAND(MyCmd)([Array = MoveTemp(LargeArray)](...) { ... });

// 【规则3】UObject使用弱引用
TWeakObjectPtr<AActor> WeakActor = this;
ENQUEUE_RENDER_COMMAND(MyCmd)([WeakActor](...) { 
    if (AActor* Actor = WeakActor.Get()) { ... }
});

// 【规则4】渲染资源使用引用计数
TRefCountPtr<FRHITexture> TextureRef = MyTexture;
ENQUEUE_RENDER_COMMAND(MyCmd)([TextureRef](...) { ... });
```

## 3.3 FRenderCommandFence同步

### 3.3.1 为什么需要Fence？

有时GameThread需要等待RenderThread完成某些操作：

### 3.3.2 FRenderCommandFence实现

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderCommandFence.h`  
**源码锚点**：搜索 `class FRenderCommandFence`、`BeginFence`、`Wait`。

```cpp
class FRenderCommandFence
{
public:
    FRenderCommandFence()
        : CompletionEvent(nullptr)
    {}
    
    // 开始追踪（插入Fence）
    void BeginFence()
    {
        // 【步骤1】创建完成事件
        if (!CompletionEvent)
        {
            CompletionEvent = FPlatformProcess::GetSynchEventFromPool();
        }
        CompletionEvent->Reset();
        
        // 【步骤2】发送Fence命令到RenderThread
        ENQUEUE_RENDER_COMMAND(FenceCommand)(
            [Event = CompletionEvent](FRHICommandListImmediate& RHICmdList)
            {
                // 当这个命令执行时，触发事件
                Event->Trigger();
            }
        );
    }
    
    // 等待Fence完成
    void Wait(bool bProcessGameThreadTasks = false)
    {
        if (!CompletionEvent)
        {
            return;  // 没有Fence，直接返回
        }
        
        if (IsInRenderingThread())
        {
            // 已经在RenderThread，不需要等待
            return;
        }
        
        // 【等待策略】
        if (bProcessGameThreadTasks)
        {
            // 边等待边处理GameThread任务（不完全阻塞）
            while (!CompletionEvent->Wait(0))
            {
                FTaskGraphInterface::Get().ProcessThreadUntilIdle(ENamedThreads::GameThread);
            }
        }
        else
        {
            // 完全阻塞等待
            CompletionEvent->Wait();
        }
    }
    
    // 检查是否完成（非阻塞）
    bool IsFenceComplete() const
    {
        return CompletionEvent == nullptr || CompletionEvent->Wait(0);
    }
    
private:
    FEvent* CompletionEvent;
};
```

### 3.3.3 使用示例

```cpp
// ====== 示例1：等待特定命令完成 ======
void MyClass::DoSomething()
{
    FRenderCommandFence Fence;
    
    // 发送命令
    ENQUEUE_RENDER_COMMAND(MyCommand)(
        [](FRHICommandListImmediate& RHICmdList)
        {
            // 渲染操作...
        }
    );
    
    // 插入Fence
    Fence.BeginFence();
    
    // 等待完成
    Fence.Wait();
    
    // 现在可以安全地使用结果
}

// ====== 示例2：异步检查完成 ======
class FAsyncOperation
{
    FRenderCommandFence Fence;
    bool bStarted = false;
    
public:
    void Start()
    {
        ENQUEUE_RENDER_COMMAND(AsyncOp)(
            [](FRHICommandListImmediate& RHICmdList)
            {
                // 耗时渲染操作...
            }
        );
        Fence.BeginFence();
        bStarted = true;
    }
    
    bool IsComplete()
    {
        return !bStarted || Fence.IsFenceComplete();
    }
    
    void EnsureComplete()
    {
        if (bStarted)
        {
            Fence.Wait();
            bStarted = false;
        }
    }
};

// ====== FlushRenderingCommands ======
// 最常用的同步函数，内部就是使用Fence
void FlushRenderingCommands()
{
    FRenderCommandFence Fence;
    Fence.BeginFence();
    Fence.Wait();
}
```

## 3.5 无锁队列（Lock-Free）

### 3.5.1 为什么需要无锁队列？

GT和RT之间需要高频通信（每帧数千条命令），如果使用锁：

```cpp
// ❌ 使用锁的队列
class FLockedQueue
{
    FCriticalSection Lock;
    TArray<FCommand*> Commands;
    
public:
    void Enqueue(FCommand* Cmd)
    {
        FScopeLock ScopeLock(&Lock);  // 获取锁（可能等待）
        Commands.Add(Cmd);
    }
    
    FCommand* Dequeue()
    {
        FScopeLock ScopeLock(&Lock);  // 获取锁（可能等待）
        return Commands.Pop();
    }
};

// 问题：
// 1. 锁竞争：GT和RT同时访问时，一个必须等待
// 2. 锁开销：即使无竞争，Lock/Unlock也有~100ns开销
// 3. 优先级反转：低优先级线程持有锁时，高优先级线程被阻塞
```

### 3.5.2 SPSC无锁队列原理

**SPSC（Single-Producer Single-Consumer）**：单生产者单消费者队列，最简单的无锁队列。

UE中GT→RT的通信就是SPSC场景（GT生产，RT消费）。

```cpp
// 简化的SPSC队列实现
template<typename T>
class TSPSCQueue
{
public:
    TSPSCQueue(int32 Capacity)
        : Buffer(new T[Capacity])
        , Mask(Capacity - 1)  // 假设Capacity是2的幂
        , Head(0)
        , Tail(0)
    {
    }
    
    // 生产者调用（只有一个线程调用）
    bool Enqueue(const T& Item)
    {
        int32 CurrentTail = Tail.Load(EMemoryOrder::Relaxed);
        int32 NextTail = (CurrentTail + 1) & Mask;
        
        // 检查是否已满
        if (NextTail == Head.Load(EMemoryOrder::Acquire))
        {
            return false;  // 队列满
        }
        
        // 写入数据
        Buffer[CurrentTail] = Item;
        
        // 更新Tail（使数据对消费者可见）
        Tail.Store(NextTail, EMemoryOrder::Release);
        return true;
    }
    
    // 消费者调用（只有一个线程调用）
    bool Dequeue(T& OutItem)
    {
        int32 CurrentHead = Head.Load(EMemoryOrder::Relaxed);
        
        // 检查是否为空
        if (CurrentHead == Tail.Load(EMemoryOrder::Acquire))
        {
            return false;  // 队列空
        }
        
        // 读取数据
        OutItem = Buffer[CurrentHead];
        
        // 更新Head
        Head.Store((CurrentHead + 1) & Mask, EMemoryOrder::Release);
        return true;
    }
    
private:
    T* Buffer;
    int32 Mask;
    TAtomic<int32> Head;  // 消费者更新
    TAtomic<int32> Tail;  // 生产者更新
};
```

**关键点：内存顺序**

| 操作 | 内存顺序 | 原因 |
|------|---------|------|
| 读取自己的索引 | Relaxed | 只有自己修改，无需同步 |
| 读取对方的索引 | Acquire | 需要看到对方的最新写入 |
| 更新自己的索引 | Release | 确保之前的写入对对方可见 |

### 3.5.3 性能对比

| 方案 | 入队/出队耗时 | 竞争时开销 | 适用场景 |
|------|-------------|-----------|----------|
| 锁保护队列 | ~100ns | ~1-10μs | 低频访问 |
| SPSC无锁 | ~20ns | 无竞争 | 单生产者单消费者 |
| MPSC无锁 | ~30ns | CAS重试 | 多生产者单消费者 |
| MPMC无锁 | ~50ns | 复杂 | 多生产者多消费者 |

💡 **核心要点**
- Task Graph是UE的任务调度系统，自动管理依赖和并行
- 无锁队列避免锁竞争，适合高频通信场景
- SPSC最简单最快，MPSC用于多生产者场景
- 内存顺序（Acquire/Release）是无锁编程的关键

```cpp
    if (!Resource) return;
```

## 4.5 性能优化建议

### 4.5.1 减少跨线程通信

```cpp
// ❌ 不好：频繁小命令
void Tick(float DeltaTime)
{
    for (int32 i = 0; i < 100; ++i)
    {
        ENQUEUE_RENDER_COMMAND(Update)(
            [i](FRHICommandListImmediate& RHICmdList)
            {
                UpdateSingleItem(i);
            }
        );
    }
}

// ✅ 好：批量命令
void Tick(float DeltaTime)
{
    TArray<int32> ItemsToUpdate;
    for (int32 i = 0; i < 100; ++i)
    {
        ItemsToUpdate.Add(i);
    }
    
    ENQUEUE_RENDER_COMMAND(UpdateBatch)(
        [Items = MoveTemp(ItemsToUpdate)](FRHICommandListImmediate& RHICmdList)
        {
            for (int32 i : Items)
            {
                UpdateSingleItem(i);
            }
        }
    );
}
```

### 4.5.2 避免阻塞等待

```cpp
// ❌ 不好：每帧阻塞
void Tick(float DeltaTime)
{
    ENQUEUE_RENDER_COMMAND(Update)(...);
    FlushRenderingCommands();  // 阻塞！
}

// ✅ 好：异步检查
class FMyComponent
{
    FRenderCommandFence Fence;
    
    void StartAsyncWork()
    {
        ENQUEUE_RENDER_COMMAND(Work)(...);
        Fence.BeginFence();
    }
    
    void Tick(float DeltaTime)
    {
        if (Fence.IsFenceComplete())
        {
            // 处理结果
            OnWorkComplete();
        }
    }
};
```

### 4.5.3 合理使用并行

```cpp
// 并行化的收益取决于工作量
// 太小的任务反而会因为调度开销变慢

// ❌ 工作量太小

{
    Result[i] = i * 2;  // 只有一次乘法
});

// ✅ 工作量足够

{
    Result[i] = ComplexComputation(i);  // 每项有大量计算
});

// 经验法则：每个任务至少执行1μs以上才值得并行化
```
