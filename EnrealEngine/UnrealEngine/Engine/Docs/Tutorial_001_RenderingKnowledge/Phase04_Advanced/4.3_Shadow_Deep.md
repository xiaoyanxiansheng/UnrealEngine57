# 第一部分：理解阴影系统的存在意义

> **核心目标**：从渲染管线的实际分辨率瓶颈和性能浪费出发，理解阴影系统解决了什么问题、付出了什么代价

---

## 一、一张固定Shadow Map的崩溃

### 1.1 问题场景

假设你正在开发一个开放世界项目，场景范围5km × 5km，平行光投射阴影。如果只用一张固定分辨率的Shadow Map，会面临什么问题？

```
场景范围：5000m × 5000m
Shadow Map分辨率：4096 × 4096

每像素覆盖范围 = 5000m / 4096 ≈ 1.22m/像素

结果：
├─ 近距离（相机前5m）：角色1.8m身高 = 1.5个像素 → 阴影像锯齿方块
├─ 中距离（50m）：树木10m高 = 8个像素 → 勉强可用，但锯齿严重
└─ 远距离（500m）：建筑50m = 41个像素 → 建筑阴影还算可以

核心矛盾：近距离需要高分辨率（每像素<1cm），远距离需要大覆盖范围（数公里）
一张固定纹理不可能同时满足两个需求
```

**问题1：分辨率不足**

4K Shadow Map覆盖5km场景，每像素1.22m。角色脚下的阴影变成马赛克方块，完全不可接受。

**问题2：分辨率浪费**

远距离的建筑不需要高精度阴影（反正看不清），却和近距离角色共享同一张纹理。距离越远，浪费越大。

**问题3：每帧重渲染**

即使场景中99%的物体没有移动，传统Shadow Map每帧都要从光源视角重新渲染整个场景。开放世界百万三角面 × 每帧重渲染 = GPU时间浪费严重。

### 1.2 问题量化

```
一张固定Shadow Map的成本：
├─ 分辨率：1.22m/像素（近距离不可用）
├─ 如果要近距离1cm精度：需要500,000 × 500,000 = 250,000K × 250,000K = 不现实
├─ GPU渲染：每帧重绘所有阴影投射物 = 2-3ms
├─ 静态物体浪费：场景中80%物体是静态的，每帧重绘毫无意义
└─ 内存浪费：远距离分配了高分辨率但实际用不到
```

**核心洞察**：需要一套系统，能够：
1. **近距离高分辨率，远距离低分辨率**（分辨率按需分配）
2. **只重渲染变化的部分**（缓存静态阴影）
3. **支持数公里场景**（虚拟化地址空间）
4. **近距离补全细节**（屏幕空间接触阴影）

这就是UE5阴影系统存在的意义。

---

## 二、解决方案：级联分割 + 虚拟化 + 屏幕空间

### 2.1 三管齐下

UE5用三种互补技术解决Shadow Map的分辨率 vs 覆盖范围矛盾：

```
┌───────────────────────────────────────────────────────────────┐
│ CSM（Cascaded Shadow Maps）                                   │
│ ├─ 解决：近距离高分辨率 + 远距离大覆盖                         │
│ ├─ 原理：将视锥体切分为4个级联，每级独立Shadow Map             │
│ ├─ 适用：DX11+所有平台，中低端设备                             │
│ └─ 代价：级联接缝、每帧重渲染、固定分辨率上限                   │
├───────────────────────────────────────────────────────────────┤
│ VSM（Virtual Shadow Maps）                                    │
│ ├─ 解决：无限分辨率 + 动态缓存 + 无级联接缝                    │
│ ├─ 原理：16K虚拟地址空间 + 128×128物理Page + 按需分配          │
│ ├─ 适用：DX12/Vulkan，推荐方案                                │
│ └─ 代价：128-512MB显存、需要Compute Shader、首帧冷启动         │
├───────────────────────────────────────────────────────────────┤
│ Contact Shadows（屏幕空间接触阴影）                            │
│ ├─ 解决：近距离自阴影细节（手指间、衣褶、凹槽）                │
│ ├─ 原理：在屏幕空间对Depth Buffer做Ray March                  │
│ ├─ 适用：所有平台，作为Shadow Map的补充                        │
│ └─ 代价：屏幕空间信息不完整、远距离无效                         │
└───────────────────────────────────────────────────────────────┘
```

### 2.2 技术选择矩阵

| 场景 | 推荐技术 | 原因 | GPU耗时 |
|------|---------|------|---------|
| 开放世界（平行光） | **VSM Clipmap** | 大场景、动态缓存、无接缝 | 0.5-2.0ms |
| 室内场景（多点光源） | **VSM Per-Light** | 动态分配、内存高效 | 0.5-1.5ms |
| 中低端设备 | **CSM（2-4级联）** | 低显存、全平台兼容 | 2-3ms |
| 电影级质量 | **Ray Traced Shadows** | 物理正确、柔和半影 | 8-15ms |
| 角色近距离细节 | **Contact Shadows** | 自阴影细节、极低成本 | 0.3-0.5ms |

### 2.3 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 默认阴影方案 | VSM（UE5.7默认开启） | 质量最好、缓存高效 |
| 级联分割策略 | 对数+线性混合 | 平衡近/远分辨率分配 |
| Page大小 | 128×128像素 | 平衡颗粒度与管理开销 |
| 缓存策略 | 帧间Page复用 + 失效追踪 | 静态场景80-95%命中 |
| Contact Shadow方法 | Stochastic Jittering + Bend SSS | 质量与性能的平衡 |
| Depth Bias | Constant + Slope自适应 | 自动适配不同级联距离 |

## 三、阴影系统在渲染管线中的位置

### 3.1 管线定位

```
FDeferredShadingSceneRenderer::Render()
    │
    ├── InitViews()
    │   └── SetupShadows()                    ← 阴影设置（CPU）
    │       ├── CreateWholeSceneProjectedShadow()  CSM/VSM创建
    │       ├── CreatePerObjectProjectedShadow()   Per-Object阴影
    │       └── AllocateVirtualShadowMaps()        VSM Page分配
    │
    ├── RenderShadowDepthMaps()               ← ⭐阴影深度渲染（GPU）
    │   ├── RenderShadowDepthMapAtlases()          CSM Atlas渲染
    │   ├── RenderShadowDepthAtlasNanite()         Nanite专用路径
    │   └── RenderVirtualShadowMaps()              VSM渲染
    │
    ├── RenderBasePass()                      ← GBuffer渲染
    │
    ├── RenderLights()                        ← ⭐阴影投影+光照
    │   ├── RenderShadowProjections()              CSM/VSM → 阴影遮罩
    │   ├── RenderScreenSpaceShadows()             Contact Shadows
    │   └── ApplyShadowsToLighting()               阴影 × 光照
    │
    └── PostProcessing()
```

| 阶段 | 输入 | 输出 | GPU耗时 |
|------|------|------|---------|

| RenderShadowDepthMaps | Shadow Info、Mesh列表 | ShadowDepthAtlas / VSM PhysicalPool | 1-2.5ms |
| RenderShadowProjections | ShadowDepth、GBuffer Depth | ScreenShadowMaskTexture（0~1） | 0.5-1.0ms |
| RenderScreenSpaceShadows | SceneDepth、法线 | Contact Shadow Mask | 0.3-0.5ms |

## 四、代价与限制

| 代价 | 影响 | 缓解措施 |
|------|------|---------|
| GPU时间：1.5-4ms/帧 | 60FPS预算16.6ms中的10-25% | VSM缓存降到0.5-1ms |
| 显存：128-512MB（VSM） | 中低端显卡压力大 | r.Shadow.Virtual.MaxPhysicalPages调整 |
| Self-Shadow Acne | Bias过小→表面出现黑色条纹 | Constant+Slope Bias自适应 |
| Peter Panning | Bias过大→阴影与物体分离 | 减小Bias、使用Normal Offset |
| 级联接缝（CSM） | Cascade边界可见锯齿跳变 | Fade过渡、或直接用VSM |
| Contact Shadow屏幕局限 | 屏幕外物体无法投射接触阴影 | 仅用于近距离补充 |

## 一、阴影系统架构总览

### 1.1 核心类关系

```

    │           └── VSM ID
    │
    ├── FVirtualShadowMapArray*                      VSM管理器
    │       ├── PhysicalPagePool                     物理Page池（GPU纹理）
    │       ├── PageTable                            虚拟→物理映射
    │       └── CacheManager                         帧间缓存管理
    │
    └── FShadowSceneRenderer*                        阴影场景渲染器

FSceneRenderer（渲染器）
    │
    ├── VisibleLightInfos[]                          可见光源信息
    │       └── AllProjectedShadows[]                该光源的所有阴影
    │
    ├── SortedShadowsForShadowDepthPass              按类型排序的阴影
    │       ├── ShadowMapAtlases[]                   CSM Atlas列表
    │       ├── ShadowMapCubemaps[]                  点光源CubeMap
    │       └── VirtualShadowMapShadows[]            VSM列表
    │
    └── ScreenSpaceShadowMaskTexture                 阴影遮罩纹理（最终输出）
```

### 1.2 数据流

```
GameThread                  RenderThread                  GPU
────────────────────────────────────────────────────────────────

                            (全屏Pass)                    (采样ShadowMap → 遮罩)
                                                          │
                            RenderLights()            →  Lighting Pass
                            (遮罩×光照)                   (最终光照结果)
```

## 二、FProjectedShadowInfo：阴影的数据载体

### 2.1 设计目标

`FProjectedShadowInfo`是UE5中**所有阴影的统一数据载体**。无论是CSM级联、VSM Page、Per-Object阴影还是CubeMap点光源阴影，都由这个类承载。

设计关键：一个类覆盖所有阴影类型，通过20+个位域标志区分具体类型，避免继承体系的虚函数开销。

### 2.2 核心字段分组

**来源**：`Engine/Source/Runtime/Renderer/Private/ShadowRendering.h`
**源码锚点**：搜索 `class FProjectedShadowInfo`、`SetupWholeSceneProjection`、`RenderDepth`。
**验证状态**：✅已验证

```cpp
class FProjectedShadowInfo : public FRefCountedObject
{
public:
    // ═══════════ 变换矩阵 ═══════════
    FMatrix TranslatedWorldToView;                    // 世界→光源View
    FMatrix ViewToClipInner;                          // View→Clip（内部，紧凑）
    FMatrix ViewToClipOuter;                          // View→Clip（外部，扩展）
    FMatrix TranslatedWorldToClipInnerMatrix;         // 世界→Clip（内部）
    FMatrix TranslatedWorldToClipOuterMatrix;         // 世界→Clip（外部）

    // ═══════════ 几何包围 ═══════════
    float MaxSubjectZ;                                // 投射物最大Z（光源空间）
    float MinSubjectZ;                                // 投射物最小Z
    FConvexVolume CasterOuterFrustum;                 // 投射物外部视锥
    FConvexVolume ReceiverInnerFrustum;               // 接收物内部视锥
    FSphere ShadowBounds;                             // 阴影包围球

    // ═══════════ Atlas分配 ═══════════
    uint32 X, Y;                                      // Atlas中的位置
    uint32 ResolutionX, ResolutionY;                  // 阴影分辨率
    uint32 BorderSize;                                // PCF采样Border

    // ═══════════ 类型标志（位域） ═══════════
    uint32 bAllocated : 1;                            // 已分配
    uint32 bRendered : 1;                             // 已渲染
    uint32 bDirectionalLight : 1;                     // 平行光
    uint32 bOnePassPointLightShadow : 1;              // 单Pass点光源
    uint32 bVSM : 1;                                  // VSM模式
    uint32 bWholeSceneShadow : 1;                     // 全场景阴影（vs Per-Object）
    uint32 bTranslucentShadow : 1;                    // 半透明阴影
    uint32 bNaniteGeometry : 1;                       // 包含Nanite几何体
    uint32 bContainsNaniteSubjects : 1;               // 包含Nanite投射物
    uint32 bPreShadow : 1;                            // 预计算阴影
    uint32 bSelfShadowOnly : 1;                       // 仅自阴影
    uint32 bPerObjectOpaqueShadow : 1;                // Per-Object不透明阴影
    uint32 bTransmission : 1;                         // 支持透射（SSS）
    uint32 bHairStrandsDeepShadow : 1;                // 毛发深度阴影
    uint32 bCapsuleShadow : 1;                        // 胶囊体阴影
    // ... 更多标志

    // ═══════════ VSM信息 ═══════════
    FVirtualShadowMapPerLightCacheEntry* VirtualShadowMapPerLightCacheEntry;
    FVirtualShadowMapClipmap* VirtualShadowMapClipmap;

    // ═══════════ 单Pass点光源 ═══════════
    TArray<FMatrix, TInlineAllocator<6>> OnePassShadowViewProjectionMatrices;

    const FPrimitiveSceneInfo* ParentSceneInfo;        // 父物体（Per-Object专用）
};
```

### 2.3 为什么用位域而不是继承

**位域方案**：20个`uint32 : 1`共享4字节内存，所有阴影类型使用同一个类。

**如果用继承**：

```
IShadow（基类，虚函数）
├── FCSMShadow
├── FVSMShadow
├── FPerObjectShadow
├── FCubeMapShadow
└── FContactShadow

问题：
├─ 虚函数表指针：每个实例+8字节
├─ 虚函数调用：2-3ns/次（分支预测失败）
├─ 无法内联：编译器不能内联虚函数
└─ 数组不连续：基类指针数组→缓存不友好
```

位域方案的优势：
- **零运行时开销**：`if (bVSM)` 是位测试指令，0.5ns
- **缓存友好**：同一类型的数组连续排列
- **编译器优化**：条件分支可被分支预测器学习
- **内存紧凑**：20个标志共享4字节，比20个bool（20字节）节省80%

**关于位域**：详见[2.3 GBuffer](../Phase02_Pipeline/2.3_GBuffer_Deep.md)，此处不再展开。

## 三、CSM：级联分割策略与矩阵计算

### 3.1 级联分割原理

CSM将相机视锥体按深度切分为多个级联（Cascade），每级使用独立的Shadow Map。近距离级联分辨率高，远距离级联覆盖范围大。

```
相机视锥体（俯视图）
                    ┌─────────────────────────────────────────┐
                    │                                         │
         ┌────┐    │    ┌──────────┐    ┌──────────────────┐  │
Camera →─│ C0 │────│────│    C1    │────│       C2         │──│
         └────┘    │    └──────────┘    └──────────────────┘  │
                    │                                         │
                    └─────────────────────────────────────────┘
          0-10m         10-50m              50-200m

Cascade 0: 覆盖 0-10m,   分辨率 2048×2048, 每像素 ~0.5cm
Cascade 1: 覆盖 10-50m,  分辨率 2048×2048, 每像素 ~2cm
Cascade 2: 覆盖 50-200m, 分辨率 2048×2048, 每像素 ~7cm
Cascade 3: 覆盖 200-1000m,分辨率 2048×2048, 每像素 ~40cm

对比一张固定4K Shadow Map：每像素 122cm
Cascade 0的精度是固定方案的 244倍
```

### 3.2 级联距离计算

UE5使用**对数+线性混合**策略分配级联距离。纯对数分割在近距离精度最高但远距离过于稀疏；纯线性分割均匀但近距离不够。混合策略取两者优点。

```
纯对数分割公式：
    D(i) = Near × (Far / Near)^(i / N)

纯线性分割公式：
    D(i) = Near + (Far - Near) × (i / N)

UE5混合公式（λ=0.5~0.9，由r.Shadow.CSM.TransitionScale控制）：
    D(i) = λ × Log(i) + (1-λ) × Linear(i)

示例（N=4, Near=0.1m, Far=1000m, λ=0.7）：
    Cascade 0: 0.1 - 5.6m
    Cascade 1: 5.6 - 31.6m
    Cascade 2: 31.6 - 177.8m
    Cascade 3: 177.8 - 1000m
```

### 3.3 CSM矩阵计算

每个Cascade需要独立的**光源View矩阵** + **正交投影矩阵**。核心流程：

```
对于每个Cascade i：
    1. 从相机Frustum切出 [D(i-1), D(i)] 范围的子Frustum
    2. 计算子Frustum的8个顶点（世界空间）
    3. 将8个顶点变换到光源空间（光源View矩阵）
    4. 计算光源空间AABB
    5. 用AABB构建正交投影矩阵（CSM用正交，无透视）
    6. Snap到Shadow Map像素网格（避免阴影闪烁）
    7. 组合：WorldToShadow = Projection × View
```

**关键优化——Snap到像素网格**：

如果不做Snap，相机每帧微小移动会导致Shadow Map像素网格偏移，阴影边缘出现亚像素级抖动。Snap将投影矩阵对齐到Shadow Map纹素大小的整数倍，消除抖动。

```
TexelWorldSize = (MaxX - MinX) / Resolution
MinX = floor(MinX / TexelWorldSize) * TexelWorldSize
MaxX = ceil(MaxX / TexelWorldSize) * TexelWorldSize
```

### 3.4 Shadow Map Atlas

4个Cascade的Shadow Map打包到一张Atlas纹理中：

```
Shadow Depth Atlas (4096×4096)
┌─────────────┬──────────┐
│             │          │
│   Cascade 0 │ Cascade 1│
│  2048×2048  │ 2048×2048│
│             │          │
├─────────────┼──────────┤
│             │          │
│   Cascade 2 │ Cascade 3│
│  2048×2048  │ 2048×2048│
│             │          │
└─────────────┴──────────┘

格式：PF_ShadowDepth（16位或32位深度）
内存：4096×4096×2字节 = 32MB（16位）
```

## 四、VSM：虚拟地址空间 + 物理Page池 + 动态缓存

### 4.1 VSM解决了CSM的什么问题

| CSM的问题 | VSM的解决 |
|----------|----------|
| 固定分辨率上限（4K Atlas） | 16K×16K虚拟地址空间 |
| 级联接缝 | 统一地址空间，无接缝 |
| 每帧重渲染所有Cascade | 80-95%缓存命中，只渲染变化部分 |
| 固定级联数量 | Clipmap级别自动适配 |
| 远距离分辨率浪费 | 按需分配Page，按距离自适应 |

### 4.2 VSM架构

```
虚拟地址空间（每个VSM 16384×16384 像素）
┌──────────────────────────────────────────┐
│  Page (0,0)  Page (1,0)  Page (2,0)  ... │  ← 每Page 128×128像素
│  Page (0,1)  Page (1,1)  Page (2,1)  ... │
│  ...                                     │
│  128×128 Pages = 16384×16384 像素         │
└──────────────────────────────────────────┘
                    │
                    │ Page Table（虚拟→物理映射）
                    ▼
Physical Page Pool（实际GPU显存）
┌────────────────────────────────────────────────────┐
│ [Page A] [Page B] [Page C] [Page D] [Free] [Free] │
│  128×128  128×128  128×128  128×128                │
│  已缓存   已缓存   需渲染   已缓存                   │
└────────────────────────────────────────────────────┘
Pool大小：r.Shadow.Virtual.MaxPhysicalPages × 128×128 × 2B/像素
典型：8192 Pages = 256MB
```

### 4.3 按需分配

不是所有虚拟Page都需要物理内存。只有**相机可见区域投影到Shadow Map空间**后覆盖的Page才会被分配。

```
示例：16384×16384虚拟空间 = 128×128 = 16384个Page
实际可见区域覆盖的Page数量 ≈ 500-2000个（取决于场景复杂度）
分配率 = 2000 / 16384 ≈ 12%

节省 88% 的物理内存
```

### 4.4 动态缓存

VSM的核心性能优势来自**帧间缓存复用**。静态物体的阴影深度在首次渲染后缓存到物理Page中，后续帧直接复用。

```
帧间缓存流程：
Frame N:                              Frame N+1:
┌──────────────┐                      ┌──────────────┐
│ Page A: 渲染  │  ───缓存──────→     │ Page A: 复用  │ ← 静态，直接用
│ Page B: 渲染  │  ───缓存──────→     │ Page B: 复用  │ ← 静态，直接用
│ Page C: 渲染  │  ───失效──────→     │ Page C: 重渲染│ ← 有物体移动
│ Page D: 渲染  │  ───缓存──────→     │ Page D: 复用  │ ← 静态，直接用
└──────────────┘                      └──────────────┘

缓存命中率：75%（3/4 Page复用）
GPU节省：3/4 × 原始渲染时间

典型缓存命中率：
├─ 纯静态场景：95-98%
├─ 半静态（角色移动）：85-90%
├─ 动态场景（车辆行驶）：60-75%
└─ 完全动态：20-30%
```

### 4.5 Clipmap（平行光专用）

平行光使用Clipmap代替CSM级联：围绕相机位置的多层级正方形区域，每层级覆盖范围翻倍、分辨率减半。

```
Clipmap层级（俯视图，以相机为中心）：

        Level 2: 200m×200m（低细节）
    ┌─────────────────────────────┐
    │   Level 1: 50m×50m（中细节）│
    │   ┌─────────────────┐       │
    │   │ Level 0: 10m×10m│       │
    │   │ （高细节，每像素  │       │
    │   │   ~0.6mm）      │       │
    │   └─────────────────┘       │
    └─────────────────────────────┘

相机移动时：
├─ 新暴露的区域：分配新Page，渲染深度
├─ 已离开的区域：Page回收到Free Pool
└─ 不变的区域：直接复用缓存（80-95%命中）
```

## 五、Contact Shadows：屏幕空间Ray March

### 5.1 为什么Shadow Map不够

即使是最高分辨率的VSM，Shadow Map从光源视角渲染，某些微小凹槽的自阴影依然无法正确表达：

```
光源方向
    ↘
     ↘       物体表面
      ↘   ┌──┐
       ↘  │  │  ← 凹槽（手指间隙、衣褶）
        ↘ └──┘
         ↘─────── 表面

Shadow Map从光源看：凹槽太小，只有1-2个像素，无法产生正确阴影
Contact Shadow从相机看：在屏幕空间对Depth Buffer做短距离Ray March
    可以捕获凹槽的自阴影细节
```

### 5.2 实现原理

**来源**：`Engine/Source/Runtime/Renderer/Private/Shadows/ScreenSpaceShadows.cpp`
**源码锚点**：搜索 `RenderScreenSpaceShadows`、`FScreenSpaceShadowsCS`、`FScreenSpaceShadowsBendCS`。
**验证状态**：✅已验证

UE5.7提供两种Contact Shadow算法：

```
enum class ContactShadowsMethod
{
    StochasticJittering,    // 随机抖动Ray March（默认）
    BendSSS                 // Bend Screen-Space Shadows（高质量）
};
```

**StochasticJittering**：沿光源方向在屏幕空间步进，每步检查Depth Buffer。加入随机偏移避免带状瑕疵。

**BendSSS**：更高级的屏幕空间阴影算法，使用弯曲光线和多方向采样，质量更高但开销更大。

```
Contact Shadow Ray March流程：
    1. 从像素世界坐标出发
    2. 沿光源方向在屏幕空间步进（8-16步）
    3. 每步将步进点投影到屏幕空间
    4. 比较步进点深度与Depth Buffer深度
    5. 如果步进点比Depth Buffer更远 → 被遮挡 → 阴影
    6. 加入时间抖动（Temporal Jittering）减少噪声

参数：
├─ ContactShadowLength：步进长度（世界空间或屏幕空间）
├─ ContactShadowCastingIntensity：投射强度
└─ ContactShadowNonCastingIntensity：非投射强度
```

## 六、Shadow Depth Bias：Self-Shadow Acne vs Peter Panning

### 6.1 Self-Shadow Acne是什么

```
光源方向
    ↘
     ↘     Shadow Map像素边界
      ↘    │
       ↘   │  表面
────────↘──│──────────────────
        A↘ │B
           ↘
            ↘

像素A和像素B在Shadow Map中记录不同深度值
但在实际表面上，A和B深度几乎相同
由于Shadow Map分辨率有限，像素B被"自己的阴影"遮挡 → 出现条纹状黑色瑕疵
```

### 6.2 Bias策略

**Constant Bias**：将Shadow Map深度值整体偏移一个固定量

```
原始深度：ShadowDepth = LightSpaceZ
偏移后：  ShadowDepth = LightSpaceZ - ConstantBias
```

**Slope Bias**：根据表面相对于光源的斜率动态调整偏移量。斜率越大（表面越斜对光源），需要的偏移越大。

```
Slope = max(abs(ddx(depth)), abs(ddy(depth)))
ShadowDepth = LightSpaceZ - ConstantBias - Slope × SlopeBias
```

### 6.3 Bias参数平衡

| 参数 | 过小 | 过大 | 推荐范围 |
|------|------|------|---------|
| ConstantBias | Self-Shadow Acne（条纹瑕疵） | Peter Panning（阴影悬空） | 0.001-0.01 |
| SlopeBias | 斜面Self-Shadow | 斜面阴影分离 | 1.0-3.0 |

UE5根据CSM级联距离自动缩放Bias：远距离级联的Bias更大（因为投影范围更大，深度精度更低）。

```
实际Bias = 基础Bias × 2^CascadeIndex
Cascade 0: Bias × 1
Cascade 1: Bias × 2
Cascade 2: Bias × 4
Cascade 3: Bias × 8
```

## 七、模板策略模式：光源类型的编译期分派

### 7.1 设计需求

ShadowDepthPass需要为不同光源类型（平行光、点光源、聚光灯）设置不同的Shader参数和渲染状态。如果用运行时`if`或虚函数切换，每个DrawCall都有分支开销。

### 7.2 UE5的模板策略实现

**来源**：`Engine/Source/Runtime/Renderer/Private/ShadowDepthRendering.cpp`
**源码锚点**：搜索 `GetShadowDepthPassShaders`、`FDirectionalLightShadowPolicy`、`FPointLightShadowPolicy`。
**验证状态**：✅已验证

```cpp
// UE5为不同光源类型定义Policy
struct FDirectionalLightShadowPolicy
{
    static constexpr bool bUsePointLightProjection = false;
    static constexpr bool bUseOnePassPointLight = false;
};

struct FPointLightShadowPolicy
{
    static constexpr bool bUsePointLightProjection = true;
    static constexpr bool bUseOnePassPointLight = false;
};

struct FOnePassPointLightShadowPolicy
{
    static constexpr bool bUsePointLightProjection = true;
    static constexpr bool bUseOnePassPointLight = true;
};

// 模板化的Shader参数设置
template<typename LightPolicy>
void SetShadowDepthShaderParameters(FRHICommandList& RHICmdList, ...)
{
    if constexpr (LightPolicy::bUseOnePassPointLight)
    {
        // 点光源CubeMap专用参数（6个View矩阵）
        SetShaderValue(RHICmdList, ..., OnePassShadowViewProjectionMatrices);
    }

    if constexpr (LightPolicy::bUsePointLightProjection)
    {
        // 点光源投影参数
        SetShaderValue(RHICmdList, ..., LightPosition);
    }
    else
    {
        // 平行光参数（正交投影）
        SetShaderValue(RHICmdList, ..., ShadowDepthParams);
    }
}
```

**编译器优化效果**：

```
对 FDirectionalLightShadowPolicy 实例化后：
├─ bUseOnePassPointLight = false → OnePassShadow代码块被编译器删除

└─ 只保留平行光参数设置代码，零分支、完全内联

对 FOnePassPointLightShadowPolicy 实例化后：
├─ bUseOnePassPointLight = true → 保留CubeMap参数代码
├─ bUsePointLightProjection = true → 保留LightPosition代码
└─ 平行光参数代码被删除
```

**性能对比**：

| 方法 | 每次调用开销 | 可否内联 |
|------|------------|---------|
| 虚函数 | 2-3ns | ❌ |
| 运行时 if | 0.5-1ns | ✅（但有分支） |
| **模板策略（constexpr if）** | **0ns** | **✅（完全内联，零分支）** |

## 一、SetupWholeSceneProjection()：CSM矩阵设置

### 1.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/ShadowSetup.cpp`

### 1.2 核心流程

```cpp
void FProjectedShadowInfo::SetupWholeSceneProjection(
    FLightSceneInfo* InLightSceneInfo,
    FViewInfo* InDependentView,
    const FWholeSceneProjectedShadowInitializer& Initializer,
    uint32 InResolutionX,
    uint32 InResolutionY,
    uint32 InBorderSize
)
{
    // 【步骤1】初始化基础信息
    LightSceneInfo = InLightSceneInfo;
    DependentView = InDependentView;
    ResolutionX = InResolutionX;
    ResolutionY = InResolutionY;
    BorderSize = InBorderSize;
    bWholeSceneShadow = true;

    // 【步骤2】从Initializer获取预计算数据
    // Initializer由上层CreateWholeSceneProjectedShadow()计算好：
    //   - WorldToLight矩阵
    //   - Frustum顶点（8个，已变换到光源空间）
    //   - Subject和Receiver的Z范围
    PreShadowTranslation = Initializer.PreShadowTranslation;

    // 【步骤3】计算光源空间AABB
    // 从Initializer获取光源空间的Frustum包围盒
    // 这个包围盒定义了正交投影矩阵的6个平面
    FBox LightSpaceBounds = Initializer.SubjectBounds;

    // 【步骤4】Snap到像素网格（消除阴影抖动）
    float TexelSizeX = (LightSpaceBounds.Max.X - LightSpaceBounds.Min.X) / InResolutionX;
    float TexelSizeY = (LightSpaceBounds.Max.Y - LightSpaceBounds.Min.Y) / InResolutionY;

    LightSpaceBounds.Min.X = FMath::FloorToFloat(LightSpaceBounds.Min.X / TexelSizeX) * TexelSizeX;
    LightSpaceBounds.Max.X = FMath::CeilToFloat(LightSpaceBounds.Max.X / TexelSizeX) * TexelSizeX;
    LightSpaceBounds.Min.Y = FMath::FloorToFloat(LightSpaceBounds.Min.Y / TexelSizeY) * TexelSizeY;
    LightSpaceBounds.Max.Y = FMath::CeilToFloat(LightSpaceBounds.Max.Y / TexelSizeY) * TexelSizeY;

    // 【步骤5】构建正交投影矩阵（反向Z）
    // UE5使用反向Z：Near=1.0, Far=0.0，浮点精度在近距离更高
    FMatrix ProjectionMatrix = FReversedZOrthoMatrix(
        LightSpaceBounds.Min.X, LightSpaceBounds.Max.X,
        LightSpaceBounds.Min.Y, LightSpaceBounds.Max.Y,
        LightSpaceBounds.Min.Z, LightSpaceBounds.Max.Z
    );

    // 【步骤6】组合最终变换矩阵
    TranslatedWorldToView = Initializer.WorldToLight;
    ViewToClipOuter = ProjectionMatrix;
    TranslatedWorldToClipOuterMatrix = ProjectionMatrix * Initializer.WorldToLight;

    // 【步骤7】记录Subject Z范围
    MaxSubjectZ = Initializer.MaxSubjectZ;
    MinSubjectZ = Initializer.MinSubjectZ;

    // 【步骤8】设置阴影包围球（用于后续剔除）
    ShadowBounds = Initializer.ShadowBounds;
}
```

### 1.3 关键细节

**为什么用PreShadowTranslation？**

开放世界中物体坐标可能很大（如10000, 20000, 5000）。32位float在万位量级的精度约为0.001，而矩阵乘法会放大这个误差。PreShadowTranslation将坐标平移到原点附近，将精度提升到0.0000001级别。

**为什么CSM用正交投影？**

平行光的光线平行，不存在透视效果。正交投影保证光线方向一致，深度值线性分布。而点光源使用透视投影（CubeMap的6个面各一个透视矩阵）。

### 3.2 VSM渲染核心流程

```cpp
void FVirtualShadowMapArray::RenderVirtualShadowMaps(
    FRDGBuilder& GraphBuilder,
    const TArray<FViewInfo>& Views
)
{
    // 【阶段1】Page标记（GPU Compute）
    // 对每个屏幕像素：
    //   1. 从GBuffer读取世界坐标
    //   2. 投影到VSM虚拟地址空间
    //   3. 计算需要的Mip Level（基于距离）
    //   4. 标记对应的虚拟Page为"需要"
    MarkPages(GraphBuilder, Views);

    // 【阶段2】Page分配
    // 遍历所有标记为"需要"的Page：
    //   1. 检查上一帧缓存是否有效 → 复用（80-95%命中）
    //   2. 缓存无效 → 从Free Pool分配新物理Page
    //   3. Free Pool为空 → 驱逐LRU Page
    AllocatePhysicalPages(GraphBuilder);

    // 【阶段3】渲染需要更新的Page
    // 只渲染缓存未命中的Page（仅5-20%）
    //   Nanite路径：GPU-Driven Cluster光栅化
    //   非Nanite路径：传统DrawCall + MeshPassProcessor
    RenderPages(GraphBuilder, Views);

    // 【阶段4】构建HZB（Hierarchical Z-Buffer）
    // 从物理Page池生成Mip Chain
    // 用于后续帧的遮挡剔除
    BuildHZB(GraphBuilder);

    // 【阶段5】合并缓存
    // 将当前帧的Page状态写入缓存
    // 供下一帧查询
    MergeStaticCaching(GraphBuilder);
}
```

### 3.3 Page分配算法

```
AllocatePhysicalPages() 流程：

对每个标记为"需要"的虚拟Page：
    │
    ├── 查PageTable：已分配物理Page？
    │   └── 是 → 返回（直接复用）
    │
    ├── 查CacheManager：上一帧有缓存？
    │   ├── 缓存有效（物体未移动）？
    │   │   └── 是 → 复用缓存Page，更新统计，返回
    │   └── 缓存失效（物体移动了）？
    │       └── 是 → 标记失效，继续分配新Page
    │
    ├── 从AllocationBitmask查找空闲Page
    │   └── 找到？
    │       ├── 是 → 分配，更新PageTable，标记NEEDS_RENDER
    │       └── 否 → Page池已满，执行LRU驱逐
    │           ├── 遍历所有Page，找最久未使用的
    │           ├── 清除其PageTable映射
    │           ├── 标记为空闲
    │           └── 分配给当前请求
    │
    └── 更新统计信息
```

### 3.4 VSM关键数据结构

**来源**：`Engine/Source/Runtime/Renderer/Private/VirtualShadowMaps/VirtualShadowMapArray.h`

```
FVirtualShadowMapArray 关键字段：

├── PhysicalPagePool（FRDGTexture）
│   格式：PF_R32_FLOAT（32位深度）
│   大小：取决于MaxPhysicalPages
│   典型：4096×4096×N Pages = 128-512MB
│
```

```hlsl
float GetVirtualShadowMapShadowFactor(float3 WorldPosition, ...)
{
    // 【步骤1】世界坐标→VSM虚拟地址
    float2 VirtualUV = WorldToVirtualShadowMapUV(WorldPosition, LightDirection);

    // 【步骤2】计算Mip级别（基于距离）
    float MipLevel = ComputeVSMMipLevel(DistanceToCamera);

    // 【步骤3】查PageTable，获取物理Page地址
    uint PhysicalAddress = PageTable[VirtualPageIndex];
    if (PhysicalAddress == INVALID_PAGE)
        return 1.0; // 无Page → 无阴影

    // 【步骤4】将虚拟UV转换为物理UV
    float2 PhysicalUV = VirtualToPhysicalUV(VirtualUV, PhysicalAddress);

    // 【步骤5】采样深度并比较
    float ShadowDepth = PhysicalPagePool.Sample(PhysicalUV);
    float ReceiverDepth = WorldToLightZ(WorldPosition);

    // 【步骤6】返回阴影因子
    return (ReceiverDepth < ShadowDepth) ? 1.0 : 0.0; // 1=光照，0=阴影
}
```

### 4.2 PCF（Percentage Closer Filtering）

Shadow Map采样的核心问题：直接比较深度值产生硬边阴影（像素级锯齿）。PCF在采样点周围多次采样并取平均，产生柔和的阴影边缘。

```hlsl
// PCF 3×3采样（简化版）
float PCFShadow3x3(Texture2D ShadowMap, SamplerComparisonState Sampler, float3 ShadowCoord)
{
    float Shadow = 0.0;
    float TexelSize = 1.0 / ShadowMapResolution;

    // 3×3采样网格
    for (int y = -1; y <= 1; y++)
    {
        for (int x = -1; x <= 1; x++)
        {
            float2 Offset = float2(x, y) * TexelSize;

            // SampleCmpLevelZero：硬件比较采样
            // 比较ShadowCoord.z（像素深度）与ShadowMap中的深度
            // 返回0（在阴影中）或1（在光照中）
            Shadow += ShadowMap.SampleCmpLevelZero(
                Sampler,
                ShadowCoord.xy + Offset,
                ShadowCoord.z
            );
        }
    }

    return Shadow / 9.0; // 平均9个采样点
}
```

**UE5实际使用的是优化版PCF**（`ShadowPercentageCloserFiltering.ush`），利用硬件双线性过滤在5×5区域用4次采样实现等效效果。

### 4.3 PCSS（Percentage Closer Soft Shadows）

PCSS在PCF基础上增加**Blocker Search**步骤，根据遮挡物到接收面的距离动态调整PCF半径，产生物理正确的半影效果（近处硬阴影，远处软阴影）。

```
PCSS流程：
    1. Blocker Search：在Shadow Map中搜索遮挡物
       采样NxN区域，找到所有深度<像素深度的texel
       计算平均遮挡物深度 = AvgBlockerDepth

    2. Penumbra计算：
       PenumbraWidth = LightSize × (ReceiverDepth - AvgBlockerDepth) / AvgBlockerDepth
       （物理正确：遮挡物越远→半影越大）

    3. PCF采样：
       使用PenumbraWidth作为PCF半径
       近距离遮挡：小半径→硬阴影
       远距离遮挡：大半径→软阴影
```

**性能对比**：

| 方法 | 采样数 | GPU耗时/像素 | 质量 |
|------|--------|-------------|------|
| 硬阴影（1次采样） | 1 | ~1ns | 锯齿严重 |
| PCF 3×3 | 9 | ~4ns | 柔和边缘 |
| PCF 5×5（优化） | 4（双线性） | ~3ns | 柔和边缘（推荐） |
| PCSS | 32-64 | ~15-30ns | 物理正确半影 |

## 三、性能优化建议

### 3.1 VSM优化（推荐方案）

| 操作 | 命令 | 预期节省 |
|------|------|---------|
| 确保VSM已启用 | r.Shadow.Virtual.Enable 1 | CSM→VSM可节省1-2ms |
| 启用缓存 | r.Shadow.Virtual.Cache 1 | 缓存命中可节省80% |
| 调整Page池大小 | r.Shadow.Virtual.MaxPhysicalPages | 根据显存调整 |

### 3.2 CSM优化（低端设备）

| 操作 | 命令 | 预期节省 |
|------|------|---------|
| 减少级联数 | r.Shadow.CSM.MaxCascades 2 | 节省1-1.5ms |
| 降低分辨率 | r.Shadow.MaxResolution 2048 | 节省0.5-1ms |
| 缩短阴影距离 | r.Shadow.DistanceScale 0.5 | 节省0.5ms |

### 3.3 通用优化

| 操作 | 说明 | 预期节省 |
|------|------|---------|
| 减少阴影投射物 | 远距离小物体关闭Cast Shadow | 减少DrawCall |
| LOD for shadows | 远距离用低LOD渲染阴影 | 减少顶点数 |
| 关闭半透明阴影 | r.Shadow.TranslucentShadow 0 | 节省0.2-0.5ms |
| Contact Shadow适度使用 | 只对主光源启用 | 节省0.2ms |

### 3.4 优化案例

**场景**：开放世界，5km×5km，200+光源，目标60FPS

```
优化前（CSM模式）：
├─ Shadow Depth: 2.5ms
├─ Shadow Projection: 0.8ms
├─ Contact Shadows: 0.5ms
└─ 总计：3.8ms

优化步骤：
1. 启用VSM + 缓存 → Shadow Depth: 2.5ms → 0.5ms     (-2.0ms)
2. 减少Per-Object Shadow → Projection: 0.8ms → 0.5ms  (-0.3ms)
3. Contact Shadow只对平行光 → 0.5ms → 0.3ms            (-0.2ms)

优化后（VSM模式）：
├─ Shadow Depth: 0.5ms
├─ Shadow Projection: 0.5ms
├─ Contact Shadows: 0.3ms
└─ 总计：1.3ms

节省：2.5ms → 可用于其他特效或提升帧率
```

## 一、与Nanite的关系

### 1.1 Nanite Shadow深度渲染

Nanite几何体在ShadowDepth渲染时走专用GPU-Driven路径：

```
传统Mesh → FShadowDepthPassMeshProcessor → DrawCall → VS/PS管线
Nanite Mesh → Nanite::RasterBins → GPU Compute → Cluster光栅化 → 直接写深度
```

**优势**：
- Nanite的Cluster粒度LOD在阴影渲染时也生效，远距离自动降低三角面
- GPU-Driven剔除：在ShadowDepth渲染中也执行Instance Culling和Cluster Culling
- 无DrawCall瓶颈：百万Cluster在一次Dispatch中完成

### 1.2 VSM + Nanite协同

VSM的Page渲染天然适配Nanite：

```
每个需要渲染的VSM Page → 相当于一个小视口（128×128）
Nanite可以为每个Page独立执行GPU Culling
只渲染该Page视锥内可见的Cluster

结果：
├─ 传统Mesh：每个Page需要独立的DrawCall集合
└─ Nanite：所有Page的Cluster在一次Dispatch中批量处理
```

---

## 二、与MegaLights的关系

### 2.1 大规模光源的阴影策略

MegaLights处理数百上千个光源时，不可能为每个光源都渲染完整的Shadow Map。MegaLights使用**随机采样 + VSM**策略：

```
MegaLights阴影流程：
1. 随机选择一小批光源（每像素4-8个）
2. 查询VSM获取阴影（复用VSM缓存）
3. 时空降噪（SVGF）平滑随机采样结果
```

VSM的缓存机制对MegaLights至关重要——如果每个光源每帧重渲染阴影，数百个光源的开销不可接受。

详见[3.4 MegaLights](../Phase03_UE5Features/3.4_MegaLights_Deep.md)。

---

## 三、与RenderGraph的关系

### 3.1 阴影Pass在RDG中的声明

阴影渲染的所有Pass通过RenderGraph声明，享受RDG的自动资源管理和屏障优化：

```
RDG声明阶段：
├─ ShadowDepthTexture = GraphBuilder.CreateTexture(...)
├─ AddPass("ShadowDepth Cascade0", ..., [](FRHICommandList& RHICmdList) { ... })
├─ AddPass("ShadowDepth Cascade1", ...)
├─ AddPass("VSM MarkPages", ...)
├─ AddPass("VSM AllocatePages", ...)
├─ AddPass("VSM RenderPages", ...)
├─ AddPass("ShadowProjection", ...)
└─ AddPass("ContactShadows", ...)

RDG自动处理：
├─ 资源状态转换（DepthWrite → ShaderResource）
├─ Transient Aliasing（ShadowDepthTexture可被后续Pass复用内存）
└─ 依赖图排序（确保ShadowDepth在ShadowProjection之前执行）
```

详见[2.5 RenderGraph](../Phase02_Pipeline/2.5_RenderGraph_Deep.md)。
