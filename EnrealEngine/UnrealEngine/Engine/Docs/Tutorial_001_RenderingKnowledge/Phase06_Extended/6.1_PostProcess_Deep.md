# 第一部分：理解后处理系统的存在意义

> **核心目标**：从渲染管线输出的HDR线性颜色不可直接显示的问题出发，理解后处理系统解决了什么问题、付出了什么代价

---

## 一、线性HDR颜色的显示困境

### 1.1 问题场景

BasePass + Lighting计算完成后，SceneColor纹理中存储的是**HDR线性颜色**（格式R11G11B10F），值域从0到65504。但显示器只能显示0-1范围的sRGB颜色。直接截断会怎样？

```
SceneColor（HDR线性）：
├─ 太阳直射区域：亮度 = 50000.0（远超显示器范围）→ 截断到1.0 = 纯白
├─ 室内暗处：亮度 = 0.002（太暗）→ 截断到0.002 = 几乎全黑
├─ 正常皮肤：亮度 = 0.8（看似正常）→ 但线性颜色看起来偏暗（缺少Gamma校正）
└─ 高光反射：亮度 = 120.0 → 截断到1.0 = 丢失所有反射细节

结果：高亮区域过曝成纯白块，暗部全黑无细节，中间调偏暗
```

**问题1：动态范围丢失**

HDR场景的亮度比可达1:100000+。显示器的亮度比约1:1000。直接映射会丢失大量高光和暗部细节。

**问题2：缺少知觉映射**

人眼对亮度的感知是非线性的（Weber-Fechner定律）。线性空间的0.5看起来不是"中灰"，而是偏暗。需要Gamma/sRGB校正才能匹配人眼感知。

**问题3：缺少影视级视觉效果**

真实相机拍摄的画面包含：镜头眩光（Bloom）、景深虚化（DOF）、运动模糊（Motion Blur）、胶片颗粒（Film Grain）、色彩调校（Color Grading）等。裸渲染输出没有这些效果，画面看起来"CG感"很重。

### 1.2 问题量化

```
直接显示HDR线性颜色的代价：
├─ 高光丢失：所有>1.0的像素变成纯白 → 约15-30%屏幕区域（有直射光时）
├─ 暗部丢失：所有<0.01的像素近似纯黑 → 约10-20%屏幕区域（室内场景）
├─ 色彩偏暗：线性空间0.5的感知亮度≈0.22（sRGB下）→ 整体画面偏暗
├─ 无辉光效果：高亮物体周围没有扩散光晕 → 缺少真实感
└─ 无自动曝光：从室内走到室外不会有亮度适应过程 → 不自然
```

**核心洞察**：需要一套系统，能够：
1. **将HDR映射到显示器范围**（Tone Mapping）：保留高光和暗部细节
2. **模拟相机/人眼特性**（Bloom、DOF、Motion Blur）：增加真实感
3. **艺术家控制色彩**（Color Grading）：实现影视级画面风格
4. **自动适应亮度变化**（Eye Adaptation）：室内外过渡自然
5. **允许插入自定义效果**（Blendable Material）：项目级风格化

这就是UE5后处理系统存在的意义。

---

## 二、解决方案：后处理管线的四个阶段

### 2.1 从HDR到最终输出

UE5的后处理管线按时间顺序分为四个阶段，每个阶段解决不同问题：

```
┌───────────────────────────────────────────────────────────────┐
│ 阶段1：几何空间效果（Pre-Tonemap，HDR线性空间）               │
│ ├─ 解决：需要深度/运动信息的2.5D效果                          │
│ ├─ 内容：Motion Blur → DOF → 自定义材质(BeforeDOF/AfterDOF)  │
│ ├─ 输入格式：HDR R11G11B10F，值域0-65504                     │
│ └─ 特点：在TAA之前，渲染分辨率执行                            │
├───────────────────────────────────────────────────────────────┤
│ 阶段2：时域处理 + Bloom（HDR线性空间）                         │
│ ├─ 解决：抗锯齿、超分辨率、辉光扩散                           │
│ ├─ 内容：TAA/TSR → Downsample → Bloom → Eye Adaptation       │
│ ├─ 特点：TAA/TSR可能改变分辨率（Primary → Display）           │
│ └─ 自定义材质(BeforeBloom)在Bloom前插入                       │
├───────────────────────────────────────────────────────────────┤
│ 阶段3：Tone Mapping（HDR → LDR转换点）                        │
│ ├─ 解决：HDR到显示器范围的映射                                │
│ ├─ 内容：CombineLUT → Tonemap(ACES + LUT + Film Grain)       │
│ ├─ 输出格式：LDR A2B10G10R10 或 RGBA8，值域0-1               │
│ └─ 自定义材质(ReplacingTonemapper)可替换整个Tonemap           │
├───────────────────────────────────────────────────────────────┤
│ 阶段4：最终处理（Post-Tonemap，LDR空间）                       │
│ ├─ 解决：空域抗锯齿、UI合成、最终风格化                       │
│ ├─ 内容：FXAA/SMAA → 自定义材质(AfterTonemapping) → Upscale  │
│ ├─ 输入格式：LDR 0-1（已Gamma/sRGB校正）                     │
│ └─ 特点：最常用的自定义材质插入点                              │
└───────────────────────────────────────────────────────────────┘
```

### 2.2 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| DOF在TAA前执行 | 渲染分辨率 | DOF需要精确深度，TAA后分辨率可能变化 |
| Bloom在Tonemap前执行 | HDR空间 | 高亮区域的扩散需要>1.0的值参与计算 |
| Color Grading用3D LUT | 32×32×32纹理 | Texture3D单次采样实现任意颜色映射，性能优于数学公式 |
| Film Grain用Halton序列 | 低差异伪随机 | 比rand()更均匀，TAA友好 |
| 自定义材质通过Blendable插入 | 7个固定位置 | 可控性强，避免用户在错误位置插入导致问题 |

## 三、后处理系统在渲染管线中的位置

### 3.1 调用链定位

```
FDeferredShadingSceneRenderer::Render()
    ├─ RenderPrePass()               ← Depth Prepass
    ├─ RenderBasePass()              ← GBuffer填充
    ├─ RenderLighting()              ← 延迟光照 → 输出HDR SceneColor
    ├─ RenderTranslucency()          ← 半透明渲染
    └─ AddPostProcessingPasses()     ← 【后处理入口】本章重点
        ├─ DOF/MotionBlur/自定义材质
        ├─ TAA/TSR
        ├─ Bloom/EyeAdaptation
        ├─ Tonemap
        ├─ FXAA/SMAA/自定义材质
        └─ PrimaryUpscale → ViewFamilyTexture（最终输出）
```

**关键点**：后处理是渲染管线的最后阶段，输入是完成所有3D渲染后的HDR SceneColor，输出是可直接显示的LDR画面。

### 3.2 输入输出

| 方向 | 数据 | 格式 | 来源/去向 |
|------|------|------|----------|
| **输入** | SceneColor | R11G11B10F HDR | Lighting Pass输出 |
| **输入** | SceneDepth | D32F | Depth Prepass |
| **输入** | GBufferVelocity | RG16F | BasePass运动矢量 |
| **输入** | CustomDepth/Stencil | D32F + S8 | 自定义深度Pass |
| **输出** | ViewFamilyTexture | A2B10G10R10 LDR | 送显/Backbuffer |

## 四、代价与限制

### 4.1 性能开销

| Pass | 示例耗时趋势 | 占比趋势 | 是否可禁用 |
|------|---------------|----------|----------|
| DOF | 示例中通常较高 | 高 | `r.DepthOfFieldQuality 0` 可用于验证影响 |
| Motion Blur | 示例中等 | 中 | `r.MotionBlur.Max 0` 可用于验证影响 |
| TAA/TSR | 示例中等 | 中 | 可禁用验证，但画质和稳定性会变化 |
| Bloom | 示例中通常较高 | 高 | `r.BloomQuality 0` 可用于验证影响 |
| Tonemap | 示例较低但通常保留 | 低到中 | HDR→LDR路径通常需要保留 |
| FXAA | 示例较低 | 低 | TAA/TSR已覆盖时可按项目验证 |
| 自定义材质 | 随材质复杂度增长 | 变长 | 按数量、采样和分支实测 |
| **总计** | **随分辨率和效果组合变化** | **100%** | 优化幅度需项目实测 |

### 4.2 设计限制

1. **BlendableLocation是固定的**：只有7个插入点，不能在任意位置插入自定义Pass
2. **全屏像素级**：后处理是O(Width×Height)计算，分辨率对性能影响是平方关系
3. **2D信息不完整**：屏幕空间只有可见像素的信息，被遮挡的像素无法访问
4. **执行顺序固定**：Pass顺序由引擎硬编码，不能通过配置改变

💡 **核心要点**：后处理系统将HDR线性渲染结果转换为可显示的LDR画面，同时添加影视级视觉效果。性能开销随分辨率、平台、开启的Pass和自定义材质复杂度变化，Bloom、DOF、TAA/TSR、自定义材质通常是优先检查对象。

### 1.3 设计模式

**1. RenderGraph声明式（核心）**

所有Pass通过RDG API声明，自动管理资源生命周期和依赖：

```cpp
// 典型Pass签名
FScreenPassTexture AddXXXPass(
    FRDGBuilder& GraphBuilder,      // RDG构建器
    const FViewInfo& View,          // 视图信息
    const FXXXInputs& Inputs)       // Pass专用输入
{
    // 声明参数、创建纹理、添加Pass
    // RDG自动处理依赖和资源回收
    return OutputTexture;
}
```

**2. Chain of Responsibility（材质链）**

同一Location的多个材质按Priority排序后链式执行：

```cpp
FScreenPassTexture Output = SceneColor;
for (const UMaterialInterface* Material : MaterialChain)
{
    Output = AddPostProcessMaterialPass(GraphBuilder, View, Inputs, Material);
}
```

**3. Strategy（算法选择）**

根据配置动态选择不同实现（FFT Bloom vs Gaussian Bloom、TSR vs TAA vs ThirdParty等）。

## 二、BlendableLocation：材质插入点系统

### 2.1 枚举定义

**来源**：`Engine/Source/Runtime/Engine/Classes/Engine/BlendableInterface.h`（搜索 token：`enum EBlendableLocation`、`BL_SceneColorBeforeDOF`、`BL_SceneColorAfterTonemapping`）

### 2.3 插入点选择指南

| Location | 输入格式 | 分辨率 | 典型用途 | 性能影响 |
|----------|---------|--------|---------|---------|
| **BeforeDOF** | HDR线性 | 渲染分辨率 | 全息扫描线、3D空间扭曲（需要精确深度） | 低 |
| **AfterDOF** | HDR线性+虚化 | 渲染分辨率 | 屏幕水滴、利用DOF虚化效果 | 低 |
| **TranslucencyAfterDOF** | 半透明层 | 渲染分辨率 | 修改半透明层本身（特殊） | 低 |
| **SSRInput** | HDR线性 | 显示分辨率 | SSR背板合成（VR/绿幕） | 低 |
| **BeforeBloom** | HDR线性 | 显示分辨率 | 控制Bloom输入（只让特定物体发光） | 低 |
| **ReplacingTonemapper** | HDR线性 | 显示分辨率 | 完全自定义Tone Mapping算法 | 高 |
| **AfterTonemapping** | LDR sRGB | 显示分辨率 | **最常用**：颜色调整、描边、Vignette | 通常较低 |

**选择原则**：
- 大多数情况优先评估`AfterTonemapping`（LDR处理简单，通常成本更低）
- 需要HDR值（>1.0）参与计算时选`BeforeBloom`
- 需要精确深度信息时选`BeforeDOF`
- 完全自定义色调映射时选`ReplacingTonemapper`

### 2.4 重要注意事项

**BeforeDOF/AfterDOF在渲染分辨率执行**：如果启用TSR，渲染分辨率可能是显示分辨率的50-85%。材质在低分辨率下执行，效果会被TSR放大。

**ReplacingTonemapper只执行最高优先级的1个材质**：不像其他Location可以链式执行多个材质。这是因为Tonemap负责HDR→LDR转换，多个替换互相矛盾。

**UE5.4重命名**：旧名`BL_BeforeTranslucency`已弃用，改为`BL_SceneColorBeforeDOF`，语义更准确。旧资产通过序列化别名保持兼容。

💡 **核心要点**：7个BlendableLocation覆盖了后处理管线的关键节点。AfterTonemapping是最常用的插入点（LDR、通常成本更低），BeforeDOF/AfterDOF适合需要精确深度时使用。

## 四、Halton序列：Film Grain的随机偏移

### 4.1 问题背景

Film Grain（胶片颗粒）需要每帧生成不同的随机噪声偏移。但`rand()`生成的完全随机数会导致分布不均匀（某些区域聚集，某些区域空缺），且对TAA不友好。

**Halton序列**是一种**低差异序列**（Low Discrepancy Sequence），在保持随机性的同时保证分布均匀。

### 4.2 原理

Halton序列基于van der Corput序列：将整数按某个质数基底展开，然后反转小数部分。

```
以Base=2为例，Index=5：
  5的二进制 = 101
  反转 → 0.101（二进制）= 1×0.5 + 0×0.25 + 1×0.125 = 0.625

序列前8个值（Base=2）：
  Index 1: 0.5
  Index 2: 0.25
  Index 3: 0.75
  Index 4: 0.125
  Index 5: 0.625
  Index 6: 0.375
  Index 7: 0.875
  Index 8: 0.0625
  → 自动均匀分布在[0,1]，无需额外处理
```

### 4.3 UE中的实现

**来源**：`Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessTonemap.h`（搜索 token：`GrainRandomFromFrame`、`FrameNumber % 8`）

```cpp
static void GrainRandomFromFrame(FVector3f* RESTRICT const Constant, uint32 FrameNumber)
{
    Constant->X = Halton(FrameNumber & 1023, 2);   // X坐标：质数基底2
    Constant->Y = Halton(FrameNumber & 1023, 3);   // Y坐标：质数基底3
}
```

**关键设计**：
- **`FrameNumber & 1023`**：位与运算限制范围为0-1023（2^10），序列每1024帧循环，避免浮点精度问题
- **质数2和3**：不同维度使用不同质数基底，保证XY维度独立
- **RESTRICT**：告诉编译器指针无别名，允许SIMD优化

**Halton()函数**来自`Math/Halton.h`（引擎数学库），实现标准van der Corput算法。

### 4.4 应用场景

Tonemap Pass在每帧使用Halton偏移量作为Film Grain噪声的UV偏移：

```hlsl
// Tonemap Pixel Shader（简化）
float2 GrainOffset = GrainRandomFull.xy;  // 从Halton序列获取
float Grain = FilmGrainNoise(UV + GrainOffset);
LDRColor += Grain * FilmGrainIntensity;
```

这样每帧的噪声模式都不同，但分布均匀，TAA可以对其进行时域平滑。

💡 **核心要点**：Halton序列通过质数基底的数位反转生成均匀分布的伪随机数，用于Film Grain的帧间随机偏移。比rand()分布更均匀，比预计算纹理内存更省。

## 五、TInlineAllocator：材质链的栈内存优化

### 5.1 问题背景

后处理材质链（`FPostProcessMaterialChain`）在`AddPostProcessingPasses()`中被频繁创建。每个BlendableLocation需要收集对应的材质列表。如果使用默认`TArray`，小数组也可能触发堆分配，在热路径中会累积额外开销。

### 5.2 解决方案

**来源**：`Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessMaterial.h`（搜索 token：`using FPostProcessMaterialChain`、`TInlineAllocator<10>`）

```cpp
using FPostProcessMaterialChain = TArray<const UMaterialInterface*, TInlineAllocator<10>>;
```

`TInlineAllocator<10>`在TArray对象内部预留10个元素的栈空间（80字节，10个指针×8字节）。只有超过10个元素时才回退到堆分配。

```
内存布局：
┌──────────────────────────────────┐
│ FPostProcessMaterialChain对象     │  ← 栈上
│  ├─ InlineData[10]               │  ← 80字节内联空间
│  ├─ HeapData = nullptr           │  ← 未超10个时不使用
│  └─ Num = 3                      │
└──────────────────────────────────┘

多数项目的单个 Location 材质链较短，10个内联容量通常能覆盖常见后处理材质组合。
```

### 5.3 性能对比

| 操作 | 默认TArray（堆分配） | TInlineAllocator<10> | 提升 |
|------|---------------------|---------------------|------|
| **构造** | 小数组可能触发堆分配 | 小数组留在栈内存 | 平台相关，需实测 |
| **添加少量元素** | 可能触发realloc | 内联容量内无堆分配 | 通常更稳 |
| **析构** | 可能需要堆释放 | 内联容量内随对象回收 | 通常更轻 |

`GetPostProcessMaterialChain()`在AddPostProcessingPasses中对每个Location调用一次，约5-7次/帧。

💡 **核心要点**：TInlineAllocator<10>将常见的小数组分配留在栈内存，减少热路径中的堆分配。10个容量覆盖大多数后处理材质链场景，极端项目仍需按实际材质数量验证。

```cpp
// BeforeDOF材质 → DOF → AfterDOF材质 → TAA/TSR → MotionBlur
// → BeforeBloom材质 → Downsample → Bloom → Tonemap
// → FXAA/SMAA → AfterTonemapping材质 → Upscale → 最终输出

// 【步骤4】简化流程（bPostProcessingEnabled = false，搜索 Minimal PostProcessing）

// 【步骤5】后续处理（搜索 Visualize / PrimaryUpscale / SecondaryUpscale）

```

### 3.1.3 关键细节

**关键变量**：

1. **SceneColor / SceneColorSlice**
   - 作用：后处理链的核心变量，每个Pass读取并更新它
   - TAA/TSR之后从`FScreenPassTexture`变为`FScreenPassTextureSlice`（支持Texture Array切片）
   - 生命周期：从HDR R11G11B10F开始，Tonemap后变为LDR A2B10G10R10

2. **PassSequence（TOverridePassSequence）**
   - 作用：管理Pass的启用/禁用和Override输出
   - `PassSequence.IsEnabled(EPass::Tonemap)` 检查是否启用
   - `PassSequence.AcceptOverrideIfLastPass()` 如果是最后一个Pass则直接输出到ViewFamilyTexture

3. **TAAConfig（EMainTAAPassConfig）**
   - 值：TSR / TAA / ThirdParty / Disabled
   - 影响：决定TAA/TSR区域使用哪个实现
   - TSR时渲染分辨率可能低于显示分辨率

**常见陷阱**：

- ⚠️ **陷阱1**：BeforeDOF材质使用SceneDepth，但DOF修改了场景外观
  - 问题：DOF虚化后深度语义不变，但视觉结果变了
  - 解决：如果需要"已虚化"的画面，改用AfterDOF

- ⚠️ **陷阱2**：BeforeBloom材质在TSR之后执行
  - 问题：如果TSR开启，BeforeBloom材质在**显示分辨率**执行，不是渲染分辨率
  - 影响：如果材质有精细的像素级效果，分辨率可能不够

## 3.2 AddPostProcessMaterialPass()

### 3.2.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessMaterial.cpp`
**函数**：`AddPostProcessMaterialPass()`（搜索 token：`FScreenPassTexture AddPostProcessMaterialPass`、`GetMaterialInfo`、`GraphBuilder.AddPass`、`DrawRectangle`）
**版本**：UE5.7

**函数签名**：

```cpp
FScreenPassTexture AddPostProcessMaterialPass(
    FRDGBuilder& GraphBuilder,
    const FViewInfo& View,
    const FPostProcessMaterialInputs& Inputs,
    const UMaterialInterface* MaterialInterface)
```

**职责**：执行单个后处理材质，处理输入纹理绑定、Depth/Stencil配置、Blend状态、输出纹理分配，最终通过全屏四边形渲染。

**性能影响**：
- 耗时趋势：随材质采样数、分支、Blend/Stencil状态和输出分辨率变化
- 热点路径：按需调用，但多个 Blendable 材质会链式累积

### 3.2.2 核心流程

```cpp
// 【步骤1】获取材质Shader（搜索 GetMaterialInfo）
const FMaterial* Material = nullptr;
const FMaterialRenderProxy* MaterialRenderProxy = nullptr;
TShaderRef<FPostProcessMaterialVS> VertexShader;
TShaderRef<FPostProcessMaterialPS> PixelShader;
GetMaterialInfo(MaterialInterface, FeatureLevel, bPathTracingEnabled, Inputs,
    Material, MaterialRenderProxy, MaterialShaderMap, VertexShader, PixelShader);

// 【步骤2】配置Depth/Stencil状态（搜索 GetMaterialCustomDepthPolicy / GetMaterialStencilState）
const EMaterialCustomDepthPolicy CustomStencilPolicy = 
    GetMaterialCustomDepthPolicy(MaterialRenderProxy, Material);

if (CustomStencilPolicy == EMaterialCustomDepthPolicy::Enabled &&
    HasBeenProduced(Inputs.CustomDepthTexture))
{
    DepthStencilTexture = Inputs.CustomDepthTexture;
    DepthStencilState = GetMaterialStencilState(Material);
}

// 【步骤3】配置Blend状态（搜索 IsMaterialBlendEnabled / GetMaterialBlendState）
if (IsMaterialBlendEnabled(Material))
{
    BlendState = GetMaterialBlendState(Material);
}

// 【步骤4】判断是否需要Composite（搜索 bCompositeWithInput）
const bool bCompositeWithInput = 
    DepthStencilState != DefaultDepthStencilState ||
    BlendState != DefaultBlendState || bMayDiscard;

// 【步骤5】分配输出纹理（搜索 bInputReadsFromOutput / bAllowSceneColorInputAsOutput / CreateTexture）
FScreenPassRenderTarget Output = Inputs.OverrideOutput;
if (!Output.IsValid())
{
    if (!bInputReadsFromOutput && !bForceIntermediateTarget &&
        Inputs.bAllowSceneColorInputAsOutput)
    {
        // 快速路径：原地修改SceneColor
        Output = FScreenPassRenderTarget(SceneColor, ERenderTargetLoadAction::ELoad);
    }
    else
    {
        // 标准路径：创建新纹理
        FRDGTextureDesc OutputDesc = ...;
        Output = FScreenPassRenderTarget(
            GraphBuilder.CreateTexture(OutputDesc, TEXT("PostProcessMaterial")), ...);
    }
}

// 【步骤6】Prime输出纹理（Blend/Stencil需要，搜索 bPrimeOutputColor / AddDrawTexturePass）
if (bPrimeOutputColor)
{
    AddDrawTexturePass(GraphBuilder, View, SceneColor, Output);
    Output.LoadAction = ERenderTargetLoadAction::ELoad;
}

// 【步骤7】添加RDG Pass执行Shader（搜索 FPostProcessMaterialParameters / GraphBuilder.AddPass）
FPostProcessMaterialParameters* PassParameters = 
    GraphBuilder.AllocParameters<FPostProcessMaterialParameters>();
PassParameters->View = View.ViewUniformBuffer;
PassParameters->SceneTextures = SceneTextures;
PassParameters->PostProcessInput[0] = GetScreenPassTextureSliceInput(SceneColorViewport);
PassParameters->RenderTargets[0] = Output.GetRenderTargetBinding();

GraphBuilder.AddPass(
    RDG_EVENT_NAME("PostProcessMaterial %s", *Material->GetAssetName()),
    PassParameters, ERDGPassFlags::Raster,
    [VertexShader, PixelShader, MaterialRenderProxy, Material, &View, ...]
    (FRHICommandList& RHICmdList)
    {
        // 绑定PSO（Pipeline State Object）
        FGraphicsPipelineStateInitializer GraphicsPSOInit;
        GraphicsPSOInit.BlendState = BlendState;
        GraphicsPSOInit.DepthStencilState = DepthStencilState;
        SetGraphicsPipelineState(RHICmdList, GraphicsPSOInit, MaterialStencilRef);

        // 设置Shader参数（材质参数 + Pass参数）
        SetShaderParametersMixedVS(RHICmdList, VertexShader, *PassParameters, ...);
        SetShaderParametersMixedPS(RHICmdList, PixelShader, *PassParameters, ...);

        // 全屏四边形绘制
        DrawRectangle(RHICmdList, 0, 0, ViewportWidth, ViewportHeight, ...);
    });
```

### 3.2.3 关键细节

**关键变量**：

1. **bAllowSceneColorInputAsOutput**
   - 作用：允许原地修改SceneColor（避免额外纹理拷贝）
   - 条件：材质不读取PostProcessInput0，且无Blend/DepthTest
   - 引擎自动检测：如果材质的Shader中引用了PostProcessInput_0_Texture，则强制创建中间纹理

2. **bCompositeWithInput**
   - 触发条件：需要Blend混合或Depth/Stencil测试
   - 影响：为true时必须Prime输出纹理（先拷贝输入到输出）

3. **EMaterialCustomDepthPolicy**
   - Disabled：不使用CustomDepth
   - Enabled：使用CustomDepth做模板测试
   - 在Nanite平台上，模板测试需要在Shader内手动执行（`bManualStencilTest`）

**常见陷阱**：

- ⚠️ **陷阱1**：材质读取PostProcessInput0，但bAllowSceneColorInputAsOutput=true
  - 问题：UE会自动检测并强制创建中间纹理，但如果检测失败会导致读写同一纹理（闪烁/错误）
  - 解决：引擎在输出分配阶段检查 `bInputReadsFromOutput`、`bForceIntermediateTarget` 和 `Inputs.bAllowSceneColorInputAsOutput`，开发者通常不需要手动设置

- ⚠️ **陷阱2**：Blend Mode设置为Translucent
  - 问题：需要Prime输出（先拷贝输入），增加一次额外全屏拷贝，成本需按分辨率实测
  - 解决：如果不需要半透明混合，保持Opaque

**源文件**：`Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessMaterial.cpp`

## 3.3 AddTonemapPass()

### 3.3.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessTonemap.cpp`
**函数**：`AddTonemapPass()`（搜索 token：`FScreenPassTexture AddTonemapPass`、`FTonemapInputs`、`TonemapperPermutation::FDesktopDomain`、`FTonemapPS`、`GrainRandomFromFrame`）
**版本**：UE5.7

**函数签名**：

```cpp
FScreenPassTexture AddTonemapPass(
    FRDGBuilder& GraphBuilder,
    const FViewInfo& View,
    const FTonemapInputs& Inputs)
```

**职责**：将HDR场景颜色映射到LDR显示范围，同时执行Bloom合成、Eye Adaptation曝光、Color Grading（3D LUT）、Film Grain和Sharpen。

**性能影响**：
- 耗时趋势：取决于输出分辨率、LUT、Film Grain、Sharpen、Local Exposure 和输出格式
- 热点路径：通常每帧调用，Gamma-only路径和特殊输出会改变执行内容
- 主要检查点：3D LUT采样、Film Grain、Sharpen、Local Exposure 分支

### 3.3.2 核心流程

```cpp
// 【步骤1】验证输入（搜索 Inputs.SceneColor.IsValid / ColorGradingTexture）
check(Inputs.SceneColor.IsValid());
if (!Inputs.bGammaOnly)
    check(Inputs.ColorGradingTexture);  // 非GammaOnly需要LUT

// 创建默认曝光Buffer（如果未提供）
FRDGBufferRef EyeAdaptationBuffer = Inputs.EyeAdaptationBuffer;
if (!EyeAdaptationBuffer)
{
    const float DefaultExposure = GetEyeAdaptationFixedExposure(View);
    EyeAdaptationBuffer = CreateStructuredBuffer(GraphBuilder, ...);
}

// 【步骤2】分配输出纹理（搜索 OutputDesc / EDisplayOutputFormat / Inputs.bOutputInHDR）
FRDGTextureDesc OutputDesc = FRDGTextureDesc::Create2D(...);

// 根据输出设备选择格式
const EDisplayOutputFormat OutputDevice = ...;
if (OutputDevice == EDisplayOutputFormat::HDR_LinearEXR)
    OutputDesc.Format = PF_A32B32G32R32F;       // 32位浮点（EXR导出）
else if (Inputs.bOutputInHDR)
    OutputDesc.Format = PF_FloatRGBA;            // 16位浮点（HDR显示器）
else
    OutputDesc.Format = PF_A2B10G10R10;          // 10位整数（SDR，最常见）

// 【步骤3】配置Shader Permutation（搜索 TonemapperPermutation::FDesktopDomain）
TonemapperPermutation::FDesktopDomain DesktopPermutationVector;
TonemapperPermutation::FCommonDomain CommonDomain =
    TonemapperPermutation::BuildCommonPermutationDomain(
        View, Inputs.bGammaOnly, bLocalExposureEnabled, Inputs.bMetalMSAAHDRDecode);
DesktopPermutationVector.Set<TonemapperPermutation::FCommonDomain>(CommonDomain);
DesktopPermutationVector.Set<TonemapperPermutation::FTonemapperOutputDeviceSRGB>(bOutputDeviceSRGB);
DesktopPermutationVector = TonemapperPermutation::RemapPermutation(DesktopPermutationVector, View.GetFeatureLevel());

TShaderMapRef<FTonemapPS> PixelShader(View.ShaderMap, DesktopPermutationVector);

// 【步骤4】填充Shader参数（搜索 FTonemapParameters / ColorGradingLUT / EyeAdaptationBuffer）
FTonemapPS::FParameters* PassParameters = GraphBuilder.AllocParameters<FTonemapPS::FParameters>();
PassParameters->Tonemap = CommonParameters;
CommonParameters.ColorGradingLUT = Inputs.ColorGradingTexture;  // 3D LUT
CommonParameters.EyeAdaptationBuffer = GraphBuilder.CreateSRV(EyeAdaptationBuffer);

if (bFilmGrain)
{
    GrainRandomFromFrame(&CommonParameters.FilmGrain.GrainRandomFull, OutputFrameIndexMod8);
}

PassParameters->RenderTargets[0] = Output.GetRenderTargetBinding();

// 【步骤5】添加Pass执行（搜索 AddDrawScreenPass / FTonemapVS / FTonemapPS）
TShaderMapRef<FTonemapVS> VertexShader(View.ShaderMap);
AddDrawScreenPass(
    GraphBuilder,
    RDG_EVENT_NAME("Tonemap %dx%d", OutputWidth, OutputHeight),
    View, OutputViewport, SceneColorViewport,
    FScreenPassPipelineState(VertexShader, PixelShader, BlendState, DepthStencilState),
    PassParameters, DrawFlags,
    [VertexShader, PixelShader, PassParameters]
    (FRHICommandList& RHICmdList)
    {
        SetShaderParameters(RHICmdList, VertexShader, VertexShader.GetVertexShader(), PassParameters->Tonemap);
        SetShaderParameters(RHICmdList, PixelShader, PixelShader.GetPixelShader(), *PassParameters);
    });

return Output.Texture;
```

### 3.3.3 Tonemap Shader核心逻辑（HLSL简化）

```hlsl
float4 TonemapPS(float2 UV : TEXCOORD0) : SV_Target
{
    // 【步骤1】采样HDR SceneColor
    float3 HDRColor = SceneColorTexture.SampleLevel(Sampler, UV, 0).rgb;

    // 【步骤2】合成Bloom（Half分辨率bilinear采样）
    float3 BloomColor = BloomTexture.SampleLevel(Sampler, UV, 0).rgb;
    HDRColor += BloomColor * BloomIntensity;

    // 【步骤3】应用曝光（Eye Adaptation）
    float Exposure = EyeAdaptationBuffer[0].x;
    HDRColor *= Exposure;

    // 【步骤4】Tone Mapping曲线（ACES FilmicTonemap）
    float3 LDRColor = ACESFilmicTonemap(HDRColor);  // HDR → LDR (0-1)

    // 【步骤5】Color Grading（3D LUT采样）
    LDRColor = ColorGradingTexture.SampleLevel(Sampler, LDRColor * LUTScale + LUTOffset, 0).rgb;

    // 【步骤6】Film Grain（可选，Halton偏移）
    #if USE_FILM_GRAIN
        float Grain = FilmGrainNoise(UV + GrainRandomFull.xy);
        LDRColor += Grain * FilmGrainIntensity;
    #endif

    // 【步骤7】Sharpen（可选，拉普拉斯锐化）
    #if USE_SHARPEN
        float3 Neighbors = SampleNeighbors(UV);
        LDRColor += (LDRColor - Neighbors) * SharpenDiv6;
    #endif

    return float4(LDRColor, Alpha);
}
```

### 3.3.4 关键细节

**关键变量**：

1. **ColorGradingTexture**（3D Texture 32×32×32）
   - 作用：Color Grading查找表，由CombineLUTPass预生成
   - 原理：输入LDR颜色作为3D UV坐标，采样得到映射后颜色
   - 优势：单次Texture3D采样实现任意颜色变换，比数学公式快

2. **EyeAdaptationBuffer**（StructuredBuffer<float4>）
   - `[0].x` = 当前曝光值
   - `[0].w` = 上一帧曝光值（平滑过渡用）
   - 更新：由AddBasicEyeAdaptationPass或AddHistogramEyeAdaptationPass计算

3. **输出格式选择**
   - `PF_A2B10G10R10`：SDR输出，10位精度（最常见）
   - `PF_FloatRGBA`：HDR显示器输出，16位浮点
   - `PF_A32B32G32R32F`：EXR导出，32位浮点

**常见陷阱**：

- ⚠️ **陷阱1**：bGammaOnly=true时LUT不生效
  - 原因：GammaOnly跳过整个Tonemap + LUT流程，只做Gamma校正
  - 解决：确保bGammaOnly=false（默认）

- ⚠️ **陷阱2**：Bloom Intensity过高导致过曝
  - 原因：Bloom在Tonemap**之前**叠加，高亮度值被Tonemap压缩不够
  - 解决：降低BloomIntensity或调整Tone Mapping曲线

💡 **核心要点**：AddTonemapPass在一个全屏Pass中完成HDR→LDR的完整流程：曝光 → ACES Tonemap → 3D LUT Color Grading → Film Grain → Sharpen。通过Shader Permutation控制可选功能的开关，避免运行时分支。

```cpp
    EBlendableLocation Location = ...;
```

```
    ↓ ──────────────────────────────┐
    ↓ AddTonemapPass               │
    ← SceneColorSlice (HDR)        │  三个输入合并
    ← EyeAdaptationBuffer         │
    ← ColorGradingTexture (3D LUT) │
    ↓ ──────────────────────────────┘
TonemapOutput (LDR A2B10G10R10，显存占用随分辨率变化)
    ↓ FXAA/SMAA
FinalColor (LDR)
    ↓
ViewFamilyTexture (最终输出)
```

**RDG自动优化**：
1. **Transient Resource**：BloomDownsample各级复用同一块显存
2. **Barrier合并**：相邻Pass的资源Transition合并为一次
3. **Async Compute**：无依赖的Pass可在Async Compute Queue并行执行

💡 **核心要点**：后处理在RenderThread上通过RDG添加Pass，实际GPU成本由屏幕分辨率、启用Pass和材质复杂度决定。TAA/TSR是分辨率分界线，Tonemap是HDR→LDR转换点。RDG自动管理资源生命周期和依赖。

### 1.1 Material Editor配置

**步骤1**：创建材质
1. Content Browser → 右键 → Material
2. 命名为`M_PP_MyEffect`

```
最简颜色反相：

┌─────────────────────────────┐
│ SceneTexture:PostProcessInput0 │  采样SceneColor
└────────┬────────────────────┘
         ↓ RGB
    ┌───────────────┐
```

```hlsl
float4 MainPS(FPostProcessMaterialVSToPS In) : SV_Target0
```


### 1.3 常见材质技巧

**访问场景深度**：

**访问运动矢量**：

## 四、性能优化建议

### 4.1 渲染设置优化

| 策略 | 操作 | 预期影响 |
|------|------|----------|
| 禁用非必要Pass | `r.MotionBlur.Max 0`、`r.DepthOfFieldQuality 0` | 移除对应屏幕Pass，收益按场景实测 |
| 降低Bloom质量 | `r.BloomQuality 3` | 减少Bloom链成本，收益按半径和分辨率实测 |
| 启用TAAU | `r.TemporalAA.Upsampling 1` + `r.ScreenPercentage 80` | 降低部分屏幕Pass像素量，画质和成本需一起评估 |
| 使用FFT Bloom | 项目设置中切换 | 大半径Bloom场景可能更合适，需项目验证 |

### 4.2 材质优化

| 优化 | 说明 |
|------|------|
| 减少Texture Sample数量 | Bandwidth常是像素级成本来源，目标值需按效果复杂度设定 |
| 使用Gather代替多次Sample | 一次Gather返回2×2区域4个值 |
| 使用LUT代替复杂数学 | `pow(x, 2.2)`用1D LUT代替 |
| Early-out黑色区域 | 亮度<0.01时直接返回 |
| 合并多个小材质 | 减少DrawCall和纹理切换 |

### 4.3 架构优化

| 优化 | 说明 |
|------|------|
| 减少PostProcessVolume数量 | 重叠Volume导致每个材质都执行 |
| 选择正确的BlendableLocation | AfterTonemapping通常成本更低，避免无必要使用BeforeDOF |
| 使用Blendable Priority排序 | 避免不必要的中间纹理创建 |
| 不读取PostProcessInput0时 | bAllowSceneColorInputAsOutput=true避免拷贝 |

### 4.4 分辨率对性能的影响

后处理是像素级计算，分辨率影响是平方关系：

| 分辨率 | 像素量趋势 | 后处理影响 |
|--------|------------|------------|
| 720p   | 较低 | 适合验证屏幕Pass是否受像素量限制 |
| 1080p  | 基准 | 常用性能对比基准 |
| 1440p  | 明显增加 | 全屏Pass成本通常上升 |
| 4K     | 大幅增加 | 需要重点评估TSR/TAAU与质量档 |

**优化建议**：使用TSR/TAAU时，可以让部分后处理在较低渲染分辨率执行，再超分到目标输出；画质、稳定性和成本需要一起评估。

### 4.5 Shader复杂度目标

| 材质类型 | 指令数目标 | 说明 |
|---------|-----------|------|
| 简单后处理（颜色调整） | <200指令 | 绿色区域 |
| 中等复杂（Blur、边缘检测） | 200-500指令 | 黄色区域 |
| 复杂效果（Glitch、扭曲） | 500-800指令 | 橙色区域 |
| ⚠️ 需优化 | >800指令 | 红色区域 |

使用`r.ShaderComplexity 1`在视口中可视化Shader复杂度。

💡 **核心要点**：后处理优化的三个杠杆：（1）禁用不必要的Pass，（2）降低渲染分辨率并配合TSR/TAAU，（3）优化材质Shader复杂度。具体收益必须用目标平台和目标场景实测。

# 第五部分：与其他系统的关系

---

## 一、与RenderGraph的关系

后处理是RDG的**重度用户**。所有Pass通过`GraphBuilder.AddPass()`声明，RDG自动处理：

1. **依赖分析**：Tonemap引用DOF的输出 → RDG自动建立依赖
2. **资源生命周期**：BloomDownsample在Tonemap后不再使用 → RDG自动释放
3. **Transient Resource**：BloomDownsample多级复用同一块显存 → 减少显存占用
4. **Barrier合并**：相邻Pass的资源状态切换合并为一次 → 减少GPU等待

**如果没有RDG**：需要手动管理~30个纹理的创建、Barrier、释放，极易出错。

---

## 二、与TAA/TSR的关系

TAA/TSR在后处理管线中是**分辨率分界线**：

```
BeforeDOF / AfterDOF 材质 → 渲染分辨率（可能50-85%显示分辨率）
    ↓ TAA/TSR（超分辨率）
BeforeBloom / AfterTonemapping 材质 → 显示分辨率
```

**关键影响**：
- BeforeDOF/AfterDOF材质在**低分辨率**执行，但效果被TSR放大
- BeforeBloom/AfterTonemapping材质在**显示分辨率**执行
- `BL_SSRInput`独立于DRS，通过`POST_PROCESS_MATERIAL_SSRINPUT`宏控制

**实践建议**：如果材质有精细的像素级效果（如1像素描边），应选择AfterTonemapping而非BeforeDOF。

---

## 三、与材质系统的关系

后处理材质复用引擎的完整材质编译管线：

```
Material Editor → UMaterial → FMaterial → FMaterialShaderMap
    ↓ GetShader<FPostProcessMaterialPS>(PermutationVector)
TShaderRef<FPostProcessMaterialPS>
    ↓ 绑定到PSO
DrawRectangle（全屏四边形）
```

**关键连接点**：
- `MaterialDomain == MD_PostProcess`：编译裁剪条件
- `BlendableLocation`：影响编译宏（BEFORE_TONEMAP / SSRINPUT）
- `FPostProcessMaterialParameters`：RDG Pass参数结构，自动生成Shader中的Uniform声明

**Substrate支持**：UE5.7的后处理材质Shader注入了`SUBSTRATE_INLINE_SHADING`和`SUBSTRATE_DEFERRED_SHADING`宏，支持新的Substrate材质模型。

💡 **核心要点**：后处理系统深度依赖RDG（资源管理）、TAA/TSR（分辨率分界）和材质系统（Shader编译管线）。理解这三个系统的交互是掌握后处理的关键。
