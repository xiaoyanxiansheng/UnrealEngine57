# 第一部分：为什么需要自定义Shader

> **核心目标**：搞清楚什么时候必须写自定义Shader，什么时候用材质编辑器就够了。如果你正在犹豫"要不要写C++自定义Shader"，这一节帮你做决定。

---

## 一、材质编辑器做不到什么

### 1.1 材质编辑器的能力边界

UE5的材质编辑器（Material Editor）通过节点图+材质表达式，已经能覆盖绝大多数视觉效果需求。但它有明确的能力边界：

| 能力 | 材质编辑器 | 自定义Shader |
|------|-----------|-------------|
| BaseColor/Roughness/Normal等GBuffer输出 | ✅ 原生支持 | 不需要 |
| 后处理效果（通过PostProcessMaterial） | ✅ 部分支持 | 需要更复杂逻辑时 |
| CustomExpression节点（嵌入HLSL片段） | ✅ 支持 | 不需要 |
| **读写任意RenderTarget** | ❌ 无法指定输出目标 | ✅ 可由RDG/Renderer代码控制 |
| **在渲染管线中插入新Pass** | ❌ 无法控制Pass顺序 | ✅ 通过RDG AddPass |
| **访问GPUScene数据/前帧历史** | ❌ 受限 | ✅ 直接绑定Buffer |
| **Dispatch ComputeShader** | ❌ 不支持 | ✅ 支持自定义CS调度 |
| **自定义Vertex输入** | ❌ 受VertexFactory限制 | ✅ 自定义VF |
| **多Pass联动（如生成+消费）** | ❌ 单Pass模型 | ✅ RDG编排多Pass |

### 1.2 典型的"必须写自定义Shader"场景

**场景1：自定义全屏后处理效果**

材质编辑器的PostProcessMaterial虽然能做后处理，但受限于单Pass、无法精确控制输入/输出纹理、无法读取自定义Buffer。当你需要：

- 读取GBuffer中的特定通道做自定义边缘检测
- 多Pass迭代（如自定义模糊：横向Pass + 纵向Pass）
- 读取前一帧的渲染结果做时序累积

这些都需要FGlobalShader + RDG。

**场景2：GPU数据处理（ComputeShader）**

材质编辑器完全不支持ComputeShader。当你需要：

- GPU粒子位置更新
- 自定义剔除逻辑
- 大规模数据并行处理（如体积雾计算）

只能通过FGlobalShader（SF_Compute频率）实现。

**场景3：在渲染管线中注入自定义Pass**

引擎的渲染管线是固定顺序（PrePass → BasePass → Lighting → PostProcess），如果你需要在两个Pass之间插入自定义逻辑——比如在BasePass之后、Lighting之前做一个自定义的屏幕空间效果——只能通过RDG AddPass注入。

### 1.3 做决定的流程图

```
需要实现一个渲染效果
    │
    ├─ 能否用材质节点图实现？
    │   ├─ 是 → 用材质编辑器（最简单）
    │   └─ 否 ↓
    │
    ├─ 能否用CustomExpression节点嵌入HLSL？
    │   ├─ 是 → 用CustomExpression（次简单）
    │   └─ 否 ↓
    │
    ├─ 是否需要ComputeShader / 多Pass / 自定义RenderTarget？
    │   ├─ 是 → 写FGlobalShader + RDG（本章内容）
    │   └─ 否 ↓
    │
    └─ 是否需要自定义网格渲染Pass（如自定义DepthPass）？
        └─ 是 → 写FMeshMaterialShader + MeshPassProcessor（更复杂）
```

## 二、Shader类型体系概览

### 2.1 三种Shader类型

UE5的Shader类型体系是一个继承树：

```
                    FShader（基类）
                       │
          ┌────────────┼─────────────┐
          │            │             │
    FGlobalShader  FMaterialShader  FComputeShader
          │            │                   │
          │       FMeshMaterialShader       │
          │                                │
    ┌─────┴────────┐              ┌────────┘
    │              │              │
FScreenPassVS  FOneColorPS   自定义CS
```

| 类型 | 依赖材质 | 依赖VertexFactory | 实例数量 | 典型用途 |
|------|---------|------------------|---------|---------|
| **FGlobalShader** | ❌ | ❌ | 1个/平台 | 后处理、全屏效果、工具 |
| FMaterialShader | ✅ | ❌ | N个（每材质1个） | Decal、Light Function |
| FMeshMaterialShader | ✅ | ✅ | N×M个 | BasePass、ShadowDepth |

**本章聚焦FGlobalShader**——最简单、最常用、最适合入门的类型。

### 2.2 为什么从FGlobalShader开始

FGlobalShader的特点：

- **全局唯一**：每个平台只有1个实例（或每个变体1个），不依赖材质参数
- **启动时自动加载**：注册到`FGlobalShaderMap`，运行时通过`TShaderMapRef`查找
- **生命周期长**：引擎启动到关闭
- **参数固定**：不从材质图获取参数，而是C++直接设置

这意味着：创建一个FGlobalShader，只需要关心"C++类 + HLSL文件 + 参数绑定"三件事，不需要理解材质编译器、VertexFactory等复杂子系统。

## 三、创建流程鸟瞰

### 3.1 五步完整链路

```
【开发时】                        【编译时】                       【运行时】

步骤1：C++类声明(.h)              步骤3：Shader编译                步骤5：RDG调度执行
├─ DECLARE_GLOBAL_SHADER      ├─ 虚拟路径 → 物理路径         ├─ TShaderMapRef获取实例

        ▼                               ▼
```

> **核心目标**：在看源码之前，先建立正确的概念模型。每个环节"是什么"和"为什么要这样设计"，理解了设计意图再看代码会事半功倍。

## 一、从.usf到GPU指令的完整数据流

### 1.1 数据流全景

```
.usf文件（HLSL方言）
    │
    ▼ 虚拟路径解析
物理磁盘上的.usf文件
    │
    ▼ 预处理器（#include展开 + 宏替换）
完整HLSL文本（包含Platform.ush/Common.ush等展开内容）
    │
    ▼ dxc.exe编译
平台Shader字节码 + ParameterMap
    │
    ▼ DDC缓存
%LocalAppData%/UnrealEngine/Common/DerivedDataCache/
    │
    ▼ FGlobalShaderMap加载
FRHIPixelShader*（GPU资源句柄）
    │
    ▼ RDG AddPass + SetShaderParameters
GPU Command Queue → GPU执行
```

### 1.2 两个关键时间点

- **编译时**（引擎启动 / 修改.usf后）：.usf → DXIL → DDC → FGlobalShaderMap
- **运行时**（每帧）：TShaderMapRef查找 → 设置参数 → DrawCall

编译通常发生在启动、Cook、热重载或.usf变更后；运行时主要做Shader实例查找、参数提交和Pass调度。把编译成本前移并通过DDC缓存复用，是UE Shader系统减少每帧开销的关键设计。

## 二、虚拟路径系统

### 2.1 为什么不直接用物理路径

HLSL中的`#include`使用虚拟路径而非物理路径：

```hlsl
#include "/Engine/Private/Common.ush"
```

而非：

```hlsl
#include "D:/Unreal/Engine/Shaders/Private/Common.ush"  // ❌ 不使用
```

原因：

1. **跨机器一致性**：不同开发者的引擎安装路径不同，虚拟路径保证一致
2. **模块化**：插件可以注册自己的虚拟路径，不会与引擎路径冲突
3. **DDC缓存**：虚拟路径是DDC Key的一部分，物理路径变化不影响缓存

### 2.2 虚拟路径映射规则

**源码锚点**：
- 路径：`Engine/Source/Runtime/RenderCore/Public/ShaderCore.h`，搜索`AddShaderSourceDirectoryMapping`（API声明）
- 路径：`Engine/Source/Runtime/RenderCore/Private/ShaderCore.cpp`，搜索`void AddShaderSourceDirectoryMapping`（映射表写入）
- 路径：`Engine/Source/Runtime/Launch/Private/LaunchEngineLoop.cpp`，搜索`AddShaderSourceDirectoryMapping(TEXT("/Engine")`和`AddShaderSourceDirectoryMapping(TEXT("/Project")`（引擎/项目默认映射）

```
虚拟路径                              物理路径
/Engine          →  <引擎目录>/Engine/Shaders
/Plugin/MyPlugin →  <插件目录>/MyPlugin/Shaders
/Project         →  <项目目录>/Shaders
```

映射通过`AddShaderSourceDirectoryMapping()`注册，通常在模块`StartupModule()`中调用：

```cpp
AddShaderSourceDirectoryMapping(
    TEXT("/Plugin/MyPlugin"),    // 虚拟路径前缀
    PluginShaderDir              // 物理路径
);
```

注册后，`IMPLEMENT_GLOBAL_SHADER`中指定的路径 `/Plugin/MyPlugin/Private/MyShader.usf` 会被解析到物理磁盘上的实际文件。

## 三、参数绑定模型

### 3.1 C++ struct ↔ HLSL变量的对应关系

自定义Shader最容易出错的环节就是参数绑定。核心规则：**C++ struct成员名必须与HLSL变量名完全一致**（包括大小写）。

```
C++ 侧（FParameters结构体）              HLSL 侧（.usf文件）
─────────────────────────────           ─────────────────────────
SHADER_PARAMETER(FVector4f, TintColor)  ←→  float4 TintColor;
SHADER_PARAMETER(float, Intensity)      ←→  float Intensity;
SHADER_PARAMETER_RDG_TEXTURE(           ←→  Texture2D InputTexture;
    Texture2D, InputTexture)
SHADER_PARAMETER_SAMPLER(               ←→  SamplerState InputSampler;
    SamplerState, InputSampler)
```

类型映射表：

| C++类型 | HLSL类型 | 大小 |
|---------|---------|------|
| `float` | `float` | 4字节 |
| `FVector2f` | `float2` | 8字节 |
| `FVector4f` | `float4` | 16字节 |
| `FMatrix44f` | `float4x4` | 64字节 |
| `int32` | `int` | 4字节 |
| `uint32` | `uint` | 4字节 |
| `FLinearColor` | `float4` | 16字节 |

### 3.2 两种绑定方式：现代vs Legacy

**现代方式（推荐用于新代码）**：`SHADER_USE_PARAMETER_STRUCT`

- 自动生成构造函数
- 自动绑定所有参数
- 编译期验证
- 1行代码完成

**Legacy方式（不推荐）**：手动`FShaderParameter::Bind()`

- 需要手动为每个参数调用Bind
- 容易拼写错误
- 代码冗长

本文档全部使用现代方式。

## 四、变体系统

### 4.1 什么是Shader变体

一个Shader可能有多个版本，每个版本通过`#define`宏开关不同的功能分支：

```hlsl
// 变体0：USE_GAMMA=0, USE_DESAT=0
// 变体1：USE_GAMMA=0, USE_DESAT=1
// 变体2：USE_GAMMA=1, USE_DESAT=0
// 变体3：USE_GAMMA=1, USE_DESAT=1
```

每个变体是一个**独立编译**的Shader二进制。2个bool维度 = 2² = 4个变体。

### 4.2 为什么需要变体而不是运行时分支

GPU上的动态分支（`if`语句）在分支发散明显时可能带来额外成本。通过编译期`#if`消除分支，编译器可以：

- 死代码消除（完全移除未使用的分支）
- 常量折叠（编译期计算已知值）
- 寄存器优化（减少寄存器压力）

代价是编译时间和DDC存储空间增加，但运行时性能更优。

### 4.3 变体裁剪

并非所有变体组合都有意义。`ShouldCompilePermutation()`允许在编译期跳过无意义的组合：

```
10个bool维度 → 理论变体数：2¹⁰ = 1024
裁剪掉互斥组合和无意义组合 → 实际可能只保留几十到数百个
收益：减少编译任务、DDC记录和运行时可选组合；具体幅度取决于裁剪规则
```

---

## 五、DDC缓存

### 5.1 DDC是什么

DDC（Derived Data Cache）是UE5的编译产物缓存系统。Shader编译结果（DXIL字节码 + ParameterMap）被保存到DDC，下次启动时直接加载，无需重新编译。

### 5.2 DDC Key计算

```
DDC Key = Hash(
    ShaderType名称 +       // "FMyShaderPS"
    .usf文件内容Hash +     // 文件修改即失效
    平台 +                 // SP_PCD3D_SM5
    PermutationId +        // 变体ID
    编译器版本 +           // dxc版本
    配置宏                 // .ini中的全局宏
)
```

只要任何一个因素变化，DDC Key就变化，触发重新编译。

### 5.3 编译成本对比

| 场景 | 成本特征 | 说明 |
|------|----------|------|
| 首次编译（DDC未命中） | 通常明显慢 | 需要预处理、dxc编译、反射和保存缓存 |
| DDC命中加载 | 通常明显快 | 主要是读取与反序列化字节码/参数映射 |
| 变体数量增加 | 编译和缓存成本随之上升 | 维度越多，越依赖`ShouldCompilePermutation()`裁剪 |

具体耗时取决于平台、CPU核心数、Shader复杂度、编译器版本、DDC位置和是否并行编译；不要把某台机器的倍率当作项目结论。

## 一、DECLARE_GLOBAL_SHADER宏展开

### 3.1.1 函数概览

**来源**：`Engine/Source/Runtime/RenderCore/Public/GlobalShader.h`
**搜索 token**：`#define DECLARE_GLOBAL_SHADER`
**版本**：UE5.7

**职责（一句话）**：为Shader类自动生成StaticType静态成员、工厂函数、编译环境修改函数等反射数据，使Shader可以被引擎自动发现和管理。

用户只需写1行：

```cpp
class FMyShaderPS : public FGlobalShader
{
    DECLARE_GLOBAL_SHADER(FMyShaderPS);
};
```

宏自动生成Shader类型反射、工厂函数和编译环境转发所需的样板代码。

---

### 3.1.2 核心流程

`DECLARE_GLOBAL_SHADER`实际展开为`DECLARE_SHADER_TYPE`，生成以下内容：

```cpp
class FMyShaderPS : public FGlobalShader
{
public:
    // 【生成1】类型别名
    using ShaderMetaType = FGlobalShaderType;
    using ShaderMapType  = FGlobalShaderMap;

    // 【生成2】全局唯一的类型描述对象
    static ShaderMetaType StaticType;

    // 【生成3】工厂函数——反序列化构造（从DDC加载时调用）
    static FShader* ConstructSerializedInstance()
    {
        return new FMyShaderPS();
    }

    // 【生成4】工厂函数——编译后构造（首次编译完成后调用）
    static FShader* ConstructCompiledInstance(
        const ShaderMetaType::CompiledShaderInitializerType& Initializer)
    {
        return new FMyShaderPS(Initializer);
    }

    // 【生成5】编译环境修改——插入变体定义
    static void ModifyCompilationEnvironmentImpl(
        const FShaderPermutationParameters& Parameters,
        FShaderCompilerEnvironment& OutEnvironment)
    {
        FPermutationDomain PermutationVector(Parameters.PermutationId);
        PermutationVector.ModifyCompilationEnvironment(OutEnvironment);
        FMyShaderPS::ModifyCompilationEnvironment(Parameters, OutEnvironment);
    }
};
```

**关键设计决策**：使用静态函数指针而非虚函数

- StaticType是一个全局静态变量，在模块加载时（`main()`之前）自动构造
- 工厂函数通过函数指针传递，避免VTable查找开销
- 所有信息在编译期确定，避免依赖运行时虚函数分发

---

### 3.1.3 关键细节

**StaticType的自动注册**

StaticType是一个全局变量，利用C++全局变量的构造顺序，在`main()`之前执行。构造函数中，StaticType将自己插入到一个全局单链表`FShaderType::TypeList`中：

```cpp
FGlobalShaderType::FGlobalShaderType(/* 参数 */)
    : FShaderType(EShaderTypeForDynamicCast::Global, /* ... */)
{
    GlobalBeginCreateShaderType();           // 加锁
    FShaderType::GetTypeList()->AddHead(this); // 插入链表头
    GlobalEndCreateShaderType();             // 解锁
}
```

这个链表是后续`CompileGlobalShaderMap()`遍历所有Shader类型的入口。

**为什么需要加锁？**

多个模块可能同时加载（多DLL并行初始化），必须保护链表操作。UE5通过`GlobalBeginCreateShaderType()`/`GlobalEndCreateShaderType()`提供互斥保护。

## 三、SHADER_PARAMETER_STRUCT参数绑定

### 3.3.1 函数概览

**来源**：
- `Engine/Source/Runtime/RenderCore/Public/ShaderParameterStruct.h`，搜索`SHADER_USE_PARAMETER_STRUCT_INTERNAL`
- `Engine/Source/Runtime/RenderCore/Public/ShaderParameterMacros.h`，搜索`BEGIN_SHADER_PARAMETER_STRUCT`

**版本**：UE5.7

**职责（一句话）**：`SHADER_USE_PARAMETER_STRUCT`自动生成Shader构造函数，`BEGIN_SHADER_PARAMETER_STRUCT`定义参数结构体，两者配合实现"声明即绑定"——参数在编译后自动映射到HLSL寄存器。

---

### 3.3.2 核心流程

**完整的参数声明模式**：

```cpp
class FMyShaderPS : public FGlobalShader
{
    DECLARE_GLOBAL_SHADER(FMyShaderPS);

    // 【关键宏1】自动生成构造函数，从ParameterMap绑定所有参数
    SHADER_USE_PARAMETER_STRUCT(FMyShaderPS, FGlobalShader);

    // 【关键宏2】定义参数结构体
    BEGIN_SHADER_PARAMETER_STRUCT(FParameters, )
        // 标量参数
        SHADER_PARAMETER(FVector4f, TintColor)
        SHADER_PARAMETER(float, Intensity)

        // RDG纹理（RenderGraph管理生命周期）
        SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture)
        SHADER_PARAMETER_SAMPLER(SamplerState, InputSampler)

        // 输出目标
        RENDER_TARGET_BINDING_SLOTS()
    END_SHADER_PARAMETER_STRUCT()
};
```

**SHADER_USE_PARAMETER_STRUCT展开为**：

```cpp
FMyShaderPS(const ShaderMetaType::CompiledShaderInitializerType& Initializer)
    : FGlobalShader(Initializer)
{
    BindForLegacyShaderParameters<FParameters>(
        this,
        Initializer.PermutationId,
        Initializer.ParameterMap,
        true   // bShouldBindEverything
    );
}
```

`BindForLegacyShaderParameters`遍历FParameters的所有成员，从编译后的`ParameterMap`查找每个参数对应的GPU寄存器位置，完成绑定。

**参数类型宏一览**：

| 宏 | C++类型 | HLSL类型 | 用途 |
|----|---------|---------|------|
| `SHADER_PARAMETER(Type, Name)` | 标量/向量 | 对应HLSL类型 | 常量参数 |
| `SHADER_PARAMETER_ARRAY(Type, Name, [N])` | 定长数组 | `Type[N]` | 数组参数 |
| `SHADER_PARAMETER_TEXTURE(Texture2D, Name)` | `FRHITexture*` | `Texture2D` | 非RDG纹理 |
| `SHADER_PARAMETER_RDG_TEXTURE(Texture2D, Name)` | `FRDGTextureRef` | `Texture2D` | RDG纹理 |
| `SHADER_PARAMETER_SAMPLER(SamplerState, Name)` | `FRHISamplerState*` | `SamplerState` | 采样器 |
| `SHADER_PARAMETER_UAV(RWTexture2D<float4>, Name)` | `FRHIUnorderedAccessView*` | `RWTexture2D` | 读写纹理 |
| `SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D, Name)` | `FRDGTextureUAVRef` | `RWTexture2D` | RDG UAV |
| `RENDER_TARGET_BINDING_SLOTS()` | - | SV_Target | 像素Shader输出目标 |

---

### 3.3.3 关键细节

**内存布局与对齐**

FParameters结构体在内存中按C++规则对齐，但GPU Constant Buffer有自己的对齐规则（16字节对齐）。UE5的宏系统自动处理两者的差异。

```
FParameters内存布局：
Offset 0:   TintColor (FVector4f)    [16字节]
Offset 16:  Intensity (float)        [4字节]
Offset 20:  Padding                  [12字节，对齐到16]
Offset 32:  InputTexture (指针)      [8字节，资源参数单独处理]
Offset 40:  InputSampler (指针)      [8字节]
```

标量参数（TintColor、Intensity）被打包到Constant Buffer，资源参数（InputTexture、InputSampler）通过独立的寄存器绑定。

**编译期验证**

如果HLSL中缺少某个FParameters成员声明，编译器会报错（因为`bShouldBindEverything=true`）。这是相对于Legacy手动Bind的最大优势——**拼写错误在编译时就会被发现**。

|------|------|----------------------|
| `SHADER_USE_PARAMETER_STRUCT` | `Engine/Source/Runtime/RenderCore/Public/ShaderParameterStruct.h` | `SHADER_USE_PARAMETER_STRUCT_INTERNAL`、`BindForLegacyShaderParameters<FParameters>` / 构造函数生成 |

`SHADER_USE_PARAMETER_STRUCT` + `BEGIN_SHADER_PARAMETER_STRUCT`实现"声明即绑定"：C++ struct成员名与HLSL变量名一致即可自动映射，编译期验证防止拼写错误。推荐所有新Shader使用此模式。

## 四、SetShaderParameters()参数提交

### 3.4.1 函数概览

**来源**：`Engine/Source/Runtime/RenderCore/Private/ShaderParameterStruct.cpp`
**搜索 token**：`void SetShaderParameters(`、`ExtractShaderParameterResources`
**版本**：UE5.7

**职责（一句话）**：将C++ FParameters结构体中的数据，按照编译后的Bindings映射关系，批量拷贝到`FRHIBatchedShaderParameters`，等待提交到GPU。

**性能影响**：

- 成本随参数数量、资源类型和RHI后端变化
- 热点路径：是（每次提交该Shader参数时都会调用）

---

### 3.4.2 核心流程

```cpp
void SetShaderParameters(
    FRHIBatchedShaderParameters& BatchedParameters,
    const FShaderParameterBindings& Bindings,
    const FShaderParametersMetadata* ParametersMetadata,
    const void* InParametersData)
{
    // 【步骤1】将FParameters指针转为字节数组
    TConstArrayView<uint8> Data(
        (const uint8*)InParametersData,
        ParametersMetadata->GetSize()
    );

    // 【步骤2】遍历所有标量参数，拷贝到BatchedParameters
    for (const auto& Param : Bindings.Parameters)
    {
        const uint8* Source = Data.GetData() + Param.ByteOffset;
        BatchedParameters.SetShaderParameter(
            Param.BufferIndex,   // Constant Buffer索引
            Param.BaseIndex,     // CB内偏移
            Param.ByteSize,      // 参数大小
            Source               // 源数据
        );
    }

    // 【步骤3】提取资源参数（Texture/UAV/Sampler）
    ExtractShaderParameterResources(
        BatchedParameters, Bindings, Data,
        EUniformBufferBindingFlags::Shader
    );
}
```

**核心思想**：一次遍历+批量拷贝。

标量参数通过memcpy拷贝到Constant Buffer对应偏移；资源参数通过独立的SetShaderTexture/SetShaderSampler绑定。所有操作收集到`FRHIBatchedShaderParameters`后，一次性提交到RHI——避免了多次RHI调用的开销。

---

### 3.4.3 关键细节

**FShaderParameterBindings**：编译后生成的映射表

```cpp
struct FShaderParameterBindings
{
    struct FParameter
    {
        uint16 BufferIndex;   // 对应GPU Constant Buffer的索引
        uint16 BaseIndex;     // CB内偏移（float4为单位）
        uint16 ByteSize;      // 参数大小（字节）
        uint16 ByteOffset;    // 在FParameters struct中的字节偏移
    };

    TArray<FParameter> Parameters;         // 标量参数列表
    TArray<FResourceParameter> ResourceParameters;  // 资源参数列表
};
```

**两层偏移的必要性**：

- `ByteOffset`：FParameters结构体内的C++偏移（源地址）
- `BaseIndex`：GPU Constant Buffer内的偏移（目标地址）
- 两者可能不一致，因为C++对齐规则与GPU CB对齐规则不同

**批量提交的性能优势**：

| 方式 | 特征 | 说明 |
|------|------|------|
| 批量提交（SetShaderParameters） | 一次遍历Bindings并写入批处理参数 | 适合现代参数结构体路径 |
| 逐个提交（Legacy API） | 多次手动绑定 | 更容易出错，也更难统一优化 |
| 实际差异 | 取决于参数数量和RHI后端 | 用Unreal Insights或平台GPU工具实测 |

---

### 3.4.4 源码索引

| 内容 | 文件 | 搜索 token / 事实类型 |
|------|------|----------------------|
| `SetShaderParameters()`实现 | `Engine/Source/Runtime/RenderCore/Private/ShaderParameterStruct.cpp` | `void SetShaderParameters(`、`SetShaderParameters(BatchedParameters` / 参数拷贝 |
| `SetShaderParameters()`模板API | `Engine/Source/Runtime/RenderCore/Public/ShaderParameterStruct.h` | `inline void SetShaderParameters`、`FRHIBatchedShaderParameters` / 调用封装 |
| 资源参数提取 | `Engine/Source/Runtime/RenderCore/Private/ShaderParameterStruct.cpp` | `ExtractShaderParameterResources` / Texture/UAV/Sampler绑定 |
| 参数绑定表 | `Engine/Source/Runtime/RenderCore/Public/Shader.h` | `class FShaderParameterBindings`、`struct FParameter` / 编译后映射 |

### 💡 核心要点

`SetShaderParameters()`遍历编译后的Bindings映射表，将FParameters中的标量数据批量拷贝到Constant Buffer、资源参数独立绑定。一次提交替代多次RHI调用，是每帧DrawCall的热点路径。

## 五、FPixelShaderUtils::AddFullscreenPass() RDG全屏Pass

### 3.5.1 函数概览

### 3.5.2 核心流程

**函数签名**：

```cpp
template<typename TShaderClass>

);
```

**内部实现（伪代码）**：

```cpp
template<typename TShaderClass>
void FPixelShaderUtils::AddFullscreenPass(
    FRDGBuilder& GraphBuilder, ...)
{
    // 【步骤1】清理FParameters中未使用的RDG资源引用
    ClearUnusedGraphResources(PixelShader, Parameters);

    // 【步骤2】通过RDG添加Raster Pass
    GraphBuilder.AddPass(
        Forward<FRDGEventName>(PassName),
        Parameters,
        ERDGPassFlags::Raster,
        [PixelShader, Parameters, Viewport, BlendState, ...](FRHICommandList& RHICmdList)
        {
            // 【步骤3】设置视口
            RHICmdList.SetViewport(
                Viewport.Min.X, Viewport.Min.Y, 0.0f,
                Viewport.Max.X, Viewport.Max.Y, 1.0f
            );

            // 【步骤4】创建PSO
            FGraphicsPipelineStateInitializer PSOInit;
            InitFullscreenPipelineState(RHICmdList, GlobalShaderMap, PixelShader, PSOInit);

            if (BlendState)
                PSOInit.BlendState = BlendState;
            // ... 其他可选状态

            SetGraphicsPipelineState(RHICmdList, PSOInit, 0);

            // 【步骤5】设置Shader参数
            SetShaderParameters(RHICmdList, PixelShader, PixelShader.GetPixelShader(), *Parameters);

            // 【步骤6】绘制全屏三角形
            DrawFullscreenTriangle(RHICmdList, 1);
        }
    );
}
```

**为什么用全屏三角形而不是全屏四边形？**

全屏三角形只需3个顶点（vs四边形4个顶点+6个索引），且没有对角线接缝问题。对于覆盖整个屏幕的Pass，三角形比四边形少一半顶点处理开销。

---

### 3.5.3 关键细节

**与AddDrawScreenPass的区别**

引擎提供两套全屏Pass API：

| API | 所在模块 | 特点 | 适用场景 |
|-----|---------|------|---------|
| `FPixelShaderUtils::AddFullscreenPass` | RenderCore | 轻量，只需PS+Viewport | 简单全屏效果 |
| `AddDrawScreenPass` | Renderer | 支持InputViewport/OutputViewport对齐 | Screen Pass管线（后处理链） |

`AddDrawScreenPass`功能更多（支持`FScreenPassTextureViewport`做UV对齐），但依赖Renderer模块。如果你的插件不想依赖Renderer，用`FPixelShaderUtils::AddFullscreenPass`即可。

**使用模式**：

```cpp
// 获取Shader实例
TShaderMapRef<FMyShaderPS> PixelShader(View.ShaderMap);

// 分配参数
auto* Parameters = GraphBuilder.AllocParameters<FMyShaderPS::FParameters>();
Parameters->TintColor = FVector4f(1.0f, 0.5f, 0.0f, 1.0f);
Parameters->InputTexture = SomeRDGTexture;
Parameters->InputSampler = TStaticSamplerState<SF_Bilinear>::GetRHI();
Parameters->RenderTargets[0] = FRenderTargetBinding(OutputTexture, ERenderTargetLoadAction::ENoAction);

// 一行添加Pass
FPixelShaderUtils::AddFullscreenPass(
    GraphBuilder,
    View.ShaderMap,
    RDG_EVENT_NAME("MyCustomEffect"),
    PixelShader,
    Parameters,
    View.ViewRect
);
```

## 六、CompileGlobalShaderMap()编译流程

### 3.6.1 函数概览

**来源**：`Engine/Source/Runtime/Engine/Private/ShaderCompiler/ShaderCompiler.cpp`
**搜索 token**：`void CompileGlobalShaderMap(`、`VerifyGlobalShaders`
**版本**：UE5.7

**职责（一句话）**：遍历所有已注册的FGlobalShaderType，为每个Shader的每个变体创建编译任务，并行编译后存入FGlobalShaderMap。

**性能影响**：

- 发生在启动、Cook、Shader热重载或缓存失效阶段，不是每帧渲染热点
- 首次编译、DDC命中、共享DDC、Cooked缓存文件四种路径的耗时差异很大，必须按项目环境实测

---

### 3.6.2 核心流程

```cpp
void CompileGlobalShaderMap(EShaderPlatform Platform, ...)
{
    // 【步骤1】服务端/NullRHI等不需要渲染的场景提前返回
    if (FPlatformProperties::IsServerOnly() || !FApp::CanEverRender())
        return;

    // 【步骤2】刷新或创建GlobalShaderMap
    // 真实代码会先尝试ODSC、Cooked缓存文件、DDC section等加载路径。
    GGlobalShaderMap[Platform] = new FGlobalShaderMap(Platform);

    // 【步骤3】编辑器可编译时，构造FGlobalShaderMapId并提交DDC请求
    // DDC miss的section会在后续VerifyGlobalShaders中补编译。

    // 【步骤4】确认缺失Shader并提交编译任务
    VerifyGlobalShaders(Platform, TargetPlatform, bLoadedFromCacheFile);

    // 【步骤5】必要时创建RHI Shader资源
    GGlobalShaderMap[Platform]->BeginCreateAllShaders();
}
```

**流程图**：

```
引擎启动
    ↓
CompileGlobalShaderMap()
    ↓
检查DDC缓存 ─── 命中 → 加载缓存记录 → 完成
    │
    └─ 未命中 ↓
遍历FShaderType::TypeList链表
    ├─ 为每个ShaderType × 每个PermutationId创建编译任务
    └─ ShouldCompilePermutation()裁剪不需要的变体
    ↓
Shader编译工作队列
    ├─ 预处理（#include展开）
    ├─ dxc等平台编译器生成目标字节码
    └─ 反射提取ParameterMap
    ↓
将编译结果存入GGlobalShaderMap
    ↓
保存到DDC
    ↓
完成
```

---

### 3.6.3 关键细节

**并行编译优化**：

UE5的Shader编译由`FShaderCompilingManager`提交任务并等待完成；具体并行度受ShaderCompileWorker数量、平台编译器、任务粒度和机器配置影响。不要把某台机器的编译倍率当作通用结论。

**recompileshaders命令**：

修改.usf后无需重启编辑器，可以在控制台输入：

```
recompileshaders changed
```

引擎会检测哪些.usf文件被修改，只重新编译受影响的Shader。

### 3.6.4 源码索引

| 内容 | 文件 | 搜索 token / 事实类型 |
|------|------|----------------------|
| `CompileGlobalShaderMap()` | `Engine/Source/Runtime/Engine/Private/ShaderCompiler/ShaderCompiler.cpp` | `void CompileGlobalShaderMap(`、`TryLoadCookedGlobalShaderMap` / 全局ShaderMap入口 |
| 缺失Shader验证 | 同上 | `VerifyGlobalShaders`、`BeginCompileShader` / 编译任务创建 |
| 编译任务提交 | 同上 | `FShaderCompilingManager::SubmitJobs`、`FinishCompilation` / 任务调度 |
| DDC请求 | 同上 | `GetCache().Get`、`GetGlobalShaderMapKey` / 缓存读取 |

### 💡 核心要点

`CompileGlobalShaderMap()`负责创建或刷新全局ShaderMap，先尝试Cooked缓存/ODSC/DDC等加载路径，再通过`VerifyGlobalShaders()`补齐缺失的GlobalShader和变体。DDC命中能避免重复编译，但实际耗时由项目、平台和缓存环境决定。

## 七、TShaderPermutationDomain变体管理

### 3.7.1 函数概览

**来源**：`Engine/Source/Runtime/RenderCore/Public/ShaderPermutation.h`
**搜索 token**：`struct TShaderPermutationDomain`、`SHADER_PERMUTATION_BOOL`、`SHADER_PERMUTATION_INT`
**版本**：UE5.7

**职责（一句话）**：管理多维度Shader变体，提供PermutationId ↔ 各维度值的双向转换，以及编译时自动插入`#define`宏。

---

### 3.7.2 核心流程

**定义变体维度**：

```cpp
// 每个维度是一个类，定义宏名和取值范围
class FDim_UseGamma : SHADER_PERMUTATION_BOOL("USE_GAMMA_CORRECTION");
class FDim_UseDesat : SHADER_PERMUTATION_BOOL("USE_DESATURATION");
class FDim_Quality  : SHADER_PERMUTATION_INT("QUALITY_LEVEL", 3);  // 0,1,2

// 组合为排列域
using FPermutationDomain = TShaderPermutationDomain<
    FDim_UseGamma,   // 2个值
    FDim_UseDesat,   // 2个值
    FDim_Quality     // 3个值
>;
// 总变体数 = 2 × 2 × 3 = 12
```

**PermutationId编解码**（类似多进制数）：

```
PermutationId = UseGamma * 1 + UseDesat * 2 + Quality * 4

例如：UseGamma=1, UseDesat=0, Quality=2
  → Id = 1*1 + 0*2 + 2*4 = 9
```

**编译时自动插入#define**：

当编译PermutationId=9时，`ModifyCompilationEnvironment`自动插入：

```hlsl
#define USE_GAMMA_CORRECTION 1
#define USE_DESATURATION 0
#define QUALITY_LEVEL 2
```

HLSL中通过`#if`使用这些宏：

```hlsl
#if USE_GAMMA_CORRECTION
    Color.rgb = pow(Color.rgb, 1.0 / 2.2);
#endif

#if QUALITY_LEVEL >= 2
    Color = ApplyHighQualityAA(Color);
#endif
```

**运行时选择变体**：

```cpp
FMyShaderPS::FPermutationDomain PermutationVector;
PermutationVector.Set<FMyShaderPS::FDim_UseGamma>(bEnableGamma);
PermutationVector.Set<FMyShaderPS::FDim_UseDesat>(bEnableDesat);
PermutationVector.Set<FMyShaderPS::FDim_Quality>(QualityLevel);

TShaderMapRef<FMyShaderPS> PixelShader(GlobalShaderMap, PermutationVector);
```

---

### 3.7.3 关键细节

**变体裁剪：ShouldCompilePermutation**

```cpp
static bool ShouldCompilePermutation(
    const FGlobalShaderPermutationParameters& Parameters)
{
    FPermutationDomain PermVec(Parameters.PermutationId);

    // 互斥裁剪：Gamma和Desat不能同时开启
    if (PermVec.Get<FDim_UseGamma>() && PermVec.Get<FDim_UseDesat>())
        return false;

    // 依赖裁剪：Quality=0时不需要Desat
    if (PermVec.Get<FDim_Quality>() == 0 && PermVec.Get<FDim_UseDesat>())
        return false;

    return true;
}
```

**裁剪效果示例**：

| 策略 | 变体数（理论12个） | 影响 |
|------|------------------|------|
| 无裁剪 | 12个 | 所有组合都会进入编译候选 |
| 互斥裁剪 | 9个 | 减少无意义组合 |
| 互斥+依赖裁剪 | 7个 | 进一步减少编译任务和DDC记录 |

**SHADER_PERMUTATION_BOOL展开**：

```cpp
#define SHADER_PERMUTATION_BOOL(DefineName) \
public: \
    using Type = bool; \
    static constexpr const TCHAR* Name = TEXT(DefineName); \
    static constexpr int32 PermutationCount = 2;
```

---

### 3.7.4 源码索引

| 内容 | 文件 | 搜索 token / 事实类型 |
|------|------|----------------------|
| `TShaderPermutationDomain`空包 | `Engine/Source/Runtime/RenderCore/Public/ShaderPermutation.h` | `struct TShaderPermutationDomain`、`PermutationCount = 1` / 空域 |
| `TShaderPermutationDomain`递归域 | 同上 | `struct TShaderPermutationDomain<TDimension, Ts...>`、`Tail` / 多维编码 |
| `SHADER_PERMUTATION_BOOL` | 同上 | `#define SHADER_PERMUTATION_BOOL` / bool维度 |
| `SHADER_PERMUTATION_INT` | 同上 | `#define SHADER_PERMUTATION_INT` / int维度 |
| `FShaderPermutationNone` | 同上 | `using FShaderPermutationNone` / 无变体默认域 |

### 💡 核心要点

`TShaderPermutationDomain`通过变参模板管理多维度变体，编译期自动计算总变体数并插入`#define`。`ShouldCompilePermutation()`允许裁剪无意义的变体组合，减少编译时间和DDC大小。

```hlsl
float4 TintColor;

void MainPS(
    in noperspective float4 UVAndScreenPos : TEXCOORD0,
    out float4 OutColor : SV_Target0

    // 采样输入
```

## 三、三类高频问题排查

### 4.10 问题1：Shader未找到

**症状**：

```
LogShaders: Error: Failed to find shader type 'FTinyTintPS'
```

**排查清单**：

| # | 检查项 | 常见原因 |
|---|--------|---------|
| 1 | `IMPLEMENT_GLOBAL_SHADER`是否在.cpp中 | 忘记写或写在.h中（链接错误） |
| 2 | 虚拟路径是否注册 | `AddShaderSourceDirectoryMapping`未调用 |
| 3 | .usf文件是否存在 | 路径拼写错误 |
| 4 | 模块是否被加载 | .Build.cs / .uproject中未引用模块 |
| 5 | IMPLEMENT和DECLARE的类名是否一致 | 拼写不一致导致StaticType未定义 |

**快速诊断**：

```cpp
// 在模块StartupModule中添加验证
UE_LOG(LogTemp, Log, TEXT("TinyTint Shader registered: %s"),
    FTinyTintPS::StaticType.GetName());
```

如果输出正确的Shader名，说明注册成功。

### 4.11 问题2：参数绑定失败

**症状**：

```
LogShaders: Warning: Shader parameter 'TintColor' not bound in FTinyTintPS
```

或效果看起来全黑/全白（参数未生效）。

**排查清单**：

| # | 检查项 | 常见原因 |
|---|--------|---------|
| 1 | C++参数名与HLSL是否完全一致 | 大小写不匹配（`TintColor` vs `tintColor`） |
| 2 | 参数是否在HLSL中被使用 | 未使用的参数会被编译器优化掉 |
| 3 | 类型是否匹配 | `FVector4f` ↔ `float4`，`float` ↔ `float` |
| 4 | RDG纹理是否有效 | `nullptr`会导致GPU崩溃 |

**防止参数被优化掉**：

如果参数在某些`#if`分支中未使用，编译器可能将其优化掉。在HLSL中确保参数在所有变体分支中都被引用，哪怕只是`(void)TintColor;`。

### 4.12 问题3：变体缺失

**症状**：

```
Assertion failed: Shader.IsValid()
// 或
LogShaders: Error: Permutation not found for FTinyTintPS
```

**排查清单**：

| # | 检查项 | 常见原因 |
|---|--------|---------|
| 1 | `ShouldCompilePermutation`是否裁剪过度 | 裁剪了运行时需要的变体 |
| 2 | 运行时请求的变体组合是否被编译 | 变体维度设置与裁剪逻辑矛盾 |

**安全写法**：运行时选择变体前先检查有效性：

```cpp
TShaderMapRef<FTinyTintPS> PixelShader(GlobalShaderMap, PermutationVector);
if (!PixelShader.IsValid())
{
    // Fallback到基础变体
    FTinyTintPS::FPermutationDomain FallbackVector;
    FallbackVector.Set<FTinyTintPS::FDim_UseDesaturation>(false);
    PixelShader = TShaderMapRef<FTinyTintPS>(GlobalShaderMap, FallbackVector);
}
```

# 第五部分：与其他系统的关系

> **核心目标**：理解自定义Shader在整个渲染管线中的位置，以及何时应该选择其他方案。

---

## 一、与RenderGraph的关系

### 5.1 FGlobalShader必须通过RDG调度

自定义Shader不能直接"运行"——它必须被添加到RDG图中，由RDG调度执行。

```
FGlobalShader（定义Shader逻辑）
    ↓ 通过FPixelShaderUtils::AddFullscreenPass
FRDGBuilder（构建渲染图）
    ↓ 编译阶段
FRDGPass（渲染Pass节点）
    ↓ 执行阶段
FRHICommandList（RHI命令）
    ↓
GPU执行
```

RDG负责：
- 资源生命周期管理（Transient Aliasing）
- Pass间依赖排序
- 资源状态转换（Texture Layout Transitions）
- 未使用资源清理（ClearUnusedGraphResources）

**不推荐**直接使用RHI命令绘制，除非你明确需要绕过RDG（极少数场景）。

---

## 二、与材质系统的关系

### 5.2 FGlobalShader vs 材质Shader

| 维度 | FGlobalShader | 材质Shader |
|------|--------------|-----------|
| 参数来源 | C++代码直接设置 | 材质编辑器节点图 |
| 变体来源 | TShaderPermutationDomain | 材质表达式生成 |
| 编译时机 | 引擎启动时 | 材质保存时 |
| 实例数量 | 1个/平台（或1个/变体） | 每个材质1个 |
| 修改方式 | 改.usf + recompileshaders | 改节点图 + Apply |

**什么时候用材质系统**：当效果可以用节点图表达，且不需要自定义Pass顺序/RenderTarget时。

**什么时候用FGlobalShader**：当需要比材质系统更细的管线控制——自定义Pass、读写特定Buffer、ComputeShader、多Pass联动。

---

## 三、与后处理管线的关系

### 5.3 注入后处理管线

自定义Shader最常见的注入点是后处理管线（详见[6.1 PostProcess详解](./6.1_PostProcess_Deep.md)）。

**注入方式**：

1. **Subsystem回调**：通过`IRendererModule::GetPostOpaqueRenderDelegate()`注册回调
2. **SceneViewExtension**：实现`ISceneViewExtension`接口，在`PrePostProcessPass_RenderThread`中注入
3. **修改PostProcessing.cpp**：直接在引擎后处理链中添加Pass（不推荐，除非你在修改引擎）

推荐方式2，因为它不需要修改引擎代码，通过插件即可实现。

---

## 四、什么时候不该写自定义Shader

### 5.4 反模式清单

| 想要的效果 | 不该写自定义Shader | 应该用 |
|-----------|-------------------|--------|
| 修改材质外观 | ❌ | 材质编辑器 |
| 简单后处理（Tint/Vignette） | ❌ | PostProcessMaterial |
| 嵌入一段HLSL计算 | ❌ | CustomExpression节点 |
| 修改光照模型 | ❌ | Substrate / Shading Model |
| 粒子视觉效果 | ❌ | Niagara |

**只有当以上方案都不满足需求时**，才考虑写自定义Shader。自定义Shader的维护成本远高于材质节点图。
