### 1.1 Dispatch机制 - 与DrawCall的本质差异

#### 1.1.1 背景：为什么需要Dispatch？

**Graphics Shader问题**：
- 只能处理"像素"或"顶点"（固定输入格式）
- 数据必须通过Mesh或RenderTarget输入
- 无法处理任意结构的数据（如粒子数组、物理网格）

**ComputeShader解决方案**：
- 可处理任意数据（StructuredBuffer、RWTexture、RWBuffer）
- 自定义线程数量和维度（1D/2D/3D）
- 通过Dispatch调度，而非Draw

---

#### 1.1.2 DrawCall vs Dispatch对比

**DrawCall（Graphics Shader）**：

```cpp
// 绘制100个三角形
RHICmdList.DrawIndexedPrimitive(
    IndexBuffer,
    0,              // BaseVertexIndex
    0,              // MinIndex
    NumVertices,    // NumVertices
    0,              // StartIndex
    NumTriangles,   // NumPrimitives
    1               // NumInstances
);

// GPU执行：
// VS执行次数 = NumVertices × NumInstances = 300次
// PS执行次数 = NumPixels（由光栅化决定，可能10万+）
```

**特点**：
- ✅ 顶点数由IndexBuffer决定
- ✅ 像素数由光栅化决定（不可控）
- ❌ 无法直接处理任意数据

---

**DispatchComputeShader（Compute Shader）**：

```cpp
// 处理1024个粒子
RHICmdList.DispatchComputeShader(
    32,  // ThreadGroupCountX（线程组数量）
    1,   // ThreadGroupCountY
    1    // ThreadGroupCountZ
);

// GPU执行：
// 假设NumThreads = [32, 1, 1]（每组32线程）
// 总线程数 = 32组 × 32线程/组 = 1024线程
// 每个线程处理1个粒子
```

**特点**：
- ✅ 线程数完全可控（精确指定）
- ✅ 可处理任意数据结构
- ✅ 没有光栅化（纯计算）

---

#### 1.1.3 Dispatch线程映射机制

**线程ID计算**：

```hlsl
// ComputeShader入口函数
[numthreads(32, 1, 1)]  // 每组32个线程
void MainCS(
    uint3 GroupID : SV_GroupID,                 // 线程组ID（由Dispatch参数决定）
    uint3 GroupThreadID : SV_GroupThreadID,     // 组内线程ID（0-31）
    uint3 DispatchThreadID : SV_DispatchThreadID, // 全局线程ID
    uint GroupIndex : SV_GroupIndex             // 组内线程索引（1D）
)
{
    // DispatchThreadID = GroupID × [32,1,1] + GroupThreadID
    
    // 示例：处理第100个粒子
    // GroupID = (3, 0, 0)，GroupThreadID = (4, 0, 0)
    // DispatchThreadID = (3×32 + 4, 0, 0) = (100, 0, 0)
    
    uint ParticleIndex = DispatchThreadID.x;
    Particles[ParticleIndex].Position += Particles[ParticleIndex].Velocity * DeltaTime;
}
```

**线程映射图**：

```
Dispatch(32, 1, 1)  ← 启动32个线程组
    ↓
┌─────────────────────────────────────────────────┐
│ 线程组0 (GroupID=0)                              │
│ ├─ 线程0  (DispatchThreadID=0)  → Particle[0]  │
│ ├─ 线程1  (DispatchThreadID=1)  → Particle[1]  │
│ ├─ ...                                          │
│ └─ 线程31 (DispatchThreadID=31) → Particle[31] │
├─────────────────────────────────────────────────┤
│ 线程组1 (GroupID=1)                              │
│ ├─ 线程0  (DispatchThreadID=32) → Particle[32] │
│ └─ ...                                          │
├─────────────────────────────────────────────────┤
│ ...                                              │
├─────────────────────────────────────────────────┤
│ 线程组31 (GroupID=31)                            │
│ └─ 线程31 (DispatchThreadID=1023) → Particle[1023] │
└─────────────────────────────────────────────────┘
```

#### 1.1.4 性能影响：Dispatch配置优化

> ⚠️ **性能数据说明**：以下为RTX 3070典型场景估算值

**问题**：如何选择ThreadGroupCount？

**原则1：对齐Warp/Wave大小**

```cpp
// ❌ 不好：31个线程组（浪费1个线程）
uint NumParticles = 992;
uint ThreadGroupCountX = (NumParticles + 31) / 32;  // = 31组
RHICmdList.DispatchComputeShader(ThreadGroupCountX, 1, 1);
// 总线程 = 31 × 32 = 992（正好）

// ✅ 好：对齐到64（2个Warp）
uint NumParticles = 1000;
uint ThreadGroupCountX = (NumParticles + 63) / 64;  // = 16组
RHICmdList.DispatchComputeShader(ThreadGroupCountX, 1, 1);
// 总线程 = 16 × 64 = 1024（浪费24线程，但Warp利用率100%）
```

**性能对比**：

| 配置 | 线程组数 | 浪费线程 | Warp利用率 | 实际性能 |
|------|---------|---------|-----------|---------|
| 31组×32线程 | 31 | 0 | 96.9% | 基准 |
| 16组×64线程 | 16 | 24 | 100% | **快15%** |

**原因**：Warp是GPU调度最小单位（NVIDIA=32线程，AMD=64线程），未对齐会导致空闲。

---

**原则2：避免单维度过大**

```cpp
// ❌ 不好：单维度65536组
DispatchComputeShader(65536, 1, 1);  // 可能超出硬件限制

// ✅ 好：分解为2D
uint GroupsX = 256;
uint GroupsY = 256;  // 256 × 256 = 65536
DispatchComputeShader(GroupsX, GroupsY, 1);
```

**硬件限制**：

| 平台 | MaxThreadGroupCountX | MaxThreadGroupCountY | MaxThreadGroupCountZ |
|------|---------------------|---------------------|---------------------|
| D3D12 | 65535 | 65535 | 65535 |
| Vulkan | 65535 | 65535 | 65535 |
| Metal | 限制更严格（查文档） | - | - |

### 1.2 NumThreads配置 - 线程组大小优化

#### 1.2.1 背景：什么是线程组？

**定义**：线程组（ThreadGroup）是GPU调度的基本单位，组内线程可共享GroupShared内存。

```hlsl
// [numthreads(X, Y, Z)]
// X × Y × Z = 每组总线程数（通常64-1024）
[numthreads(8, 8, 1)]  // 8×8 = 64线程/组
void ProcessTexture(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    // 处理纹理像素
    OutputTexture[DispatchThreadID.xy] = InputTexture[DispatchThreadID.xy] * 2.0;
}
```

---

#### 1.2.2 NumThreads配置原则

**原则1：总数应为Warp/Wave倍数**

```hlsl
// ✅ 推荐配置（64线程 = 2个Warp）
[numthreads(8, 8, 1)]   // 8×8 = 64
[numthreads(64, 1, 1)]  // 64×1 = 64
[numthreads(4, 4, 4)]   // 4×4×4 = 64

// ⚠️ 可接受（128线程 = 4个Warp）
[numthreads(16, 8, 1)]  // 16×8 = 128
[numthreads(128, 1, 1)] // 128×1 = 128

// ❌ 不推荐（非Warp倍数，浪费资源）
[numthreads(10, 10, 1)] // 10×10 = 100（浪费28线程）
[numthreads(7, 7, 1)]   // 7×7 = 49（浪费15线程）
```

---

**原则2：根据任务类型选择维度**

| 任务类型 | 推荐配置 | 理由 |
|---------|---------|------|
| **1D数据**（粒子、数组） | `[numthreads(64, 1, 1)]` | 连续内存访问，Cache友好 |
| **2D纹理**（图像处理） | `[numthreads(8, 8, 1)]` | 2D局部性，利用Texture Cache |
| **3D体素**（体积光、流体） | `[numthreads(4, 4, 4)]` | 3D局部性 |
| **小数据量**（<256元素） | `[numthreads(256, 1, 1)]` | 单组完成，减少Dispatch开销 |

---

**原则3：考虑寄存器压力**

```hlsl
// ❌ 不好：每线程使用100个寄存器
[numthreads(256, 1, 1)]  // 256线程/组
void HeavyCompute(...)
{
    float temp[25];  // 大量局部变量（占用寄存器）
    // 复杂计算...
}
// SM占用率 = 50%（寄存器不足）

// ✅ 好：减少线程数或优化寄存器使用
[numthreads(64, 1, 1)]   // 减少到64线程/组
void HeavyCompute(...)
{
    // 相同计算，但SM占用率提升到80%
}
```

**SM占用率（Occupancy）**：

| 每线程寄存器数 | 256线程/组 | 128线程/组 | 64线程/组 |
|--------------|-----------|-----------|----------|
| 32个 | 100% | 100% | 100% |
| 64个 | 50% | 100% | 100% |
| 128个 | ❌ 无法启动 | 50% | 100% |

---

#### 1.2.3 实际案例：纹理生成优化

**场景**：生成1024×1024的程序化噪声纹理

**方案对比**：

```hlsl
// 方案A：[numthreads(32, 32, 1)] = 1024线程/组
[numthreads(32, 32, 1)]
void GenerateNoiseA(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    OutputTexture[DispatchThreadID.xy] = PerlinNoise(DispatchThreadID.xy);
}
// Dispatch(32, 32, 1) → 32×32 = 1024个线程组
```

```hlsl
// 方案B：[numthreads(8, 8, 1)] = 64线程/组
[numthreads(8, 8, 1)]
void GenerateNoiseB(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    OutputTexture[DispatchThreadID.xy] = PerlinNoise(DispatchThreadID.xy);
}
// Dispatch(128, 128, 1) → 128×128 = 16384个线程组
```

**性能测试结果**（RTX 3070）：

| 方案 | 线程组大小 | 线程组数 | 耗时 | SM占用率 |
|------|-----------|---------|------|---------|
| A | 1024 | 1024 | 2.5ms | 65% |
| B | 64 | 16384 | **1.8ms** | **90%** |

**结论**：方案B快38%，因为：
1. 64线程/组更灵活调度（1024线程/组占用过多资源）
2. SM占用率更高（90% vs 65%）
3. 线程组数更多，GPU并行度更高

### 1.3 同步原语 - GroupShared内存和Barrier

#### 1.3.1 背景：为什么需要同步？

**问题场景**：计算数组的和（Reduction）

```hlsl
// ❌ 错误：没有同步导致竞态条件
groupshared float SharedSum;

[numthreads(64, 1, 1)]
void SumReduction(uint GroupIndex : SV_GroupIndex)
{
    if (GroupIndex == 0)
        SharedSum = 0;  // 线程0初始化
    
    // ⚠️ 问题：其他线程可能在初始化完成前就读取SharedSum
    SharedSum += InputData[GroupIndex];  // 竞态条件！
    
    if (GroupIndex == 0)
        OutputData[0] = SharedSum;
}
```

**解决方案**：使用Barrier同步

```hlsl
// ✅ 正确：使用GroupMemoryBarrierWithGroupSync
groupshared float SharedData[64];

[numthreads(64, 1, 1)]
void SumReduction(uint GroupIndex : SV_GroupIndex)
{
    // 【步骤1】每个线程加载数据到SharedMemory
    SharedData[GroupIndex] = InputData[GroupIndex];
    
    // 【步骤2】等待所有线程完成加载
    GroupMemoryBarrierWithGroupSync();  // ← 同步点
    
    // 【步骤3】归约（只有线程0执行）
    if (GroupIndex == 0)
    {
        float Sum = 0;
        for (uint i = 0; i < 64; ++i)
            Sum += SharedData[i];
        OutputData[0] = Sum;
    }
}
```

---

#### 1.3.2 同步原语类型

**UE5/HLSL提供的Barrier**：

```hlsl
// 1. GroupMemoryBarrierWithGroupSync()
// 作用：等待组内所有线程 + 刷新GroupShared内存
// 适用：组内线程协作
GroupMemoryBarrierWithGroupSync();

// 2. GroupMemoryBarrier()
// 作用：只刷新GroupShared内存，不等待线程
// 适用：手动控制同步
GroupMemoryBarrier();

// 3. DeviceMemoryBarrierWithGroupSync()
// 作用：等待组内所有线程 + 刷新全局内存（UAV）
// 适用：多组协作写入全局Buffer
DeviceMemoryBarrierWithGroupSync();

// 4. DeviceMemoryBarrier()
// 作用：只刷新全局内存
DeviceMemoryBarrier();

// 5. AllMemoryBarrierWithGroupSync()
// 作用：最强同步（GroupShared + 全局内存 + 等待线程）
// 适用：复杂多级同步
AllMemoryBarrierWithGroupSync();
```

**性能对比**：

| Barrier类型 | 同步范围 | 耗时（估算） | 使用频率 |
|------------|---------|------------|---------|
| GroupMemoryBarrierWithGroupSync | 组内 | ~10 cycles | ✅ 最常用 |
| DeviceMemoryBarrierWithGroupSync | 全局 | ~50 cycles | ⚠️ 慎用 |
| AllMemoryBarrierWithGroupSync | 全部 | ~60 cycles | ❌ 尽量避免 |

---

#### 1.3.3 GroupShared内存最佳实践

**内存限制**：

| 平台 | GroupShared限制 |
|------|---------------|
| D3D12 (SM5.0+) | 32KB/组 |
| Vulkan | 32KB/组 |
| Metal | 32KB/组（部分设备16KB）|

**示例：优化纹理模糊**

```hlsl
// ❌ 不好：重复采样（每个像素采样9次）
[numthreads(8, 8, 1)]
void BlurBad(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    float4 Sum = 0;
    for (int y = -1; y <= 1; ++y)
        for (int x = -1; x <= 1; ++x)
            Sum += InputTexture[DispatchThreadID.xy + int2(x, y)];
    OutputTexture[DispatchThreadID.xy] = Sum / 9.0;
}
// 每组64线程，总采样 = 64 × 9 = 576次
```

```hlsl
// ✅ 好：使用GroupShared缓存
groupshared float4 SharedTile[10][10];  // 缓存10×10区域

[numthreads(8, 8, 1)]
void BlurGood(uint3 GroupThreadID : SV_GroupThreadID, uint3 DispatchThreadID : SV_DispatchThreadID)
{
    // 【步骤1】加载到SharedMemory（含边界）
    SharedTile[GroupThreadID.y + 1][GroupThreadID.x + 1] = InputTexture[DispatchThreadID.xy];
    
    // 边界线程额外加载周围像素
    if (GroupThreadID.x == 0)
        SharedTile[GroupThreadID.y + 1][0] = InputTexture[DispatchThreadID.xy + int2(-1, 0)];
    // ... 其他边界 ...
    
    // 【步骤2】同步
    GroupMemoryBarrierWithGroupSync();
    
    // 【步骤3】从SharedMemory计算（无纹理采样）
    float4 Sum = 0;
    for (int y = 0; y < 3; ++y)
        for (int x = 0; x < 3; ++x)
            Sum += SharedTile[GroupThreadID.y + y][GroupThreadID.x + x];
    OutputTexture[DispatchThreadID.xy] = Sum / 9.0;
}
// 每组64线程，总采样 = 10×10 = 100次（减少82%）
```

**性能提升**：

| 方案 | 纹理采样次数 | L1 Cache命中率 | 耗时 |
|------|------------|--------------|------|
| 无SharedMemory | 576次/组 | 60% | 基准 |
| 有SharedMemory | 100次/组 | 95% | **快3倍** |

## 💡 本节核心要点

**5秒版**：ComputeShader通过Dispatch精确控制线程数，NumThreads配置影响SM占用率（推荐64倍数），GroupShared内存配合Barrier实现组内高速协作（比全局内存快10倍）。

**关键词**：Dispatch调度、DispatchThreadID、NumThreads对齐Warp、GroupShared缓存、GroupMemoryBarrierWithGroupSync、SM占用率

**与Unity对比**：UE5和Unity的ComputeShader语法完全一致（都是HLSL），差异在C++调用层（UE5用RDG，Unity用CommandBuffer）。

### 2.2 ComputeShader vs Graphics Shader核心差异

#### 2.2.1 对比表

| 维度 | Graphics Shader | ComputeShader |
|------|----------------|--------------|
| **调用方式** | DrawPrimitive/DrawIndexed | DispatchComputeShader |
| **输入** | VertexBuffer + IndexBuffer | 任意Buffer/Texture |
| **输出** | RenderTarget（只写） | UAV（可读写）|
| **线程数** | 由几何体决定（不可控） | 完全可控（Dispatch参数）|
| **管线阶段** | VS→HS→DS→GS→RS→PS | 单阶段（CS）|
| **典型用途** | 渲染图形 | 通用计算 |

---

#### 2.2.2 内存访问模式对比

**Graphics Shader**：

```hlsl
// PixelShader只能写入RenderTarget
void MainPS(
    in float2 UV : TEXCOORD0,
    out float4 OutColor : SV_Target0  // ← 只写，不能读
)
{
    OutColor = InputTexture.Sample(Sampler, UV);
    
    // ❌ 不能做：读取OutputTexture
    // ❌ 不能做：随机写入其他像素
}
```

**ComputeShader**：

```hlsl
// ComputeShader可以读写UAV
RWTexture2D<float4> OutputTexture;  // ← UAV，可读写

[numthreads(8, 8, 1)]
void MainCS(uint3 DispatchThreadID : SV_DispatchThreadID)
{
    uint2 Coord = DispatchThreadID.xy;
    
    // ✅ 可以读取
    float4 CurrentValue = OutputTexture[Coord];
    
    // ✅ 可以写入当前位置
    OutputTexture[Coord] = CurrentValue * 2.0;
    
    // ✅ 可以写入其他位置
    OutputTexture[Coord + uint2(1, 0)] = float4(1, 0, 0, 1);
```

```cpp
template<typename ParameterStructType, typename ExecuteLambdaType>
void FRDGBuilder::AddPass(
    FRDGEventName&& Name,
    ParameterStructType* Parameters,
    ERDGPassFlags Flags,  // ERDGPassFlags::Compute
    ExecuteLambdaType&& ExecuteLambda)
{
    // 【步骤1】创建Pass对象
    FRDGPass* Pass = new FRDGPass(Name, Parameters, Flags);
    
    // 【步骤2】分析参数依赖（哪些资源被读/写）
    for (auto& UAV : Parameters->UAVs)
    {
        Pass->AddUAVAccess(UAV, ERHIAccess::UAVCompute);
    }
    
    // 【步骤3】插入UAV Barrier（如果需要）
    if (PreviousPass->WritesTo(UAV) && Pass->ReadsFrom(UAV))
    {
        InsertUAVBarrier(UAV);  // 防止Read-After-Write Hazard
    }
    
    // 【步骤4】记录Execute Lambda
    Pass->ExecuteLambda = ExecuteLambda;
    
    // 【步骤5】添加到Pass列表
    Passes.Add(Pass);
}
```
