# 第一部分：理解控制台系统的存在意义

> **核心目标**：从渲染开发中的实际痛点出发，理解控制台系统解决了什么问题

---

## 一、修改一个渲染参数的痛苦

### 1.1 问题场景

假设你正在调试阴影质量：Shadow Map分辨率从2048改到4096，看看画面差异。如果没有控制台系统，你需要：

```
修改代码中的常量：MaxShadowResolution = 4096
    ↓
重新编译（C++ → 30-120秒）
    ↓
重启编辑器（加载资源 → 20-60秒）
    ↓
打开关卡、进入PIE（10-20秒）
    ↓
观察画面效果
    ↓
不满意？回到第一步

总耗时：60-200秒/次迭代
一个下午调10次参数 → 浪费30分钟在等编译
```

**问题1：迭代速度不可接受**

渲染调试需要频繁微调参数（Shadow Bias、Screen Percentage、Lumen质量等级...），每次改动等60-200秒不现实。

**问题2：参数组合爆炸**

Nanite Visualize有10+模式、Lumen Visualize有5+模式、Shadow有CSM/VSM两条路径。如果要逐一测试，需要编译数十次。

**问题3：发布版本无法调试**

发布后的构建不可能重编译。线上出了渲染问题，没有运行时调参手段等于盲人摸象。

### 1.2 问题量化

```
典型渲染参数调试的迭代频率需求：
├─ Shadow Bias调整：需要5-20次微调才能消除Acne/Peter Panning
├─ Screen Percentage：需要对比50%/75%/100%/200%四档画面
├─ Lumen质量：需要对比不同ScreenProbe密度、Trace长度
├─ 性能分析：需要逐一开关各个Pass（Shadow/GI/Reflection/PostProcess）
└─ 总计：一个调试会话中修改参数50-100次

如果每次60秒编译：50次 × 60秒 = 50分钟纯等待
如果有控制台命令：50次 × 1秒输入 = 50秒

效率差距：60倍
```

**核心需求**：一套系统，能够：
1. **运行时修改参数**（无需重新编译、无需重启）
2. **平均哈希路径执行**（数千命令中通过名称快速定位）
3. **线程安全**（GameThread修改、RenderThread读取）
4. **自动注册**（写一个static变量就完成注册，无需手动初始化）

这就是UE5控制台命令系统存在的意义。

## 二、解决方案：运行时参数调控

### 2.1 控制台系统做什么

```
┌──────────────────────────────────────────────────────────────┐
│ 用户输入                                                      │
│   按 ~ 打开Console → 输入 r.Shadow.MaxResolution 4096        │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ IConsoleManager（全局单例）                                    │
│   ├── TMap<FString, IConsoleObject*>    命令/变量注册表        │
│   ├── FindConsoleObject("r.Shadow.MaxResolution")             │
│   │       └── O(1) 哈希查找 → 找到 IConsoleVariable*         │
│   └── Set(4096)    修改变量值                                 │
│           └── 如果标记ECVF_RenderThreadSafe                   │
│               → 同步Shadow Copy到RenderThread                 │
└──────────────────────────────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ 渲染代码（RenderThread）                                      │
│   int32 MaxRes = CVarShadowMaxRes.GetValueOnRenderThread();   │
│   // 返回4096（Shadow Copy已更新）                            │
│   // 下一帧立即生效，无需重启                                  │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 存储结构 | TMap<FString, IConsoleObject*> | 平均哈希查找；交互输入足够快，热路径仍应缓存 |
| 注册方式 | 静态全局变量自动注册 | 低侵入，无需手动Init/Shutdown |
| 线程语义 | Shadow Copy（双份值） | GameThread读写[0]、RenderThread读[1]；读取路径不需要锁 |
| 类型安全 | Delegate系统 | 编译期检查参数类型 |
| 优先级系统 | SetBy层级（Constructor < Ini < Console） | 高优先级覆盖低优先级 |

## 四、三类控制台对象

### 4.1 对象分类

| 类型 | 基类 | 用途 | 注册方式 | 典型场景 |
|------|------|------|---------|---------|
| **Console Variable** | IConsoleVariable | 存储一个值（int/float/bool/string） | TAutoConsoleVariable | `r.Shadow.MaxResolution 4096` |
| **Console Variable Ref** | IConsoleVariable | 绑定一个C++变量 | FAutoConsoleVariableRef | `r.Nanite.MaxNodes` 绑定到 GNaniteMaxNodes |
| **Console Command** | IConsoleCommand | 执行一个操作 | FAutoConsoleCommand | `stat GPU`、`recompileshaders` |

### 4.2 区别

**Variable vs Command**：
- Variable存储一个值，可以反复读取，`r.ScreenPercentage`就是Variable
- Command执行一次操作然后结束，`recompileshaders`就是Command

**Variable vs VariableRef**：
- Variable在ConsoleManager内部分配存储空间
- VariableRef绑定到一个已有的C++变量，修改CVar时直接修改那个变量

```
TAutoConsoleVariable<int32>（内部存储）：
    ConsoleManager内部 → [值=4096] ← 读取需要调用GetValueOn*()
    
FAutoConsoleVariableRef（引用绑定）：
    int32 GMaxResolution = 4096;   ← C++全局变量
    CVar修改时直接写入 GMaxResolution
    读取时直接访问 GMaxResolution，调用开销最低
```

## 二、IConsoleManager：全局单例管理器

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `struct IConsoleManager`
**实现**：`Engine/Source/Runtime/Core/Private/HAL/ConsoleManager.h`，搜索 `class FConsoleManager`

### 2.2 核心接口

```cpp
struct IConsoleManager
{
    // 获取全局单例
    static IConsoleManager& Get();

    // ═══ 注册 ═══
```

### 2.3 FConsoleManager内部存储

**来源**：`Engine/Source/Runtime/Core/Private/HAL/ConsoleManager.h`，搜索 `class FConsoleManager`

```cpp
class FConsoleManager : public IConsoleManager
{
    // 核心存储：命令名 → 控制台对象指针
```

```
查找 "r.Shadow.MaxResolution"：
1. Hash = GetTypeHash("r.Shadow.MaxResolution")  → 计算哈希值
2. 进入哈希容器定位候选槽位或桶
3. 比较候选键并返回对象指针

具体耗时取决于构建配置、字符串长度、缓存命中和平台；热路径仍应缓存 `IConsoleVariable*` 或使用 `TAutoConsoleVariable<T>`。
```

## 三、命令注册：FAutoConsoleCommand

### 3.1 设计目标

FAutoConsoleCommand通过C++静态初始化机制实现**自动注册**：声明一个static变量，构造函数中注册命令，析构函数中反注册。无需手动Init/Shutdown。

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `class FAutoConsoleCommand`

### 3.2 支持的Delegate类型

```cpp
// 搜索 DECLARE_DELEGATE：控制台命令Delegate类型声明
DECLARE_DELEGATE( FConsoleCommandDelegate );                                           // 无参
DECLARE_DELEGATE_OneParam( FConsoleCommandWithArgsDelegate, const TArray<FString>& );  // 带参数
DECLARE_DELEGATE_OneParam( FConsoleCommandWithWorldDelegate, UWorld* );                 // 带World
DECLARE_DELEGATE_TwoParams( FConsoleCommandWithWorldAndArgsDelegate, const TArray<FString>&, UWorld* );
DECLARE_DELEGATE_TwoParams( FConsoleCommandWithArgsAndOutputDeviceDelegate, const TArray<FString>&, FOutputDevice& );
DECLARE_DELEGATE_ThreeParams( FConsoleCommandWithWorldArgsAndOutputDeviceDelegate, const TArray<FString>&, UWorld*, FOutputDevice& );
DECLARE_DELEGATE_OneParam( FConsoleCommandWithOutputDeviceDelegate, FOutputDevice& );  // 带输出设备
```

### 3.3 注册模式速查

| 需求 | Delegate类型 | 创建方式 | 场景 |
|------|-------------|---------|------|
| 无参数 | FConsoleCommandDelegate | `CreateStatic([](){})` | 简单触发器 |
| 带参数 | FConsoleCommandWithArgsDelegate | `CreateStatic([](const TArray<FString>& Args){})` | 接收用户输入 |

### 4.1 设计目标

TAutoConsoleVariable<T>是控制台变量的主要注册方式。它在内部分配TConsoleVariableData<T>存储值，支持GameThread/RenderThread双份Shadow Copy。

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `class TAutoConsoleVariable`

### 4.2 模板类定义

```cpp
template <class T>
class TAutoConsoleVariable : public FAutoConsoleObject
{
public:
    // 注册构造（搜索 TAutoConsoleVariable(const TCHAR* Name）
    TAutoConsoleVariable(const TCHAR* Name, const T& DefaultValue, const TCHAR* Help, uint32 Flags = ECVF_Default)
        : TAutoConsoleVariable(IConsoleManager::Get().RegisterConsoleVariable(Name, DefaultValue, Help, Flags))
    {}

    // 带变化回调的注册构造（搜索 SetOnChangedCallback）
    TAutoConsoleVariable(const TCHAR* Name, const T& DefaultValue, const TCHAR* Help,
                         const FConsoleVariableDelegate& Callback, uint32 Flags = ECVF_Default)
```

### 5.1 设计目标

FAutoConsoleVariableRef将CVar绑定到一个已有的C++全局变量。修改CVar时直接修改该变量，读取时无需调用GetValue函数。

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `class FAutoConsoleVariableRef`

### 5.2 使用模式

```cpp
    }
```

### 5.3 TAutoConsoleVariable vs FAutoConsoleVariableRef

| 特性 | TAutoConsoleVariable<T> | FAutoConsoleVariableRef |
|------|------------------------|----------------------|
| 存储位置 | 内部TConsoleVariableData | 外部C++变量 |
| 读取方式 | GetValueOnGameThread() | 直接读取全局变量 |
| 读取开销 | 函数调用，通常很低 | 直接内存读取，调用开销最低 |
| 线程安全 | Shadow Copy保证 | ⚠️ 需要手动保证（注意ECVF_RenderThreadSafe的限制） |
| 适用场景 | 大多数CVar | 性能敏感的热路径变量 |

⚠️ **FAutoConsoleVariableRef + ECVF_RenderThreadSafe的限制**：`ECVF_RenderThreadSafe` 的源码注释明确说明，引用变量使用该标志时默认按 RenderThread 访问假设处理。也就是说，`FAutoConsoleVariableRef` 的外部变量不是“任意线程安全变量”；如果需要 GameThread、RenderThread、任务线程都可能读取，应优先使用 `TAutoConsoleVariable<T>` 的 `GetValueOnGameThread()` / `GetValueOnRenderThread()` / `GetValueOnAnyThread()`，或自行设计同步边界。

## 七、线程安全：ECVF_RenderThreadSafe与Shadow Copy

### 7.1 问题：GameThread写、RenderThread读

```
GameThread：                    RenderThread：
    CVar.Set(42)                    int v = CVar.GetValue()  
    // 修改值                        // 读取值
    
如果没有同步，RenderThread可能读到：
├─ 旧值（写入未完成）
├─ 部分写入值（64位值的高32位和低32位不一致）
└─ 结果：渲染异常、闪烁，严重时可能触发稳定性问题
```

### 7.2 Shadow Copy方案

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `class TConsoleVariableData`

TConsoleVariableData<T>维护两份值：

```cpp
template <class T>
class TConsoleVariableData
{
    // ShadowedValue[0] = GameThread的值
    // ShadowedValue[1] = RenderThread的值
    T ShadowedValue[2];                           // [0]=GameThread语义，[1]=RenderThread语义

    T GetValueOnGameThread() const
    {
        return ShadowedValue[0];                  // 直接读[0]
    }

    T GetValueOnRenderThread() const
    {
        return ShadowedValue[1];                  // 直接读[1]
    }

    T GetValueOnAnyThread(bool bForceGameThread = false) const
    {
        return ShadowedValue[GetShadowIndex(bForceGameThread)];  // 根据当前线程选择
    }

    static uint32 GetShadowIndex(bool bForceGameThread = false)
    {
        if (bForceGameThread) return 0;
        return IsInParallelGameThread() || IsInGameThread() ? 0 : 1;
    }
};
```

### 7.3 同步流程

```
GameThread：Set("r.xxx", 42)
    │
    ├── ShadowedValue[0] = 42               ← 立即更新GameThread值
    │
    └── if (Flags & ECVF_RenderThreadSafe)
            │
            ├── IConsoleThreadPropagation::OnCVarChange(ShadowedValue[1], 42)
            │       └── 入队RenderThread命令
            │
            └── 在下一次RenderThread Tick时：
                    ShadowedValue[1] = 42    ← 延迟更新RenderThread值

RenderThread：GetValueOnRenderThread()
    └── return ShadowedValue[1]              ← 读取的是同步后的值

特性：
├─ GameThread写入立即可见（对GameThread自身）
├─ RenderThread在下一次Tick后可见（1帧延迟）
├─ 读取路径不走锁、原子操作或虚函数分发
└─ 性能开销：直接数组访问，量级很低
```

### 7.4 ECVF标志速查

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `enum EConsoleVariableFlags`

| 标志 | 值 | 含义 | 典型场景 |
|------|-----|------|---------|
| `ECVF_Default` | 0x0 | 默认 | GameThread-only变量 |
| `ECVF_Cheat` | 0x1 | Shipping构建中隐藏 | 调试专用命令 |
| `ECVF_ReadOnly` | 0x4 | 控制台不可修改 | 引擎内部变量 |
| `ECVF_RenderThreadSafe` | 0x20 | Shadow Copy同步到RenderThread | RenderThread读取的渲染CVar应使用 |
| `ECVF_Scalability` | 0x40 | 可被画质等级控制 | Shadow Quality等 |
| `ECVF_ScalabilityGroup` | 0x80 | 控制其他Scalability CVar | sg.ShadowQuality |
| `ECVF_GeneralShaderChange` | 0x200 | 修改后重新编译所有Shader | 全局Shader开关 |
| `ECVF_MobileShaderChange` | 0x400 | 修改后重编译移动端Shader | 移动端专用开关 |
| `ECVF_DesktopShaderChange` | 0x800 | 修改后重编译桌面端Shader | 桌面端专用开关 |

**SetBy优先级**（低→高）：

```
ECVF_SetByConstructor    (0x00) ← 代码中的默认值
ECVF_SetByScalability    (0x02) ← Scalability.ini
ECVF_SetByGameSetting    (0x03) ← 游戏设置
ECVF_SetByProjectSetting (0x04) ← 项目设置
ECVF_SetByDeviceProfile  (0x07) ← 设备Profile
ECVF_SetByConsoleVariablesIni (0x0A) ← ConsoleVariables.ini
ECVF_SetByCommandline    (0x0D) ← 命令行参数
ECVF_SetByCode           (0x0E) ← 代码Set()
ECVF_SetByConsole        (0x0F) ← 控制台输入（最高优先级）

规则：高优先级可以覆盖低优先级，反之不行
```

## 一、RegisterConsoleCommand()：命令注册

### 1.1 函数概览

**来源**：`Engine/Source/Runtime/Core/Private/HAL/ConsoleManager.cpp`
**函数**：`FConsoleManager::RegisterConsoleCommand()`，搜索 `FConsoleManager::RegisterConsoleCommand`
**职责**：将命令包装为IConsoleCommand对象，插入全局TMap

### 1.2 核心流程

```cpp
// 以FConsoleCommandWithArgsDelegate为例（搜索 FConsoleCommandWithArgsDelegate）
IConsoleCommand* FConsoleManager::RegisterConsoleCommand(
    const TCHAR* Name,
    const TCHAR* Help,
    const FConsoleCommandWithArgsDelegate& Command,
    uint32 Flags)
{
    // 【步骤1】创建命令对象（包装Delegate）
    // FConsoleCommandWithArgs持有用户的Lambda/函数指针
    FConsoleCommandWithArgs* NewCommand = new FConsoleCommandWithArgs(Command, Help, (EConsoleVariableFlags)Flags);

    // 【步骤2】添加到全局TMap
    return AddConsoleObject(Name, NewCommand)->AsCommand();
}
```

### 1.3 AddConsoleObject实现

**来源**：`Engine/Source/Runtime/Core/Private/HAL/ConsoleManager.cpp`，搜索 `FConsoleManager::AddConsoleObject`

```cpp
IConsoleObject* FConsoleManager::AddConsoleObject(const TCHAR* Name, IConsoleObject* Obj)
{
    check(Name);
    check(Obj);

    // 【步骤1】加锁（支持多线程注册）
    UE::TScopeLock ScopeLock(ConsoleObjectsSynchronizationObject);

    // 【步骤2】检查是否已存在同名对象
    IConsoleObject* ExistingObj = ConsoleObjects.FindRef(Name);

    if (ExistingObj)
    {
        if (ExistingObj->TestFlags(ECVF_Unregistered))
        {
            // 之前反注册过，复用（DLL卸载/重加载场景）
            // ... 复用逻辑 ...
        }
        else if (ExistingObj->TestFlags(ECVF_CreatedFromIni))
        {
            // ini文件预创建的，拷贝值后替换
            // ... 拷贝逻辑 ...
        }
        else
        {
            // 重复注册，警告
            UE_LOG(LogConsoleManager, Warning, TEXT("Console object '%s' already registered"), Name);
        }
    }
    else
    {
        // 【步骤3】新对象，插入TMap
        ConsoleObjects.Add(Name, Obj);   // O(1) 哈希插入
    }

    return Obj;
}
```

### 1.4 关键细节

**线程安全**：AddConsoleObject使用`UE::TScopeLock`保护并发访问。多个模块可以在不同线程的静态初始化阶段同时注册命令。

**DLL热重载**：如果一个命令被反注册（ECVF_Unregistered），下次注册同名命令时会复用已有的IConsoleObject对象。这对DLL热重载（编辑器中重编译Plugin）至关重要。

**ini预创建**：如果ini文件中引用了一个还未注册的CVar，引擎会创建一个ECVF_CreatedFromIni的占位对象。等代码中注册该CVar时，ini的值会被拷贝过来。

### 💡 核心要点

命令注册的关键路径是：`new FConsoleCommandXXX(Delegate)` → `TMap.Add(Name, Ptr)`。构造函数自动完成这一切，用户只需声明一个static变量。

### 2.2 核心流程

```cpp
bool FConsoleManager::ProcessUserConsoleInput(
    const TCHAR* InInput,
    FOutputDevice& Ar,
    UWorld* InWorld)
{
    // 【步骤1】处理内置命令（搜索 dumpcvars / dumpccmds / setcvar / unsetcvar）
    // dumpcvars、dumpccmds、setcvar、unsetcvar 等特殊命令
    if (HandleBuiltInCommands(InInput, Ar))
        return true;

    // 【步骤2】解析第一个Token（命令名/CVar名，搜索 GetTextSection）
    FString Param1 = GetTextSection(InInput);
    // 输入 "r.Shadow.MaxResolution 4096"
    // Param1 = "r.Shadow.MaxResolution"
    // 剩余 = "4096"

    // 【步骤3】处理帮助查询（搜索 bCommandEndedInQuestion）
    // 如果输入以 ? 结尾，显示帮助信息
    if (Param1.EndsWith(TEXT("?")))
    {
        // 显示CVar帮助文本
        return true;
    }

    // 【步骤4】处理 @platform/deviceprofile 语法（搜索 Param1.Contains(TEXT("@"))）
    // 例如：r.Shadow.Quality@iOS = 2

    // 【步骤5】查找控制台对象（搜索 FindConsoleObject(*Param1)）
    IConsoleObject* CObj = FindConsoleObject(*Param1);
    
    if (!CObj)
    {
        // 未找到命令
        return false;
    }

    // 【步骤6】分支：命令 vs 变量
    IConsoleCommand* CCmd = CObj->AsCommand();
    if (CCmd)
    {
        // 是命令 → 解析参数并执行（搜索 CCmd->Execute）
        TArray<FString> Args;
        // 将剩余输入按空格分割为参数数组
        while (/* 还有参数 */)
        {
            Args.Add(NextToken);
        }
        
        // 执行命令
        CCmd->Execute(Args, InWorld, Ar);
        return true;
    }

    IConsoleVariable* CVar = CObj->AsVariable();
    if (CVar)
    {
        // 是变量 → 设置值或显示当前值
        if (/* 有参数 */)
        {
            CVar->Set(*ValueString, ECVF_SetByConsole);  // 最高优先级
        }
        else
        {
            // 无参数 → 显示当前值
            Ar.Logf(TEXT("%s = %s"), *Param1, *CVar->GetString());
        }
        return true;
    }

    return false;
}
```

### 2.3 关键细节

**优先级**：通过控制台输入的Set操作使用`ECVF_SetByConsole`（0x0F），这是最高优先级。可以覆盖所有其他来源（ini、代码、画质等级...）设置的值。

**内置命令**：以下命令不需要注册，直接在ProcessUserConsoleInput中处理：
- `dumpcvars`：导出所有CVar到日志
- `dumpccmds`：导出所有命令到日志
- `setcvar`：设置CVar（支持更复杂的语法）
- `help`：搜索命令/变量

### 💡 核心要点

输入处理的关键流程是：解析Token → FindConsoleObject(O(1)) → 判断类型 → Execute(命令) 或 Set(变量)。整个过程在GameThread执行。

## 三、FindConsoleObject()：TMap查找

### 3.1 函数概览

**来源**：`Engine/Source/Runtime/Core/Private/HAL/ConsoleManager.cpp`
**函数**：`FConsoleManager::FindConsoleObject()`，搜索 `FConsoleManager::FindConsoleObject`
**职责**：在TMap中查找命令/变量对象

### 3.2 核心流程

```cpp
IConsoleObject* FConsoleManager::FindConsoleObject(
    const TCHAR* Name,
    bool bTrackFrequentCalls) const
{
    // 【步骤1】无过滤查找
    IConsoleObject* CVar = FindConsoleObjectUnfiltered(Name);

    // 【步骤2】过滤ini预创建对象
    if (CVar && CVar->TestFlags(ECVF_CreatedFromIni))
    {
        return nullptr;  // ini预创建的占位对象对外不可见
    }

    // 【步骤3】频繁调用追踪（Debug构建）
#if TRACK_CONSOLE_FIND_COUNT
    if (bTrackFrequentCalls && CVar)
    {
        ++CVar->FindCallCount;

        if (CVar->FindCallCount == 500)
        {
            UE_LOG(LogConsoleManager, Warning,
                TEXT("Performance warning: Console object '%s' has %d FindConsoleObject() calls (consider caching)"),
                Name, CVar->FindCallCount);
        }
    }
#endif

    return CVar;
}

// 实际查找逻辑（搜索 FConsoleManager::FindConsoleObjectUnfiltered）
IConsoleObject* FConsoleManager::FindConsoleObjectUnfiltered(const TCHAR* Name) const
{
    UE::TScopeLock ScopeLock(ConsoleObjectsSynchronizationObject);  // 读锁

    return ConsoleObjects.FindRef(Name);  // O(1) TMap查找
}
```

### 3.3 关键细节

**频繁调用警告**：在Debug构建中，如果一个CVar被FindConsoleObject查找超过500次，引擎会输出性能警告。这是提醒你：**缓存查找结果，不要在循环中反复查找**。

```cpp
// ❌ 性能差：每次调用都查找
void OnTick()
{
    IConsoleVariable* CVar = IConsoleManager::Get().FindConsoleVariable(TEXT("r.xxx"));
    int32 Value = CVar->GetInt();  // 每帧重复查找会累积字符串哈希和比较成本
}

// ✅ 性能好：缓存引用
static IConsoleVariable* CachedCVar = IConsoleManager::Get().FindConsoleVariable(TEXT("r.xxx"));
void OnTick()
{
    int32 Value = CachedCVar->GetInt();  // 指针直接访问，避免重复查找
}
```

### 💡 核心要点

查找的核心是`TMap::FindRef(Name)`这类哈希路径。但引擎依然警告频繁调用（500次阈值），因为字符串哈希、锁和比较的常数成本会在热循环里累积。热路径应缓存指针。

## 四、TConsoleVariableData：Shadow Copy实现

### 4.1 函数概览

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `class TConsoleVariableData`
**职责**：存储CVar的双份值（GameThread + RenderThread），让读取端不走锁

### 4.2 核心实现

```cpp
template <class T>
class TConsoleVariableData
{
public:
    TConsoleVariableData(const T DefaultValue)
    {
        for (uint32 i = 0; i < UE_ARRAY_COUNT(ShadowedValue); ++i)
        {
            ShadowedValue[i] = DefaultValue;  // 初始化两份
        }
    }

    T GetValueOnGameThread() const
    {
        cvarCheckCode(ensure(GetShadowIndex() == 0));  // Debug：确认在GameThread
        return ShadowedValue[0];                        // 直接读[0]
    }

    T GetValueOnRenderThread() const
    {
        cvarCheckCode(ensure(IsInParallelRenderingThread()));  // Debug：确认在RenderThread
        return ShadowedValue[1];                               // 直接读[1]
    }

    T GetValueOnAnyThread(bool bForceGameThread = false) const
    {
        return ShadowedValue[GetShadowIndex(bForceGameThread)];  // 根据当前线程选择
    }

private:
    T ShadowedValue[2];  // [0]=GameThread语义，[1]=RenderThread语义

    static uint32 GetShadowIndex(bool bForceGameThread = false)
    {
        if (bForceGameThread) return 0;
        return IsInParallelGameThread() || IsInGameThread() ? 0 : 1;
    }
};
```

### 4.3 IConsoleThreadPropagation

**来源**：`Engine/Source/Runtime/Core/Public/HAL/IConsoleManager.h`，搜索 `struct IConsoleThreadPropagation`

当GameThread修改带ECVF_RenderThreadSafe标记的CVar时，引擎通过IConsoleThreadPropagation接口将变化传播到RenderThread：

```cpp
struct IConsoleThreadPropagation
{
    virtual void OnCVarChange(int32& Dest, int32 NewValue) = 0;
    virtual void OnCVarChange(float& Dest, float NewValue) = 0;
    virtual void OnCVarChange(bool& Dest, bool NewValue) = 0;
    virtual void OnCVarChange(FString& Dest, const FString& NewValue) = 0;
};
```

RenderThread注册一个IConsoleThreadPropagation实现。GameThread每次Set()时检查ECVF_RenderThreadSafe标志，如果存在则调用OnCVarChange将新值入队到RenderThread。

### 4.4 性能分析

```
读取性能（量级示意，读取路径不走锁）：
├─ GetValueOnGameThread()     → 数组访问 ShadowedValue[0]
├─ GetValueOnRenderThread()   → 数组访问 ShadowedValue[1]
└─ GetValueOnAnyThread()      → 分支判断 + 数组访问

写入性能（较慢但不在热路径）：
├─ GameThread Set()            → 直接赋值
├─ RenderThread同步            → 入队命令 + 延迟赋值
└─ 仅在用户输入时触发，不影响帧率

对比方案：
├─ 互斥锁（FRWLock）  → 每次读取需要同步原语
├─ 原子操作（TAtomic） → 支持类型受限，且仍有原子读写成本
└─ Shadow Copy          → 适合渲染CVar这种“少写多读”的路径
```

### 💡 核心要点

Shadow Copy方案的精髓是**用空间换时间**：多存一份数据，换取读取端不走锁的稳定路径。对于每帧可能反复读取的渲染CVar来说，这是 UE 在 Core 层提供的常用解法。

## 一、与RenderThread的关系

### 1.1 跨线程执行

控制台命令在GameThread执行，但渲染数据只能在RenderThread访问。两种跨线程模式：

**模式1：ENQUEUE_RENDER_COMMAND（命令模式）**

```cpp
static FAutoConsoleCommand CCmdRenderOp(
    TEXT("MyDebug.RenderOp"), TEXT("help"),
    FConsoleCommandDelegate::CreateStatic([]()
    {
        // GameThread → 入队 → RenderThread执行
        ENQUEUE_RENDER_COMMAND(MyCommand)(
            [](FRHICommandListImmediate& RHICmdList)
            {
                // 在RenderThread中访问渲染数据
            });
    })
);
```

**模式2：ECVF_RenderThreadSafe（变量模式）**

```cpp
static TAutoConsoleVariable<int32> CVar(
    TEXT("r.MyVar"), 0, TEXT("help"),
    ECVF_RenderThreadSafe  // Shadow Copy自动同步
);

// RenderThread中直接读取
void RenderFunction()
{
    int32 Value = CVar.GetValueOnRenderThread();  // 读取RenderThread副本
}
```

### 1.2 选择建议

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 持久参数（阴影分辨率、质量等级...） | TAutoConsoleVariable + ECVF_RenderThreadSafe | 帧间持久，读取RenderThread副本 |
| 一次性操作（DumpGBuffer、FreezeRendering...） | FAutoConsoleCommand + ENQUEUE | 执行一次即结束 |
| 高频读取的开关 | FAutoConsoleVariableRef + 全局变量 | 调用开销最低，但需要自行守住线程假设 |

---

## 二、与RenderGraph的关系

### 2.1 CVar控制RDG Pass开关

渲染管线中大量使用CVar控制Pass的开启/关闭：

```cpp
// 典型模式：CVar作为RDG Pass的条件
void FDeferredShadingSceneRenderer::Render(FRDGBuilder& GraphBuilder)
{
    // VSM开关
    if (CVarEnableVirtualShadowMaps.GetValueOnRenderThread())
    {
        RenderVirtualShadowMaps(GraphBuilder, ...);
    }
    else
    {
        RenderShadowDepthMaps(GraphBuilder, ...);
    }

    // Contact Shadow开关
    if (CVarContactShadows.GetValueOnRenderThread())
    {
        RenderScreenSpaceShadows(GraphBuilder, ...);
    }
}
```

### 2.2 CVar作为Shader参数

```cpp
// CVar值传递到Shader Uniform
PassParameters->ScreenPercentage = CVarScreenPercentage.GetValueOnRenderThread() / 100.0f;
PassParameters->ShadowBias = CVarShadowBias.GetValueOnRenderThread();
```

---

## 三、与Scalability系统的关系

### 3.1 画质等级控制CVar

标记`ECVF_Scalability`的CVar可以被画质等级系统（sg.ShadowQuality等）批量控制：

```
sg.ShadowQuality 0 (Low):
    r.Shadow.MaxResolution = 512
    r.Shadow.CSM.MaxCascades = 1
    r.Shadow.DistanceScale = 0.5

sg.ShadowQuality 3 (Epic):
    r.Shadow.MaxResolution = 4096
    r.Shadow.CSM.MaxCascades = 4
    r.Shadow.DistanceScale = 1.0
```

**优先级**：`sg.*`的SetBy优先级低于Console输入。在控制台手动设置的值不会被画质等级覆盖。
