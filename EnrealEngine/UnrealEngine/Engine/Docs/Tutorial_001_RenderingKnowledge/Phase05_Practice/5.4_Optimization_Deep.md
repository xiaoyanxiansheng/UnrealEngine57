# 第一部分：为什么性能优化总是低效

> **核心目标**：先把"优化为什么经常做错"说清楚，再建立正确的分析顺序。性能优化低效不是因为技巧不够，而是因为大多数人从来没有先测量。

---

## 一、三类最常见的优化失败

### 1.1 失败模式A：凭感觉降画质

最典型的错误流程：

```
老板说"帧率太低"
  ↓
把所有画质设置调到 Medium
  ↓
帧率确实上去了一些
  ↓
但画面变丑了，而且帧率仍然不够
  ↓
继续降画质 → 画面更丑 → 帧率仍有波动
```

这种做法的根本问题是：**你不知道瓶颈在哪**。可能 GPU 只占 40%，真正卡的是 RenderThread 的 DrawCall 过多——降画质改变的是 GPU 负载，对 CPU 瓶颈毫无帮助。

### 1.2 失败模式B：只看 FPS 数字

```
stat fps 显示 45FPS
  ↓
开始优化
  ↓
做了很多改动
  ↓
stat fps 显示 48FPS
  ↓
"提升了3帧"
  ↓
但不知道哪个改动真正有效，哪些是无用功
```

FPS 是结果，不是原因。只看 FPS 就像只看体温计不做检查——你知道发烧了，但不知道为什么发烧。

### 1.3 失败模式C：优化错误的层级

```
GPU 占 15ms（80%负载）
  ↓
但真正慢的是某个 Post Process Pass（8ms）
  ↓
你花 2 天优化了 BasePass 的 Shader 复杂度
  ↓
BasePass 从 3ms → 2.5ms（节省 0.5ms）
  ↓
总帧率几乎没变
  ↓
因为你优化的不是瓶颈
```

### 💡 核心要点

性能优化只有一个铁律：**先测量，再定位，最后才动手**。跳过前两步直接动手，成功率极低。

---

## 二、为什么"凭经验降画质"是最差策略

### 2.1 渲染帧时间不是单一数字

一帧渲染涉及多个线程和多个 Pass，它们之间的关系是：

```
帧时间 = max(GameThread, RenderThread, GPU)

其中：
GameThread  = Tick + Physics + Animation + Blueprint + ...
RenderThread = InitViews + Culling + DrawCall Setup + Command录制 + ...
GPU         = BasePass + Shadow + Lumen + Reflection + PostProcess + ...
```

降画质只改变了 GPU 部分的某些 Pass。如果瓶颈在 RenderThread（CPU侧），降画质不会有任何效果。

### 2.2 Amdahl定律的残酷现实

如果一帧 16ms 里：

- RenderThread 占 12ms（瓶颈）
- GPU 占 8ms

你把 GPU 从 8ms 优化到 4ms（提升 50%！），帧时间从 16ms 变成 ... 还是 12ms。因为帧时间取 max，瓶颈线程没变。

反过来，如果你把 RenderThread 从 12ms 优化到 8ms，帧时间从 16ms 变成 8ms，FPS 从 62 跳到 125。

**同样的精力，方向错了效果为零。**

### 2.3 正确的分析顺序

```
第1步：判断瓶颈线程
  stat unit → 看 Game / Draw / GPU 哪个最长

第2步：在瓶颈线程内定位热点
  CPU → Unreal Insights Timing View → 找最耗时函数
  GPU → stat gpu → 找最耗时 Pass

第3步：针对热点选择优化方法
  DrawCall过多 → 合并 / Instancing / GPU-Driven
  某个Pass慢 → 降低该Pass质量或优化输入

第4步：验证效果
  重新测量，确认瓶颈转移或消失
```

### 💡 核心要点

性能优化不是"让所有东西都变快"，而是**找到当前的瓶颈，只优化瓶颈**。瓶颈优化掉之后，下一个最慢的会成为新瓶颈，然后继续。

## 三、本章给出的四层分析链路

```
第1层：帧级别
  stat unit / stat fps
  回答：瓶颈是 CPU 还是 GPU？

第2层：Pass / 函数级别
  stat gpu / profilegpu / Unreal Insights
  回答：哪个 Pass 或哪个函数是热点？

第3层：DrawCall / Shader级别
```

### 💡 核心要点

四层链路的价值不在"全面"，而在**强制你每次都从最高层开始**。直接跳到第3层抓帧，效率通常不如先花 30 秒看 `stat unit`。

> **核心目标**：理解 `stat gpu`、`profilegpu`、Unreal Insights 这些工具背后在做什么。只有知道数据从哪来，你才能判断"这个数字可不可信、为什么和预期不符、什么时候需要换工具"。

## 一、stat 命令的数据流

### 1.1 stat 统计项的注册

UE 的 stat 系统基于 `DECLARE_CYCLE_STAT` 宏，在编译期注册统计项。

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderCore.h`，搜索 `DECLARE_CYCLE_STAT_EXTERN`、`STAT_InitViewsTime`、`STAT_TotalGPUFrameTime`、`STAT_TotalSceneRenderingTime`、`STATGROUP_SceneRendering`

```cpp
DECLARE_CYCLE_STAT_EXTERN(TEXT("BeginOcclusionTests"),STAT_BeginOcclusionTestsTime,STATGROUP_SceneRendering, RENDERCORE_API);
DECLARE_CYCLE_STAT_EXTERN(TEXT("InitViews"),STAT_InitViewsTime,STATGROUP_SceneRendering, RENDERCORE_API);
DECLARE_CYCLE_STAT_EXTERN(TEXT("Dynamic shadow setup"), STAT_DynamicShadowSetupTime, STATGROUP_SceneRendering, RENDERCORE_API);
DECLARE_CYCLE_STAT_EXTERN(TEXT("Total GPU rendering"),STAT_TotalGPUFrameTime,STATGROUP_SceneRendering, RENDERCORE_API);
DECLARE_CYCLE_STAT_EXTERN(TEXT("RenderViewFamily"),STAT_TotalSceneRenderingTime,STATGROUP_SceneRendering, RENDERCORE_API);
```

每个 `DECLARE_CYCLE_STAT` 做了两件事：

1. 定义一个 `TStatId`，用于运行时快速查找
2. 把该统计项注册到某个 `STATGROUP`（如 `STATGROUP_SceneRendering`）

当你在控制台输入 `stat scenerendering` 时，引擎找到 `STATGROUP_SceneRendering` 下的所有统计项，把它们的值渲染到 HUD 上。

### 1.2 stat 数据的采集

运行时通过 `SCOPE_CYCLE_COUNTER` 宏采集：

```cpp
{
    SCOPE_CYCLE_COUNTER(STAT_InitViewsTime);
    // ... InitViews 的实际代码 ...
}
```

这个宏在进入作用域时记录 `FPlatformTime::Cycles64()`，在离开时再记录一次，差值就是该统计项的耗时。

### 1.3 stat unit 的特殊性

`stat unit` 不是普通的 `STATGROUP`，而是引擎 HUD 直接计算的帧级统计。

**来源**：`Engine/Source/Runtime/Engine/Private/UnrealEngine.cpp`，搜索 `STAT_Unit`、`UEngine::RenderStatUnit`、`UEngine::ToggleStatUnit`、`STAT_UnitTime`

它显示的 `Game`、`Draw`、`GPU`、`RHIT` 分别对应：

| 显示名 | 含义 | 数据来源 |
|--------|------|----------|
| `Frame` | 帧时间 | 两帧之间的墙钟时间 |
| `Game` | GameThread 耗时 | GT Tick 开始到结束 |
| `Draw` | RenderThread 耗时 | RT 从接到命令到录制完成 |
| `GPU` | GPU 耗时 | GPU Timestamp Query 差值 |
| `RHIT` | RHIThread 耗时 | RHI 命令提交耗时 |

### 1.4 stat gpu 的数据来源

**来源**：`Engine/Source/Runtime/RenderCore/Public/ProfilingDebugging/RealtimeGPUProfiler.h`，搜索 `STATGROUP_GPU`、`DECLARE_GPU_STAT`、`SCOPED_GPU_STAT`。

```cpp
DECLARE_STATS_GROUP(TEXT("GPU"), STATGROUP_GPU, STATCAT_Advanced);
```

`stat gpu` 显示的每一行（如 `BasePass: 3.2ms`）来自渲染代码中的 `SCOPED_GPU_STAT` 或 `RDG_GPU_STAT_SCOPE`。这些宏在 GPU 时间线上插入 Timestamp Query，事件完成后由 `FRealtimeGPUProfiler` 收集并汇报给 stat 系统。

### 💡 核心要点

`stat unit` 告诉你**哪个线程是瓶颈**，`stat gpu` 告诉你**GPU 内哪个 Pass 最慢**。前者判断方向，后者锁定目标。两者的数据来源完全不同：前者是 CPU 墙钟 + GPU Query，后者纯粹是 GPU Timestamp。

## 二、stat gpu 与 SCOPED_GPU_STAT：GPU Pass 计时如何工作

### 2.1 GPU Timestamp Query 的原理

GPU 计时不像 CPU 那样简单地取时间戳。由于 GPU 指令是异步提交的，必须使用专门的 Timestamp Query：

```
CPU 侧：
  插入 BeginQuery（GPU 时间线标记点 A）
  → 提交 DrawCall / Dispatch
  插入 EndQuery（GPU 时间线标记点 B）

GPU 侧（滞后 1-2 帧）：
  执行到标记点 A → 写入时间戳 T1
  执行 DrawCall
  执行到标记点 B → 写入时间戳 T2

CPU 侧（下一帧回读）：
  T2 - T1 = 该 Pass 的 GPU 耗时
```

### 2.2 SCOPED_GPU_STAT 和 RDG_GPU_STAT_SCOPE

这两个宏是在渲染代码中插入 GPU 计时的标准方式。

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderGraphEvent.h`，搜索 `RDG_GPU_STAT_SCOPE`、`FRDGScope_GPU`、`ERDGScopeFlags::AlwaysEnable`

```cpp
#define RDG_GPU_STAT_SCOPE(GraphBuilder, StatName) \
    TRDGEventScopeGuard<FRDGScope_GPU> ... \
    ((GraphBuilder), ERDGScopeFlags::AlwaysEnable, \
     (GraphBuilder).RHICmdList.GetGPUMask(), \
     CSV_STAT_FNAME(StatName), GET_STATID(Stat_GPU_##StatName), ...);
```

```cpp
#define RDG_RHI_GPU_STAT_SCOPE(GraphBuilder, StatName) \
    RDG_GPU_STAT_SCOPE(GraphBuilder, StatName); \
    SCOPED_GPU_STAT(GraphBuilder.RHICmdList, StatName);
```

`RDG_GPU_STAT_SCOPE` 在 RenderGraph 构建阶段注册计时区间，`SCOPED_GPU_STAT` 直接在 RHI CommandList 上插入 Query。两者配合确保无论是 RDG Pass 还是非 RDG 代码路径，GPU 计时都能被采集。

### 2.3 为什么 stat gpu 有时候数据不稳定

两个常见原因：

1. **GPU Timestamp 有 1-2 帧延迟**：你看到的数据是 1-2 帧前的结果
2. **GPU 频率波动**：boost/throttle 导致同一操作的 GPU 时间不同

所以看 `stat gpu` 时，不要盯着单帧数字，要**看趋势**。

### 💡 核心要点

`stat gpu` 的数据来自 GPU Timestamp Query，由渲染代码中的 `SCOPED_GPU_STAT` / `RDG_GPU_STAT_SCOPE` 注入。这些计时是真实的 GPU 硬件时间，但有 1-2 帧延迟。

## 三、profilegpu：单帧 GPU 分析的完整链路

### 3.1 命令注册与触发

当你输入 `profilegpu` 时，引擎会：

```
│  └─ ...

└─ ...
```

### 3.3 profilegpu 的配置选项

| CVar | 作用 | 推荐值 |
|------|------|--------|
| `r.ProfileGPU.Sort` | 排序方式（0=时序, 1=inclusive, 2=exclusive） | `1`（按耗时排） |
| `r.ProfileGPU.ThresholdPercent` | 过滤低于该百分比的事件 | `1.0`（忽略<1%的事件） |
| `r.ProfileGPU.ShowStats` | 显示 DrawCall/顶点数等额外列 | `true` |
| `r.ProfileGPU.ShowExclusive` | 显示 exclusive 时间 | `true` |
| `r.ProfileGPU.Root` | 过滤显示某个子树 | `"BasePass"`（只看BasePass） |

### 3.4 profilegpu vs stat gpu

| 特性 | `stat gpu` | `profilegpu` |
|------|-----------|--------------|
| **刷新频率** | 每帧实时 | 单帧快照 |
| **粒度** | Pass 级 | Pass + DrawCall 级 |
| **开销** | 很低（始终开启） | 较高（仅采集帧） |
| **适用场景** | 快速看趋势 | 深入分析单帧 |

### 💡 核心要点

`stat gpu` 适合**持续监控**，`profilegpu` 适合**深入分析**。日常开发用前者，排查具体瓶颈用后者。

## 五、工具选择决策

```
性能问题出现
  ↓
问自己：知道瓶颈是 CPU 还是 GPU 吗？
  ↓
不知道 → stat unit（30秒就能判断）
  ↓
┌───────────────────┬────────────────────┐
│ Draw > GPU        │ GPU > Draw         │
│ (CPU瓶颈)         │ (GPU瓶颈)          │
└───────────────────┴────────────────────┘
  ↓                        ↓
Unreal Insights          stat gpu（先快速看）
  Timing View                ↓
  定位到具体函数           哪个 Pass 慢？
  ↓                        ↓
优化：                   ┌────────┴────────┐
- 减少 DrawCall          ▼                 ▼
- GPU-Driven           profilegpu        RenderDoc
- 合并 Mesh            （树状报告）      （DrawCall级）
```

| 问题类型 | 首选工具 | 第二工具 | 不要先做什么 |
|---------|---------|---------|-------------|
| **不知道瓶颈在哪** | `stat unit` | — | 不要先降画质 |
| **CPU 明确慢** | Unreal Insights | `stat scenerendering` | 不要先看 GPU |
| **GPU 明确慢** | `stat gpu` | `profilegpu` | 不要先改材质 |
| **某个 Pass 慢** | `profilegpu` | RenderDoc | 不要先猜原因 |
| **帧率不稳定** | Unreal Insights | `stat unitgraph` | 不要看平均值 |
| **内存持续增长** | Insights Memory | `stat memory` | 不要先重启 |

### 💡 核心要点

工具选择的核心不是"哪个更高级"，而是**当前问题在哪一层**。层级对了，最简单的工具就能给出答案。

## 一、`RenderViewFamily_RenderThread`：整帧渲染计时

### 1.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp`  
**源码锚点**：搜索 `RenderViewFamily_RenderThread`、`STAT_TotalSceneRenderingTime`、`Renderer->Render`。  
**职责**：整帧渲染的入口——从这里开始，一帧内所有的渲染工作（InitViews、BasePass、Lumen、PostProcess等）都在其下执行。

### 1.2 核心流程

```cpp
// 概念流程（基于 SceneRendering.cpp 中 RenderViewFamily_RenderThread / STAT_TotalSceneRenderingTime / Renderer->Render 的实际逻辑简化）
static void RenderViewFamily_RenderThread(
    FRDGBuilder& GraphBuilder,
    FSceneRenderer* Renderer,
    const FSceneRenderUpdateInputs* SceneUpdateInputs)
{
    FSceneViewFamily& ViewFamily = Renderer->ViewFamily;

    LLM_SCOPE(ELLMTag::SceneRender);

    // 【步骤1】开始整帧计时
    SCOPE_CYCLE_COUNTER(STAT_TotalSceneRenderingTime);

    // 【步骤2】进入具体渲染路径
    if (ViewFamily.EngineShowFlags.HitProxies)
    {
        Renderer->RenderHitProxies(GraphBuilder);
    }
    else
    {
        Renderer->Render(GraphBuilder);  // ← 延迟着色主路径
    }
}
```

### 1.3 关键细节

#### 细节A：`STAT_TotalSceneRenderingTime` 是你在 `stat scenerendering` 里看到的 "RenderViewFamily"

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderCore.h`，搜索 `STAT_TotalSceneRenderingTime`、`DECLARE_CYCLE_STAT_EXTERN(TEXT("RenderViewFamily")`

```cpp
DECLARE_CYCLE_STAT_EXTERN(TEXT("RenderViewFamily"),STAT_TotalSceneRenderingTime,STATGROUP_SceneRendering, RENDERCORE_API);
```

这意味着 `stat scenerendering` 中的 "RenderViewFamily" 数值 = 从 `RenderViewFamily_RenderThread` 进入到退出的 CPU 时间。

#### 细节B：它测量的是 RenderThread CPU 时间，不是 GPU 时间

这个统计项只计算 RenderThread 上的 CPU 工作时间（命令录制、状态设置等），不包含 GPU 实际执行时间。GPU 时间由 `STAT_TotalGPUFrameTime` 单独统计。

## 三、`RDG_GPU_STAT_SCOPE`：RenderGraph 中的 GPU 计时

### 3.1 宏概览

**来源**：`Engine/Source/Runtime/RenderCore/Public/RenderGraphEvent.h`，搜索 `RDG_GPU_STAT_SCOPE`、`RDG_RHI_GPU_STAT_SCOPE`、`TRDGEventScopeGuard<FRDGScope_GPU>`、`SCOPED_GPU_STAT`

```cpp
#define RDG_GPU_STAT_SCOPE(GraphBuilder, StatName) \
    TRDGEventScopeGuard<FRDGScope_GPU> ... \
    ((GraphBuilder), ERDGScopeFlags::AlwaysEnable, \
     (GraphBuilder).RHICmdList.GetGPUMask(), \
     CSV_STAT_FNAME(StatName), GET_STATID(Stat_GPU_##StatName), ...);

#define RDG_RHI_GPU_STAT_SCOPE(GraphBuilder, StatName) \
    RDG_GPU_STAT_SCOPE(GraphBuilder, StatName); \
    SCOPED_GPU_STAT(GraphBuilder.RHICmdList, StatName);
```

### 3.2 它在做什么

当渲染代码里写：

```cpp
RDG_GPU_STAT_SCOPE(GraphBuilder, BasePass);
// ... BasePass 的 RDG Pass 添加 ...
```

实际发生的是：

1. 在 RDG 构建阶段注册一个 GPU 计时区间
2. RDG 执行阶段，在该区间前后插入 GPU Timestamp Query
3. `FRealtimeGPUProfiler` 收集 Query 结果
4. 结果汇报给 `STATGROUP_GPU`，显示在 `stat gpu` 中

### 3.3 为什么这对理解 stat gpu 很重要

`stat gpu` 中每一行的名字，直接对应源码中的 `RDG_GPU_STAT_SCOPE(GraphBuilder, XXX)` 调用。也就是说：

- `stat gpu` 中的 `BasePass` 行 → 源码中 `RDG_GPU_STAT_SCOPE(GraphBuilder, BasePass)`
- `stat gpu` 中的 `Lumen` 行 → 源码中 `RDG_GPU_STAT_SCOPE(GraphBuilder, Lumen)`
- `stat gpu` 中的 `ShadowDepths` 行 → 源码中 `RDG_GPU_STAT_SCOPE(GraphBuilder, ShadowDepths)`

知道这个映射关系后，你就可以**直接在源码中搜索 stat 名称**来定位具体的渲染代码。

### 💡 核心要点

`stat gpu` 不是魔法——每一行数据都来自渲染代码中明确的 `RDG_GPU_STAT_SCOPE` 调用。知道这个映射，你就能从 stat 数字直接跳到源码。

## 五、`FHZBOcclusionTester`：HZB 遮挡剔除

### 5.1 概览

**来源**：
- `Engine/Source/Runtime/Renderer/Private/SceneRendering.h`，搜索 `class FHZBOcclusionTester`
- `Engine/Source/Runtime/Renderer/Private/SceneOcclusion.cpp`，搜索 `FHZBOcclusionTester::Submit`

`FHZBOcclusionTester` 是 UE5.7 中基于 Hierarchical Z-Buffer 的遮挡剔除实现。它利用上一帧的深度 Mip 链来判断物体是否被遮挡。

### 5.2 HZB 剔除的工作流程

```
上一帧深度缓冲
  ↓
生成 HZB Mip 链（逐级取 max depth）
  ↓
当前帧对每个 Primitive 的包围盒：
  → 查询对应 Mip 级别的深度值
  → 如果包围盒最近深度 > HZB 深度 → 物体被完全遮挡 → 剔除
  ↓
只有通过 HZB 测试的 Primitive 进入后续渲染
```

### 5.3 HZB 相关的控制变量

```bash
r.HZBOcclusion          # 1=开启 HZB 遮挡剔除
```

### 5.4 FOcclusionQueryHelpers

**来源**：`Engine/Source/Runtime/Renderer/Private/ScenePrivateBase.h`，搜索 `class FOcclusionQueryHelpers`、`GetNumBufferedFrames`、`GetQueryLookupIndex`、`GetQueryIssueIndex`

遮挡查询有帧延迟（通常 2 帧），`FOcclusionQueryHelpers` 管理查询的发起和回读帧索引，确保不会读到过期数据。

### 💡 核心要点

HZB 剔除是**CPU 侧可见性剔除最重要的优化之一**。它能在不增加 GPU 负载的前提下大幅减少 DrawCall 数量。对于有大量互相遮挡物体的场景（室内、城市），效果尤为明显。

---

# 第四部分：实践指南

> **核心目标**：把前面的工具和源码入口变成能直接落地的排查与优化流程。

## 一、5分钟瓶颈定位标准流程

### 1.1 总流程

```
第0分钟：stat unit
  看 Game / Draw / GPU / RHIT
  判断瓶颈线程

第1-2分钟：
  CPU瓶颈 → stat scenerendering → 看 InitViews / DrawCall数
  GPU瓶颈 → stat gpu → 找最慢的 Pass

第3-4分钟：
  CPU → 是 InitViews 慢？还是其他函数？
  GPU → 该 Pass 是因为分辨率？DrawCall？还是 Shader？

第5分钟：
  确定优化方向，开始实施
```

### 1.2 判断瓶颈线程的规则

```bash
stat unit

# 结果示例：
Frame: 22.2ms (45 FPS)
Game:  5.5ms
Draw:  18.3ms   ← 最长 = RenderThread CPU瓶颈
GPU:   12.7ms
RHIT:  3.1ms
```

| 情况 | 瓶颈 | 首选优化方向 |
|------|------|-------------|
| `Draw` 最长 | RenderThread（CPU） | 减少 DrawCall、GPU-Driven |
| `GPU` 最长 | GPU | 降低 Pass 开销、降画质 |
| `Game` 最长 | GameThread（CPU） | 优化 Tick、Blueprint、Physics |
| `Draw` ≈ `GPU` | 双瓶颈 | 两边都要优化 |

### 1.3 为什么第一个动作永远是 stat unit

因为它只需要 0.5 秒，就能告诉你：

- 不要浪费时间在非瓶颈线程上
- 不要用 GPU 工具去查 CPU 问题
- 不要用 CPU Profiler 去查 GPU 问题

### 💡 核心要点

标准流程的意义在于把"凭经验乱查"变成"固定动作序列"。只要流程固定，排查速度就会稳定。

## 二、CPU 瓶颈排查：RenderThread 过载

### 2.1 确认 CPU 瓶颈后的下一步

```bash
stat scenerendering

# 关键指标：
RenderViewFamily:       18.3ms    # 整帧 RT 时间
InitViews:              8.5ms     # 可见性 + DrawCall构建 ← 重点
Dynamic shadow setup:   3.2ms     # 阴影准备
BasePass:               2.1ms     # BasePass CPU侧录制
```

### 2.2 InitViews 慢（最常见的 CPU 瓶颈）

如果 `InitViews` 占 RT 时间的 40% 以上，原因通常是：

1. **Primitive 太多**：每个 Primitive 都要视锥测试 + MDC 构建
2. **DrawCall 太多**：每个 DrawCall 都要 CPU 设置状态
3. **遮挡查询回读阻塞**：GPU 还没完成 Query，CPU 等待

**优化策略**：

| 原因 | 判断方法 | 优化手段 |
|------|---------|---------|
| Primitive 太多 | `stat scenerendering` Visible Primitives > 5000 | Merge Actors / HISM / Nanite |
| DrawCall 太多 | `stat rhi` Draw Calls > 2000 | Instancing / Material 合并 / Nanite |
| MDC 构建慢 | Insights 中 `AddMeshBatch` 耗时高 | 开启 MDC 缓存（默认开） / 减少动态物体 |

### 2.3 stat rhi 关键指标

```bash
stat rhi

# 关键指标：
Draw Calls:            2345    # ← >2000 需要优化
Triangles:             567890
DrawPrimitive Calls:   345
SetShaderParameter:    5234    # ← >5000 可能拖慢 RT
```

### 2.4 当 InitViews 不是最慢时

如果 `InitViews` 正常，但 RT 仍然慢，用 Unreal Insights 钻取：

```
Timing View → RenderThread → 展开
  → 找到占比最高的函数
  → 常见热点：
    - FMeshPassProcessor::AddMeshBatch（DrawCall 构建）
    - FSceneRenderer::RenderShadowDepthMaps（阴影准备）
    - SetShaderParameters_RenderThread（参数设置）
```

### 💡 核心要点

CPU 瓶颈的本质几乎总是**"CPU 要为 GPU 准备的工作太多了"**。减少这个工作量的最有效方式是：减少 Primitive 数量、减少 DrawCall 数量、让 GPU 接管更多工作（GPU-Driven）。

## 三、GPU 瓶颈排查：从 stat gpu 到 Pass 级分析

### 3.1 stat gpu 快速定位

```bash

[GPU] Total: 16.5ms
  BasePass:              4.2ms     # GBuffer 填充
  ShadowDepths:          3.1ms     # 阴影深度
  Lumen:                 5.8ms     # ← 最慢
    ScreenProbeGather:   3.2ms
    Reflections:         1.8ms
  PostProcessing:        1.5ms
  Nanite:                0.8ms
  HZB:                   0.3ms
```

### 3.2 常见 GPU 热点及优化方向

| Pass | 典型耗时 | 超标阈值 | 优化方向 |
|------|---------|---------|---------|
| `BasePass` | 2-5ms | >6ms | Shader 简化、DrawCall 减少 |
| `ShadowDepths` | 1-3ms | >5ms | CSM 级联减少、Shadow Distance 降低 |
| `Lumen` | 3-8ms | >10ms | 降低 Probe 密度、开启 HW-RT |
| `PostProcessing` | 1-3ms | >4ms | 关闭非必要后处理 |
| `Nanite` | 0.5-2ms | >3ms | Cluster 设置、视距调整 |
| `Reflections` | 1-3ms | >4ms | 降低反射质量、减少反射捕获 |

### 3.3 profilegpu 深入分析

当 `stat gpu` 确定了热点 Pass 后，用 `profilegpu` 进一步分析：

```bash
│  ├─ FilterScreenProbes [0.7ms]

├─ Reflections [1.8ms]
│  ├─ TraceReflections [1.2ms]
│  └─ ComposeReflections [0.6ms]
└─ SurfaceCacheUpdate [0.4ms]
```

### 3.4 Scalability 排除法

当不确定哪个系统最贵时，逐个关闭来测量：

```bash
# 基线

# 关闭阴影
sg.ShadowQuality 0
stat unit  # GPU: 13.4ms → 阴影花了 3.1ms

# 恢复阴影，关闭 Lumen
sg.ShadowQuality 3

stat unit  # GPU: 10.7ms → Lumen 花了 5.8ms  ← 最大开销

# 恢复 Lumen，关闭后处理
sg.GlobalIlluminationQuality 3
sg.PostProcessQuality 0
stat unit  # GPU: 15.0ms → 后处理花了 1.5ms
```

### 💡 核心要点

GPU 优化的第一步永远是**找到最慢的 Pass**。只有知道 GPU 时间花在哪里，才能选择正确的优化方向。

## 四、DrawCall 优化：合并、实例化、GPU-Driven

### 4.1 为什么 DrawCall 是 CPU 和 GPU 的双重杀手

每个 DrawCall 的 CPU 开销（⚠️ 估算）：

```
SetShaderParameters      ~2-5μs
BindVertexBuffer         ~1μs
BindIndexBuffer          ~1μs
SetRenderTargets         ~0.5μs
DrawIndexed API调用      ~1μs
Driver验证和Command生成  ~5-10μs
────────────────────────────
总计：                    ~10-20μs/DrawCall

如果有 2000 个 DrawCall：
2000 × 15μs = 30ms！（RenderThread 瓶颈）
```

### 4.2 方法1：Static Mesh 合并

**适用场景**：大量静态小物体（路灯、栏杆、花盆等）

```
编辑器操作：
1. 选中多个 StaticMeshActor
2. 右键 → Merge Actors
3. 设置合并参数
4. 点击 Merge

效果（⚠️ 示例数据）：
├─ 合并前：1234 个 Actor = 1234 DrawCall
├─ 合并后：45 个 Merged Mesh = 45 DrawCall
└─ RenderThread 时间：18ms → 6ms
```

**注意**：
- 不要合并需要独立移动的物体
- 不要合并使用不同 Material 的物体（会打断 Batch）
- 合并后单个 Mesh 包围盒变大，可能影响剔除效率

### 4.3 方法2：Instanced Static Mesh（ISM / HISM）

**适用场景**：大量相同 Mesh 的重复放置（树木、草地、路灯）

```cpp
// ISM：所有实例 1 个 DrawCall
UInstancedStaticMeshComponent* ISM = NewObject<UInstancedStaticMeshComponent>(this);
ISM->SetStaticMesh(MyMesh);
for (int32 i = 0; i < 100; ++i)
{
    FTransform Transform;
    Transform.SetLocation(FVector(i * 100, 0, 0));
    ISM->AddInstance(Transform);
}

// HISM：>1000 个实例时使用，自动构建层次结构做剔除
UHierarchicalInstancedStaticMeshComponent* HISM =
    NewObject<UHierarchicalInstancedStaticMeshComponent>(this);
```

**效果（⚠️ 估算）**：

| 方案 | 100棵树 DrawCall | RenderThread |
|------|-----------------|-------------|
| 100 个 StaticMeshActor | 100 | ~5ms |
| 1 个 ISM Component | 1 | ~0.2ms |
| 提升 | **100倍减少** | **25倍** |

### 4.4 方法3：GPU-Driven（Nanite + GPUScene）

**适用场景**：大量高精度静态 Mesh

Nanite 将 Culling 和 DrawCall 生成完全转移到 GPU，CPU 只提交极少量的 Indirect DrawCall。

```bash
# 开启方法
右键 StaticMesh → Nanite → Enable Nanite

# 验证
stat nanite

# 效果（⚠️ 估算）：
Nanite DrawCalls: 1              # 整个场景只有 1 个 Indirect DrawCall
Nanite Instances: 2345
Nanite Visible Clusters: 12345
```

**效果对比（⚠️ 估算，100个百万面模型场景）**：

| 指标 | 传统 LOD | Nanite |
|------|---------|--------|
| DrawCall | 2000 | 1 |
| CPU 时间 | 25ms | 3ms |
| GPU 时间 | 15ms | 10ms |
| 质量 | LOD 切换可见 | 像素级自适应 |

**Nanite 限制**：
- 不支持 Skeletal Mesh
- 不支持 World Position Offset
- 不支持半透明 Material

### 4.5 方法4：Material 合并

**问题**：相同 Mesh 但不同 Material = 无法 Batch

```
优化前：
├─ 树干 Material_A
├─ 树叶 Material_B
└─ 结果：2 DrawCall

优化后：
└─ 合并 Material（Texture Atlas + UV偏移）
    └─ 结果：1 DrawCall
```

### 4.6 DrawCall 优化的优先级

| 方法 | 效果 | 工作量 | 推荐优先级 |
|------|------|--------|-----------|
| **Nanite** | 极高 | 低（开关） | P0 |
| **ISM/HISM** | 高 | 低 | P0 |
| **Merge Actors** | 高 | 中 | P1 |
| **Material 合并** | 中 | 高 | P2 |

### 💡 核心要点

DrawCall 优化的终极方向是**GPU-Driven**：让 GPU 自己决定画什么、怎么画，CPU 只管提交。Nanite 和 GPUScene 正是这个方向的实现。

## 五、Overdraw 治理

### 5.1 什么是 Overdraw

同一个像素被多次绘制。每次绘制都要执行 Pixel Shader，Overdraw 5x 意味着 GPU 做了 5 倍的无用功。

**可视化**：

```bash
ViewMode → Optimization Viewmodes → Shader Complexity (with Quadratic Overdraw)

颜色含义：
- 绿色：1-2次（理想）
- 黄色：3-5次（可接受）
- 红色：>10次（严重！）
```

### 5.2 Early-Z 与 PrePass

UE5.7 通过 PrePass 实现 Early-Z 优化：

```
渲染流程：
1. PrePass（只写 Depth，Pixel Shader极简）
   → 所有不透明物体快速绘制一遍
   
2. BasePass（写 GBuffer）
   → 利用 PrePass 的 Depth 做 Early-Z
   → 被遮挡的像素直接跳过 Pixel Shader
```

```bash
# 控制 PrePass 行为
r.EarlyZPass
# 0 = 关闭（简单场景）
# 1 = 只渲染 Masked Material（默认）
# 2 = 所有不透明物体（复杂 Shader 场景推荐）
```

**判断标准**：
- Pixel Shader < 50 条指令：关闭 PrePass（省 DrawCall）
- Pixel Shader > 200 条指令：开启 PrePass（省 Overdraw）

### 5.3 半透明物体

半透明无法利用 Early-Z（必须从后往前绘制来保证 Blend 正确），是 Overdraw 的主要来源。

**优化策略**：

| 方法 | 适用场景 |
|------|---------|
| 用 Masked 替代 Translucent | 树叶、铁丝网等有明确轮廓的透明 |
| 限制半透明层数 | `r.MaxTranslucencyPasses 4` |
| 减小半透明面积 | 缩小粒子尺寸、减少全屏遮罩 |

**Material 类型性能对比**：

| 类型 | Overdraw | Early-Z | 排序开销 |
|------|---------|---------|---------|
| **Opaque** | 最低 | ✅ | 无 |
| **Masked** | 低 | ✅ | 无 |
| **Translucent** | 高 | ❌ | 需要排序 |

### 5.4 粒子系统 Overdraw

粒子特效通常是半透明的，大量重叠 = 严重 Overdraw。

**优化方法**：
1. 减少粒子数量（Spawn Rate 降低）
2. 减少粒子尺寸
3. 使用 SubUV 动画（少量大粒子替代大量小粒子）
4. 使用 Niagara GPU 粒子（GPU Culling + Sorting）
5. 粒子 LOD（远处降低数量）

### 💡 核心要点

Overdraw 优化的优先级：**先确认是否有不必要的 Translucent（改为 Masked）**，再处理粒子，最后调 PrePass。

## 六、Shader 性能

### 6.1 Shader Complexity 可视化

```bash
ViewMode → Optimization Viewmodes → Shader Complexity

颜色含义：
- 绿色：<50条指令
- 黄色：50-200条指令
- 红色：>200条指令
- 粉红：>500条指令（性能杀手）
```

**经验阈值（⚠️ 估算，桌面平台）**：

| 指标 | 舒适区 | 警戒区 | 危险区 |
|------|--------|--------|--------|
| Pixel Shader 指令数 | < 200 | 200-350 | > 350 |
| 纹理采样数 | ≤ 6 | 7-10 | > 10 |
| 单材质 BasePass 额外耗时 | < 0.2ms | 0.2-0.6ms | > 0.6ms |

### 6.2 减少纹理采样

```hlsl
// ❌ 6次采样
float3 BaseColor = Texture2DSample(BaseColorTex, UV);
float Metallic   = Texture2DSample(MetallicTex, UV).r;
float Roughness  = Texture2DSample(RoughnessTex, UV).r;
float AO         = Texture2DSample(AOTex, UV).r;
float3 Normal    = Texture2DSample(NormalTex, UV);
float Emissive   = Texture2DSample(EmissiveTex, UV).r;

// ✅ 4次采样（Metallic/Roughness/AO 打包到一张纹理的 RGB）
float3 BaseColor = Texture2DSample(BaseColorTex, UV);
float3 MaskMap   = Texture2DSample(MaskTex, UV);  // R=Metallic, G=Roughness, B=AO
float3 Normal    = Texture2DSample(NormalTex, UV);
float Emissive   = Texture2DSample(EmissiveTex, UV).r;
```

### 6.3 避免 GPU 分支发散

```hlsl
// ❌ 动态分支（Warp 内不同线程走不同路径 = 两条路径都要执行）
if (bUseDetailNormal)
{
    Normal = SampleDetailNormal();  // 即使只有 10% 的像素需要
}

// ✅ 静态分支（编译出不同 Shader 变体，运行时只执行一条路径）
// Material Editor → Static Bool Parameter → UseDetailNormal
// StaticSwitch 节点
```

### 6.4 Shader 变体管理

```
3 个 StaticSwitch → 2^3 = 8 个变体
10 个 StaticSwitch → 2^10 = 1024 个变体
15 个 StaticSwitch → 2^15 = 32768 个变体！
```

**优化策略**：

1. 合并相关开关为质量级别枚举
2. 用 `ShouldCompilePermutation` 排除非法组合
3. 保持 StaticSwitch 数量 < 5

详细说明见 [4.2 MaterialSystem_Deep](../Phase04_Advanced/4.2_MaterialSystem_Deep.md)。

### 💡 核心要点

Shader 优化的优先顺序是：**砍无意义分支 → 减少纹理采样 → 合并重复计算 → 最后才考虑换表现方案**。

## 七、光照优化

### 7.1 Shadow Map 优化

阴影通常是 GPU 第二大开销（仅次于光照/GI）。

```bash
stat gpu
# ShadowDepths: 4.5ms  ← 偏高

# 优化选项：
r.Shadow.CSM.MaxCascades 2          # 默认4级联 → 减少到2
r.Shadow.MaxResolution 1024          # 默认2048 → 降低
r.Shadow.DistanceScale 0.5           # 阴影距离减半
```

**CSM 可视化**：

```bash
r.Shadow.CSM.VisualizeMode 1

# 颜色含义：
- 红色：级联0（最近、最高精度）
- 绿色：级联1
- 蓝色：级联2
- 黄色：级联3（最远、最低精度）
```

### 7.2 Lumen 性能平衡

```bash
stat gpu
# Lumen: 8.5ms
#   ScreenProbeGather: 4.8ms  ← 最慢
#   Reflections: 2.5ms

# 降低 Screen Probe 开销
r.Lumen.ScreenProbeGather.ScreenTraces 0        # 关闭 Screen Traces
r.Lumen.ScreenProbeGather.RadianceCache.ProbeResolution 16  # 降低分辨率

# 如果有 RTX 显卡
r.Lumen.HardwareRayTracing 1                    # HW-RT 通常比 SW Tracing 更快
r.Lumen.HardwareRayTracing.Default 1
```

**Lumen 配置对比（⚠️ 估算）**：

| 配置 | GPU 时间 | 画质影响 |
|------|---------|---------|
| Epic（默认） | ~8-12ms | 最佳 |
| 降低 Probe 密度 | ~4-6ms | 轻微降低 |
| + HW-RT | ~3-5ms | 接近最佳 |
| 关闭 Lumen | ~0ms | 无动态 GI |

### 7.3 光源数量控制

```bash
# 限制动态光源影响范围
r.LightMaxDrawDistanceScale 0.5    # 光源绘制距离减半

# 每个光源：设置合理的 Attenuation Radius
# 超出范围自动剔除，减少光照计算
```

### 💡 核心要点

光照优化的核心是**先确定哪个光照子系统最贵**（Shadow / Lumen / Reflection），再针对性调整。不要同时降低所有光照质量——那是在用画质换帧率，不是在优化。

## 九、实战案例：从 45FPS 到 80FPS 的完整过程

### 9.1 场景描述

**项目**：某城镇场景  
**问题**：帧率 45FPS，目标 60FPS  
**硬件**：RTX 3060、i7-12700K

### 9.2 第一步：stat unit 判断瓶颈（30秒）

```bash
stat unit

Frame: 22.2ms (45 FPS)
Game:  5.5ms
Draw:  18.3ms   ← RenderThread 是主要瓶颈
GPU:   12.7ms   ← GPU 也有压力
```

**结论**：RenderThread（CPU）是主要瓶颈，GPU 次要。

### 9.3 第二步：stat scenerendering 定位 CPU 热点（1分钟）

```bash
stat scenerendering

RenderViewFamily:       18.3ms
InitViews:              8.5ms     ← 占 46%
Dynamic shadow setup:   3.2ms

stat rhi
Draw Calls:             3456      ← 远超 1000 的目标
SetShaderParameter:     12345     ← 太高
```

**结论**：DrawCall 过多是 CPU 瓶颈的根因。

### 9.4 第三步：stat gpu 定位 GPU 热点（30秒）

```bash
stat gpu

BasePass:        4.2ms
ShadowDepths:    3.5ms     ← 偏高
Lumen:           3.8ms
PostProcess:     1.2ms
```

**结论**：GPU 没有单一极端热点，但阴影偏高。

### 9.5 第四步：实施优化（2小时）

**优化1：Merge Actors（最大收益）**

```
选中所有场景装饰（花盆、路灯、栏杆等）→ Merge Actors

效果：
├─ DrawCall：3456 → 890（减少 74%）
├─ RenderThread：18.3ms → 8.5ms（减少 54%）
└─ 工作量：30分钟
```

**优化2：ISM 替换重复物体**

```
100 个相同路灯 → 1 个 ISM Component

效果：
├─ DrawCall：890 → 620
├─ RenderThread：8.5ms → 7.2ms
└─ 工作量：20分钟
```

**优化3：部分物体开启 Nanite**

```
建筑、雕塑等高精度静态 Mesh → Enable Nanite

效果：
├─ DrawCall：620 → 345
├─ RenderThread：7.2ms → 5.8ms
├─ GPU BasePass：4.2ms → 2.8ms
└─ 工作量：15分钟
```

**优化4：Shadow Map 降级**

```bash
r.Shadow.CSM.MaxCascades 2
r.Shadow.MaxResolution 1024

效果：
├─ GPU ShadowDepths：3.5ms → 1.8ms
├─ 画质影响：远处阴影精度降低（多数场景可接受）
└─ 工作量：5分钟
```

### 9.6 第五步：验证效果（30秒）

```bash
stat unit

Frame: 12.5ms (80 FPS) ✅
Game:  5.5ms
Draw:  5.8ms（优化前 18.3ms）
GPU:   9.7ms（优化前 12.7ms）

stat rhi
Draw Calls: 345（优化前 3456）⬇️ 减少 90%
```

### 9.7 优化总结

| 优化项 | CPU 效果 | GPU 效果 | 工作量 |
|-------|---------|---------|--------|
| Merge Actors | RT -54% | - | 30min |
| ISM | RT -15% | - | 20min |
| Nanite | RT -20% | BasePass -33% | 15min |
| Shadow 降级 | - | Shadow -49% | 5min |
| **总计** | **18.3ms → 5.8ms** | **12.7ms → 9.7ms** | **~70min** |

**帧率**：45FPS → 80FPS（+78%）  
**投入产出比**：极高

### 💡 核心要点

这个案例的关键不是"做了什么优化"，而是**每一步都先测量、再动手**。从 `stat unit` 到 `stat scenerendering` 到 `stat rhi`，每一步都在缩小范围，从未盲目操作。

# 第五部分：与其他系统的关系

> **核心目标**：知道性能问题什么时候该在渲染层解决，什么时候根因其实在别的系统。

---

## 一、与 Nanite / GPUScene 的关系

Nanite 和 GPUScene 是渲染性能优化最重要的底层支撑：

- **Nanite**：消除传统 LOD 切换、把 Culling 和光栅化转移到 GPU
- **GPUScene**：把 Instance 数据镜像到 GPU，让 GPU 自己做 Instance Culling

**当你看到 CPU 瓶颈（DrawCall 过多、InitViews 慢）时**，首先考虑的应该是：

1. 能否给静态 Mesh 开启 Nanite？
2. 能否用 HISM 代替独立 Actor？
3. GPUScene 是否正常工作？

而不是先去优化 Shader 或降画质。

### 💡 核心要点

Nanite 和 GPUScene 把**最大的 CPU 瓶颈（DrawCall）从根本上消除**。在考虑其他优化之前，先确认 GPU-Driven 是否已经充分利用。

---

## 二、与 Lumen 的关系

Lumen 通常是 GPU 最大的单一开销。但 Lumen 的性能不只取决于自身设置，还受材质影响：

- 复杂材质的 Surface Cache 更新更慢
- 高光材质触发的反射 Trace 更多
- Roughness 过低的大面积材质让 Screen Probe 更吃力

所以当 `stat gpu` 中 Lumen 偏高时，不一定是 Lumen 设置不对，可能是材质导致的。

**排查顺序**：

1. 先看 Lumen 子系统哪个最慢（ScreenProbe? Reflections? SurfaceCache?）
2. 如果是 Reflections 慢 → 检查场景中是否有大面积低 Roughness 材质
3. 如果是 SurfaceCache 慢 → 检查是否有大量动态材质更新

### 💡 核心要点

Lumen 性能不是孤立的——它和材质、场景布局紧密耦合。优化 Lumen 有时意味着优化材质，而不是降低 Lumen 质量。

---

## 三、与 RenderGraph 的关系

RenderGraph 对性能的影响主要在两个方面：

1. **Transient Aliasing**：RDG 自动复用 RT 内存，减少显存占用和带宽
2. **Pass 合并**：RDG 可以自动合并兼容的 Pass，减少状态切换

你不需要直接"优化 RenderGraph"，但理解它有助于解释一些现象：

- 为什么某些 Pass 在 profilegpu 中显示为 0ms（被合并了）
- 为什么删掉一个 Pass 不一定能省 RT 内存（Aliasing 已经在复用了）

### 💡 核心要点

RenderGraph 是渲染性能的**基础设施**，不是优化目标。理解它主要是为了**正确解读 profiling 数据**。

---

## 四、什么时候不该继续优化渲染

当出现以下情况时，性能问题可能不在渲染层：

| 现象 | 可能的真正原因 |
|------|-------------|
| `Game` > `Draw` 且 `Game` > `GPU` | GameThread 瓶颈：Blueprint、Physics、Tick |
| 帧率间歇性卡顿（平时60，偶尔30） | GC、资源 Streaming、Shader 编译 |
| 内存持续增长 | 资源泄漏、Texture Streaming 配置 |
| Loading 慢 | 资源打包策略、IO 瓶颈 |

**跳出渲染层的信号**：

```
stat unit 显示：
  Game: 25ms    ← 最长
  Draw: 8ms
  GPU:  10ms

→ 不要在渲染层浪费时间
→ 转去查 Blueprint Profiler / 物理系统 / Tick 逻辑
```

### 💡 核心要点

最成熟的优化能力不是"总能在渲染里找到答案"，而是**知道什么时候该退出渲染层**。

## A.5 实战检查清单

### 瓶颈定位检查清单

```
[ ] stat unit 已执行，已判断瓶颈线程
[ ] 如果 CPU 瓶颈：stat scenerendering 已执行
[ ] 如果 GPU 瓶颈：stat gpu 已执行
[ ] 已确定最慢的 Pass 或函数
[ ] 未跳层直接动手
```

### CPU 优化检查清单

```
[ ] DrawCall 数量是否超标（stat rhi）
[ ] InitViews 是否偏高（stat scenerendering）
[ ] 是否有大量可合并的静态 Mesh
[ ] 是否有大量重复 Mesh 可用 ISM/HISM
[ ] 静态 Mesh 是否可开启 Nanite
```

### GPU 优化检查清单

```
[ ] 最慢的 Pass 是哪个（stat gpu）

[ ] Shader Complexity 是否有明显热点
[ ] Shadow/Lumen 设置是否合理
[ ] 是否有严重 Overdraw（半透明/粒子）
```

### 验证检查清单

```
[ ] 优化后重新执行 stat unit

```
