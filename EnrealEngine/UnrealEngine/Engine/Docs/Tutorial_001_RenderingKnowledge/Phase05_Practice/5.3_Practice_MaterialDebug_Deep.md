# 第一部分：为什么材质调试总是低效

> **核心目标**：先把“问题长什么样”说清楚，再建立正确的排查顺序。材质问题之所以难，不是因为工具少，而是因为大家经常从错误的一层开始查。

---

## 一、四类最常见故障

### 1.1 编译失败：材质直接变成默认外观

最典型的现象不是“有一点不对”，而是**整个材质回退成默认材质**。在编辑器里通常表现为：

```
改了一个节点
  ↓
点击 Apply
  ↓
日志刷出一串 HLSL / Material 错误
  ↓
预览球可能仍有缓存结果
  ↓
场景里的真实物体变成默认材质 / 灰格 / 明显错误
```

这类问题的根因通常不在“美术审美”，而在**材质表达违反编译约束**：

- 在错误的材质域里读取SceneTexture
- Substrate图产生非法组合
- 纹理采样类型不匹配
- 自定义插值器超预算
- 某个表达式缺失必需输入

### 1.2 场景表现错误：预览球是对的，进关卡就错

这是第二高频问题，也是最容易误判成“引擎Bug”的问题。

典型症状：

| 症状 | 真正可能出错的层 |
|------|------------------|
| BaseColor看起来发灰 | 不是BaseColor本身，可能是曝光、AO、Lumen间接光照 |
| 金属看起来像塑料 | Roughness / Metallic输出错误，或反射环境不足 |
| 法线方向怪异 | Normal空间不对、纹理压缩不对、切线空间有问题 |
| 某物体和相同材质别的物体不一致 | 运行时材质实例覆盖、组件级Override、Nanite路径差异 |

**关键现实**：预览球只验证“材质图能否生成一个结果”，不验证这个结果在真实场景里的光照、实例、Pass绑定是否正确。

### 1.3 参数不生效：你改了值，但真正被渲染的不是那个值

这类问题在项目里非常常见，因为UE材质链路经常有多层覆盖：

```
母材质 UMaterial
  ↓
材质实例 UMaterialInstanceConstant
  ↓
运行时动态实例 UMaterialInstanceDynamic
  ↓
组件级 OverrideMaterials
  ↓
蓝图 / C++ 每帧再次写入参数
```

你在编辑器里改的是第1层，但场景实际使用的是第4层或第5层，于是现象就是：

- 细节面板里数值变了，画面不变
- 场景里一会儿生效，一会儿被改回去
- 只有某个Actor不对，别的都对

### 1.4 性能异常：不是“材质复杂”，而是“复杂在哪里”

“这个材质太重了”是低质量结论。高质量结论应该是：

- 是像素指令数过高？
- 还是纹理采样过多？
- 还是Substrate层数太深？
- 还是BasePass本身不贵，但反射/Lumen因为它变贵？
- 还是只有某一类实例触发了复杂分支？

同样是“红色热力图”，背后可能是完全不同的问题。

### 💡 核心要点

材质问题可以压缩成四类：**编译失败、场景表现错误、参数不生效、性能异常**。只要先判断自己属于哪一类，后面的排查路径就会立刻收敛。

## 二、为什么“看起来像材质问题”往往不真是材质问题

### 2.1 材质只负责“输出”，不负责“最终观感”

从渲染链路看，材质更多是在输出中间结果：

```
Material Graph
  ↓
BaseColor / Normal / Roughness / Metallic / Opacity / Substrate Closure
  ↓
BasePass写入 GBuffer / Substrate Buffer
  ↓
Lighting / Lumen / Reflection / PostProcess 消费这些输出
  ↓
最终屏幕颜色
```

所以“最终看起来不对”不等于“材质图不对”。

例如：

- 颜色发灰：可能是ACES / 曝光 / Lumen
- 反射发脏：可能是反射环境或Roughness
- 细节发糊：可能是Mipmap、Streaming、VT页缺失
- 看起来闪烁：可能是TAA、Lumen、Temporal反走样，而不是材质节点

### 2.2 材质调试失败的根本原因：没有分层

很多人一上来就做下面这种事：

```
症状：地面材质发黑
  ↓
开始疯狂改材质图
  ↓
把乘法、Lerp、法线都试一遍
  ↓
问题仍然存在
  ↓
最后发现只是场景里SkyLight没更新
```

这类低效来自于**没先分层**。正确顺序应该是：

1. 先看材质有没有编译错误
2. 再看材质输出缓冲是否正确
3. 再看运行时实例参数是否被覆盖
4. 最后才看Lighting / Lumen / Reflection

### 💡 核心要点

材质调试不是“盯着节点图看”，而是先把**材质输出**和**最终光照结果**分开。只要这两个层混在一起，排错一定慢。

---

## 三、本章给出的四层调试链路

我们把整个材质调试流程收敛成四层：

```
第1层：编辑器层
  节点预览、Live Preview、Find in Material、Stats

第2层：编译层
  CacheShaders、CompileErrors、HLSL Code、Material IR

第3层：场景层
  VisualizeBuffer、Shader Complexity、Substrate视图、Lighting隔离

第4层：运行时层
  MID参数覆盖、组件Override、蓝图/C++写值、GPU Pass耗时
```

任何问题都从第1层开始，**逐层向下**，不要跳层。

### 3.1 为什么这种顺序有效

因为它和真实的数据流是一致的：

```
节点图不对
  一定会影响编译层

编译层不对
  一定会影响场景层

场景层不对
  不一定是运行时层

运行时层不对
  往往节点图本身根本没问题
```

这就是为什么“先查实例覆盖，再改节点图”能省很多时间。

### 💡 核心要点

本章不是“材质技巧合集”，而是一套**按层排查的方法论**。你后面看到的每个工具，都会被放回这四层链路里解释。

## 一、从节点图到场景像素的数据流

### 1.1 完整链路

```
Material Editor 节点图
  ↓
FMaterialEditor 命令（Apply / Preview / Stats / Search）
  ↓
UMaterial::CacheShaders()
  ↓
FHLSLMaterialTranslator 递归翻译表达式树
  ↓
FMaterialShaderMap / Shader编译
  ↓
材质资源绑定到 BasePass / 其他Pass
  ↓
GBuffer / Substrate Buffer
  ↓
Lighting / Lumen / Reflections / PostProcess
  ↓
屏幕最终结果
```

### 1.2 对调试最重要的两个分界点

#### 分界点A：编译成功之前，所有问题都不是“运行时问题”

只要材质还在报编译错误，就不要先看：

- Lumen
- 反射
- 曝光
- GPU Profile

因为材质根本没有稳定进入渲染路径。

#### 分界点B：GBuffer/Substrate输出正确之后，就不该继续在材质图里硬查

如果：

- BaseColor看对了
- WorldNormal看对了
- Roughness看对了
- ShadingModel / Substrate结果也对

但最终屏幕仍不对，那么大概率问题已经在**光照、反射、后处理**，不是材质图。

### 💡 核心要点

材质调试真正重要的不是“会多少节点”，而是知道**哪一层已经证明没问题**。一旦某层被证明正确，就要果断往后走，不要回头浪费时间。

| 按钮 / 功能 | 作用 | 适合处理什么问题 |

## 三、场景侧工具：Buffer可视化、Shader Complexity、Substrate

### 3.1 为什么先看Buffer，不先看最终颜色

最终颜色混合了太多系统。材质调试更应该先看中间缓冲：

- `BaseColor`
- `WorldNormal`
- `Roughness`
- `Metallic`
- `ShadingModel`
- `SceneDepth`

这套Buffer可视化目标由`FBufferVisualizationData`注册。

**来源**：`Engine/Source/Runtime/Engine/Private/BufferVisualizationData.cpp`，搜索 `FBufferVisualizationData::Initialize`、`FBufferVisualizationData::ConfigureConsoleCommand`、`r.BufferVisualizationOverviewTargets`

```cpp

    TEXT("r.BufferVisualizationOverviewTargets"),
    TEXT("BaseColor,Specular,SubsurfaceColor,WorldNormal,...,Opacity,SceneDepth,Roughness,Metallic,ShadingModel,..."),
    *ConsoleDocumentationOverviewTargets,
    ECVF_Default
);
```

### 3.2 `VisualizeBuffer`和`ShaderComplexity`本质上是ViewMode

它们不是“某个材质插件功能”，而是引擎级ViewMode。

**来源**：
- `Engine/Source/Runtime/Engine/Public/ShowFlagsValues.inl`，搜索 `SHOWFLAG_FIXED_IN_SHIPPING(0, VisualizeBuffer`、`SHOWFLAG_ALWAYS_ACCESSIBLE(VisualizeSubstrate`、`SHOWFLAG_FIXED_IN_SHIPPING(0, ShaderComplexity`
- `Engine/Source/Runtime/Engine/Private/ShowFlags.cpp`，搜索 `SetVisualizeBuffer`、`SetVisualizeSubstrate`、`SetShaderComplexity`

```cpp
SHOWFLAG_FIXED_IN_SHIPPING(0, VisualizeBuffer, ...)
SHOWFLAG_ALWAYS_ACCESSIBLE(VisualizeSubstrate, ...)
SHOWFLAG_FIXED_IN_SHIPPING(0, ShaderComplexity, ...)

EngineShowFlags.SetVisualizeBuffer(ViewModeIndex == VMI_VisualizeBuffer);
EngineShowFlags.SetVisualizeSubstrate(ViewModeIndex == VMI_VisualizeSubstrate);
EngineShowFlags.SetShaderComplexity(ViewModeIndex == VMI_ShaderComplexity ...);
```

也就是说，**它们是渲染管线原生的观察窗口**，不是编辑器“效果图”。

### 3.3 Substrate视图的意义

如果项目使用Substrate，那么“只看BaseColor/Roughness”有时不够，因为真正影响光照成本和表现的是：

- Closure数量
- 分层关系
- 材质分类路径

`SubstrateVisualize.cpp`会在`VisualizeSubstrate`开启时注入专门的调试Pass。

**来源**：`Engine/Source/Runtime/Renderer/Private/Substrate/SubstrateVisualize.cpp`，搜索 `ShouldRenderSubstrateDebugPasses`、`AddSubstrateDebugPasses`、`View.Family->EngineShowFlags.VisualizeSubstrate`

### 3.4 `Shader Complexity`能回答什么，不能回答什么

它能回答：

- 哪些屏幕区域像素着色更重
- 哪些材质覆盖区域存在热点
- 改某个分支后热点有没有下降

它不能回答：

- 为什么Lumen Pass更慢
- 为什么反射Pass更慢
- 为什么材质在某平台一定更慢

所以它适合做**第一道性能筛查**，但不是最终归因工具。

### 💡 核心要点

场景层最核心的习惯是：**先看中间缓冲，再看最终颜色；先看ViewMode，再看肉眼感觉。**

## 四、运行时实例链路：为什么参数改了没反应

### 4.1 MID路径的本质

运行时动态改参数通常走`UMaterialInstanceDynamic`：

**来源**：`Engine/Source/Runtime/Engine/Private/Materials/MaterialInstanceDynamic.cpp`，搜索 `UMaterialInstanceDynamic::InitializeMID`、`SetVectorParameterValue`、`SetScalarParameterValue`

```cpp
void UMaterialInstanceDynamic::InitializeMID(UMaterialInterface* ParentMaterial)
void UMaterialInstanceDynamic::SetVectorParameterValue(FName ParameterName, FLinearColor Value)
void UMaterialInstanceDynamic::SetScalarParameterValue(FName ParameterName, float Value)
```

这里最重要的事实是：

**改MID参数不会触发材质重编译。**

它只是修改实例参数数据，并让后续渲染读取新值。

### 4.2 这会带来哪类误判

#### 误判1：你在改“母材质默认值”，但场景使用的是MID覆盖值

结果：编辑器里参数改了，场景不变。

#### 误判2：你在蓝图BeginPlay里创建了MID，但Construction Script里又给组件重新设了材质

结果：运行时你改的是一个MID，场景实际渲染的是另一个MaterialInterface。

#### 误判3：每帧Tick都在写参数

结果：你手动在细节面板里改值后，下一帧立刻被代码改回去。

### 4.3 为什么材质实例问题常常比节点图更常见

项目进入中后期以后，主材通常比较稳定，真正频繁变化的是：

- 实例参数
- 组件Override
- 蓝图逻辑
- 序列动画
- Niagara / MPC驱动

所以“参数不生效”的排查优先级通常应该高于“图是不是又接错了”。

### 💡 核心要点

运行时层的核心原则是：**先确认当前Actor真正绑定的是谁，再确认最后写入参数的是谁。** 很多“材质Bug”本质上都是实例链路Bug。

### 3.1 函数概览


### 3.3 关键细节

#### 细节A：失败时会明确说明“会回退默认材质”

`HandleCacheShadersForResourcesErrors()`里直接构造错误字符串：

**来源**：`Engine/Source/Runtime/Engine/Private/Materials/Material.cpp`，搜索 `HandleCacheShadersForResourcesErrors`、`CurrentResource->GetCompileErrors`、`Default Material will be used in game`

```cpp
);

}
```

这就是为什么**真正该看的不是“最后材质变粉了”，而是前面那段CompileErrors**。

#### 细节B：编译错误并不都来自HLSL编译器

很多错误在进入最终Shader编译之前，就已经在Material翻译阶段被拦下来了。也就是说：

- 有些错误是表达式规则错误
- 有些错误是材质域约束错误
- 有些错误是Substrate图结构错误

## 四、MID参数更新：为什么运行时参数不会触发重编译

### 4.1 函数概览

**来源**：`Engine/Source/Runtime/Engine/Private/Materials/MaterialInstanceDynamic.cpp`  
**关键入口**：
- `InitializeMID()`
- `SetVectorParameterValue()`
- `SetScalarParameterValue()`

**职责**：在不重编译Shader的前提下，给实例级参数提供运行时覆盖能力。

### 4.2 核心流程


- 改参数不应导致重新编译卡顿
- 如果你改参数后看到明显编译行为，说明可能改到的是Static开关或母材质本体

#### 细节B：实例调试要先查绑定对象

因为一个Actor最终可能走的是：

- 母材质
- MIC
- MID
- 组件Override

MID问题和编译问题是两条完全不同的链路。**前者查“当前绑定的是谁”，后者查“当前编译出了什么”。**

# 第四部分：实践指南

> **核心目标**：把前面的工具和源码入口，变成能直接落地的排查流程。你遇到问题时，应该能拿着本节按步骤做，而不是重新思考一遍方法论。

---

## 一、10分钟标准排查流程

### 1.1 总流程

```
第1分钟：确认症状属于哪一类
  编译失败 / 表现错误 / 参数不生效 / 性能异常

第2-3分钟：编辑器层
  Search / 节点预览 / Force Refresh / Stats

第4-6分钟：编译层
  Apply → 看 CompileErrors → 看 HLSL / Platform Stats

第7-8分钟：场景层
  VisualizeBuffer → Shader Complexity → 必要时 VisualizeSubstrate

第9-10分钟：运行时层
  查实例绑定 / MID / Override / 蓝图写值
```

### 1.2 第一个动作永远不是“改节点”

你应该先做的是：

1. `Apply`
2. 看有没有报错
3. 看`Stats`
4. 看`BaseColor` / `WorldNormal` / `Roughness`

如果这四步都还没做，就直接改图，成功率通常很低。

### 💡 核心要点

标准流程的意义，在于把“凭经验乱查”变成“固定动作序列”。只要流程固定，排查速度就会稳定。

---

## 二、编译失败：错误信息如何落到具体节点

### 2.1 先看哪三处

#### 第一处：Material Editor消息区

它离节点图最近，适合快速跳回问题位置。

#### 第二处：`Stats` / `Platform Stats`

适合看：

- 是否某个派生实例单独失败
- 是否只有某种质量级别失败

#### 第三处：Output Log

适合看完整的`CompileErrors`串，因为`Material.cpp`会把Translator和编译器汇总后的错误一起吐出来。

### 2.2 五类高频错误与根因

以下错误都来自真实源码里的`Errorf()`信息：

**来源**：`Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.cpp`

| 错误信息 | 典型根因 | 首选修复动作 |
|----------|----------|-------------|
| `Only transparent or postprocess materials can read from scene depth.` | 在不允许的材质域里读取SceneDepth | 先确认Material Domain，再确认该节点是否必须存在 |
| `SceneTexture expressions cannot be used in opaque materials...` | Opaque材质使用了错误的SceneTexture读取 | 先改为允许的路径，或把逻辑转移到后处理材质 |
| `Gather does not support virtual textures.` | 对VT做了不支持的Gather采样 | 改采样方式，不要继续在Sampler节点上硬凑 |
| `Maximum number of custom vertex interpolators exceeded.` | 自定义插值器预算超限 | 合并插值、下放到像素阶段、减少中间通道 |
| `Substrate material errors encountered.` / `Substrate - Material has no BSDF` | Substrate图结构非法、没有有效BSDF输出 | 先检查Substrate主链路是否完整闭合 |

### 2.3 “从错误字符串跳到节点”的方法

#### 方法A：搜参数名 / 注释名

如果错误涉及某个参数名或函数名，先用`Find in Material`定位。

#### 方法B：对疑似路径启用节点预览

如果错误是结构性问题，比如Substrate层没有闭合，直接预览某个中间节点往往更快。

#### 方法C：最小化图

把嫌疑链路逐段断开，先让材质恢复编译，再逐步接回。原则是：

```
先恢复“可编译”
  ↓
再恢复“正确”
```

### 2.4 一个典型排查例子

```
症状：Apply后报 SceneDepth 错误
  ↓
步骤1：确认材质不是 Post Process / Transparent
  ↓
步骤2：Search "SceneDepth" 或搜 SceneTexture 节点
  ↓
步骤3：预览该分支，确认只是某个局部效果需要深度
  ↓
步骤4：先断开这段逻辑让材质恢复编译
  ↓
步骤5：决定改域还是改实现，而不是继续让错误存在
```

### 2.5 `Fixup mismatched samplers`什么时候有用

这个命令来自`MaterialEditorActions.cpp`。它不是万能修复器，但当你做了以下操作后很有价值：

- 把一批Texture Object改回Texture Sample
- 复制了旧节点链路
- 替换了纹理资源类型

它能帮你快速修正常见采样器类型不匹配，而不用每个节点手工点。

### 💡 核心要点

编译失败阶段最重要的不是“会多少错误信息”，而是**先恢复到稳定可编译状态**。只有先把错误清掉，后面的视觉验证才有意义。

## 三、预览正常、场景异常：如何定位是材质输出还是光照问题

### 3.1 第一步：切到Buffer可视化

建议顺序：

1. `BaseColor`
2. `WorldNormal`
3. `Roughness`
4. `Metallic`
5. `ShadingModel`

如果这5个目标都对，而最终观感不对，就不要继续只改材质图。

### 3.2 具体判断方法

#### 看`BaseColor`

如果`BaseColor`已经偏灰，问题就在材质图或纹理输入。

如果`BaseColor`正确、最终颜色偏灰，优先查：

- 曝光
- 后处理
- AO
- Lumen间接光照

#### 看`WorldNormal`

如果法线可视化方向错乱，优先查：

- 法线纹理导入设置
- 法线强度计算
- 切线空间开关

不要先去怀疑反射系统。

#### 看`Roughness` / `Metallic`

金属像塑料、塑料像金属，80%的情况都能在这两个Buffer里直接定性。

### 3.3 第二步：切`Shader Complexity`

如果视觉不对同时性能还怪，切Shader Complexity能回答两个问题：

- 这个区域是否真的更贵
- 贵的是整个材质，还是只有某一块遮罩 / 某一层叠加

### 3.4 第三步：如果项目用Substrate，再看`VisualizeSubstrate`

Substrate问题经常不是“BaseColor错”，而是：

- 层数太深
- 混合方式不对
- 某层在不该参与时仍进入完整着色路径

这时继续盯普通GBuffer往往不够。

### 3.5 一个常见误判案例

```
症状：清漆车漆看起来脏、偏暗
  ↓
BaseColor 正常
Roughness 正常
Metallic 正常
  ↓
最后发现是反射环境不足 + 场景曝光压低
  ↓
材质本体并没有问题
```

### 💡 核心要点

“预览球对、场景错”时，最重要的动作不是重新做材质，而是**先把材质输出和光照结果分离**。Buffer可视化就是干这个的。

## 四、参数不生效：材质实例与运行时覆盖排查

### 4.1 先回答三个问题

1. 当前Actor真正绑定的是哪个`MaterialInterface`？
2. 这个对象是母材质、MIC还是MID？
3. 最后一次写入参数的是编辑器、蓝图还是C++？

如果这三个问题没答清，几乎不可能高效排查。

### 4.2 常见故障模式

#### 模式A：改了母材质默认值，但实例里有覆盖

现象：

- 母材质里参数变了
- 某个实例无变化

处理：

- 打开实例，检查该参数是否被Override
- 看组件是否直接绑定了实例而不是母材质

#### 模式B：运行时创建了MID，但没把它重新设回组件

现象：

- 蓝图里`Create Dynamic Material Instance`
- 随后改参数
- 场景不变

根因通常是：创建出来了，但真正渲染的组件没用这个MID。

#### 模式C：Tick里每帧回写

现象：

- 你在细节面板里调一下参数，下一帧就弹回

处理：

- 搜蓝图 / C++ 有没有每帧写值
- 特别检查Timeline、Sequencer、Construction Script、Niagara驱动

### 4.3 推荐排查顺序

```
Actor Details
  ↓ 看材质槽
组件 OverrideMaterials
  ↓ 看当前绑定对象
蓝图里是否 Create MID
  ↓
参数是否由 Tick / Timeline / Sequencer 持续写入
  ↓
最后才回到母材质图本体
```

### 4.4 一个高效技巧：用极端值验证是否真的生效

不要一上来改`0.4 -> 0.5`这种肉眼难判断的值。先改成极端值：

- Roughness：`0` 或 `1`
- Metallic：`0` 或 `1`
- BaseColor：纯红 / 纯绿
- Emissive：明显发光值

如果极端值都看不见变化，问题八成不在“调得不够明显”，而在“根本没写到实际实例”。

### 💡 核心要点

参数不生效问题的本质是**绑定对象与最后写入者不一致**。别急着改图，先把“谁在渲染、谁在写值”这两个身份查清。

## 五、性能异常：从Stats到GPU Pass

### 5.1 先分清“编译复杂”和“运行复杂”

`Stats` / `Platform Stats`更适合回答：

- 代表性Shader有多少指令
- 纹理采样数是多少
- 某个实例是否比别的实例更重

`profilegpu`更适合回答：

- BasePass到底花了多少ms
- 反射/Lumen是否因该材质变贵
- 问题是材质本身，还是被材质放大的后续Pass

### 5.2 一套实用预算（⚠️ 估算值）

以下是桌面项目里可用的经验阈值：

| 指标 | 舒适区 | 警戒区 | 危险区 |
|------|--------|--------|--------|
| Pixel Shader指令数 | `< 200` | `200-350` | `> 350` |
| 纹理采样数 | `<= 6` | `7-10` | `> 10` |
| Shader Complexity可见热点面积 | 小且局部 | 中等 | 大片红白 |
| 单材质导致的BasePass额外耗时 | `< 0.2ms` | `0.2-0.6ms` | `> 0.6ms` |

这些不是官方硬阈值，但足够用于日常优先级判断。

### 5.3 优化顺序不要反

正确顺序：

1. 先砍无意义分支
2. 再减少重复纹理采样
3. 再合并重复计算
4. 最后才考虑换表现方案

错误顺序：

1. 先压缩纹理
2. 先把材质做丑
3. 仍然不知道真正热点在哪里

### 5.4 什么情况该怀疑Substrate层数

如果材质具备这些特征：

- 多层清漆 / 污渍 / 覆雪 / 湿润叠加
- 复杂条件混合
- 一堆Substrate节点串接

那就要主动怀疑：

- Closure数量
- Tile分类是否进入复杂路径
- 某些遮罩是否本可提前剔除却没有剔除

### 5.5 `profilegpu`该怎么看

重点看：

- `BasePass`
- `Substrate`相关Pass（如果启用）
- `Lumen Reflections` / `Lumen Screen Probe`（当高光或间接光疑似受材质影响）
- `PostProcessMaterial`（如果问题材质其实是后处理材质）

### 5.6 一个典型性能排查例子

```
症状：湿地面材质打开后帧率掉 12 FPS
  ↓
Stats：像素指令数 420，纹理采样 11
  ↓
Shader Complexity：热点只集中在地面区域
  ↓
profilegpu：BasePass 明显上涨，Lumen Reflections 也上涨
  ↓
结论：不是“全局GPU变慢”，而是高光材质同时放大了BasePass和反射成本
  ↓
处理：减少重复湿润层采样，简化高频法线混合，必要时减少清漆层参与
```

### 💡 核心要点

性能调试的核心不是“看到红就优化”，而是先回答**贵在哪个Pass、贵的是哪类计算、是不是材质放大了别的系统成本**。

## 六、RenderDoc与高级验证

### 6.1 什么时候值得抓帧

只有在下面三类问题时，抓帧收益高：

1. Buffer可视化看起来正常，但最终结果仍异常
2. `profilegpu`显示BasePass正常，后续Pass异常，需要看真实输入
3. Substrate / 后处理材质路径复杂，单靠编辑器窗口难以确认

### 6.2 抓帧时重点看什么

#### 看BasePass输出

确认：

- BaseColor
- Normal
- Roughness
- Metallic
- 深度

是否与编辑器可视化一致。

#### 看后续消费Pass

如果是Lumen或反射问题，继续看：

- 这些Pass拿到的GBuffer是不是正确
- 是否某个材质输出在后续Pass被当成了意料之外的输入

### 6.3 Material IR / HLSL更适合回答什么

| 工具 | 更适合回答的问题 |
|------|------------------|
| 节点预览 | 中间结果是不是已经偏了 |
| HLSL Code | 翻译后的表达式是否符合预期 |
| Material IR Dump | 图结构和翻译模块是否异常复杂 |
| RenderDoc | 真正进入GPU的结果是什么 |

### 💡 核心要点

RenderDoc不是第一步，而是**当编辑器、Stats、ViewMode都不足以解释现象时的最终验证工具**。

# 第五部分：与其他系统的关系

> **核心目标**：知道材质问题什么时候该继续在材质里查，什么时候必须切换到别的系统。没有这条边界，调试会无限延长。

---

## 一、与BasePass / MeshDrawCommand的关系

材质真正进入渲染，不是“停留在材质编辑器里”，而是被BasePass消费。

```
Material / MaterialInstance
  ↓
ShaderMap / FShader
  ↓
MeshPassProcessor 选择合适的 Pass Shader
  ↓
MeshDrawCommand 绑定PSO与资源
  ↓
BasePass 输出 GBuffer / Substrate Buffer
```

因此下面这些现象，应该开始联想到BasePass / MDC，而不是只看材质图：

- 只有某一类网格错
- 某个Pass里材质根本没被正确消费
- DrawCall层面的绑定对象不对

### 💡 核心要点

材质要想真正“显示出来”，必须先成功进入BasePass / MeshDrawCommand链路。**编辑器预览正确并不代表这条链路正确。**

---

## 二、与GBuffer / Substrate的关系

如果项目走传统GBuffer路径，那么材质调试重点是：

- BaseColor
- Normal
- Roughness
- Metallic
- ShadingModel

如果项目走Substrate路径，那么除了最终观感，还要关注：

- Closure数量
- 分层结构
- 分类路径

也就是说：

```
传统材质问题
  更常问：输出值对不对？

Substrate材质问题
  更常问：输出值对不对 + 结构是不是过深/非法/过贵？
```

### 💡 核心要点

Substrate让材质调试从“值调试”扩展到了“结构调试”。这就是为什么UE5.7里看HLSL和看IR都变得更有意义。

---

## 三、与Lumen / Nanite的关系

### 3.1 与Lumen

Lumen会消费材质输出参与：

- 间接光照
- 反射
- Surface Cache捕获

所以当你看到：

- 高光结果怪
- 间接光照色偏
- 某些表面在Lumen下异常发灰

要先问：

1. 材质输出是否正确？
2. Lumen消费这些输出时是否被别的系统影响？

不要直接把“Lumen结果不对”写成“材质问题”，也不要反过来。

### 3.2 与Nanite

如果只有Nanite对象不对，要额外确认：

- 预览网格和真实网格是否一致
- 该材质是否在Nanite物体上触发了不同表现路径
- 这是不是几何/法线/切线问题，而不是材质逻辑问题

### 💡 核心要点

Lumen和Nanite经常让材质问题“表面上更像材质”。真正高效的做法是：**先确认材质输出，再确认这些系统如何消费它。**

---

## 四、什么时候不该继续在材质里找

当出现以下任一条件时，你应该停止继续在材质图里硬查：

- `BaseColor / Normal / Roughness / Metallic`都正确
- 运行时绑定对象已经确认正确
- 编译错误已经清零
- `Stats`和`Shader Complexity`没有显示异常

这时优先切到：

- 光照系统
- 曝光与后处理
- 反射与Lumen
- 网格法线 / 切线 / UV
- 资源Streaming

### 💡 核心要点

最成熟的调试能力不是“总能在材质里找到答案”，而是**知道什么时候该退出材质层**。

| Apply后报错 | CompileErrors / Stats | HLSL Code / Search | 不要先抓帧 |
| 预览球对、场景错 | VisualizeBuffer | Unlit / Shader Complexity | 不要先重做整张材质图 |
| 参数不生效 | Actor材质槽 / MID检查 | 蓝图 / C++写值路径 | 不要先改母材质默认值 |
| 画面对但太贵 | Stats | `profilegpu` / Shader Complexity | 不要先无脑删效果 |
| Substrate层异常 | VisualizeSubstrate | HLSL / IR Dump | 不要只看BaseColor |

## A.3 高频错误速查

| 错误关键字 | 优先怀疑 |
|-----------|---------|
| `SceneDepth` | 材质域不对 |
| `SceneTexture` | Opaque / Domain约束 |
| `Gather does not support virtual textures` | VT采样方式不合法 |
| `custom vertex interpolators exceeded` | 插值器预算溢出 |
| `Substrate material errors encountered` | Substrate结构非法 |
| `Material has no BSDF` | Substrate主输出链断裂 |
| `Could not resolve referenced texture` | 纹理引用缺失或资源无效 |
