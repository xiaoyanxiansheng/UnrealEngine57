# 第一部分：理解BasePass的存在意义

## 1.1 从一个实际问题开始

假设你是一个渲染程序员，需要渲染一个包含10000个物体的场景。每个物体都有自己的材质（BaseColor、Normal、Roughness等参数）。

### 1.1.1 前向渲染的困境

**最直观的方法（前向渲染）**：

```
对于场景中的每个物体：
    对于每个影响该物体的光源：
        计算材质参数
        计算光照
        累加到最终颜色
```

**问题**：
```
假设：10000个物体，100个光源

每帧需要的Shader执行次数 = 10000 × 100 = 1,000,000次

如果多个光源重叠（Overdraw）：
- 同一像素被渲染多次
- 每次都要重新采样材质纹理
- GPU时间爆炸！
```

**前向渲染的致命缺陷**：
| 问题 | 影响 |
|------|------|
| **光源数量限制** | 每增加一个光源，渲染开销线性增加 |
| **Overdraw浪费** | 被遮挡的像素也执行了完整的光照计算 |
| **材质重复采样** | 同一像素的材质被多次采样 |

### 1.1.2 延迟渲染的解决方案

**核心思想**：把"计算材质"和"计算光照"分开！

```
【第一步：BasePass】
对于场景中的每个物体：
    计算材质参数（BaseColor、Normal、Roughness...）
    存储到GBuffer（多个RenderTarget）
    
【第二步：LightingPass】
对于屏幕上的每个像素：
    从GBuffer读取材质参数
    对于每个影响该像素的光源：
        计算光照贡献
    累加到最终颜色
```

**为什么更快？**

```
┌─────────────────────────────────────────────────────────────────┐
│ 前向渲染：每个物体 × 每个光源 = O(物体数 × 光源数)               │
│                                                                 │
│     物体1    物体2    物体3    ...    物体10000                 │
│       │        │        │              │                        │
│       ↓        ↓        ↓              ↓                        │
│    ┌──────────────────────────────────────────────┐             │
│    │ 光源1计算 + 光源2计算 + ... + 光源100计算     │ ← 每个物体│
│    └──────────────────────────────────────────────┘             │
│                                                                 │
│    总次数：10000 × 100 = 1,000,000 次Shader执行                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 延迟渲染：物体数 + 像素数 × 光源数 = O(物体数) + O(像素数 × 光源)│
│                                                                 │
│  【BasePass】                                                   │
│     物体1    物体2    物体3    ...    物体10000                 │
│       │        │        │              │                        │
│       ↓        ↓        ↓              ↓                        │
│    ┌──────────────────────────────────────────────┐             │
│    │        只计算材质参数，存入GBuffer            │ ← 1次/物体 │
│    └──────────────────────────────────────────────┘             │
│                                                                 │
│  【LightingPass】                                               │
│    ┌──────────────────────────────────────────────┐             │
│    │  全屏Quad + 每个光源                          │             │
│    │  从GBuffer读取材质 → 计算光照                 │             │
│    └──────────────────────────────────────────────┘             │
│                                                                 │
│    总次数：10000（材质）+ 1920×1080×光源影响比例                │
│    相比前向渲染，复杂场景快5-10倍！                              │
└─────────────────────────────────────────────────────────────────┘
```

### 1.1.3 BasePass的核心职责

**BasePass = 延迟渲染的第一阶段**

```
┌─────────────────────────────────────────────────────────────────┐
│                        BasePass职责                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  输入：                                                          │
│    • 场景中所有可见的Opaque/Masked物体                           │
│    • 每个物体的材质（Material）                                  │
│    • 视图变换矩阵（ViewProjection）                              │
│                                                                 │
│  处理：                                                          │
│    • 顶点变换（VS）                                              │
│    • 材质参数计算（PS）                                          │
│    • 写入GBuffer                                                │
│                                                                 │
│  输出：                                                          │
│    • GBufferA: WorldNormal（世界空间法线）                       │
│    • GBufferB: BaseColor + Metallic（基础色 + 金属度）           │
│    • GBufferC: Specular + Roughness（高光 + 粗糙度）             │
│    • GBufferD: CustomData（自定义数据，如Subsurface）            │
│    • GBufferE: PrecomputedShadow + AO                           │
│    • Depth: 场景深度                                             │
│                                                                 │
│  💡 注意：BasePass不计算光照！光照在后续的LightingPass中计算     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 1.2 第二个问题：如何高效管理DrawCall？

BasePass需要渲染场景中的每个可见物体。但直接渲染有性能问题。

### 1.2.1 朴素实现的问题

```cpp
// ❌ 朴素实现（类似Unity的传统做法）
void RenderBasePass()
{
    for (每个可见物体 Object)
    {
        // 【每帧都要做】
        Material* mat = Object->GetMaterial();
        Shader* shader = mat->GetShader();           // 1. 选择Shader
        PSO* pso = CreatePSO(shader, blendState...); // 2. 构建PSO
        
        BindPSO(pso);                                // 3. 绑定PSO
        BindTextures(mat->Textures);                 // 4. 绑定纹理
        BindVertexBuffer(Object->VB);                // 5. 绑定VB
        BindIndexBuffer(Object->IB);                 // 6. 绑定IB
        
        DrawIndexed(Object->IndexCount);             // 7. DrawCall
    }
}
```

**问题分析**：

| 操作 | 单次耗时 | 10000物体总耗时 | 问题 |
|------|---------|----------------|------|
| 选择Shader | ~0.01ms | 100ms | 每帧重复 |
| 构建PSO | ~0.02ms | 200ms | 每帧重复 |
| 绑定PSO | ~0.001ms | 10ms | - |
| 绑定纹理 | ~0.002ms | 20ms | - |
| DrawCall | ~0.001ms | 10ms | - |
| **总计** | - | **~340ms** | **远超16ms帧预算！** |

### 1.2.2 UE的解决方案：MeshDrawCommand缓存

**核心思想**：把"准备工作"和"实际渲染"分开！

```
┌─────────────────────────────────────────────────────────────────┐
│                    UE的两阶段设计                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【Setup阶段】（场景加载时/物体变化时 - 只执行一次）             │
│                                                                 │
│     for (每个物体 Object)                                       │
│     {                                                           │
│         选择Shader                                              │
│         构建PSO                                                 │
│         缓存为 MeshDrawCommand ← 保存！                         │
│     }                                                           │
│                                                                 │
│  【Render阶段】（每帧执行）                                      │
│                                                                 │
│     for (每个缓存的 MeshDrawCommand)                            │
│     {                                                           │
│         直接提交！← 无需重新计算                                │
│     }                                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**性能对比**：

| 方式 | Setup开销 | 每帧开销 | 适用场景 |
|------|----------|---------|---------|
| **朴素实现** | 0ms | 340ms | ❌ |
| **MeshDrawCommand缓存** | 340ms（一次性） | ~3ms | ✅ 静态场景 |

## 1.3 架构概览：BasePass在渲染管线中的位置

```
┌─────────────────────────────────────────────────────────────────┐
│                   FDeferredShadingSceneRenderer::Render()        │
│                          （主渲染函数）                           │
└──────────────────────┬──────────────────────────────────────────┘
                       │
          ┌────────────┴────────────┐
          ↓                         ↓
┌─────────────────────┐   ┌─────────────────────┐
│   RenderPrePass()   │   │  RenderBasePass()   │ ← 本章重点
│    (Depth Prepass)  │   │   (材质写入GBuffer)  │
└──────────┬──────────┘   └──────────┬──────────┘
           │                         │
           └────────────┬────────────┘
                        ↓
              ┌─────────────────────┐
              │  RenderLights()     │
              │  (光照计算)          │
              └──────────┬──────────┘
                         │
                         ↓
              ┌─────────────────────┐
              │  后处理、合成        │
              └─────────────────────┘
```

**BasePass与其他Pass的关系**：

| Pass | 输入 | 输出 | 与BasePass关系 |
|------|-----|------|---------------|
| **PrePass** | 场景几何 | Depth | BasePass读取（Early-Z优化） |
| **BasePass** | 场景几何+材质 | GBuffer | **本章主题** |
| **LightingPass** | GBuffer | 光照结果 | 读取BasePass输出 |
| **ReflectionPass** | GBuffer | 反射 | 读取BasePass输出 |

## 1.4 小结：为什么需要理解BasePass

| 问题 | BasePass的解决方案 |
|------|-------------------|
| 前向渲染的光源限制 | 延迟渲染：材质和光照分离 |
| DrawCall准备开销大 | MeshDrawCommand缓存 |
| 需要支持复杂材质 | GBuffer存储所有材质参数 |
| 需要支持大量物体 | MeshPassProcessor并行处理 |

**💡 核心要点**

BasePass是延迟渲染的第一阶段，职责是将场景中所有可见物体的材质参数写入GBuffer。UE通过MeshDrawCommand缓存机制，将DrawCall的"准备工作"和"实际渲染"分离，静态场景的CPU开销从340ms降低到3ms。

## 2.1 GBuffer：材质参数的存储格式

### 2.1.1 什么是GBuffer？

```
│  │   X    │   Y    │   Z    │ Data   │                          │
```

```
GBuffer内存 = 分辨率 × RT数量 × 字节数

示例（1920×1080，5个GBuffer）：
  GBufferA: 1920 × 1080 × 4 = 8.3MB
  GBufferB: 1920 × 1080 × 4 = 8.3MB
  GBufferC: 1920 × 1080 × 4 = 8.3MB
  GBufferD: 1920 × 1080 × 4 = 8.3MB
  GBufferE: 1920 × 1080 × 4 = 8.3MB
  Depth:    1920 × 1080 × 4 = 8.3MB
  
  总计：约50MB（1080p）
        约200MB（4K）
```

## 2.2 MeshDrawCommand：渲染命令的数据结构

### 2.2.1 为什么需要MeshDrawCommand？

回顾1.2节的问题：每帧重复计算Shader选择、PSO构建是浪费。

**MeshDrawCommand = 预计算好的渲染命令**：

```cpp
// MeshDrawCommand包含执行一次DrawCall需要的所有信息
struct FMeshDrawCommand
{
    // 【PSO相关】（最重要！）
    FGraphicsMinimalPipelineStateId CachedPipelineId;  // PSO ID
    
    // 【Shader参数】
    FMeshDrawShaderBindings ShaderBindings;  // Uniform Buffer、纹理绑定
    
    // 【几何数据】
    TArrayView<FVertexInputStream> VertexStreams;  // 顶点缓冲绑定
    FRHIIndexBuffer* IndexBuffer;                   // 索引缓冲
    uint32 FirstIndex;                              // 起始索引
    uint32 NumPrimitives;                           // 三角形数量
    uint32 NumInstances;                            // 实例数量
    
    // 【排序】
    FMeshDrawCommandSortKey SortKey;  // 用于优化渲染顺序
};
```

### 2.2.2 MeshDrawCommand的生命周期

```
┌─────────────────────────────────────────────────────────────────┐
│                  MeshDrawCommand生命周期                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【创建时机】                                                    │
│    • 场景加载时：静态Mesh加入场景                                │
│    • 物体变化时：材质变化、变换更新                              │
│    • 视图变化时：动态Mesh每帧重建                                │
│                                                                 │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2.3 MeshPassProcessor：DrawCommand的生成器

### 2.3.1 设计模式

**MeshPassProcessor采用"策略模式"**：每种Pass有自己的Processor，负责为该Pass生成MeshDrawCommand。

```
┌─────────────────────────────────────────────────────────────────┐
│                  MeshPassProcessor继承体系                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    ┌────────────────────┐                       │
│                    │ FMeshPassProcessor │ ← 抽象基类            │
│                    │  (定义接口)         │                       │
│                    └─────────┬──────────┘                       │
│                              │                                  │
│           ┌──────────────────┼──────────────────┐               │
│           │                  │                  │               │
│           ↓                  ↓                  ↓               │
│  ┌────────────────┐ ┌────────────────┐ ┌────────────────┐      │
│  │FBasePassMesh   │ │FDepthPassMesh  │ │FShadowDepth    │      │
│  │   Processor    │ │   Processor    │ │   Processor    │      │
│  │ (BasePass专用) │ │ (Prepass专用)  │ │ (阴影专用)     │      │
│  └────────────────┘ └────────────────┘ └────────────────┘      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3.2 核心接口：AddMeshBatch

```
我的记录：
材质维度：它是“材质分段（Section）”级别的
模型维度：它是“实例（Instance）”的集合
Drawcall 说： “用这个 Shader，绑定这个顶点缓冲区，画这 100 个三角形。” MeshDrawCommand (指令层 / 真正的 Drawcall) RHICmdList.DrawIndexedPrimitive(
FMeshBatch 说： “我是这组几何体，我要用这个材质，请帮我画出来。”
```

每个MeshPassProcessor必须实现`AddMeshBatch`：

```cpp
// 伪代码：展示核心逻辑
void FBasePassMeshProcessor::AddMeshBatch(
    const FMeshBatch& MeshBatch,        // 输入：Mesh信息
    const FPrimitiveSceneProxy* Proxy   // 输入：Primitive信息
)
{
    // 【步骤1】获取材质
    const FMaterial* Material = MeshBatch.MaterialRenderProxy->GetMaterial();
    
    // 【步骤2】过滤：只处理Opaque/Masked材质
    if (Material->GetBlendMode() != BLEND_Opaque && 
        Material->GetBlendMode() != BLEND_Masked)
    {
        return;  // 透明材质不走BasePass
    }
    
    // 【步骤3】获取Shader
    FMeshMaterialShader* VertexShader = Material->GetShader<FBasePassVS>();
    FMeshMaterialShader* PixelShader = Material->GetShader<FBasePassPS>();
    
    // 【步骤4】构建MeshDrawCommand
    BuildMeshDrawCommands(
        MeshBatch,
        Material,
        VertexShader,
        PixelShader,
        // ... 其他参数
    );
}
```

**关键设计：虚函数与多态**

这里使用了C++的虚函数机制。基类`FMeshPassProcessor`定义了`AddMeshBatch`为纯虚函数，每个子类（如`FBasePassMeshProcessor`）提供自己的实现。

```cpp
// 基类声明
class FMeshPassProcessor
{
public:
    // 纯虚函数 - 子类必须实现
    virtual void AddMeshBatch(...) = 0;
};
```

**虚函数开销分析**：
- 虚函数调用：~3ns（需要查找VTable）
- AddMeshBatch只在Setup阶段调用，不是每帧热路径
- 因此虚函数开销可以忽略

## 3.3 RenderBasePassView：提交MeshDrawCommand

### 3.3.1 函数概览

**来源**：`Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp`  
**源码锚点**：搜索 `RenderBasePassInternal`、`EMeshPass::BasePass`、`SubmitMeshDrawCommands`。

### 3.3.2 核心流程

```cpp
void RenderBasePassView(FRHICommandList& RHICmdList, FViewInfo& View)
{
    // 【步骤1】获取该View缓存的MeshDrawCommand列表
    //   这些Command在Setup阶段（场景加载时）已经构建好
    FMeshDrawCommandList& CommandList = View.MeshDrawCommands[EMeshPass::BasePass];
    
    // 【步骤2】按SortKey排序（优化PSO切换）
    //   排序规则：MaterialID → VertexFactory → Depth
    CommandList.Sort();
    
    // 【步骤3】遍历提交
    for (FMeshDrawCommand& Command : CommandList)
    {
        // 3.1 设置PSO（如果变化了）
        if (Command.CachedPipelineId != CurrentPSO)
        {
            RHICmdList.SetGraphicsPipelineState(Command.CachedPipelineId);
            CurrentPSO = Command.CachedPipelineId;
        }
        
        // 3.2 绑定Shader参数
        Command.ShaderBindings.SetOnCommandList(RHICmdList);
        
        // 3.3 绑定顶点/索引缓冲
        RHICmdList.SetStreamSource(0, Command.VertexStreams[0].VertexBuffer, 0);
        
        // 3.4 DrawCall
        RHICmdList.DrawIndexedPrimitive(
            Command.IndexBuffer,
            0,
            0,
            Command.NumVertices,
            Command.FirstIndex,
            Command.NumPrimitives,
            Command.NumInstances
        );
    }
}
```

### 3.3.3 性能分析

**排序的重要性**：

```
未排序：
  Command1: Material_A → SetPSO(A)
  Command2: Material_B → SetPSO(B)  ← PSO切换！
  Command3: Material_A → SetPSO(A)  ← PSO切换！
  Command4: Material_B → SetPSO(B)  ← PSO切换！
  
  PSO切换次数：4次
  
排序后：
  Command1: Material_A → SetPSO(A)
  Command3: Material_A → 不切换    ← 跳过！
  Command2: Material_B → SetPSO(B)
  Command4: Material_B → 不切换    ← 跳过！
  
  PSO切换次数：2次
  性能提升：50%
```

**每帧开销估算**（10000个MeshDrawCommand）：

| 操作 | 单次耗时 | 总耗时 |
|------|---------|-------|
| 排序 | - | ~0.5ms |
| PSO切换 | ~0.01ms | ~1ms（假设100次切换） |
| 参数绑定 | ~0.0005ms | ~0.5ms |
| DrawCall | ~0.0001ms | ~1ms |
| **总计** | - | **~3ms** |

**💡 核心要点**

RenderBasePassView的核心工作是遍历预缓存的MeshDrawCommand并提交给RHI。通过排序减少PSO切换，10000个物体的CPU开销控制在3ms左右。

```cpp
// Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp

void FBasePassMeshProcessor::AddMeshBatch(
    const FMeshBatch& MeshBatch,
    uint64 BatchElementMask,
    const FPrimitiveSceneProxy* PrimitiveSceneProxy,
    int32 StaticMeshId
)
{
    // 【步骤1】获取材质
    const FMaterialRenderProxy* MaterialRenderProxy = MeshBatch.MaterialRenderProxy;
    const FMaterial* Material = MaterialRenderProxy->GetMaterialNoFallback(FeatureLevel);
    
    if (Material == nullptr)
    {
        return;
    }
    
    // 【步骤2】判断混合模式
    //   BasePass只处理Opaque和Masked
    EBlendMode BlendMode = Material->GetBlendMode();
    if (BlendMode != BLEND_Opaque && BlendMode != BLEND_Masked)
    {
        return;
    }
    
    // 【步骤3】判断Shading Model
    //   不同的Shading Model可能需要不同的Shader变体
    EMaterialShadingModel ShadingModel = Material->GetShadingModel();
    
    // 【步骤4】获取Shader
    //   根据材质和VertexFactory获取编译好的Shader
    FMaterialShaderMap* ShaderMap = Material->GetRenderingThreadShaderMap();
    TShaderMapRef<FBasePassVS> VertexShader(ShaderMap);
    TShaderMapRef<FBasePassPS> PixelShader(ShaderMap);
    
    // 【步骤5】构建PSO初始化器
    FGraphicsPipelineStateInitializer PSOInit;
    PSOInit.BoundShaderState.VertexShaderRHI = VertexShader.GetVertexShader();
    PSOInit.BoundShaderState.PixelShaderRHI = PixelShader.GetPixelShader();
    PSOInit.BoundShaderState.VertexDeclarationRHI = MeshBatch.VertexFactory->GetDeclaration();
    
    // 设置混合状态（Opaque不混合）
    PSOInit.BlendState = TStaticBlendState<>::GetRHI();
    
    // 设置深度状态（读取Prepass深度，不写入）
    PSOInit.DepthStencilState = TStaticDepthStencilState<
        false,                    // 不写深度
        CF_DepthNearOrEqual       // 深度测试：<=
    >::GetRHI();
    
    // 【步骤6】调用基类方法构建MeshDrawCommand
    BuildMeshDrawCommands(
        MeshBatch,
        BatchElementMask,
        PrimitiveSceneProxy,
        *MaterialRenderProxy,
        *Material,
        PassDrawRenderState,
        VertexShader,
        PixelShader,
        PSOInit,
        // ...
    );
}
```

### 3.4.3 BuildMeshDrawCommands：模板方法

`BuildMeshDrawCommands`是基类`FMeshPassProcessor`提供的模板方法：

```cpp
// 模板方法允许不同类型的Shader作为参数
template<typename VertexShaderType, typename PixelShaderType>
void FMeshPassProcessor::BuildMeshDrawCommands(
    const FMeshBatch& MeshBatch,
    // ... 其他参数
    const VertexShaderType& VertexShader,
    const PixelShaderType& PixelShader,
    const FGraphicsPipelineStateInitializer& PSOInit
)
{
    // 【创建MeshDrawCommand】
    FMeshDrawCommand& Command = DrawListContext->AddCommand(
        MeshBatch,
        BatchElementMask
    );
    
    // 【设置PSO】
    Command.CachedPipelineId = GetGraphicsPipelineStateId(PSOInit);
    
    // 【设置Shader绑定】
    Command.ShaderBindings.Initialize(VertexShader, PixelShader);
    
    // 【设置几何数据】
    Command.VertexStreams = MeshBatch.VertexFactory->GetStreams();
    Command.IndexBuffer = MeshBatch.Elements[0].IndexBuffer;
    Command.FirstIndex = MeshBatch.Elements[0].FirstIndex;
    Command.NumPrimitives = MeshBatch.Elements[0].NumPrimitives;
    Command.NumInstances = MeshBatch.Elements[0].NumInstances;
    
    // 【计算排序键】
    Command.SortKey = CalculateSortKey(MeshBatch, Material, VertexShader);
}
```

**C++知识：模板函数**

这里使用了模板函数，允许不同类型的Shader作为参数。编译器会为每种Shader类型组合生成对应的代码，实现编译期多态，没有运行时开销。

## 3.5 Nanite BasePass特殊流程

### 3.5.1 为什么Nanite需要特殊处理？

Nanite使用软件光栅化，几何处理在Prepass的Compute Shader中完成。BasePass只需要执行材质Shading。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Nanite渲染流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【Prepass阶段 - 已完成】                                        │
│    1. Cluster剔除（GPU Driven）                                 │
│    2. 软件光栅化（Compute Shader）                              │
│    3. 输出VisibilityBuffer（记录每个像素属于哪个三角形）         │
│                                                                 │
│  【BasePass阶段 - 本章重点】                                     │
│    1. 读取VisibilityBuffer                                      │
│    2. 按MaterialID分类像素                                      │
│    3. 对每种Material执行Compute Shader Shading                  │
│    4. 输出GBuffer                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.5.2 Nanite::DrawBasePass核心流程

**来源**：`Engine/Source/Runtime/Renderer/Private/Nanite/NaniteShading.cpp`

```cpp
// 简化的流程
void Nanite::DrawBasePass(...)
{
    // 【步骤1】按Material分类像素
    ClassifyMaterials(GraphBuilder, VisibilityBuffer);
    
    // 【步骤2】对每种Material执行Shading
    //   关键：使用DispatchComputeShader，不是DrawCall！

    {
        // 获取该Material覆盖的像素列表

            MaterialId,
            Pixels,
            GBufferTargets
        );
    }
}
```

### 3.5.3 性能关键：预排序优化

**来源**：`Engine/Source/Runtime/Renderer/Private/Nanite/NaniteShading.cpp`，搜索 `GNaniteComputeMaterialsSort`、`DispatchComputeShaderBundle`。

```cpp
// Nanite使用预排序策略减少PSO切换
if (GNaniteComputeMaterialsSort != 0)
{
    Commands.Sort([](auto& A, auto& B)
    {
        // 【排序优先级】
        // 1. BoundTargetMask（UAV输出目标）
        // 2. ComputeShader（PSO）
        // 3. ShadingBin（材质实例）
        
        if (A.Pipeline->BoundTargetMask != B.Pipeline->BoundTargetMask)
            return A.Pipeline->BoundTargetMask < B.Pipeline->BoundTargetMask;
            
        if (A.Pipeline->ComputeShader != B.Pipeline->ComputeShader)
            return A.Pipeline->ComputeShader < B.Pipeline->ComputeShader;
            
        return A.ShadingBin < B.ShadingBin;
    });
}
```

| 策略 | PSO切换次数 | GPU状态切换 | 性能 |
|------|-----------|------------|------|
| 无排序 | 1000次 | 500次（50%缓存命中） | Baseline |
| 预排序 | 1000次 | 50次（95%缓存命中） | +30% |

```
│  PrePass：                                                      │
│    • 只写深度（无材质计算）                                      │
│    • 建立Early-Z                                                │
│                                                                 │
│  BasePass：                                                     │
│    • 只读深度（深度测试，不写入）                                │
│    • 利用Early-Z跳过被遮挡像素                                  │
│    • 专注材质计算                                               │
│                                                                 │
│  【优化效果】                                                    │
│    无PrePass：BasePass = 25ms（大量Overdraw）                   │
│    有PrePass：PrePass 3ms + BasePass 12ms = 15ms               │
│    节省：40%                                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
