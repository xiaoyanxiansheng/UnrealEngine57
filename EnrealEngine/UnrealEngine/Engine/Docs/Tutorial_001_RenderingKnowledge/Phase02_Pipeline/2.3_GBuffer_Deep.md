# 第一部分：理解GBuffer的存在意义

## 1.1 从一个实际问题开始

在 [2.2 BasePass](./2.2_BasePass_Deep.md) 中我们已经知道，延迟渲染把"计算材质"和"计算光照"分成了两个阶段。**但问题来了**：

> BasePass计算出的材质参数（法线、颜色、金属度、粗糙度……），**如何传递给后续的LightingPass？**

### 1.1.1 最朴素的想法

"直接传一个大结构体不就行了？"

```
BasePass输出 ──(FGBufferData结构体)──> LightingPass输入
```

**问题**：这在CPU世界很自然，但GPU不是这样工作的。

GPU的渲染是**逐像素并行**的。BasePass的Pixel Shader处理完一个像素后就结束了，它不能"把结构体传给下一个Pass"——因为下一个Pass是**独立的全屏DrawCall**，每个像素的Shader实例是独立的。

两个Pass之间唯一的通信方式是：**通过纹理（RenderTarget）**。

### 1.1.2 解决方案：把材质参数"画"到纹理上

```
┌─────────────────────────────────────────────────────────────────┐
│  BasePass（Pixel Shader）                                        │
│                                                                 │
│  对于屏幕上的每个像素：                                           │
│    1. 计算材质参数（Normal, BaseColor, Metallic, Roughness...）  │
│    2. 把这些参数编码到多个RenderTarget的RGBA通道中                │
│                                                                 │
│  输出：一组纹理，每个像素存储了该位置的材质参数                    │
│        这组纹理就是 GBuffer（Geometry Buffer）                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LightingPass（全屏Pixel Shader）                                │
│                                                                 │
│  对于屏幕上的每个像素：                                           │
│    1. 从GBuffer纹理中采样，解码出材质参数                         │
│    2. 用材质参数 + 光源信息 计算光照                              │
│    3. 输出最终颜色                                               │
└─────────────────────────────────────────────────────────────────┘
```

**GBuffer = 延迟渲染中Pass之间的"通信协议"。**

### 1.1.3 为什么不用一个RT就够了？

一个标准PBR材质需要的参数：

| 参数 | 分量数 | 精度要求 |
|------|--------|---------|
| WorldNormal | float3 | 中高（光照方向敏感） |
| BaseColor | float3 | 中（sRGB 8位足够） |
| Metallic | float1 | 低（8位足够） |
| Specular | float1 | 低（8位足够） |
| Roughness | float1 | 低（8位足够） |
| ShadingModelID | uint | 极低（4位） |
| AO | float1 | 低 |
| CustomData | float4 | 中（特殊材质） |
| PrecomputedShadow | float4 | 低 |
| **总计** | **~19个分量** | — |

一个R8G8B8A8纹理只有4个通道。**19个分量 ÷ 4通道 ≈ 5个RT**。这就是UE传统GBuffer使用5个RT的原因。

---

## 1.2 GBuffer设计的核心取舍

GBuffer不是"放得下就行"——每一个设计决策都是在精度、带宽、灵活性之间权衡。

### 1.2.1 带宽是最大的敌人

```
GBuffer带宽计算（1080p，60fps）：

写入带宽（BasePass）：
  1920 × 1080 × 4字节 × 5个RT × 60fps ≈ 2.5 GB/s

读取带宽（LightingPass）：
  1920 × 1080 × 4字节 × 5个RT × 60fps ≈ 2.5 GB/s

总计：约5 GB/s 仅用于GBuffer读写

RTX 4090显存带宽：1008 GB/s
GBuffer占用比例：约0.5%（1080p）

4K（3840×2160）下：
  带宽 × 4 ≈ 20 GB/s → 约2%
```

1080p下GBuffer带宽不是瓶颈，但4K下开始显著。这就是为什么UE不断优化GBuffer布局（Substrate用3个RT替代5个）。

### 1.2.2 精度 vs 带宽的取舍

| 存储格式 | 每像素字节 | 精度 | 用途 |
|---------|-----------|------|------|
| R8G8B8A8（UNORM） | 4 | 256级/通道 | 大多数GBuffer |
| R10G10B10A2 | 4 | 1024级/通道 | 法线（更高精度） |
| R16G16B16A16F | 8 | 半精度浮点 | Substrate TopLayer |
| R16G16F | 4 | 半精度浮点 | Velocity |

UE的策略：**能用8位就不用16位**。绝大多数PBR参数（Metallic、Roughness、Specular）在8位精度下肉眼无法察觉差异。

## 1.3 架构概览：GBuffer在渲染管线中的位置

```
┌─────────────────────────────────────────────────────────────────┐
│              FDeferredShadingSceneRenderer::Render()              │
└──────────────────────────┬──────────────────────────────────────┘
                           │
         ┌─────────────────┼─────────────────┐
         ↓                 ↓                 ↓
┌────────────────┐ ┌────────────────┐ ┌────────────────┐
│  PrePass       │ │  BasePass      │ │  LightingPass  │
│  (写Depth)     │ │  (写GBuffer)   │ │  (读GBuffer)   │
│                │ │                │ │                │
│  输出：         │ │  输出：         │ │  输入：         │
│  SceneDepth    │ │  GBufferA~E   │ │  GBufferA~E   │
│                │ │  Velocity     │ │  SceneDepth   │
│                │ │  SceneDepth   │ │  Lights       │
└────────────────┘ └────────────────┘ │                │
                                      │  输出：         │
                                      │  SceneColor   │
                                      └────────────────┘
                                               │
                                               ↓
                                      ┌────────────────┐
                                      │  PostProcess   │
                                      │  (TAA/Bloom等) │
                                      └────────────────┘

【GBuffer的角色】
  • 生产者：BasePass（写入5个RT）
  • 消费者：LightingPass、SSAO、SSR、Lumen等（读取）
  • 管理者：RenderGraph（分配/释放RT资源）
```

**💡 核心要点**

GBuffer是延迟渲染中BasePass和LightingPass之间的"通信纹理"。设计的核心矛盾是：存更多数据需要更多RT（带宽增加），用更少RT则需要更精巧的编码（复杂度增加）。UE选择了5个R8G8B8A8的RT作为基本方案，在带宽和灵活性之间取得平衡。

# 第二部分：背景知识

## 2.1 延迟渲染的完整数据流

在进入GBuffer布局之前，先建立完整的数据流认知：

```
【BasePass Pixel Shader 内部】

材质节点计算
    │
    ↓
FGBufferData GBuffer;  ← 结构体，存储所有材质参数
GBuffer.WorldNormal = normalize(TangentToWorld(NormalMap));
GBuffer.BaseColor = BaseColorTexture.Sample(...);
GBuffer.Metallic = MetallicValue;
GBuffer.Roughness = RoughnessValue;
GBuffer.ShadingModelID = SHADINGMODELID_DEFAULT_LIT;
    │
    ↓
EncodeGBuffer(GBuffer, OutA, OutB, OutC, OutD, OutE, OutVelocity);
    │                    ↓     ↓     ↓     ↓     ↓      ↓
    │                   RT0   RT1   RT2   RT3   RT4  Velocity
    │
    ↓
GPU硬件写入RenderTarget

─────────────── 帧内时间线 ───────────────

【LightingPass Pixel Shader 内部】

采样GBuffer纹理
    │
    ↓
FGBufferData GBuffer = DecodeGBufferData(A, B, C, D, E, ...);
    │
    ↓
float3 Lighting = 0;
for each Light:
    Lighting += CalculateBxDF(GBuffer, Light);
    │
    ↓
输出 SceneColor
```

## 2.2 编码背后的数学：为什么法线只用3个通道

### 2.2.1 法线编码：从float3到RGB

法线是单位向量，范围[-1, 1]。但纹理通道（R8G8B8A8_UNORM）只能存[0, 1]。

**UE5.7的编码**（当前默认方式）：

```hlsl
// 编码：[-1,1] → [0,1]
float3 EncodeNormal(float3 N)
{
    return N * 0.5 + 0.5;
}

// 解码：[0,1] → [-1,1]
float3 DecodeNormal(float3 N)
{
    return N * 2.0 - 1.0;
}
```

这是最简单的线性映射，每个分量用8位存储，精度为 1/128 ≈ 0.0078。

**精度分析**：

| 编码方式 | 存储 | 角度误差 | 指令数 | 备注 |
|---------|------|---------|-------|------|
| 线性映射（当前） | RGB 24位 | ~1.4° | 2 | 简单快速 |
| 八面体编码 | RG 16位 | ~0.06° | ~15 | 精度高，省1通道 |

UE5.7选择线性映射而非八面体编码，原因是：
1. 线性映射只要2条指令（MAD），性能极佳
2. 1.4°误差在1080p下肉眼不可见
3. GBufferA的B通道虽然"浪费"了，但整体RT数量没变（仍然是5个R8G8B8A8）

### 2.2.2 sRGB编码：让8位精度发挥最大价值

BaseColor存储在GBufferC，格式是 `R8G8B8A8_UNORM_SRGB`。

**为什么用sRGB？**

人眼对暗部细节更敏感。如果用线性编码，8位中大量精度浪费在人眼不敏感的亮部：

```
线性8位：暗部(0~0.01)只有2-3个级别 → 暗部色带明显
sRGB 8位：暗部(0~0.01)有30+个级别  → 暗部平滑

sRGB编码：存储值 = pow(线性值, 1/2.2)
sRGB解码：线性值 = pow(存储值, 2.2)

关键：GPU硬件内置sRGB转换，零额外开销！
```

**💡 核心要点**

GBuffer的编码策略不是随意选择的——法线用线性映射（性能优先），BaseColor用sRGB（感知精度优先），PBR参数直接存（精度足够）。每个选择都是精度、性能、带宽的三方权衡。

### 3.1.1 GBufferB.a 的位运算编码

GBufferB的Alpha通道是GBuffer中最精巧的设计——**一个字节存两个数据**。

**问题**：ShadingModelID（16种材质，需要4位）和SelectiveOutputMask（4个标志位）都需要存储，但只剩一个8位通道。

**解决方案**：用位运算把两个4位数据打包到一个字节中。

```hlsl
// 来源：DeferredShadingCommon.ush

// 【编码】将两个4位数据合并到一个字节
float EncodeShadingModelIdAndSelectiveOutputMask(
    uint ShadingModelId,        // 低4位：材质类型（0~15）
    uint SelectiveOutputMask)   // 高4位：输出标志
{
    // 【步骤1】位运算合并
    //   ShadingModelId & 0x0F → 保留低4位
    //   SelectiveOutputMask   → 已经在高4位（值为0, 16, 32, ...）
    uint Value = (ShadingModelId & SHADINGMODELID_MASK) | SelectiveOutputMask;

    // 【步骤2】归一化到[0,1]以存入UNORM纹理
    return (float)Value / (float)0xFF;
}

// 【解码】从一个字节分离出ShadingModelID
uint DecodeShadingModelId(float InPackedChannel)
{
    // 【步骤1】反归一化到0~255
    uint Value = (uint)round(InPackedChannel * (float)0xFF);

    // 【步骤2】提取低4位
    return Value & SHADINGMODELID_MASK;  // SHADINGMODELID_MASK = 0x0F
}

// 【解码】从一个字节分离出SelectiveOutputMask
uint DecodeSelectiveOutputMask(float InPackedChannel)
{
    uint Value = (uint)round(InPackedChannel * (float)0xFF);
    return Value & ~SHADINGMODELID_MASK;  // ~0x0F = 0xF0，保留高4位
}
```

**位运算可视化**：

```
示例：ShadingModelID=1(DefaultLit), SelectiveOutputMask=32(bOutputVelocity)

编码：
  ShadingModelId    = 0000 0001  (1)
  Mask              = 0x0F       (0000 1111)
  ShadingModelId & Mask          = 0000 0001
  SelectiveOutputMask            = 0010 0000  (32)
  合并(OR)                       = 0010 0001  (33)
  归一化                         = 33 / 255 = 0.129

解码ShadingModelID：
  反归一化 round(0.129 × 255) = 33 = 0010 0001
  & 0x0F                            = 0000 0001 = 1 ✅

解码SelectiveOutputMask：
  反归一化 round(0.129 × 255) = 33 = 0010 0001
  & 0xF0                            = 0010 0000 = 32 ✅
```

**关于C++位运算**：上面用到的 `&`（按位与）、`|`（按位或）、`~`（按位取反）是GPU上的单周期指令，性能开销可忽略。这种"一个通道存多个数据"的技巧在GPU编程中非常常见，本质是用计算换带宽。

### 3.1.2 GBufferD：多态的CustomData

GBufferD是唯一"含义不固定"的RT——它的RGBA四个通道代表什么，取决于当前像素的ShadingModelID。

这是一种**数据多态**：同一块内存，根据类型标识（ShadingModelID）承载不同含义。

```
ShadingModelID → CustomData含义

  1  DefaultLit      → (0, 0, 0, 0)         不使用
  2  Subsurface      → (SubsurfaceColor.rgb, Opacity)
  3  PreintSkin      → (SubsurfaceColor.rgb, ProfileIndex/255)
  4  ClearCoat       → (ClearCoat, ClearCoatRoughness, 0, 0)
  5  SubsurfaceProf  → (SubsurfaceColor.rgb, Opacity)
  6  TwoSidedFoliage → (SubsurfaceColor.rgb, Opacity)
  7  Hair            → (Scatter, Backlit, 0, 0)
  8  Cloth           → (FuzzColor.rgb, ClothAmount)
  9  Eye             → (IrisNormal.rg, IrisMask, IrisDistance)
  10 SingleLayerWater→ 不使用（独立Water Pass）
  11 ThinTranslucent → (TransmittanceColor.rgb, 0)
```

**设计思想**：与其为每种材质类型增加一个RT（带宽爆炸），不如共享一个RT，通过ShadingModelID在运行期选择解释方式。代价是LightingPass需要分支——但80%以上的像素是DefaultLit，分支预测命中率很高。

## 3.2 EncodeGBuffer源码解读

**来源**：`Engine/Shaders/Private/DeferredShadingCommon.ush`  
**函数**：`EncodeGBuffer()`

### 3.2.1 函数概览

```hlsl
void EncodeGBuffer(
    FGBufferData GBuffer,           // 输入：完整材质数据
    out float4 OutGBufferA,         // 输出：RT0 法线
    out float4 OutGBufferB,         // 输出：RT1 PBR参数
    out float4 OutGBufferC,         // 输出：RT2 BaseColor
    out float4 OutGBufferD,         // 输出：RT3 CustomData
    out float4 OutGBufferE,         // 输出：RT4 预计算阴影
    out float4 OutGBufferVelocity,  // 输出：运动矢量
    float QuantizationBias = 0      // TAA量化抖动（-0.5~0.5）
)
```

### 3.2.2 核心流程（四段式）

**编码分8步完成**，下面逐步解读：

```hlsl
// 【步骤1】平台分支 — Mobile vs Desktop
#if SHADING_PATH_MOBILE
    // Mobile路径：简化为2个RT，不支持CustomData/Velocity
    MobileEncodeGBuffer(GBuffer, OutGBufferA, OutGBufferB, OutGBufferC, OutGBufferD);
    OutGBufferE = 0;
    OutGBufferVelocity = 0;
    return;
#endif

// 以下全部是Desktop路径 ───────────────────

// 【步骤2】Unlit快速路径
if (GBuffer.ShadingModelID == SHADINGMODELID_UNLIT)
{
    // Unlit材质不参与延迟光照，只需要在GBufferB标记自己是Unlit
    OutGBufferA = 0;
    SetGBufferForUnlit(OutGBufferB);  // 写入ShadingModelID=0
    OutGBufferC = 0;
    OutGBufferD = 0;
    OutGBufferE = 0;
    // Unlit的颜色直接在BasePass输出到SceneColor，不走GBuffer
    return;
}
// → 带宽优化：Unlit像素只写1个RT（GBufferB），节省约80%写入带宽

// 【步骤3】编码GBufferA — 法线
OutGBufferA.rgb = EncodeNormal(GBuffer.WorldNormal);
// EncodeNormal实现：return N * 0.5 + 0.5;
// 输入[-1,1] → 输出[0,1]，线性映射

OutGBufferA.a = GBuffer.PerObjectGBufferData;
// Per-Object标志位，用于SSAO/SSS等屏幕空间效果加速

// 【步骤4】编码GBufferB — PBR参数 + 材质类型
OutGBufferB.r = GBuffer.Metallic;
OutGBufferB.g = GBuffer.Specular;
OutGBufferB.b = GBuffer.Roughness;
OutGBufferB.a = EncodeShadingModelIdAndSelectiveOutputMask(
    GBuffer.ShadingModelID,
    GBuffer.SelectiveOutputMask
);
// 位运算编码，详见3.1.1节

// 【步骤5】编码GBufferC — BaseColor + AO
OutGBufferC.rgb = EncodeBaseColor(GBuffer.BaseColor);
// EncodeBaseColor实现：直接返回BaseColor
// 实际的sRGB转换由RT格式（R8G8B8A8_UNORM_SRGB）在硬件层面完成

// Alpha通道根据编译配置存储不同数据：
#if GBUFFER_HAS_DIFFUSE_SAMPLE_OCCLUSION
    // SSGI模式：存储间接光采样遮蔽Bitmask
    OutGBufferC.a = float(GBuffer.DiffuseIndirectSampleOcclusion) / 255.0f;
#elif ALLOW_STATIC_LIGHTING
    // Lightmap模式：存储AO×间接光（对数编码压缩HDR范围）
    OutGBufferC.a = EncodeIndirectIrradiance(
        GBuffer.IndirectIrradiance * GBuffer.GBufferAO
    ) + QuantizationBias * (1.0 / 255.0);
    // QuantizationBias：TAA生成的抖动值，打破8位量化产生的色带
#else
    // 纯动态光照：直接存储AO
    OutGBufferC.a = GBuffer.GBufferAO;
#endif

// 【步骤6】编码GBufferD — CustomData
OutGBufferD = GBuffer.CustomData;
// 直接存储，含义由ShadingModelID决定（见3.1.2节）

// 【步骤7】编码GBufferE — 预计算阴影
OutGBufferE = GBuffer.PrecomputedShadowFactors;

// 【步骤8】编码Velocity — 运动矢量
#if WRITES_VELOCITY_TO_GBUFFER
    OutGBufferVelocity = GBuffer.Velocity;
    // 屏幕空间运动矢量，R16G16F格式
    // 用于TAA、运动模糊、TSR
#else
    OutGBufferVelocity = 0;
#endif
```

### 3.2.3 关键细节

**IndirectIrradiance的对数编码**：

Lightmap烘焙的间接光照度范围很大（0~数百），但只有8位存储空间。UE使用对数编码把HDR范围压缩到[0,1]：

```hlsl
// 编码（存储时）：
float EncodeIndirectIrradiance(float Irradiance)
{
    const float LogBlackPoint = 0.00390625;  // 2^(-8)
    return (log2(Irradiance + LogBlackPoint) + 8.0) / 16.0;
    // 范围[0, 256] → [0, 1]
}

// 解码（读取时）：
float DecodeIndirectIrradiance(float Encoded)
{
    return exp2(Encoded * 16.0 - 8.0) - 0.00390625;
}
```

**为什么用对数而不是线性？**

| 编码 | 暗部0.001的精度 | 亮部100的精度 | 总动态范围 |
|------|----------------|--------------|-----------|
| 线性 | 仅1级（0或1/255） | 足够 | [0, 1] |
| 对数 | 30+级 | 足够 | [0, 256] |

对数编码在暗部有更多精度级别——与人眼特性一致。

**关于constexpr/static const**：上面代码中的 `LogBlackPoint = 0.00390625` 是编译期常量。HLSL中使用 `static const` 声明，编译器会在编译期计算完结果直接内联，运行时零开销。这与C++端的 `constexpr` 作用相同——UE的C++端大量使用constexpr定义GBuffer相关常量（如`SHADINGMODELID_MASK = 0x0F`），避免运行期计算。

### 3.2.4 源码索引

|------|------|------|
| `EncodeGBuffer()` | DeferredShadingCommon.ush | 完整编码 |
| `EncodeNormal()` | DeferredShadingCommon.ush | 法线编码 |
| `EncodeBaseColor()` | DeferredShadingCommon.ush | BaseColor编码 |

**💡 核心要点**

## 3.3 DecodeGBufferData源码解读

**来源**：`Engine/Shaders/Private/DeferredShadingCommon.ush`  
**函数**：`DecodeGBufferData()`

### 3.3.1 函数概览

```hlsl
FGBufferData DecodeGBufferData(
    float4 InGBufferA,          // RT0采样结果
    float4 InGBufferB,          // RT1采样结果
    float4 InGBufferC,          // RT2采样结果（sRGB自动转线性）
    float4 InGBufferD,          // RT3采样结果
    float4 InGBufferE,          // RT4采样结果
    float4 InGBufferF,          // GBufferF（各向异性，可选）
    float4 InGBufferVelocity,   // Velocity采样结果
    float CustomNativeDepth,    // CustomDepth
    uint CustomStencil,         // CustomStencil
    float SceneDepth,           // 从DepthBuffer采样的深度
    bool bGetNormalizedNormal,  // 是否归一化法线
    bool bChecker               // Checkerboard模式（SSS优化）
)
```

### 3.3.2 核心流程

```hlsl
FGBufferData GBuffer = (FGBufferData)0;  // 零初始化

// 【步骤1】解码法线
GBuffer.WorldNormal = DecodeNormal(InGBufferA.xyz);
// DecodeNormal: return N * 2 - 1;

if (bGetNormalizedNormal)
{
    GBuffer.WorldNormal = normalize(GBuffer.WorldNormal);
    // 重新归一化：硬件双线性过滤会破坏法线长度
    // 开销：1条rsqrt + 3条mul ≈ 4周期
    //
    // 何时可以跳过？
    //   - SSAO等后处理：只关心方向，不要求精确长度
    //   - 性能敏感的移动端
}

GBuffer.PerObjectGBufferData = InGBufferA.a;

// 【步骤2】解码PBR参数
GBuffer.Metallic  = InGBufferB.r;
GBuffer.Specular  = InGBufferB.g;
GBuffer.Roughness = InGBufferB.b;

// 【步骤3】解码ShadingModelID和OutputMask（位运算反向）
GBuffer.ShadingModelID = DecodeShadingModelId(InGBufferB.a);
GBuffer.SelectiveOutputMask = DecodeSelectiveOutputMask(InGBufferB.a);

// 【步骤4】解码BaseColor
GBuffer.BaseColor = DecodeBaseColor(InGBufferC.rgb);
// GBufferC是sRGB纹理，GPU在采样时已自动将sRGB转为线性
// DecodeBaseColor直接返回采样值（已是线性空间）

// 【步骤5】解码AO / IndirectIrradiance
#if GBUFFER_HAS_DIFFUSE_SAMPLE_OCCLUSION
    GBuffer.DiffuseIndirectSampleOcclusion = 255 * InGBufferC.a;
    GBuffer.GBufferAO = saturate(
        1.0 - float(countbits(GBuffer.DiffuseIndirectSampleOcclusion))
              * rcp(float(INDIRECT_SAMPLE_COUNT))
    );
    GBuffer.IndirectIrradiance = 1;
#elif ALLOW_STATIC_LIGHTING
    GBuffer.GBufferAO = 1;
    GBuffer.IndirectIrradiance = DecodeIndirectIrradiance(InGBufferC.a);
#else
    GBuffer.GBufferAO = InGBufferC.a;
    GBuffer.IndirectIrradiance = 1;
#endif

// 【步骤6】解码CustomData（条件读取）
GBuffer.CustomData = HasCustomGBufferData(GBuffer.ShadingModelID)
    ? InGBufferD : 0;
// 优化：DefaultLit像素（约80%）跳过GBufferD读取
// HasCustomGBufferData返回true的材质：
//   Subsurface, PreintSkin, ClearCoat, SubsurfaceProfile,
//   TwoSidedFoliage, Hair, Cloth, Eye

// 【步骤7】解码PrecomputedShadowFactors
#if ALLOW_STATIC_LIGHTING
    GBuffer.PrecomputedShadowFactors =
        HasPrecShadowMask(GBuffer) ? InGBufferE
        : (HasZeroPrecShadowMask(GBuffer) ? 0 : 1);
#else
    GBuffer.PrecomputedShadowFactors = 1;
#endif

// 【步骤8】计算DiffuseColor和SpecularColor
GBuffer.StoredBaseColor = GBuffer.BaseColor;
GBuffer.StoredMetallic = GBuffer.Metallic;
GBuffer.StoredSpecular = GBuffer.Specular;

// Eye材质特殊处理
if (GBuffer.ShadingModelID == SHADINGMODELID_EYE)
{
    GBuffer.Metallic = 0;       // 眼睛永远非金属
    GBuffer.Specular = 0.25;    // 降低镜面（湿润但不过亮）
}

// SpecularColor（F0）计算
GBuffer.SpecularColor = ComputeF0(GBuffer.Specular, GBuffer.BaseColor, GBuffer.Metallic);
// ComputeF0实现：
//   float DielectricF0 = 0.08 * Specular;    // 电介质约0.04
//   return lerp(DielectricF0, BaseColor, Metallic);
// 物理含义：
//   非金属(Metallic=0)：F0 = 灰色(0.04) — 所有电介质约4%反射
//   金属(Metallic=1)  ：F0 = BaseColor  — 金属反射带颜色

// DiffuseColor计算（能量守恒）
GBuffer.DiffuseColor = GBuffer.BaseColor * (1 - GBuffer.Metallic);
// 物理含义：金属没有漫反射（光全部变成镜面反射）

// SubsurfaceProfile的Checkerboard优化
if (UseSubsurfaceProfile(GBuffer.ShadingModelID))
{
    AdjustBaseColorAndSpecularColorForSubsurfaceProfileLighting(
        GBuffer.BaseColor, GBuffer.SpecularColor, GBuffer.Specular, bChecker);
    // 奇偶帧交替渲染SSS，节省50%计算量
}

return GBuffer;
```

### 3.3.3 性能分析

| 步骤 | 指令数（估算） | 条件 | 备注 |
|------|-------------|------|------|
| 法线解码 | 3 | 必执行 | MAD + normalize |
| PBR参数 | 0 | 必执行 | 直接赋值 |
| ShadingModelID | 6 | 必执行 | round + AND |
| BaseColor | 0 | 必执行 | sRGB硬件转换 |
| AO/Irradiance | 5 | 分支 | 对数解码 |
| CustomData | 0~1 | 条件读取 | 80%像素跳过 |
| F0/Diffuse | 8 | 必执行 | lerp + mul |
| **总计** | **~22指令** | — | 对比纹理采样~400周期 |

**结论**：解码逻辑的ALU开销（~22指令）远小于5次纹理采样的开销（~2000周期），不是性能瓶颈。

**💡 核心要点**

DecodeGBufferData做了两件事：①还原材质参数（反向编码），②计算派生数据（DiffuseColor、SpecularColor）。条件跳过GBufferD读取是重要的带宽优化——80%的DefaultLit像素不需要读取这个RT。

## 3.4 FGBufferData：Shader端的材质数据结构

**来源**：`Engine/Shaders/Private/DeferredShadingCommon.ush`

```hlsl
struct FGBufferData
{
    // ===== 几何 =====
    float3 WorldNormal;
    float3 WorldTangent;           // 各向异性材质
    float  Anisotropy;

    // ===== PBR参数 =====
    float3 BaseColor;
    float  Metallic;
    float  Specular;
    float  Roughness;

    // ===== 派生数据（解码时计算） =====
    float3 DiffuseColor;           // BaseColor * (1 - Metallic)
    float3 SpecularColor;          // lerp(0.04, BaseColor, Metallic)

    // ===== 光照辅助 =====
    float  GBufferAO;
    float  IndirectIrradiance;

    // ===== 材质类型 =====
    uint   ShadingModelID;         // 0~15
    uint   SelectiveOutputMask;

    // ===== 特殊材质 =====
    float4 CustomData;             // 含义随ShadingModelID变化

    // ===== 预计算 =====
    float4 PrecomputedShadowFactors;

    // ===== 运动/深度 =====
    float4 Velocity;
    float  Depth;
    float  CustomDepth;
    uint   CustomStencil;

    // ===== 备份（SSS用） =====
    float3 StoredBaseColor;
    float  StoredMetallic;
    float  StoredSpecular;

    // ===== Per-Object =====
    float  PerObjectGBufferData;

    // ===== SSGI =====
    uint   DiffuseIndirectSampleOcclusion;
};
```

这个结构体是 **EncodeGBuffer的输入** 和 **DecodeGBufferData的输出**。BasePass的Pixel Shader填充它，LightingPass的Pixel Shader使用它。

## 3.5 CPU端：GBuffer资源分配

GBuffer的RT资源由RenderGraph管理。CPU端的关键流程：

```cpp
// 简化的GBuffer分配流程（RenderThread）

void FSceneTextures::InitializeViewFamily(FRDGBuilder& GraphBuilder, ...)
{
    // 【步骤1】根据配置确定GBuffer格式
    EPixelFormat GBufferAFormat = PF_R8G8B8A8;
    EPixelFormat GBufferCFormat = PF_R8G8B8A8;  // 会标记sRGB
    // ...

    // 【步骤2】通过RDG创建纹理（声明式，延迟分配）
    SceneTextures.GBufferA = GraphBuilder.CreateTexture(
        FRDGTextureDesc::Create2D(Extent, GBufferAFormat,
            FClearValueBinding::Black,
            TexCreate_RenderTargetable | TexCreate_ShaderResource),
        TEXT("GBufferA")
    );
    // 类似地创建GBufferB/C/D/E

    // 【步骤3】RenderGraph在执行阶段从池中分配实际GPU资源
    // 帧结束后资源归还池中复用
}
```

**与直接分配的区别**：RenderGraph的声明式创建不会立即分配GPU内存。它先收集所有Pass的资源需求，分析依赖关系后，再决定哪些资源可以复用。这能显著减少显存占用。

## 3.6 Substrate新架构简介

**启用条件**：`r.Substrate=1`（UE5.7可选）

Substrate是UE5.4+引入的新材质系统，核心改变是：**用动态Slab组合替代固定的ShadingModelID**。

```
传统GBuffer：
  GBufferB.a 的4位 → 16种ShadingModel → 固定
  ClearCoat+Subsurface？不支持！

Substrate：
  Material Slab可动态组合
  ClearCoat层 + Subsurface层 + 自定义层 → 自由叠加
```

**布局对比**：

| 维度 | 传统5RT | Substrate 3RT |
|------|--------|--------------|
| RT数量 | 5 | 3 |
| 内存(1080p) | 40 MB | 40 MB（2个float16 RT） |
| 法线精度 | 8位/通道 | 16位/通道（float16） |
| 材质灵活性 | 16种固定ID | 动态Slab组合 |
| Shader复杂度 | 简单switch | 复杂动态分支 |
| 向后兼容 | ✅ | ❌ 需重新烘焙Lightmap |

**何时使用Substrate？**

- ✅ 需要复杂材质组合（ClearCoat+Subsurface）
- ✅ 需要高精度法线（水面、玻璃）
- ❌ 旧项目迁移（成本高）
- ❌ 移动端（Shader变体增加）
- ❌ 简单项目（DefaultLit足够）

**💡 核心要点**

Substrate是传统GBuffer的演进方向。它用更少的RT实现更灵活的材质，但代价是Shader更复杂。对于大多数项目，传统5RT方案仍然是默认且稳定的选择。

## 4.4 常见问题诊断

### 问题1：法线显示异常（最常见）

**症状**：光照效果错误，物体看起来"平"或方向不对

**诊断**：
```bash
vis GBufferA    # 检查法线通道
```

**常见原因**：
1. 法线贴图未连接到材质的Normal输入 → GBufferA全是(0.5,0.5,1) = 朝上
2. 切线空间法线被错误当成世界空间 → 法线随视角旋转
3. FBX导入时Tangent计算错误 → 检查导入设置

### 问题2：Metallic/Roughness全是极端值

**症状**：物体全反射或完全无反射

**诊断**：
```bash
vis GBufferB    # 查看R(Metallic)和B(Roughness)通道
```

**常见原因**：
- 材质编辑器中Metallic/Roughness输入未连接（默认0或1）
- ORM贴图通道映射错误（Roughness通常在G通道）

### 问题3：BasePass > 5ms（性能问题）

**诊断步骤**：
1. `profilegpu` — 确认BasePass是瓶颈
2. `stat rhi` — 查看DrawCall数量（>10000需要优化）
3. `r.ShaderComplexity 1` — 可视化Shader复杂度（红色=昂贵）

**优化方向**：
- DrawCall过多 → 启用Nanite或实例化
- Shader过复杂 → 简化材质节点
- 带宽瓶颈 → 降低分辨率或减少RT

# 第五部分：与其他系统的关系

## 5.1 与BasePass的关系

```
BasePass ──(写入)──> GBuffer

BasePass是GBuffer的唯一生产者。
BasePass的Pixel Shader调用EncodeGBuffer()写入5个RT。
传统几何和Nanite共享同一套GBuffer。

渲染顺序：
  1. Nanite BasePass（Compute Shader写GBuffer）
  2. 传统BasePass（光栅化写GBuffer）
  3. 两者输出合并在同一套GBuffer中
```

## 5.2 与LightingPass的关系

```
GBuffer ──(读取)──> LightingPass

LightingPass是GBuffer的主要消费者。
LightingPass的全屏Pixel Shader调用DecodeGBufferData()读取材质参数，
然后根据ShadingModelID选择对应的BxDF函数计算光照。

关键：LightingPass中的switch(ShadingModelID)分支
  - DefaultLit(1)：标准PBR（GGX + Lambert）
  - Subsurface(2)：次表面散射
  - ClearCoat(4)：双层BRDF
  - Cloth(8)：布料BRDF（Charlie分布）
  - Eye(9)：眼球渲染（视差虹膜）
```

## 5.3 与其他消费者

GBuffer不只被LightingPass读取，还有：

| 系统 | 读取的GBuffer | 用途 |
|------|-------------|------|
| SSAO | GBufferA(Normal) + Depth | 屏幕空间环境光遮蔽 |
| SSR | GBufferA(Normal) + GBufferB(Roughness) + Depth | 屏幕空间反射 |
| Lumen | 全部 | 全局光照（Surface Cache） |
| TAA/TSR | Velocity + Depth | 时间超分辨率 |
| 自定义后处理 | SceneTexture节点访问 | 自定义效果 |

## 5.4 与RenderGraph的关系

```
RenderGraph管理GBuffer的生命周期：

  声明阶段：BasePass声明写入GBufferA~E
  声明阶段：LightingPass声明读取GBufferA~E
             ↓
  编译阶段：RenderGraph分析依赖，确定：
            - BasePass必须在LightingPass之前执行
            - GBuffer RT在BasePass开始时分配
            - GBuffer RT在最后一个读取者完成后可释放/复用
             ↓
  执行阶段：自动插入Resource Barrier
            GBufferA: RenderTarget → ShaderResource
```

RenderGraph自动处理了GBuffer的资源状态转换（Barrier），你不需要手动管理。
