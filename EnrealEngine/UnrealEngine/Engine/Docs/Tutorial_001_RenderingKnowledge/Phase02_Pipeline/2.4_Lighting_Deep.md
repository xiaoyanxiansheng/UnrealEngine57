## 🎯 Unity对比：光照架构差异

| 维度 | UE5.7（默认桌面路径） | Unity URP/HDRP | 差异说明 |
|------|-------|----------------|----------|
| **主要路径** | Deferred（延迟） | URP: Forward+ / HDRP: 可选Deferred | UE更适合大量动态光源场景 |
| **光源剔除** | Clustered 3D Grid | URP: Tiled（2D Grid） | UE的3D剔除多一个Z轴细分，更精确 |
| **光源数量上限** | Standard: ~1000 / MegaLights: 100万+ | URP: ~300 / HDRP: ~500 | UE支持更大规模场景 |
| **Shader调用** | 全屏Pass（1 DrawCall） | URP: 逐光源DrawCall | UE批量处理性能更好 |
| **阴影集成** | Virtual Shadow Maps（VSM） | Shadow Cascades | UE的VSM与Nanite深度集成 |

**关键差异**：UE的Clustered Deferred在Z轴上进一步细分，减少了无效光照计算。Unity URP的Forward+是2D Tiled剔除，对深度变化大的场景优化较弱。

## 第一部分：理解光照系统的存在意义（问题驱动）

> **核心问题**：为什么需要Clustered Deferred这么复杂的系统？能不能用简单的Forward渲染？

---

### 1.1 问题：Forward渲染为什么不够用？

#### 1.1.1 从需求出发：现代游戏的光照复杂度

**典型场景案例**：
```
【场景A】室内办公室（50×50×3米）
- 天花板主灯20个 + 台灯50个 + 窗户光5个 + 屏幕发光30个
- 总计：105个光源（75个需要实时计算）

【场景B】夜晚街道（200×50×10米）
- 路灯100个 + 橱窗80个 + 车灯20-40个 + 霓虹灯150个
- 总计：350-370个光源（全部实时计算）

问题：传统Forward渲染能否支撑？
```

#### 1.1.2 Forward渲染的核心瓶颈

Forward渲染的根本问题是**重复计算**——对每个物体的每个光源，都要重复读取材质参数、重复计算BRDF、重复计算阴影：

```
Forward渲染的复杂度模型：

For each object (N个):
    For each light (L个):
        读取材质参数()     ← 重复N×L次
        计算BRDF()         ← 重复N×L次
        计算阴影()         ← 重复N×L次
        累加光照()

总计算量 = O(N × L)

代入场景B数据：
- 物体数量 N = 500
- 光源数量 L = 350
- 总光照计算次数 = 500 × 350 = 175,000次
- DrawCall数量 ≈ 175,000（不可接受！）
```

**场景B实测性能**（⚠️ 估算值）：

| 渲染方案 | GPU耗时 | 帧率 | 主要瓶颈 |
|---------|---------|------|---------|
| Standard Forward | ~120ms | <10 FPS ❌ | DrawCall爆炸（175K） |
| Unity URP Forward+ | ~22ms | 45 FPS ⚠️ | Z轴无细分，远近光源混合 |

**结论**：Forward渲染的O(N×L)复杂度在大量光源场景下完全不可接受。

#### 1.1.3 Forward的三个根本性问题

```
问题1：Overdraw（过度绘制）
─────────────────────────────────
同一像素被多个物体覆盖时，前面的计算全部浪费
- 典型Overdraw率：2-4倍（50-75%计算浪费）
- Forward无法提前知道哪个物体最终可见

问题2：材质参数重复读取
─────────────────────────────────
每个光源都要从纹理中读取BaseColor/Normal/Roughness
- 350个光源 = 读取350次相同的纹理数据
- 纹理带宽成为严重瓶颈

问题3：DrawCall线性增长
─────────────────────────────────
每个物体每个光源一个Pass
- 500物体 × 350光源 = 理论175K个DrawCall
- 即使限制为8光源/Pass，仍需44个Pass
- CPU-GPU通信成为瓶颈
```

#### 1.1.4 为什么Unity URP选择Forward+而非Deferred？

Unity的技术选择有其合理性，核心在于**目标市场不同**：

```
Unity的选择逻辑：
├─ 目标市场：移动端占比高（50%+收入来自移动）
├─ 硬件限制：移动端带宽敏感（LPDDR5: 68 GB/s）
├─ 架构需求：跨平台统一代码（降低维护成本）
└─ 光源需求：<500个光源（中小规模场景）

结论：Forward+（2D Tiled剔除）是合理选择
✅ 跨平台统一
✅ 带宽友好
✅ MSAA+透明物体原生支持
❌ 光源上限较低
❌ Z轴剔除缺失

UE的选择逻辑：
├─ 目标市场：PC/主机为主（移动端单独Mobile Renderer）
├─ 硬件条件：带宽充足（GDDR6: 448-760 GB/s）
├─ 架构需求：支持大规模场景（1000+光源）
└─ 技术集成：Nanite/Lumen必须依赖GBuffer

结论：Clustered Deferred 是 UE 当前桌面/主机目标场景下的主流选择
✅ 支持1000+光源
✅ 与Nanite/Lumen深度集成
✅ Z轴精确剔除
❌ 需要GBuffer带宽
❌ 不原生支持MSAA和透明物体
```

**两者选择都合理，取决于目标市场与技术路线。**

### 1.2 设计演进：从Forward→Deferred→Clustered

理解四代光照技术的演进，是理解UE5 Clustered Deferred的关键。

#### 1.2.1 四代技术对比总览

| 代际 | 技术方案 | 复杂度 | 核心思想 | 场景B耗时(⚠️) |
|------|---------|--------|---------|-------------|
| 第一代 | Standard Forward | O(N×L) | 暴力循环 | ~120ms ❌ |
| 第二代 | Standard Deferred | O(N+L) | GBuffer解耦材质与光照 | ~36ms ⚠️ |
| 第三代a | Tiled Deferred（2D） | O(N+T×L_avg) | 2D屏幕分块+光源剔除 | ~26ms ⚠️ |
| 第三代b | **Clustered Deferred（3D）** | O(N+C×L_avg) | **3D空间分块+Z轴对数剔除** | **~17ms ✅** |
| 第四代 | MegaLights（BVH） | O(N+P×logL) | BVH加速+重要性采样 | ~12ms ✅ |

> N=物体数, L=光源数, T=Tile数, C=Cluster数, P=像素数, L_avg=每Cluster平均光源数

#### 1.2.2 第一代 → 第二代：Deferred渲染的核心突破

**Standard Deferred的核心思想**：把"材质写入"和"光照计算"分成两个独立Pass。

```
Forward渲染（耦合）                  Deferred渲染（解耦）
─────────────────                   ─────────────────────

For each 物体:                      【Pass 1: GBuffer Pass】
  For each 光源:                    For each 物体:
    读材质 ← 重复！                     写材质到GBuffer  ← 只写一次！
    算光照 ← 重复！
    算阴影 ← 重复！                 【Pass 2: Lighting Pass】
                                    For each 光源:
复杂度: O(N × L)                        读GBuffer  ← 已缓存
                                        算光照     ← 每像素只算一次
                                    
                                    复杂度: O(N + L)
```

**关键突破**：GBuffer充当了"材质参数缓存"。BaseColor、Normal、Roughness等材质参数只在GBuffer Pass写入一次，后续所有光源共享同一份数据。

**性能提升**：场景B（350光源）从120ms → 36ms（提升3.3倍）

**但仍有问题**：Lighting Pass中每个光源一个DrawCall（350个DrawCall），且每个光源全屏计算，大量像素不受该光源影响却仍参与计算。

#### 1.2.3 第二代 → 第三代a：Tiled剔除（Unity HDRP方案）

**核心思想**：把屏幕切分成2D Tile，每个Tile只计算影响它的光源。

```
屏幕2D Tile划分（16×16像素/Tile）
┌─┬─┬─┬─┬─┬─┐
│3│5│2│0│1│0│  ← 每个Tile内的光源数量
├─┼─┼─┼─┼─┼─┤
│4│8│6│2│0│0│  ← 远近光源全混在一起
├─┼─┼─┼─┼─┼─┤     （Z轴没有细分！）
│2│5│3│1│0│0│
└─┴─┴─┴─┴─┴─┘

优点：从全屏350光源 → 平均20光源/Tile
缺点：Z轴没有细分，远处和近处光源混在一起
     → 狭长走廊场景40%计算浪费在远处光源上
```

**性能**：场景B从36ms → 26ms（提升1.4倍），但Z轴缺失导致深度变化大的场景优化不足。

#### 1.2.4 第三代b：Clustered Deferred（UE5方案）

**核心突破**：在2D Tile基础上增加Z轴细分，形成3D Cluster。

```
从2D Tile到3D Cluster：
                                     Z轴（16层对数分布）
                                          ↑
2D Tiled（侧视图）           3D Clustered（侧视图）
┌──────────────┐             ┌──┬──┬───┬────┬─────┐
│              │             │  │  │   │    │     │  Slice 0-4（近）
│  远近光源    │             ├──┼──┼───┼────┼─────┤
│  全混在一起  │    →→→      │  │  │   │    │     │  Slice 5-10（中）
│              │             ├──┼──┼───┼────┼─────┤
│              │             │  │  │   │    │     │  Slice 11-15（远）
└──────────────┘             └──┴──┴───┴────┴─────┘
 平均20光源/Tile              平均5-8光源/Cluster
```

**Z轴对数分布**：
```
为什么用对数而非线性分布？

线性分布（16层，0-1000m）：
Slice 0:    0 - 62.5m    ← 近处太粗糙！一层包含大量物体
Slice 15: 937.5 - 1000m  ← 远处太浪费！几乎没有物体

对数分布（16层，0-1000m）：
Slice 0:   0.1 -  0.2m   ← 近处密集细分
Slice 1:   0.2 -  0.4m
Slice 2:   0.4 -  0.8m
Slice 3:   0.8 -  1.6m
  ...每层覆盖范围翻倍...
Slice 10:  51  - 102m
Slice 15: 819  - 1000m   ← 远处合并（反正物体少）

原因：透视投影下近处物体密集（前10米占50%屏幕），
     远处物体稀疏（后990米只占50%屏幕）
```

**参数配置**（1920×1080）：
| 参数 | 值 | 计算方式 |
|------|------|--------|
| TileSize | 16×16像素 | 配置：`r.LightCulling.TileSizeX/Y` |
| NumTilesX | 120 | ceil(1920/16) |
| NumTilesY | 68 | ceil(1080/16) |
| NumSlices | 16 | 配置：`r.LightCulling.NumZSlices` |
| **TotalClusters** | **130,560** | 120 × 68 × 16 |

**性能**：场景B从26ms → 17ms（提升1.5倍），Z轴剔除贡献了40%的计算节省。

#### 1.2.5 第四代：MegaLights（2024年，UE5.5+）

**核心突破**：支持百万光源级别。

传统Clustered Deferred的Build Light Grid阶段复杂度是O(Clusters × Lights)，超过1000光源后成为瓶颈。MegaLights用三项创新突破了这个限制：

```
1. BVH加速结构（空间划分树）
   ───────────────────────
   光源组织成BVH树，查询复杂度从O(Lights)降到O(log Lights)

2. 重要性采样
   ───────────────────────
   不计算所有光源，只采样最重要的8-16个
   蒙特卡洛积分误差<2%

3. GPU Ray Marching
   ───────────────────────
   遍历BVH查找光源，替代传统Light Grid读取

性能：100万光源 @ ~12ms @ RTX 4090（⚠️ 估算值）
状态：✅ UE5.5+官方支持
限制：⚠️ 需要RTX 3000+硬件，部分光源类型不完全支持
详见：[3.4 MegaLights新特性](../Phase03_UE5Features/3.4_MegaLights_Deep.md)
```

#### 1.2.6 演进总结

```
技术决策树（场景B：350光源 @ 1080p）

Forward            120ms  ■■■■■■■■■■■■■■■■■■■■■■■■  ❌ 不可用
                          │
Standard Deferred   36ms  ■■■■■■■                     ⚠️ 勉强（GBuffer解耦材质）
                          │
Tiled Deferred      26ms  ■■■■■                       ⚠️ 可用（2D剔除）
                          │
Clustered Deferred  17ms  ■■■                         ✅ 理想（+Z轴剔除）
                          │
MegaLights          12ms  ■■                          ✅ 最优（BVH+采样，百万光源）
```

### 1.3 UE5.7的技术选择：为什么是Clustered Deferred

UE5选择Clustered Deferred不仅是性能考量，更因为它是整个渲染架构的**核心枢纽**：

```
Clustered Deferred是整个管线的枢纽：

Nanite → 提供高精度深度     ─┐
                              ├─→ Build Light Grid（精确3D剔除）
GBuffer → 缓存材质参数      ─┘          │
                                         ↓
                              Clustered Lighting Pass
                                         │
                              ├─→ SceneColor（直接光照）
                              └─→ Lumen GI（间接光照输入）

VSM → 提供Shadow Mask Bits ─→ 阴影查询（位运算，不需采样）
```

| 需求驱动因素 | Forward方案 | Clustered Deferred方案 | 决策原因 |
|-----------|-----------|--------------------------|---------|
| 1000+光源需求 | 不支持（<100）❌ | 1000（Standard）/100万（MegaLights）✅ | AAA游戏城市场景需求 |
| Nanite集成 | 百万三角形×光源=天文DrawCall ❌ | GBuffer解耦，与三角形数无关 ✅ | 虚拟几何必需 |
| Lumen集成 | 需额外Material Pass ⚠️ | GBuffer天然复用（0开销）✅ | 动态GI必需 |
| PC带宽充足 | 无GBuffer开销 | ~116MB/帧占<1%带宽 ✅ | 硬件能力支撑 |

### 1.4 代价权衡：带宽、内存、复杂度分析

#### 1.4.1 GBuffer带宽开销

```
GBuffer带宽计算（1920×1080，每帧）：

GBufferA（Normal + ShadingModelID）：     ~16.6 MB
GBufferB（Metallic/Specular/Roughness）：   ~8.3 MB
GBufferC（BaseColor + AO）：               ~8.3 MB
SceneDepth（Depth + Stencil）：            ~8.3 MB
──────────────────────────────────────────────────
GBuffer写入总计：                          ~41.5 MB
Lighting Pass读取：                        ~41.5 MB
写入+读取总计：                            ~83 MB

对比GPU带宽（RTX 3080: 760 GB/s）：
83 MB ÷ 760,000 MB/s × 1000 = 0.11ms（占16.67ms帧预算的0.7%）

结论：GBuffer带宽开销在PC平台可忽略不计
```

**为什么移动端不行？**

```
移动端带宽限制：
LPDDR5: 68 GB/s（是PC的1/11）
83 MB ÷ 68,000 MB/s × 1000 = 1.22ms（占帧预算的7.3%）
加上功耗和发热问题 → 不适合移动端

解决方案：UE5有单独的Mobile Renderer（Forward路径）
```

#### 1.4.2 Light Grid内存开销

```
Light Grid数据结构：

┌─ CulledLightGridHeader ──────────────────────────┐
│ 每个Cluster一条记录：{NumLights, LightListOffset} │
│ 8 bytes × 130,560 = ~1.0 MB                      │
└───────────────────────────────────────────────────┘
              │
              │ LightListOffset指向↓
              ▼
┌─ CulledLightIndices ─────────────────────────────┐
│ 全局光源索引列表（平均5个光源/Cluster）            │
│ 4 bytes × 130,560 × 5 = ~2.5 MB                  │
└───────────────────────────────────────────────────┘

总内存：1.0 + 2.5 = ~3.5 MB

对比Unity URP（2D Tiled）：
- Tile数：8,160（无Z维度）× 平均20光源 = ~0.64 MB
- UE用5倍内存换取4倍计算量减少（20→5光源/区域）

结论：内存换性能 ✅（3.5 MB对于PC/主机完全可接受）
```

#### 1.4.3 权衡总结

| 维度 | 投入（代价） | 产出（收益） |
|-----|-----------|-----------|
| **GBuffer带宽** | ~83 MB/帧（占PC带宽<1%） | 材质参数只写一次，避免重复读取 |
| **Light Grid内存** | ~3.5 MB | 光源剔除精确，减少75%计算 |
| **代码复杂度** | ~4500行C++/HLSL | 支持1000+光源 @ 60 FPS |
| **总效果** | 适度资源投入 | **Forward 120ms → Clustered 17ms（7倍提升）** |

### 2.1 延迟渲染原理与Forward对比

#### 2.1.1 Forward vs Deferred的根本区别

```
Forward渲染：
──────────────────────────────────────────────────
                    Pass 1          Pass 2         Pass 3
Object A ──→ [Light1+Light2+...] 
Object B ──→ [Light1+Light2+...]   (每个物体一个Pass)
Object C ──→ [Light1+Light2+...]
...
问题：相同像素可能被Object A、B、C重复覆盖→ Overdraw

Deferred渲染：
──────────────────────────────────────────────────
                  GBuffer Pass              Lighting Pass
                  (写材质)                  (算光照)
Object A ──┐                          ┌──→ Light 1 ──┐
Object B ──┼──→ GBuffer ──────────────┼──→ Light 2 ──┼──→ SceneColor
Object C ──┘    (材质缓存)            └──→ Light 3 ──┘
...                                        ...
优点：每个像素的材质只写入一次，光照计算不受物体数量影响
```

**复杂度对比**：

| 维度 | Forward | Deferred |
|------|---------|----------|
| 材质读取次数 | N × L 次 | N + P 次（P=像素数） |
| 光照计算次数 | N × L 次 | P × L_avg 次 |
| Overdraw影响 | 致命（重复计算） | 无影响（深度测试自动处理） |
| DrawCall数 | N × L 个 | N + 1 个 |
| 透明物体 | ✅ 原生支持 | ❌ 需要额外Forward Pass |
| MSAA | ✅ 原生支持 | ❌ 需要替代方案（TAA） |

#### 2.1.2 Deferred渲染为什么解决了Overdraw？

这是Deferred最核心的优势，值得深入理解：

```
Forward的Overdraw问题：

像素(100,200)被3个物体覆盖（远到近：A、B、C）
─────────────────────────────────────────────────
第1轮：绘制A → 计算A的100个光源光照 → 写入FrameBuffer
第2轮：绘制B → 计算B的100个光源光照 → 覆盖FrameBuffer（A白算了！）
第3轮：绘制C → 计算C的100个光源光照 → 覆盖FrameBuffer（B也白算了！）

浪费：200次光照计算（2/3浪费）

Deferred的解决方式：

GBuffer Pass：
─────────────────────────────────────────────────
第1轮：绘制A → 写入GBuffer → 深度测试记录DepthA
第2轮：绘制B → 深度测试通过（B更近）→ 覆盖GBuffer → DepthB
第3轮：绘制C → 深度测试通过（C最近）→ 覆盖GBuffer → DepthC

GBuffer最终只保留最近的物体C的材质参数！

Lighting Pass：
─────────────────────────────────────────────────
读取GBuffer(100,200) → 只有C的参数 → 只计算C的光照 → 写入SceneColor

节省：200次光照计算（A、B完全跳过）
```

**关键洞察**：Deferred渲染的GBuffer Pass本质是一个"可见性测试缓存"——利用硬件深度测试自动保留最终可见的像素，后续Lighting Pass不再有任何Overdraw。

#### 2.1.3 Deferred渲染的局限性

Deferred不是完美方案，它有两个根本限制：

```
限制1：透明物体
─────────────────────────────────────────────────
GBuffer只能存一个表面的参数（通过深度测试选择最近的）
透明物体需要同时看到前后多个表面 → GBuffer无法存储多层
解决：UE5混合方案
  ├─ 不透明物体：Deferred路径
  └─ 透明物体：Forward路径（单独Pass）

限制2：MSAA不兼容
─────────────────────────────────────────────────
GBuffer是全屏纹理，不是几何体 → MSAA无法对GBuffer进行多重采样
解决：UE5使用TAA（时间域抗锯齿）替代MSAA
  ├─ 效果：TAA质量不亚于4xMSAA
  └─ 优势：无额外带宽开销
```

### 2.2 Clustered Deferred完整流程

理解完整的渲染流程是掌握光照系统的关键。以一个典型场景（100个光源，1920×1080）为例：

#### 2.2.1 三步流程总览

```
┌────────────────────────────────────────────────────────────────┐
│                    Clustered Deferred 完整流程                  │
│                                                                │
│  ┌──────────────┐    ┌──────────────┐    ┌─────────────────┐  │
│  │ Pass 1:      │    │ Pass 2:      │    │ Pass 3:         │  │
│  │ GBuffer Pass │───→│ Build Light  │───→│ Clustered       │  │
│  │              │    │ Grid         │    │ Lighting Pass   │  │
│  │ 写入材质参数 │    │ 3D空间光源   │    │ 全屏像素级      │  │
│  │              │    │ 剔除         │    │ 光照计算        │  │
│  └──────────────┘    └──────────────┘    └─────────────────┘  │
│    ~8ms               ~0.5ms(CPU)          ~6ms               │
│    (几何复杂度)        +~0.5ms(GPU)        (光源密度)          │
│                                                                │
│  输入：场景物体        输入：视锥体+光源    输入：GBuffer       │
│  输出：GBuffer         输出：Light Grid     +Light Grid        │
│       (4张纹理)              (3D纹理)      +ForwardLightData  │
│                                            输出：SceneColor    │
└────────────────────────────────────────────────────────────────┘
```

#### 2.2.2 步骤1：GBuffer Pass（写入材质参数）

```
时间：~8ms（取决于几何复杂度，与光源数量无关）

流程：
┌─────────────────────────────────────────┐
│ For each 可见物体:                       │
│   执行Vertex Shader（变换顶点到屏幕空间） │
│   执行Pixel Shader（写入材质参数到GBuffer）│
│   硬件深度测试（自动保留最近表面）        │
└─────────────────────────────────────────┘

输出GBuffer（4张纹理）：
┌──────────────────────────────────────────────────────────┐
│ GBufferA │ WorldNormal(xyz) + ShadingModelID(w)          │
│ GBufferB │ Metallic(r) + Specular(g) + Roughness(b)     │
│ GBufferC │ BaseColor(rgb) + AO(a)                        │
│ Depth    │ 线性深度(24bit) + Stencil(8bit)               │
└──────────────────────────────────────────────────────────┘

关键：深度测试自动处理Overdraw，每个像素只保留最近物体的材质
```

#### 2.2.3 步骤2：Build Light Grid（3D空间光源剔除）

```
时间：~1ms（CPU粗剔除 + GPU精剔除）

这是Clustered Deferred的核心创新——3D空间光源分配：

┌────────────────────────────────────────────────────────┐
│ CPU阶段：粗剔除（Frustum Culling）                      │
│ 100个光源 → 剔除视锥外 → 60个可见光源                   │
│                                                        │
│ GPU阶段：精剔除（Compute Shader并行）                   │
│ 对130,560个Cluster并行处理：                            │
│   For each Cluster:                                    │
│     构建Cluster的3D AABB包围盒                          │
│     For each 可见光源:                                  │
│       if 光源球体与Cluster AABB相交:                    │
│         将光源索引添加到该Cluster的光源列表              │
│                                                        │
│ 输出：Light Grid（每个Cluster的光源列表）               │
└────────────────────────────────────────────────────────┘

Light Grid内存结构：
┌──────────────────────────────────────────────────────┐
│ Header表（1.0 MB）                                    │
│ Cluster 0: {光源数:5, 偏移:0}                         │
│ Cluster 1: {光源数:0, 偏移:5}      ← 空Cluster(78%)  │
│ Cluster 2: {光源数:8, 偏移:5}                         │
│ ...                                                   │
│                                                       │
│ 索引表（2.5 MB）                                      │
│ [0-4]:    12, 45, 67, 89, 123      ← Cluster 0的光源 │
│ [5-12]:   23, 56, 78, 90, ...      ← Cluster 2的光源 │
│ ...                                                   │
└──────────────────────────────────────────────────────┘

关键：~78%的Cluster是空的（无光源影响），后续可以直接跳过
```

#### 2.2.4 步骤3：Clustered Lighting Pass（全屏像素级光照）

```
时间：~6ms（取决于光源密度和BRDF复杂度）
DrawCall：1个（全屏Quad）
并行度：2,073,600像素同时计算（1080p）

流程（每个像素独立执行）：
┌──────────────────────────────────────────────────────────────┐
│ 1. 计算3D Cluster Index                                      │
│    PixelXY → TileXY（÷16）                                   │
│    Depth → SliceZ（对数映射）                                 │
│    → 合成为线性Cluster索引                                    │
│                                                              │
│ 2. 读取Light Grid Header                                     │
│    → 获取该Cluster的光源数量和列表偏移                         │
│                                                              │
│ 3. 早期退出检查（关键优化！）                                  │
│    if 光源数=0 或 材质是Unlit → 直接返回黑色                   │
│    → ~80%的像素在此提前退出                                   │
│                                                              │
│ 4. 读取GBuffer获取材质参数                                    │
│    → BaseColor, Normal, Roughness, Metallic 等                │
│                                                              │
│ 5. 遍历Cluster的光源列表，累加光照                             │
│    For each light in Cluster:                                │
│      读取光源数据（位置、颜色、衰减等）                        │
│      计算BRDF（Cook-Torrance PBR）                            │
│      查询阴影（VSM Shadow Mask，位运算）                      │
│      累加到结果                                               │
│                                                              │
│ 6. 输出到SceneColor（Additive Blending累加）                  │
└──────────────────────────────────────────────────────────────┘
```

#### 2.2.5 性能分解总结

| Pass | GPU耗时(⚠️) | 占比 | 主要瓶颈 |
|------|--------|------|---------|
| GBuffer Pass | ~8ms | 57% | 几何复杂度、DrawCall |
| Build Light Grid | ~0.5ms | 4% | 光源数量 |
| Clustered Lighting | ~6ms | 43% | 光源密度、BRDF复杂度 |
| **总计** | **~14.5ms** | **100%** | **69 FPS ✅** |

### 2.3 Light Grid数据结构详解

#### 2.3.1 3D Grid划分原理

```
X维度：屏幕水平方向
Y维度：屏幕垂直方向     → XY构成2D Tile（与Unity Tiled相同）
Z维度：深度方向（对数分布） → Z是Clustered比Tiled多出的关键维度

┌──────────────── 屏幕空间 ────────────────┐
│                                          │
│   120 Tiles × 68 Tiles = 8,160 Tiles     │  ← 2D Tiled到此为止
│                                          │
│   × 16 Z-Slices = 130,560 Clusters       │  ← Clustered多了这步
│                                          │
└──────────────────────────────────────────┘
```

**Z轴对数分布的直观理解**：

```
想象一个100米长的走廊：

线性分布（每6.25米一层）：
├──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤──┤
  0  6  12 19 25 31 38 44 50 56 62 69 75 81 88 94 100
前10米只有1-2层 ← 近处精度不够！

对数分布：
├┤┤┤─┤──┤────┤────────┤────────────────┤──────────────────────┤
0 0.2 0.8 1.6 3.2  6.4    12.8       25.6              51.2        100
前10米有6-7层 ← 近处精度充足！

原理：log2分布让每一层覆盖前一层的2倍深度范围
     → 自然适配透视投影的特性
```

#### 2.3.2 Light Grid的两级存储

**为什么分Header和Indices两级？**

```
方案1：直接存储（浪费内存）
─────────────────────────────────────
每个Cluster固定分配MaxLights个槽位
130,560 Clusters × 64个槽位 × 4 bytes = 33.4 MB
问题：78%的Cluster是空的，浪费了78%的内存！

方案2：两级存储（UE5的选择）
─────────────────────────────────────
Header：每个Cluster只存{数量, 偏移}（8 bytes固定）
Indices：只有非空Cluster才占用索引空间

节省：33.4 MB → 3.5 MB（10.5倍！）

两级读取流程：
Cluster Index → Header[Index] → {Num: 5, Offset: 42}
                                     │
                                     └→ Indices[42..46] → 光源ID列表
```

这种设计的本质是**间接寻址**——用一次额外的内存读取，换取大量的内存节省。对GPU来说，StructuredBuffer的随机访问很快（2-3 cycles），这个权衡完全值得。

#### 2.3.3 Cluster Index计算公式

```
输入：像素坐标(PixelX, PixelY) + 深度Depth

步骤1：计算Tile坐标
  TileX = PixelX / TileSize       （整数除法）
  TileY = PixelY / TileSize

步骤2：计算深度Slice（对数映射）
  SliceZ = floor(log2(Depth × ScaleFactor + 1) × BiasFactor)
  SliceZ = clamp(SliceZ, 0, NumSlices - 1)

步骤3：3D → 1D线性化
  ClusterIndex = (SliceZ × NumTilesY + TileY) × NumTilesX + TileX

性能：O(1)常数时间，~10-15 GPU cycles
特点：纯数学计算，无分支，无内存访问，完美并行
```

### 3.1 CPU端：RenderLights → 光源分类 → 路径选择

#### 3.1.1 RenderLights做了什么？

RenderLights是光照阶段的总入口，它不直接计算光照，而是**分类和分发**：

```
RenderLights() 的职责：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  1. 接收输入                                                │

### 3.3 GPU端：ClusteredShadingPixelShader

**入口函数**：`ClusteredShadingPixelShader()`  
**文件**：`Engine/Shaders/Private/ClusteredDeferredShadingPixelShader.usf`

这是在GPU上每个像素执行的核心着色器。理解它的流程等于理解整个光照计算。

#### 3.3.1 执行流程

```
ClusteredShadingPixelShader 执行流程（每像素执行）：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

输入：SvPosition（像素坐标）、UVAndScreenPos（屏幕UV）
输出：OutColor（光照结果，累加到SceneColor）

┌─ 步骤1：准备基础数据 ────────────────────────── ~10 cycles ┐
│ • 从GBuffer读取ShadingModelID（材质类型）                    │
│ • 计算Dither噪声值（减少Band效应，用于软阴影）              │
│ • 获取VR眼睛索引（单屏为0）                                 │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤2：计算Cluster Index ──────────────────── ~15 cycles ┐
│ • 从SceneDepth纹理读取深度值                                │
│ • 屏幕XY ÷ 16 → TileXY                                    │
│ • 深度 → 对数映射 → SliceZ                                 │
│ • (SliceZ × NumTilesY + TileY) × NumTilesX + TileX         │
│   → ClusterIndex                                            │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤3：读取Light Grid Header ─────────────── ~3 cycles  ┐
│ • CulledLightGridHeader[ClusterIndex]                       │
│ → {NumLights, LightListOffset}                              │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤4：早期退出检查 ★ 关键优化 ──────────────────────── ┐
│ if (NumLights == 0 || 材质是Unlit)                          │
│     → 输出黑色，立即返回                                    │
│                                                             │
│ 效果：~78%空Cluster + ~5%Unlit像素 → ~80%像素提前退出       │
│ 性能提升：Lighting Pass从~10ms → ~6ms（40%提升）            │
└─────────────────────────────────────────────────────────────┘
              ↓ （只有20%像素继续执行）
┌─ 步骤5：重建世界空间位置 ──────────────────── ~10 cycles ┐
│ • 从ScreenPosition + Depth重建TranslatedWorldPosition      │
│ • 使用ScreenToTranslatedWorld矩阵                          │
│ • 计算Camera向量（View方向）                                │
│                                                             │
│ 为什么用TranslatedWorldPosition？                           │
│ → 避免浮点精度问题（相对于Camera的坐标，不是绝对世界坐标）  │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤6：读取完整GBuffer数据 ──────────────── ~20 cycles  ┐
│ • GetScreenSpaceData(ScreenUV)                              │
│ → BaseColor, Normal, Roughness, Metallic, AO, ...           │
│                                                             │
│ 为什么在步骤1只读了ShadingModelID，这里才读完整数据？       │
│ → 步骤1只需要ShadingModelID判断是否Unlit（用于早期退出）    │
│ → 完整GBuffer数据读取开销更大，应在确定需要时才读取          │
│ → 80%的像素在步骤4退出，不需要读完整GBuffer                  │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤7：遍历光源列表，累加光照 ──── ~50-200 cycles/光源 ┐
│ For i = 0 to NumLights-1:                                   │
│   读取光源索引: CulledLightIndices[Offset + i]              │
│   读取光源数据: ForwardLightData[LightIndex]                │
│   计算BRDF:     GetDynamicLighting(GBuffer, Light, ...)     │
│   查询阴影:     ShadowMaskBits位运算                        │
│   累加结果:     CompositedLighting += LightColor             │
│                                                             │
│ 平均5-8个光源/Cluster → 250-1600 cycles                    │
└─────────────────────────────────────────────────────────────┘
              ↓
┌─ 步骤8：输出结果 ────────────────────────────────────────┐
│ OutColor = CompositedLighting                               │
│ Blending模式：Additive（加法累加到SceneColor）              │
└─────────────────────────────────────────────────────────────┘
```

#### 3.3.2 早期退出：为什么能节省这么多？

早期退出（步骤4）是整个Lighting Pass性能的关键。理解为什么78%的Cluster是空的：

```
为什么大部分Cluster没有光源？

1. 光源的Attenuation Radius有限
   ─────────────────────────────────────
   一个半径5米的点光源只影响方圆5米内的Cluster
   走廊场景200米长 → 130,560个Cluster中大部分不在任何光源范围内

2. 场景光源分布不均
   ─────────────────────────────────────
   光源集中在室内/建筑区域
   天空/地面/远景的Cluster几乎全是空的

3. Frustum Culling的效果
   ─────────────────────────────────────
   CPU粗剔除已移除视锥外光源
   进一步减少了有效光源覆盖的Cluster范围

实际分布（典型场景）：
┌────────────────────────────────────────┐
│ 空Cluster（0光源）：  ~78%   → 直接跳过│
│ 稀疏Cluster（1-5）：  ~15%   → 快速处理│
│ 密集Cluster（6-20）：  ~6%   → 正常处理│
│ 过密Cluster（20+）：  ~1%   → 性能热点│
└────────────────────────────────────────┘
```

#### 3.3.3 ShadingModel分支消除

Pixel Shader中有一个设计值得理解——如何处理不同ShadingModel的光照计算：

```
问题：不同材质需要不同的BRDF计算
─────────────────────────────────────
DefaultLit：      Cook-Torrance PBR（标准）
Subsurface：      加上次表面散射项
ClearCoat：       双层反射（底层+清漆层）
Hair (Marschner)：各向异性散射
Eye：             角膜折射+瞳孔

如果运行时if-else判断 → GPU Wavefront Divergence
（同一Warp的32个像素走不同分支 → 串行执行 → 性能灾难）

UE5的解决方案：宏展开 + Stencil分离
─────────────────────────────────────
GET_LIGHT_GRID_LOCAL_LIGHTING_SINGLE_SM(SHADINGMODELID_DEFAULT_LIT, ...)
GET_LIGHT_GRID_LOCAL_LIGHTING_SINGLE_SM(SHADINGMODELID_SUBSURFACE, ...)
GET_LIGHT_GRID_LOCAL_LIGHTING_SINGLE_SM(SHADINGMODELID_HAIR, ...)

配合Substrate Stencil：
- Simple材质（80%像素）→ 走优化Shader路径
- Complex材质（20%像素）→ 走完整Shader路径
- 同一Warp内的像素大概率是相同ShadingModel → Divergence极少

对比Unity：
shader内 if(shadingModel == HAIR) → 运行时判断 → Divergence
UE5编译期消除分支 + 运行时Stencil分离 → 几乎零Divergence
```

### 3.4 核心算法：ComputeLightGridCellIndex

**文件**：`Engine/Shaders/Private/LightGridCommon.ush`

这是将屏幕像素映射到3D Cluster的核心算法，虽然只有几行逻辑，但设计精巧。

```
ComputeLightGridCellIndex 算法：

输入：PixelPos(uint2), SceneDepth(float), EyeIndex(uint)
输出：ClusterIndex(uint)

┌──────────────────────────────────────────────────────────┐
│ 步骤1：XY → Tile坐标                                     │
│   TileX = PixelX / 16  （整数除法，16=TILE_SIZE）        │
│   TileY = PixelY / 16                                    │
│                                                          │
│ 步骤2：Depth → Z Slice（对数映射）                       │
│   ZSlice = max(0, log2(Depth × Scale + 1) × Bias)        │
│   SliceIndex = min(ZSlice, NumSlices - 1)                │
│                                                          │
│ 步骤3：3D → 1D（行优先线性化）                           │
│   Index = (SliceZ × NumTilesY + TileY) × NumTilesX + X  │
│   Index += EyeIndex × TotalClusters  （VR偏移）          │
│                                                          │
│ 返回：Index                                              │
└──────────────────────────────────────────────────────────┘

性能特性：
├─ 时间复杂度：O(1)（常数时间）
├─ 空间复杂度：O(1)（无临时数组）
├─ GPU开销：~10-15 cycles（纯ALU计算）
├─ 分支：0个
├─ 内存访问：0次（只读Uniform Buffer常量）
└─ 并行性：完美（每像素独立计算，2M+像素同时执行）
```

**为什么这个算法如此之快？** 它只包含除法、对数、乘法、加法——全是ALU操作，不访问任何纹理或Buffer（只读Uniform常量），没有分支判断，每个像素完全独立。这是GPU最擅长的计算模式。

### 3.5 光照累加：GetLightGridLocalLighting

**文件**：`Engine/Shaders/Private/ClusteredDeferredShadingPixelShader.usf`

#### 3.5.1 光照累加循环

```
光照累加的核心逻辑（每像素执行）：

AccumulatedLighting = 0

For i = 0 to NumLights-1:
  ┌───────────────────────────────────────────────────────┐
  │ 1. 读取光源索引                                       │
  │    LightIndex = CulledLightIndices[Offset + i]        │
  │    （~2-3 cycles，Buffer读取）                         │
  │                                                       │
  │ 2. 读取光源完整数据                                    │
  │    LightData = ForwardLightData[LightIndex]            │
  │    包含：位置、颜色、方向、衰减、角度等               │
  │    （~5-10 cycles，256 bytes）                         │
  │                                                       │
  │ 3. 距离衰减计算                                        │
  │    计算像素到光源的距离                                │
  │    应用衰减函数（Inverse Square + Radius Falloff）     │
  │    如果衰减≈0 → 跳过此光源（二次剔除）                │
  │                                                       │
  │ 4. BRDF计算（Cook-Torrance PBR）                       │
  │    Diffuse = Lambert(BaseColor, Normal, LightDir)      │
  │    Specular = CookTorrance(Normal, View, Light,        │
  │               Roughness, Metallic)                     │
  │    （~30-80 cycles，取决于ShadingModel复杂度）         │
  │                                                       │
  │ 5. 阴影查询                                            │
  │    ShadowFactor = ShadowMaskBits & (1 << LightChannel)│
  │    位运算！不是纹理采样！（~1 cycle）                  │
  │    对比传统：SampleShadowMap() → ~20-50 cycles         │
  │                                                       │
  │ 6. 累加                                                │
  │    AccumulatedLighting += BRDF × LightColor × Shadow   │
  └───────────────────────────────────────────────────────┘

return AccumulatedLighting
```

#### 3.5.2 BRDF计算（理解Cook-Torrance即可）

不需要记住所有公式，理解物理含义：

```
Cook-Torrance BRDF = Diffuse + Specular

Diffuse项（漫反射）：
────────────────────
光线打到粗糙表面后向各方向散射
公式核心：BaseColor × max(0, dot(Normal, LightDir))
物理含义：法线朝向光源的程度 × 材质颜色

Specular项（镜面反射）：
────────────────────
光线打到光滑表面后定向反射
公式核心：D × G × F / (4 × NdotL × NdotV)
├─ D（Normal Distribution）：微表面法线分布 → 由Roughness控制
│  Roughness低 → D值集中 → 高光亮且小
│  Roughness高 → D值分散 → 高光暗且大
├─ G（Geometry/Shadow-Masking）：微表面自遮挡
│  掠射角时G值低 → 高光边缘变暗
└─ F（Fresnel）：菲涅尔效应
   掠射角时F值高 → 边缘反射强（水面边缘更亮）
   由Metallic控制基础反射率

UE5的ShadingModel系统：
────────────────────
DefaultLit：标准Cook-Torrance（最常用）
Subsurface：+ 次表面散射项（皮肤、蜡烛）
ClearCoat：双层反射（汽车漆面）
Hair：Marschner模型（各向异性散射）
Eye：特殊的角膜+瞳孔模型
```

#### 3.5.3 VSM Shadow Mask的位运算优化

UE5的阴影查询是一个显著的性能优化点：

```
传统阴影查询（Unity CSM方式）：
─────────────────────────────────────
每个光源：采样Shadow Map纹理 → ~20-50 cycles
350个光源 = 7,000-17,500 cycles/像素 ← 巨大开销！

UE5 VSM Shadow Mask方式：
─────────────────────────────────────
预计算阶段：VSM将每个光源的阴影结果压缩为1 bit
存储：uint ShadowMaskBits（32 bit → 支持32个阴影光源）

查询：ShadowFactor = (ShadowMaskBits >> LightChannel) & 1
→ 1 cycle！（位运算）

对比：
├─ 传统：每光源20-50 cycles（纹理采样）
├─ VSM：每光源1 cycle（位运算）
└─ 提升：20-50倍！

限制：只有32个光源槽位（更多光源需要fallback到采样模式）
但实际场景大部分Cluster内光源数<32，所以绝大多数走位运算路径
```

### 完整调用链总结

```
CPU (GameThread → RenderThread)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
FDeferredShadingSceneRenderer::Render()
  └─→ RenderLights()
       ├─→ Substrate::AddSubstrateStencilPass()  [Stencil标记]
       ├─→ AddClusteredDeferredShadingPass()      [注册RDG Pass]
       │    ├─ 创建Shader Permutation
       │    ├─ 绑定资源（GBuffer + LightGrid + VSM）
       │    └─ GraphBuilder.AddPass()
       └─→ RenderLight()  [逐光源：Unbatched]

GPU (RenderGraph执行)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ClusteredShadingPixelShader()  [全屏Quad，每像素执行]
  ├─→ 准备数据（ShadingModelID, Dither）
  ├─→ ComputeLightGridCellIndex()          [3D→1D索引]
  ├─→ GetCulledLightsGridHeader()          [读取光源列表]
  ├─→ 早期退出检查（80%像素在此返回）
  ├─→ 重建世界位置 + 读取完整GBuffer
  └─→ GetLightGridLocalLighting()          [光照累加循环]
       └─→ For each light in Cluster:
            ├─ ForwardLightData[LightIndex]  [读光源]
            ├─ GetDynamicLighting()           [BRDF计算]
            ├─ ShadowMaskBits位运算           [阴影查询]
            └─ 累加到CompositedLighting
```

## 💡 第三部分核心要点

### 5秒版
RenderLights()是总入口，将光源分为4类（Simple/Batched/Unbatched/MegaLights）走不同优化路径。Batched光源进入ClusteredDeferredShadingPass，注册为一个全屏RDG Pass。GPU端ClusteredShadingPixelShader每像素执行：计算Cluster Index → 读Light Grid → 80%空Cluster早期退出 → 剩余像素遍历光源列表累加BRDF结果。VSM Shadow Mask用位运算替代纹理采样，阴影查询快20-50倍。

### 实践建议

**阅读源码顺序**：
1. `RenderLights()` — 理解路径选择逻辑
2. `AddClusteredDeferredShadingPass()` — 理解资源绑定和Pass配置
3. `ClusteredShadingPixelShader()` — 理解GPU执行流程

**避免过度优化**：
- 不要修改Light Grid索引算法（已极致优化，10-15 cycles）
- 不要删除Empty Cluster检查（节省40%性能）
- 不要在Shader中添加运行时分支（破坏Permutation优化）

### 5.1 与GBuffer的集成

```
GBuffer对光照系统的意义：

Forward时代：
  物体A × Light1 → 读材质A + 算光照
  物体A × Light2 → 再读材质A + 算光照  ← 重复读取！
  物体A × Light3 → 又读材质A + 算光照  ← 重复读取！

Deferred时代（GBuffer作为中间缓存）：
  GBuffer Pass：物体A → 写入材质A到GBuffer（一次性）
  Lighting Pass：
    读取GBuffer获取材质A  ← 只读一次
    × Light1 → 算光照
    × Light2 → 算光照（复用同一份材质参数）
    × Light3 → 算光照

关键点：
├─ GBuffer读取只发生一次/像素，不随光源数增加
├─ Texture Cache命中率95%+（连续内存访问）
└─ 光源数从1个到1000个，GBuffer读取开销不变
```

**GBuffer布局对光照的影响**：

| GBuffer通道 | 内容 | 光照计算用途 |
|------------|------|------------|
| GBufferA.xyz | WorldNormal | BRDF中的N向量 |
| GBufferA.w | ShadingModelID | 选择光照计算路径 |
| GBufferB.r | Metallic | Fresnel基础反射率 |
| GBufferB.g | Specular | 非金属的反射率缩放 |
| GBufferB.b | Roughness | 微表面粗糙度（D项） |
| GBufferC.rgb | BaseColor | 漫反射颜色 |
| GBufferC.a | AO | 环境光遮蔽 |
| SceneDepth | 深度 | 重建世界位置 + 计算ClusterIndex |

### 5.2 与阴影系统的集成（VSM）

```
传统阴影查询 vs UE5 VSM Shadow Mask：

传统（Unity CSM方式）：
─────────────────────────────────────
Lighting Pass中每个光源都要采样Shadow Map纹理
├─ 开销：每光源 ~20-50 cycles（纹理采样+PCF过滤）
├─ 350个光源 = 7,000-17,500 cycles/像素
└─ 占Lighting Pass总时间的40-60%

UE5 VSM Shadow Mask：
─────────────────────────────────────
VSM Pass（在Lighting Pass之前）：
  预计算每个像素的阴影结果 → 压缩为1 bit/光源
  存储为uint ShadowMaskBits（32个光源用32 bit）

Lighting Pass中查询：
  Shadow = (ShadowMaskBits >> LightChannel) & 1
├─ 开销：每光源 1 cycle（位运算）
├─ 350个光源 = 350 cycles/像素
└─ 占Lighting Pass总时间的<5%

性能对比：
├─ 传统：7,000-17,500 cycles（阴影查询占主导）
├─ VSM Mask：350 cycles（阴影查询可忽略）
└─ 提升：20-50倍阴影查询加速

代价：
├─ VSM Pass本身有开销（~2-4ms）
├─ 但VSM与Nanite集成后可共享深度数据
└─ 整体净收益显著
```

### 5.3 与Nanite的协同

```
Nanite对光照系统的两个关键贡献：

1. 高精度深度 → 精确的Light Grid
─────────────────────────────────────
Nanite的GPU驱动光栅化产生像素级精确的深度
→ Build Light Grid阶段的3D AABB构建更精确
→ 光源剔除更精确（减少False Positive）
→ 比传统Depth Pre-Pass更高精度

2. GBuffer直出 → 无额外Geometry Pass
─────────────────────────────────────
Nanite在渲染几何体时直接输出GBuffer
→ Lighting Pass可以直接使用Nanite写的GBuffer
→ 无需额外的材质Pass
→ 百万三角形的场景，GBuffer写入开销与三角形数无关
   （Nanite的GPU剔除已保证只写可见像素）

集成收益：
├─ 避免传统Depth Pre-Pass（~1-2ms）
├─ GBuffer精度更高（Nanite的亚像素精度）
└─ 光源剔除更准确（减少无效光照计算~5-10%）
```

### 5.4 与Lumen的协同

```
Lumen（动态全局光照）与Clustered Lighting的关系：

渲染顺序：
  GBuffer Pass → Build Light Grid → Clustered Lighting → Lumen GI
                                          │
                                          └─→ SceneColor（直接光照）
                                                    │
                                          Lumen读取直接光照结果
                                          + GBuffer + Light Grid
                                                    │
                                                    ↓
                                          IndirectLighting（间接光照）

Lumen复用Clustered的数据：
├─ 复用1：Light Grid中的光源信息
│  Lumen的Surface Cache需要知道光源位置和参数
│  → 直接复用ForwardLightData Buffer（零额外开销）
│
├─ 复用2：GBuffer中的材质参数
│  Lumen需要Surface反照率计算间接光反弹
│  → 直接读取GBufferC的BaseColor（零额外开销）
│
└─ 复用3：SceneColor中的直接光照
   Lumen的Screen Probe需要直接光照作为初始采样
   → 直接读取Clustered Lighting的输出

如果没有这种集成：
├─ Lumen需要单独的Light Info Pass：+3-5ms
├─ Lumen需要单独的Material Pass：+2-3ms
└─ 总额外开销：+5-8ms
   → 集成后为零额外开销
```

### 5.5 MegaLights演进路线（UE5.5+）

```
从Clustered到MegaLights的演进：

Clustered Deferred的上限：
├─ Build Light Grid复杂度：O(Clusters × Lights)
├─ 130,560 Clusters × 1000 Lights = 1.3亿次测试
├─ 超过1000光源后Build Light Grid成为瓶颈
└─ 实测：2000光源 → Build Light Grid ~8ms（不可接受）

MegaLights的突破：
├─ BVH加速：O(Clusters × log(Lights))
│  查找光源从线性遍历变为树搜索
│  1M光源：log2(1M) ≈ 20步 → 远好于遍历1M
│
├─ 重要性采样：每像素只采样8-16个最重要的光源
│  使用蒙特卡洛积分保证结果收敛
│  误差<2%（视觉上几乎无损）
│
└─ GPU Ray Marching：遍历BVH查找光源
   替代传统Light Grid读取
   更灵活的查询模式

两个系统的关系：
├─ Clustered Deferred：UE5的默认光照路径（0-1000光源）
├─ MegaLights：大规模光源场景的扩展路径（1000+光源）
├─ 两者可共存：不同光源分类走不同路径
└─ MegaLights不替代Clustered，而是补充

当前状态（UE5.7）：
✅ Clustered Deferred：成熟稳定，所有平台支持
✅ MegaLights：官方支持（UE5.5+）
⚠️ MegaLights限制：需要RTX 3000+，部分光源类型支持不完整
```

```
Clustered Deferred在渲染管线中的位置：

Nanite GPU Driven Rendering
  ├─ 输出1：GBuffer（材质参数）
  └─ 输出2：SceneDepth（精确深度）

Lumen GI Pass
  ├─ 输入：GBuffer + Light Grid + SceneColor
  └─ 输出：IndirectLighting（间接光照）
        ↓
【合成输出】
Final Combine
  └─ SceneColor + IndirectLighting → 最终画面

### 集成收益总结

| 集成系统 | 复用数据 | 性能收益 | 如果不集成的代价(⚠️) |
|---------|---------|---------|---------------------|
| **GBuffer** | 材质参数缓存 | 避免N次重复读取 | +15ms（每光源读一次材质） |
| **VSM** | Shadow Mask Bits | 位运算替代纹理采样 | +8ms（每光源采样Shadow Map） |
| **Nanite** | 高精度Depth | Light Grid精确构建 | +1ms（需要Depth Pre-Pass） |
| **Lumen** | Light Grid + GBuffer | 零额外数据传递 | +5-8ms（需要额外Info Pass） |
| **总收益** | — | **避免29-32ms额外开销** | 不集成则帧时间翻倍 |

**结论**：Clustered Deferred不是孤立系统，而是UE5渲染管线的**核心枢纽**，深度集成Nanite/Lumen/VSM等所有主要系统。这种集成是UE5性能优势的关键所在。
