# RenderingPractice 操作教学架构（历史原则补充）

本文件保留“操作必须帮助理解”的知识节点原则。课程最高结构现以 `CURRICULUM.md` 和 `CASE_CONTINUITY.md` 为准：主案例推进是课程主轴，知识节点用于组织单篇中的局部解释。

## 核心定位

目标读者具备资深程序和 Unity TA 经验，不缺少图形学、Shader、Profiler 或实验能力；真正缺少的是 UE 的概念体系、编辑器操作语境和源码框架映射。

教程不能先堆一大段 UE 概念，再让读者集中搭场景验证；也不能只给编辑器步骤，把原理留给 `RenderingDeep` 自行补齐。

正确形态是：**知识点推动操作，操作产生现象，现象反过来固定知识点。**

## 最小教学单位：知识节点

一篇实践由多个知识节点组成。每个节点只引入一个主要新概念，并完成一次闭环：

```text
当前困惑
→ 最小原理模型
→ Unity 心智迁移
→ UE 具体操作
→ 可观察现象
→ 原理解释与边界
→ RenderingDeep 深入入口
→ 下一节点为什么自然出现
```

操作不是实验完成后的验证环节，而是概念形成过程的一部分。

## 知识节点的写法

### 当前困惑

从前一个节点自然产生一个问题，例如：World Outliner 中的 Cube 为什么不是 Renderer 直接使用的数据？同一个场景为什么还需要 View？关闭灯光后 BasePass 为什么仍在？

### 最小原理模型

只引入回答当前问题所需的新名词。第一次讲 Actor/Component 时，不提前展开 RDG；第一次讲 GT/RT 时，不提前展开 MeshDrawCommand。

模型必须说明当前对象或数据是什么、谁拥有它、为什么存在、接收什么输入、把结果交给谁。

### Unity 心智迁移

用已有经验加速理解，但明确类比边界。优先写“哪里相似”和“哪里不能直接套用”，不做术语翻译表堆砌。

### UE 具体操作

UE 特有操作首次出现时必须完整写出：菜单或面板入口、创建或修改的对象、Details 中的英文属性、Override/Apply/保存/编译/重启要求，以及操作成功后的检查方法。

操作量由当前知识点决定，不为后续节点提前搭建全部环境。

### 可观察现象

明确告诉读者应观察对象结构、画面变化、日志、统计项还是 GPU event。预期现象可以直接给出，不隐藏答案。

### 原理解释与边界

把现象翻译回当前模型，并说明当前证据还不能证明什么。证据边界是技术内容，不是学习者自评流程。

### RenderingDeep 深入入口

每个节点结束立即引用文件名、目标小节、本次建议理解到什么程度、当前可跳过的细节和可搜索源码符号。

### 下一节点衔接

用当前模型尚未回答的问题引出下一节点，而不是机械切换主题。

## 概念引入规则

- 一个术语首次出现前，必须通过当前问题说明它为什么需要存在。
- 允许提前提到后续术语名称，但必须标注“当前只需知道其角色”。
- 不允许在篇首同时投放 SceneProxy、FViewInfo、RDG、RHI、Mesh Pass 等整套名词。
- 每个节点最多承担一个主概念和少量必要支撑概念。
- 复杂系统先给职责模型，内部算法留给后续节点或 `RenderingDeep`。

## 一篇实践的整体形态

一篇实践仍然保持单文件和连续主线，但不再使用“原理大章 + 操作大章”。推荐结构：

1. 本篇主问题与最终模型。
2. 知识节点 1：建立最基础对象。
3. 知识节点 2：由节点 1 的缺口引入下一层。
4. 按依赖继续推进。
5. 全篇模型回放。
6. 下一篇入口。

## 历史原型的节点顺序

归档文件 `_Archive/01_FrameAndTools_ConceptPrototype.md` 曾按以下依赖推进：

1. Level / Actor / Component：UE 场景对象是什么。
2. Scene / Camera / View：场景与观察请求为什么分开。
3. Game Thread / Render Thread / SceneProxy：为什么需要渲染侧稳定表示。
4. CPU / GPU 时间线：一帧由谁工作，`stat unit` 看什么。
5. BasePass / GBuffer / Lighting：不透明延迟路径的数据合同。
6. Translucency：为什么透明需要另一条合成路径。
7. SceneColor / PostProcessing：最终颜色为什么是一条持续变化的流。
8. RDG / RHI / GPU event：这些阶段如何变成 GPU 工作，工具证据到哪一层。

该结构只作为局部知识节点写法参考，不再决定正式课程篇目。正式课程以环境案例进度组织。

## 深度标准

实践正文必须让读者获得可用于后续推理的模型，而不是只知道操作结果。例如不能只写“关闭灯光后 Lighting 下降”，必须解释 BasePass 发布表面合同、Lighting 消费合同，因此单个灯光不决定 BasePass 是否存在。

操作可以少，但原理链不能断。
