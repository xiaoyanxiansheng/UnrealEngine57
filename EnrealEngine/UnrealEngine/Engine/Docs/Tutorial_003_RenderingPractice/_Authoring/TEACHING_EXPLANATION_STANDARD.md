# RenderingPractice 教学解释检查表

本检查表用于编写和审核正式教程。

## 关键选择检查

对每个会影响工程的模板、开关、参数或资产方案检查：

| 问题 | 必须回答 |
|---|---|
| 控制对象 | 它控制编辑器工作流、项目配置、渲染路径、资产、平台还是运行时行为？ |
| 课程理由 | 为什么当前阶段选择它？ |
| 技术后果 | 对 Shader、构建、性能、内存、流送、质量或平台有什么影响？ |
| 替代方案 | 其他有效选择是什么，各自适合什么情况？ |
| 生命周期 | 当前选择何时生效，是否需要保存、编译或重启？ |
| 后续调整 | 哪一章会重新评估或修改？ |
| 成功检查 | 读者如何确认设置真正生效？ |

## 操作段落检查

- 是否给出了 UE 5.7 中可找到的菜单、面板和英文属性名？
- 是否说明了操作对象是 Asset、Actor、Component、Level 还是项目配置？
- 是否解释了为什么此时需要做这一步？
- 是否说明操作完成后应看到什么？
- 是否给出了最常见的失败原因和恢复方式？
- 是否说明该操作对后续主案例的作用？

## 教学边界检查

- 是否默认读者有程序和渲染经验，而不是重复基础知识？
- 是否避免假定读者熟悉 UE 的默认值和隐式行为？
- 是否把操作和技术解释放在一起？
- 是否区分课程默认、临时实验设置和生产建议？
- 是否在没有合适 Deep 章节时仍能独立讲清楚？

## 来源检查

- 技术事实是否以 UE 5.7 为准？
- 高风险或易变结论是否有源码、官方文档或实机观察支持？
- 引用是否真正帮助理解或验证？
- 是否避免将旧版本菜单、默认值或实验性功能状态写成 5.7 事实？
