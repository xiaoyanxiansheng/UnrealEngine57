# RenderingPractice 操作课程作者规范

本目录只用于生产和评审实践正文，不属于学习主路径。

## 目标读者

- 10 年左右程序经验。
- 具备多年 Unity TA、Shader、Profiler 和渲染管线经验。
- 不熟悉 UE 编辑器、资产流程、环境搭建、大世界系统和 Renderer 工具。

默认读者理解通用图形学、控制变量和代码阅读，但**不默认熟悉 UE 编辑器操作**。正文不能解释基础编程概念，却必须在 UE 特有操作首次出现时给出可直接执行的菜单路径、面板名称、属性名称、必要的 Override/重启条件和操作后的检查方法。

## 核心原则

1. **主案例驱动**：每篇推进 Room、Building、District 或 World，或解决主案例中的实际问题。
2. **操作结果明确**：开篇先说明本篇会新增什么地图、资产、系统或可见效果。
3. **首次操作完整**：UE 特有操作第一次出现时，提供菜单、面板、属性、保存/编译/重启和成功检查方式。
4. **解释紧跟操作**：关键参数立刻解释其渲染、内存、流送或工程意义。
5. **项目状态连续**：说明开始状态、修改内容、完成状态和下一篇依赖。
6. **Lab 回归主案例**：隔离实验必须说明结论如何应用回主地图。
7. **Deep 可选**：只在确实帮助理解时提供原理、源码或 Deep 延伸，不用它决定章节边界。
8. **不管理学习者**：不使用答案隔离、评级、强制报告或形式化作业流程。

## 教学解释标准

RenderingPractice 是教学正文，不是操作清单。读者虽然具备程序、Unity TA、图形学和 Shader 经验，但对 UE 的术语、编辑器、默认值和系统边界是新手。

每个会影响工程或后续结论的选择，都必须回答：

1. **它控制什么**：这个模板、选项、属性或参数属于哪一层系统。
2. **为什么这样选**：它如何服务当前案例和课程目标。
3. **产生什么影响**：它会改变哪些渲染路径、Shader、资产、内存、流送、构建或平台能力。
4. **替代方案是什么**：其他选项适合什么场景，与当前选择相比牺牲或获得什么。
5. **何时重新调整**：当前选择是永久基线、课程默认还是临时设置，后续哪一篇会修改。

例如创建工程时不能只写 `Blank / C++ / Desktop / Maximum / No Starter Content`，必须分别解释模板内容、代码扩展能力、目标平台限制、默认质量取向、预置资产污染和后续修改时机。

### 解释深度

- 通用程序和图形学知识可以压缩，不必向资深读者解释什么是类、线程、PBR 或控制变量。
- UE 特有知识必须详细：Actor/Component、Asset、Level、Mobility、RHI、Shader Model、Lumen、Nanite、World Partition 等首次出现时要建立可用模型。
- 编辑器操作不能依赖读者外部搜索；首次出现时必须写完整路径和成功检查。
- 解释应紧跟选择或操作，不把原因集中堆到文章结尾。
- 对复杂选项优先使用“选项—作用—本课程选择—替代方案—后续变化”的表格。

### 旁证与事实来源

重要结论按任务需要选择最直接的依据：

- UE 5.7 编辑器中的实际行为和可重复观察。
- UE 5.7 源码中的责任边界和默认配置。
- Epic 官方文档中的平台、功能和工作流说明。
- `RenderingDeep` 中已经校准的原理模型。

引用服务于解释和验证，不为增加引用数量而堆砌来源。正文仍应先用清楚语言讲明白，再给可选深入入口。

## 单篇标准结构

1. 本篇完成的场景结果。
2. 开始前的工程状态。
3. 需要理解的 UE 概念。
4. 按真实工作顺序排列的详细操作。
5. 操作后的可见结果和检查方式。
6. 关键参数的渲染、内存或流送意义。
7. 常见错误与恢复方式。
8. 完成后的工程状态。
9. 可选的官方文档、源码或 Deep 延伸。

## 技术事实规则

- 主版本固定为 UE 5.7。
- 技术事实必须通过 UE 5.7 实际编辑器、源码、官方资料或已验证内容校准。
- Event 名、CVar 和菜单路径不是稳定 API，正文必须声明适用环境和合理差异。
- 不把 GPU event 当作完整 C++ 调用栈。
- 不把 event 缺失直接证明为 RDG culling。
- 不把单次性能数值写成跨机器结论。
- 单变量对比必须固定相机、曝光、分辨率、画质、运行模式和主要场景状态。
- 正文应区分直接观察、合理推断和需要额外工具证明的底层事实，但不要求读者填写形式化证据表。

## 操作深度

对 UE 编辑器操作采用“首次完整、后续压缩”的规则。一个面板、资产类型、项目设置、调试工具或运行模式第一次出现时，应写出：

- 从哪个菜单或面板进入。
- 需要创建或选中什么对象。
- 属性的英文名称和所在 Details 分类。
- 是否需要勾选 Override、Apply、保存、编译或重启。
- 操作成功后应在画面、Outliner、日志或工具中看到什么。

后续文章再次使用同一操作时可以压缩并链接首次说明。

至少在以下情况给具体菜单或步骤：

- UE 名称与 Unity 经验差异大。
- 设置需要重启或重新编译。
- 错误配置会改变渲染路径。
- 工具入口不容易发现。

读者可以快速理解的普通编辑器操作应压缩成一句。

## 禁止项

- 为了关联 Deep 而加入与当前环境建设无关的知识。
- 每篇重新创建孤立场景，导致资产和经验无法延续。
- 只列点击步骤，不解释关键参数和完成后的检查方式。
- 用“课程统一”“后续需要”代替具体技术理由。
- 只解释选中的选项，不说明有意义的替代方案和取舍。
- 把临时选择写成永久最佳实践，或不说明后续何时调整。
- 把环境搭建、大世界和流送降级为渲染源码的附属内容。
- 提前批量创建没有实际问题来源的 Lab。
- 让读者离开正文搜索基础 UE 操作才能继续。
